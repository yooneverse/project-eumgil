## 1. 디자인 한줄 정의

부산이음길 UI는 `블루 기반의 공공서비스형 신뢰감` 위에 `큰 터치 영역`, `높은 가독성`, `접근성 모드 분기`를 얹은 길안내 중심 모바일 인터페이스다.

화이트 배경과 카드 구조를 기본으로 하되, 전체폭 CTA와 명확한 상태 색상으로 사용자의 다음 행동을 빠르게 결정하게 만드는 방향이 일관되게 관찰된다.

로그인 화면 디자인은 아직 확정 시안이 없으므로 본 문서에서는 별도 시각 상세를 확정하지 않는다. 다만 로그인 필수 정책에 따라 `비로그인으로 계속` CTA는 생산 디자인에 포함하지 않고, 소셜 로그인 버튼과 `AUTH-001` 이후 `ONB-001~ONB-005` 연결 상태는 후속 Figma handoff 필수 항목으로 본다.

### 3.4 주의할 점

- 디자인 시스템 보드의 컬러가 현재 기준 문서의 1차 소스다.
- 아이콘 시스템 보드의 샘플 컬러는 일부 semantic 표현과 미세한 차이가 있을 수 있으므로, 실제 구현 토큰은 본 문서의 컬러 토큰을 우선 사용한다.
- Danger는 존재하지만 공공서비스형 안정감 유지를 위해 남용하지 않는 편이 적절하다.
- 온보딩 사용자 유형 선택에서는 서비스 상위 용어를 `보행약자`로 통일하고, 세부 이동 특성은 `휠체어 사용자`, `기타 보행약자` 기준으로 표시한다. 선택 상태 색상은 서비스 전체 토큰으로 구현한다.

## 6. 레이아웃 컨벤션

### 6.1 기본 모바일 화면 구조

```text
Safe Area Top
  -> App Bar
  -> Scroll Content (카드 / 리스트 / 지도 / 폼)
  -> Fixed CTA 또는 Bottom Navigation
  -> Safe Area Bottom
```

| Bottom Navigation | 최신 FE 기준은 `홈 / 북마크 / 제보 / 마이페이지` 4탭이다. | 2026-04-27 확정 |

### 6.3 App Bar 구조

| 패턴 | 설명 |
| --- | --- |
| Back + Title | 일반 상세/폼 화면 기본 |
| Back + Title + Text Action | `다음`, `제출`, `수정`, `닫기` 등 단계형 화면 |
| Search형 App Bar | 목적지 설정, 지도 선택 등 검색 진입 화면 |

### 6.4 CTA와 Bottom Navigation

- 핵심 행동은 `하단 전체폭 CTA 1개`가 기본이다.
- 보조 행동은 CTA 아래의 ghost text나 secondary 버튼으로 배치된다.
- 탭바가 있는 화면은 CTA와 탭바가 동시에 노출되지 않도록 구조를 나누는 편이 적절하다.
- 버튼 시스템 보드 기준으로 버튼 높이는 `48 / 40 / 32 / 28`이지만, 전체 화면 내 터치 영역은 safe area와 하단 여백을 포함해 더 크게 느껴질 수 있다.

### 6.5 지도 화면 오버레이 규칙

- 상단: 검색 또는 경로 요약
- 중앙: 지도/경로선/선택 마커
- 우측: 지도 제어 버튼
- 하단: 경로/시설/제보 정보 카드

위 구조가 경로 탐색, 목적지 선택, 위치 지정, 길 안내의 공통 베이스다.

#### 실제 Android 아이콘 리소스 매핑 기준

- 2026-04-27 기준 Android 구현 아이콘의 단일 소스는 `FE/app/src/main/res/drawable/ic_*.xml`이다.
- Compose에서는 Material Icons 기본 세트와 임의 SVG를 화면 단위로 섞지 않고 `painterResource(id = R.drawable.<resource_name>)`로 로드한다.
- 같은 의미의 아이콘은 화면마다 바꾸지 않는다. 신규 아이콘이 필요하면 디자인 시스템에 먼저 추가하고 `ic_<group>_<name>.xml` 형식으로 등록한다.
- 화면 맥락에 강하게 결합된 보조선/배치 구조가 있더라도, 재사용 가능한 본체 아이콘은 우선 공용 `drawable`로 등록하고 Compose에서는 tint·크기·배치만 조합한다. 화면 전용 드로잉이 불가피한 경우에만 로컬 `Canvas` 구현을 허용하고, 인벤토리 문서에 `구현 컴포저블명`, `색상`, `접근성 전달 주체`를 함께 기록한다.
- 아이콘만 있는 버튼은 화면명세서에 `사용 목적`, `리소스명`, `접근성 라벨`을 함께 남긴다.
- 텍스트가 함께 보이는 아이콘은 장식으로 취급해 `contentDescription = null`을 기본으로 하고, 전체 컴포넌트 라벨 또는 버튼 텍스트가 의미를 전달한다.

### 9.3 FE 구현 체크리스트

- 일반 텍스트 대비는 최소 `4.5:1` 유지
- 선택 상태는 `색상 + 아이콘/보더/텍스트` 조합으로 전달
- 모든 주요 터치 영역은 최소 `44x44px`, 권장 `48px 이상`
- icon-only 버튼은 `aria-label` 또는 네이티브 접근성 라벨 필수
- 도착, 제보 완료, 경로 변경, 오류는 `aria-live` 또는 동등한 네이티브 공지 방식 고려
- 칩/세그먼트는 `aria-selected` 또는 `aria-pressed` 상태 반영
- 지도상 정보는 가능하면 리스트/카드 대체 접근 경로를 함께 제공
- 폰트 200% 확대와 모바일 시스템 font scale에서도 핵심 정보가 잘리지 않아야 함
- `prefers-reduced-motion` 또는 OS 모션 축소 설정 대응은 추가 검토 권장

### 9.4 접근성 정보 라벨 기준

- 접근성 정보 라벨의 고정 문구와 우선순위는 [2026-04-24_부산이음길_FE_접근성_정보_라벨_가이드.md](2026-04-24_부산이음길_FE_접근성_정보_라벨_가이드.md)를 따른다.
- 현재 기준 라벨은 `출입 가능`, `단차 없음`, `경사로`, `엘리베이터`, `장애인 주차`, `장애인 화장실`, `안내시설`, `장애인 객실`이다.
- 검색 결과, 시설 상세 바텀시트, 북마크 리스트에서 동일한 표기명을 유지한다.
- 검색 결과와 최근 목적지처럼 폭이 좁은 영역에서는 `+n` overflow 칩으로 접지 않고 두 줄까지 자연스럽게 감싸서 표시한다.
- 위 목록은 `긍정 접근성 정보`로 취급하므로 기본적으로 블루 계열 칩을 우선 사용한다.

## 11. 바로 개발에 쓸 수 있는 요약

### 11.1 반드시 유지해야 하는 시각 규칙 10개

1. 기본 배경은 `White`, 핵심 행동색은 `Primary 600`으로 유지한다.
2. 주요 행동은 `하단 전체폭 CTA 1개` 중심으로 정리한다.
3. 정보는 긴 문단보다 `카드 + 리스트` 구조로 쪼갠다.
4. `Blue=기본/선택`, `Green=완료/안전`, `Amber=주의`, `Red=위험` 의미를 유지한다.
5. 상태는 반드시 `색 + 아이콘 + 텍스트`로 함께 전달한다.
6. 숫자 정보는 라벨보다 크게 강조하고, 색보다 크기와 굵기로 위계를 만든다.
7. 지도 화면은 `상단 오버레이 + 우측 플로팅 컨트롤 + 하단 정보 카드` 구조를 유지한다.
8. Bottom Navigation은 `홈/북마크/제보/마이페이지` 4탭과 지도 홈, 목적지 설정, 북마크, 제보, 마이페이지 계열 화면에서 사용하고 길 안내/평가 바텀시트/인증/온보딩에서는 숨긴다.
9. 로그인/온보딩/권한/약관 같은 진입 gate 화면에서는 Bottom Navigation을 숨긴다.
10. 접근성 관련 UI는 최소 `44x44px`, 대비 `4.5:1`, 포커스 링 `2px` 기준을 지킨다.
