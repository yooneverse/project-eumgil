# 부산이음길 FE 바이브 코딩 컨벤션

- 작성일: 2026-04-13
- 정리일: 2026-04-27
- 대상: `FE/app` Android 앱 구현, AI/Codex 협업, MVP 화면 개발
- 목적: FE 개발자가 구현 전에 빠르게 읽고 바로 지킬 실전 규칙만 남긴다.
- 보관: 의사결정 근거, 레퍼런스 비교, 장기 확장 아이디어, 초기 세팅 상세 체크리스트는 `FE/docs/archive/2026-04-13_FE_코드_컨벤션_분리_참고자료.md`로 분리했다.

## 1. 현재 기준

- 온보딩은 신규 5단계 디자인(`ONB-001`~`ONB-005`)을 기준으로 한다. 기존 `OnboardingType / OnboardingLevel / LocationTerms` 3단계 예시는 과거 구현 참고로만 본다.
- 로그인은 필수다. `비로그인 MVP`, `비로그인으로 계속`, production local-first 저장 전제는 쓰지 않는다.
- 앱 시작 gate는 기존 회원 기준 `authSession -> selectedPrimaryUserType 홈 분기`로 고정한다. 신규 사용자는 `signupToken -> 온보딩 -> POST /auth/signup` 흐름으로 분리한다. `저시력자`는 `LV-HOM-01`, `보행약자`는 `MAP-01`로 연결한다.
- production 기준 source of truth는 인증 사용자 서버 데이터다.
- Room/DataStore는 cache, offline, pending, settings mirror, debug fixture 용도로만 사용한다.
- 공통 컴포넌트는 지금 필요한 범위까지만 만든다. 화면 하나에서만 쓰는 UI를 `core`로 올리지 않는다.
- `FE/docs/archive/`와 `FE/docs/debug-report/`의 2차 정리 문서는 구현 계약을 요약한 참고 문서다. 실제 규칙 변경은 이 문서와 1차 FE 계약 문서에 반영된 뒤 사용한다.

## 2. 바로 고정할 규칙 Top 15

1. `MainActivity`는 `AppNavHost`만 호출한다.
2. 패키지 구조는 `app/core/data/di/feature`로 고정한다.
3. feature 기본 파일은 `Contract/Route/Screen/ViewModel` 네 개로 시작한다.
4. 상태 패턴 용어는 `UiState + UiAction + UiEvent`로 통일한다. Android `Intent`와 혼동되는 `Intent` 명칭은 쓰지 않는다.
5. `Route`는 ViewModel 연결과 side-effect만, `Screen`은 순수 UI만 담당한다.
6. 화면은 Repository를 직접 알면 안 된다.
7. DTO와 Entity는 절대 `UiState`에 올리지 않는다.
8. 앱 시작 gate는 기존 회원 기준 `authSession -> selectedPrimaryUserType 홈 분기`로 고정한다. 신규 사용자는 `signupToken -> 온보딩 -> POST /auth/signup` 흐름으로 분리한다. `저시력자`는 `LV-HOM-01`, `보행약자`는 `MAP-01`로 연결한다.
9. 인증 세션은 설정 DataStore와 분리하고, 로그아웃/탈퇴/refresh 실패 시 즉시 삭제한다.
10. 온보딩/권한 안내/약관/설정은 DataStore mirror, 북마크/자주 가는 길은 서버 + Room cache로 간다.
11. read API는 `remote -> cache -> mock`, write API는 remote 성공 또는 local draft/pending 상태로 처리한다.
12. 공통 컴포넌트는 primitive와 지도 composite까지만 만든다.
13. 공통 컴포넌트는 `Eum` 접두어를 쓴다.
14. 사용자 노출 문자열과 접근성 라벨은 전부 리소스로 뺀다.
15. TalkBack, font scale `1.3/1.5`, 대비 기준 점검을 MVP 핵심 화면에 바로 적용한다.

## 3. 아키텍처 방향

> 부산이음길 FE는 **단일 `app` 모듈**에서 시작하고, **`app/core/data/di/feature` 패키지 구조 + feature별 `Contract/Route/ViewModel/Screen` 패턴 + 인증 세션 gate + repository 내부 remote/cache/mock/fallback 은닉 + 지도 전용 composite 컴포넌트 우선 전략**으로 간다.

## 6. Route / Screen / ViewModel / Contract

### Compose 함수

- `@Composable` 함수는 PascalCase를 쓴다.
- `Route`는 ViewModel 연결과 side-effect만 담당한다.
- `Screen`은 순수 UI 렌더링과 콜백 인자만 가진다.
- 공통 컴포넌트는 `Eum` 접두어를 쓴다.
- feature 전용 composite는 feature 도메인 명사를 앞에 둔다.

```kotlin
@Composable
fun MapRoute(
    onNavigateToSearch: () -> Unit,
    viewModel: MapViewModel = hiltViewModel(),
) { ... }

@Composable
fun MapScreen(
    uiState: MapUiState,
    onAction: (MapUiAction) -> Unit,
) { ... }

@Composable
fun EumButton(...) { ... }

@Composable
fun MapPlaceDetailSheet(...) { ... }
```

### Contract

- `Contract` 파일 하나에 `UiState`, `UiAction`, `UiEvent`를 묶는다.
- `UiState`는 화면이 그리는 데이터만 가진다.
- `UiAction`은 사용자 입력만 가진다.
- `UiEvent`는 snackbar, navigation 같은 일회성 처리만 가진다.
- `MutableStateFlow`, `MutableSharedFlow`는 ViewModel 내부에서만 사용한다.

```kotlin
data class MapUiState(
    val isLoading: Boolean = false,
    val markers: List<PlaceMarkerUiModel> = emptyList(),
    val selectedPlace: PlaceSummaryUiModel? = null,
)

sealed interface MapUiAction {
    data object Load : MapUiAction
    data class FilterToggled(val category: PlaceCategory) : MapUiAction
    data class MarkerClicked(val placeId: String) : MapUiAction
}

sealed interface MapUiEvent {
    data class ShowSnackbar(val message: String) : MapUiEvent
    data object NavigateToSearch : MapUiEvent
}
```

### Route / NavGraph

- 타입 세이프 라우트를 기본으로 쓴다.
- `Route.kt`에서 route 타입을 정의하고, 그래프 파일은 목적별로만 나눈다.
- route 이름은 화면 라벨이 아니라 도메인 동사/명사로 정한다.
- 신규 온보딩 route는 5단계 기준으로 `Splash`, `UserType`, `Permissions`, `Terms`, `ServiceIntro`를 쓴다.


## 7. 데이터 모델과 Repository

### 모델 구분

| 층위 | 규칙 | 예시 |
| --- | --- | --- |
| DTO | API 계약 그대로 반영, suffix `Dto` 필수 | `RouteSearchResponseDto` |
| Entity | Room/DataStore 저장 모델, suffix `Entity` 필수 | `BookmarkPlaceEntity` |
| Domain Model | 비즈니스 기준 모델, suffix 생략 | `Place`, `RouteOption`, `HazardReport` |
| UI Model | 화면에 맞춘 표현 모델, suffix `UiModel` 필수 | `PlaceDetailUiModel`, `BookmarkRouteItemUiModel` |

### 역할 경계

| 역할 | 반드시 하는 일 | 하면 안 되는 일 |
| --- | --- | --- |
| Repository | remote/local/mock 선택, mapper 호출, cache/fallback 정책, sync 정책 | Compose 상태 관리, `NavController` 호출, SDK UI 직접 제어 |
| RemoteDataSource | API 호출, SDK 네트워크 호출, 응답 수신 | fallback 분기, Room 저장, UI 메시지 생성 |
| LocalDataSource | Room/DataStore CRUD | API 호출, UI 모델 생성 |
| MockDataSource | 데모 fixture 반환 | 실제 저장 성공인 척 write 처리 |
| Manager | TTS, 위치, 권한, 센서, 시스템 서비스 래핑 | 비즈니스 분기, repository 대체 |

```kotlin
class PlaceRepository @Inject constructor(
    private val remote: PlaceRemoteDataSource,
    private val local: PlaceLocalDataSource,
    private val mock: PlaceMockDataSource,
    private val policy: DataSourcePolicy,
) {
    suspend fun getNearbyPlaces(query: PlaceQuery): List<Place> {
        if (policy.forceMock()) return mock.getNearbyPlaces(query)

        return runCatching {
            val remoteResult = remote.getNearbyPlaces(query)
            local.savePlaceCache(remoteResult)
            remoteResult
        }.getOrElse {
            local.getPlaceCache(query).ifEmpty {
                mock.getNearbyPlaces(query)
            }
        }
    }
}
```

### 인증 세션과 저장소

| 도메인 | 기준 저장소 | 읽기 | 쓰기 |
| --- | --- | --- | --- |
| 인증 세션 | `AuthSessionDataStore` 또는 보안 저장소 | 앱 시작 시 최우선 확인 | 로그인 성공 시 저장, 로그아웃/탈퇴/refresh 실패 시 삭제 |
| 사용자 프로필 / 설정 | remote + DataStore mirror | `GET /users/me` 우선 | `PATCH /users/me/user-type` 성공 후 local mirror 갱신 |
| 온보딩 / 권한 안내 / 약관 / 접근성 설정 / TTS 설정 | DataStore + 가입 완료 API | local 빠른 복원 후 `POST /auth/signup` 성공 여부 확인 | local 저장 후 가입 완료 요청 또는 앱 내부 설정으로 관리 |
| 북마크 / 자주 가는 길 | remote + Room cache | 서버 사용자 데이터 우선, 실패 시 cache 표시 | 서버 성공 후 Room cache 갱신, 실패 시 pending/outbox 여부 명시 |
| 최근 검색 / 제보 draft | Room | local 우선 | local write, 서버 전송 항목은 인증 성공 후 submit |
| 장소 마커 / 장소 상세 / 경로 탐색 | remote + cache + mock | remote 우선, 실패 시 cache, 없으면 mock | 읽기 전용 cache 저장 |
| 제보 제출 | remote + Room draft/outbox | draft가 있으면 로드 후 remote 제출 | 로그인 세션을 가진 앱 흐름에서 제출, 실패 시 draft/outbox 유지 |

### Mock / fallback

- 발표 필수 read API는 `remote -> local cache -> mock` 순서를 기본으로 한다.
- `forceMock`는 debug build에서만 켠다.
- write API는 실패 시 성공처럼 보이게 처리하지 않는다.
- 제보 draft와 최근 검색은 local write를 먼저 허용한다.
- 북마크/자주 가는 길은 서버 성공 후 cache를 갱신하고, 오프라인 저장을 허용할 경우 pending 상태를 명시한다.
- mock fixture는 `data/mock/fixture`에 모은다.
- mock도 domain model 기준으로 반환한다. DTO를 mock으로 직접 쓰지 않는다.

## 8. 디자인, 리소스, 접근성

### 디자인 토큰

- 토큰은 `core/designsystem/token` 아래에서만 관리한다.
- 최소 토큰은 `Color`, `Typography`, `Spacing`, `Radius`, `Elevation`, `IconSize` 여섯 종류다.
- UI 코드에서 raw hex color를 쓰지 않는다.
- UI 코드에서 raw spacing 값을 남발하지 않는다. 반복되면 토큰을 먼저 만든다.
- 온보딩 사용자 유형과 안내 모드별 강조 차이는 token 또는 semantic style로 분리한다.

### 공통 컴포넌트

- 지금 만들 공통 컴포넌트는 버튼, 아이콘 버튼, 입력, 검색바, 칩, 카드, 리스트, 시트, 다이얼로그, 스낵바, 상태 뷰, 지도 상세 시트까지다.
- map 전용 composite는 `MapTopSearchBar`, `MapFilterChipBar`, `MapPlaceDetailSheet`부터 시작한다.
- 온보딩 전용 카드, 약관 묶음, 시각장애 대화 UI는 feature 내부에 둔다.

### 문자열 / 리소스 / 아이콘

- 사용자에게 보이는 문자열은 전부 `strings.xml`에 둔다.
- TalkBack용 라벨도 문자열 리소스로 관리한다.
- 지도 시설/카테고리 아이콘은 `res/drawable/ic_place_*` 규칙으로 관리한다.
- Material 기본 아이콘은 내부 제어용에 한해 사용하고, 도메인 아이콘은 서비스 자산으로 관리한다.
- placeholder 이미지, 더미 텍스트도 preview 전용이 아니면 production 리소스 위치를 정한다.

### 접근성

- 핵심 화면은 TalkBack 라벨을 명시한다.
- 최소 터치 영역, font scale `1.3/1.5`, 고대비를 확인한다.
- 접근성 수동 점검 필수 화면은 로그인, 프로필 설정, 온보딩 5단계, 지도 메인, 시설 상세 시트, 경로 설정, 내비게이션 진행, 제보 화면이다.

## 9. 금지 목록

### UI에서 금지

- Retrofit, Room, DataStore 직접 호출
- `remember { mutableStateOf(...) }`로 서버/로컬 원본 상태를 들고 있는 것
- 하드코딩된 색상, `dp`, `sp`, 문자열 남발
- `NavController`를 `Screen`까지 끌고 들어가는 것
- 지도 SDK 타입을 feature 밖으로 노출하는 것
- `Toast`를 화면 곳곳에서 직접 호출하는 것

### ViewModel에서 금지

- `Activity`, `Context`, `NavController` 보관
- DTO/Entity를 그대로 `UiState`에 노출
- 로컬/원격/mock 분기를 ViewModel에서 직접 처리
- 화면 문구를 하드코딩한 긴 문자열 생성
- 여러 feature의 상태를 한 ViewModel에 몰아넣는 것

### 지금 도입하지 않을 것

- 멀티모듈
- `domain/usecase` 계층 강제
- 리포지토리 전부 interface + impl 분리
- 다크 모드
- 대화형 LLM UI용 전용 디자인 시스템
- 범용 지도 엔진 추상화
- 화면 하나짜리 공통 컴포넌트 승격

## 10. Preview와 테스트

### Preview

- 공통 primitive 컴포넌트는 최소 2개 preview를 둔다.
- Screen은 기본 상태와 loading / empty / error 중 1개 preview를 둔다.
- 지도 화면 preview는 실제 SDK 지도를 띄우지 않는다. placeholder surface와 fake marker data를 쓴다.
- preview 데이터는 `core/designsystem/preview` 또는 feature 내부 `preview` 패키지에 둔다.

### 테스트

- ViewModel마다 정상 로드와 실패 또는 fallback 단위 테스트를 둔다.
- Repository마다 source selection 테스트를 둔다.
- Compose UI 테스트는 앱 시작 인증 gate, 프로필 미완료 이동, 온보딩 5단계 이동, 약관 동의 버튼 활성화, 지도 필터 선택 상태를 우선한다.
- 인증/세션 테스트는 token refresh, refresh 만료 시 세션 삭제, 로그아웃 시 local mirror 삭제, 401 수신 시 재시도 또는 재로그인 이벤트를 우선한다.

## 11. 구현 전 체크리스트

- [ ] feature 폴더에 `Contract/Route/Screen/ViewModel`이 있는가?
- [ ] `Route`와 `Screen` 책임이 분리됐는가?
- [ ] `UiState`에 DTO/Entity가 직접 들어가지 않았는가?
- [ ] ViewModel이 remote/cache/mock 분기를 직접 처리하지 않는가?
- [ ] Repository가 `remote -> cache -> mock` 또는 write pending 정책을 명시하는가?
- [ ] 로그인 필수 gate가 깨지지 않았는가?
- [ ] `비로그인으로 계속` CTA가 production flow에 없는가?
- [ ] 문자열과 TalkBack 라벨이 리소스로 분리됐는가?
- [ ] raw color와 반복 spacing이 토큰 없이 들어가지 않았는가?
- [ ] Preview와 최소 테스트 기준을 작업 범위에 맞게 추가했는가?

## 12. 문서화 규칙

- FE 전용 문서는 `FE/Docs`에 둔다.
- 구현이 문서에서 벗어나면 같은 작업 주기에 문서를 갱신한다.
- 현재 구현 기준과 목표 기준이 다르면 한 문서 안에서 구분한다.
- feature별 임시 메모, 회의록, 캡처 문서는 활성 문서로 두지 않는다. 필요하면 `archive`로 분리한다.
