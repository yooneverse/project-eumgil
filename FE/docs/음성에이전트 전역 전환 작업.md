### 2.3 컨텍스트 모델 추가

기존 음성 진입점이 특정 route string 하나에 묶이지 않도록 `VoiceAssistantContext`를 별도 모델로 만들었다.

현재 포함 필드:

```kotlin
data class VoiceAssistantContext(
    val currentRoute: String? = null,
    val currentTopLevelRoute: String? = null,
    val userType: VoiceAssistantUserType? = null,
    val editingTarget: RouteEditingTarget = RouteEditingTarget.DESTINATION,
)
```

설계 의도:

- `currentRoute`: 현재 NavBackStackEntry의 route 또는 상세 route 패턴을 담는다.
- `currentTopLevelRoute`: 현재 화면이 어느 top-level 탭에 속하는지 담는다.
- `userType`: 일반/지체장애 모드 분기를 위해 사용한다.
- `editingTarget`: 검색/경로 편집 시 출발지 또는 도착지 편집 대상 정보를 보존한다.

### 2.4 VoiceAssistantAction 추가

전역 음성 명령은 `VoiceAssistantAction` sealed interface로 표현한다. 실제 navigation controller나 화면 구현 타입은 이 계약에 노출하지 않았다.

정의한 action:

```kotlin
VoiceAssistantAction.SearchPlace
VoiceAssistantAction.OpenReport
VoiceAssistantAction.OpenSavedRoutes
VoiceAssistantAction.OpenMyPage
VoiceAssistantAction.OpenMap
VoiceAssistantAction.StopNavigation
VoiceAssistantAction.ResumeNavigationGuidance
VoiceAssistantAction.UnknownCommand
```

검색은 반드시 `SearchPlace` 하나로만 표현한다. 장소 검색, 목적지 검색, 출발지 검색 같은 세부 차이는 `SearchPlace.query`와 `SearchPlace.editingTarget`으로 처리한다.

모든 action은 `requiresConfirmation`을 가진다. 기본값은 위험도가 낮은 화면 이동/검색은 `false`, 경로 안내 중단은 `true`로 잡았다.

### 2.5 UI 계약 추가

`UiState`, `UiAction`, `UiEvent`를 추가했다.

`UiState`는 현재 context, 음성 처리 상태, transcript, confirmation 대기 action, 마지막 해석 action, 오류 메시지를 담는다.

`UiAction`은 사용자의 assistant 클릭, dismiss, context 변경, transcript 변경, action 해석 완료, confirmation 수락/거절을 표현한다.

`UiEvent`는 listening 시작/중지, confirmation 요청, action 실행, 메시지 표시를 표현한다.

## 5. 후속 작업 안내

### 5.1 명령 해석 계층

후속 작업자는 STT/LLM/KWS 결과를 `VoiceAssistantAction`으로 변환하는 계층을 추가한다.

권장 방향:

- `VoiceAnalyzeResult` 또는 KWS 결과를 직접 navigation으로 보내지 않는다.
- 먼저 `VoiceAssistantAction`으로 변환한다.
- 알 수 없는 명령은 `UnknownCommand(rawCommand = ...)`로 보낸다.
- 검색 명령은 항상 `SearchPlace(query, editingTarget, requiresConfirmation)`로 보낸다.

### 5.2 전역 context 수집

앱 루트 또는 navigation host 주변에서 현재 route 정보를 수집해 `VoiceAssistantContext`로 변환한다.

주의사항:

- `VoiceAssistantContext`에는 `NavController`를 넣지 않는다.
- `currentRoute`와 `currentTopLevelRoute`를 분리해서 유지한다.
- `editingTarget`은 경로 편집 흐름에서 출발지/도착지 선택을 보존하는 데 사용한다.
- 일반/지체장애 모드 분기는 `VoiceAssistantUserType`으로 정규화해서 넘긴다.

### 5.3 action 실행 adapter

`VoiceAssistantAction`을 실제 화면 이동이나 ViewModel 호출로 바꾸는 adapter는 app/navigation 계층에 둔다.

예상 매핑:

```text
SearchPlace -> SearchRoute.Results 또는 SearchRoute.Entry 기반 검색 흐름
OpenReport -> ReportRoute.Report
OpenSavedRoutes -> TopLevelRoute.SavedRoute
OpenMyPage -> TopLevelRoute.MyPage
OpenMap -> TopLevelRoute.Map
StopNavigation -> navigation ViewModel 또는 navigation route action
ResumeNavigationGuidance -> navigation guidance TTS/action 재개
UnknownCommand -> 안내 메시지 또는 재시도 UI
```

이 매핑은 contract 파일에 넣지 않는다. contract는 "무엇을 하려는지"만 표현하고, "어떻게 이동할지"는 app/navigation 계층에서 결정한다.

### 5.4 confirmation 처리

`requiresConfirmation`이 `true`인 action은 바로 실행하지 말고 confirmation UI나 음성 확인 단계를 거친다.

권장 흐름:

```text
ActionResolved(action)
-> action.requiresConfirmation 확인
-> true면 UiEvent.RequestConfirmation(action)
-> 사용자가 수락하면 UiEvent.ExecuteAction(action)
-> false면 즉시 UiEvent.ExecuteAction(action)
```

### 9.2 VoiceAssistantOverlay 역할

`VoiceAssistantOverlay`는 전역 음성 어시스턴트 상태를 `ModalBottomSheet`로 표시하는 Composable이다.

공개 API는 아래 형태다.

```kotlin
@Composable
fun VoiceAssistantOverlay(
    uiState: UiState,
    onAction: (UiAction) -> Unit,
    visible: Boolean,
    modifier: Modifier = Modifier,
    assistantMessage: String? = null,
    pendingAction: VoiceAssistantAction? = uiState.pendingConfirmationAction,
    bottomSheetState: SheetState = rememberModalBottomSheetState(skipPartiallyExpanded = true),
)
```

설계 의도:

- ViewModel 생성/주입은 Overlay 내부에서 하지 않는다.
- 외부에서 `uiState`, `onAction`, `visible`만 넘기면 렌더링할 수 있게 했다.
- `assistantMessage`는 외부 어댑터나 LLM/STT 계층이 만든 사용자 안내 문구를 주입할 수 있도록 optional로 열어두었다.
- `pendingAction`은 confirmation 대기 action을 외부에서 명시적으로 덮어쓸 수 있게 두었다.
- AppNavHost 연결 전에도 preview나 테스트에서 독립적으로 사용할 수 있게 했다.

### 9.3 상태 매핑

Contract의 `VoiceAssistantStatus`만으로는 UI 요구사항의 `ResultReady`, `ConfirmationRequired`를 그대로 표현하기 어렵다. 그래서 Overlay 전용 visual state를 따로 두었다.

```kotlin
enum class VoiceAssistantOverlayVisualState {
    Idle,
    Listening,
    Processing,
    ResultReady,
    ConfirmationRequired,
    Error,
}
```

매핑 규칙은 `resolveVoiceAssistantOverlayVisualState(uiState)`에 있다.

```text
status == Error 또는 errorMessage 존재
-> Error

status == AwaitingConfirmation
-> ConfirmationRequired

status == Listening
-> Listening

status == Processing
-> Processing

lastResolvedAction != null
-> ResultReady

그 외
-> Idle
```

주의할 점:

- `Error`가 최우선이다. 이전에 resolved action이 남아 있어도 `errorMessage`가 있으면 오류 상태를 먼저 보여준다.
- Contract의 `AwaitingConfirmation`은 화면 요구사항의 `ConfirmationRequired`로 이름만 변환해 표시한다.
- Contract에 별도 `ResultReady` status가 없어서 `lastResolvedAction != null`인 Idle 상태를 `ResultReady`로 해석한다.

### 10.2 VoiceAssistantInterpreter 역할

`VoiceAssistantInterpreter`는 transcript와 `VoiceAssistantContext`를 받아 `VoiceAssistantAction`으로 바꾸는 인터페이스다.

기본 구현은 `RuleBasedVoiceAssistantInterpreter`로 두었다. 현재는 서버 LLM이나 STT 분석 결과에 의존하지 않고, 정해진 문구를 기준으로만 action을 반환한다.

현재 rule:

```text
"제보", "신고", "불편 제보", "제보해줘"
-> VoiceAssistantAction.OpenReport

"부산역 찾아줘", "부산역 검색해줘"
-> VoiceAssistantAction.SearchPlace(query = "부산역", editingTarget = context.editingTarget)

그 외
-> VoiceAssistantAction.UnknownCommand(rawCommand = normalizedTranscript)
```

주의사항:

- 검색 명령은 새 action을 만들지 않고 반드시 `SearchPlace`로 유지한다.
- 검색 대상은 현재 임시 rule이므로 "부산역"만 처리한다.
- `SearchPlace.editingTarget`은 interpreter 내부에서 임의로 고정하지 않고 `VoiceAssistantContext.editingTarget`을 그대로 사용한다.
- rule-based 구현은 임시 계층이다. 이후 LLM/서버/로컬 NLU로 바꾸더라도 외부 계약은 `VoiceAssistantInterpreter` 인터페이스를 유지하는 방향이 좋다.

### 10.3 VoiceAssistantViewModel 역할

`VoiceAssistantViewModel`은 transcript 입력을 interpreter에 넘기고, 해석된 `VoiceAssistantAction`을 UI event로 노출한다.

핵심 원칙:

- ViewModel은 `NavController`를 알지 않는다.
- ViewModel은 실제 화면 이동을 직접 수행하지 않는다.
- 외부로 내보내는 event는 `CloseOverlay` 또는 `DispatchAction`만 사용한다.
- confirmation이 필요한 action은 사용자 확인 전까지 `DispatchAction`을 emit하지 않는다.

이번 작업에서 ViewModel 쪽에 추가한 event/action:

```kotlin
data object ConfirmClicked : UiAction
data object CloseOverlay : UiEvent
data class DispatchAction(val action: VoiceAssistantAction) : UiEvent
```

`ConfirmClicked`는 확인이 필요한 action을 사용자가 승인했을 때 쓰는 action이다. 기존 `UiAction.ConfirmationAccepted`도 같은 의미로 처리되도록 열어두었다.

### 10.4 confirmation 처리 정책

confirmation이 필요 없는 action은 transcript 해석 직후 바로 `DispatchAction(action)`으로 emit한다.

confirmation이 필요한 action은 바로 emit하지 않고 ViewModel state에 보류한다.

```text
TranscriptChanged
-> interpreter.interpret(...)
-> action.requiresConfirmation == true
-> pendingConfirmationAction 저장
-> status = AwaitingConfirmation
-> DispatchAction 미발행
```

사용자가 확인하면 그때만 action을 내보낸다.

```text
ConfirmClicked 또는 UiAction.ConfirmationAccepted
-> pendingConfirmationAction 확인
-> DispatchAction(pendingAction) emit
-> pending state 정리
```

이 정책 때문에 다음 작업자가 adapter를 붙일 때도 confirmation required action을 중복 실행하지 않아야 한다. adapter는 `DispatchAction`을 받은 뒤에만 실제 navigation 또는 ViewModel 호출을 수행한다.

### 10.5 overlay close 정책

사용자가 overlay를 닫는 action은 ViewModel에서 `CloseOverlay` event로만 변환한다.

```text
UiAction.Dismissed
-> state 초기화
-> UiEvent.CloseOverlay emit
```

ViewModel 내부에서 route pop, dialog dismiss, bottom sheet state 변경을 직접 하지 않는다. 실제 overlay visibility 처리는 상위 UI 계층이 `CloseOverlay` event를 받아 처리해야 한다.

## 12. 추가 완료 작업: 일반/지체장애 검색 화면 mic 전역 overlay 전환

작성일: 2026-05-17
작업 범위: FE

이 섹션은 일반/지체장애 검색 화면의 mic 버튼이 기존 `search/voice` 화면으로 이동하지 않고, `AppNavHost`가 소유한 전역 `VoiceAssistantOverlay`를 열도록 전환한 작업 기록이다.

섹션 11의 전역 진입 정책 중 검색 화면 연결부를 후속 작업자가 빠르게 확인할 수 있도록 별도로 정리한다.

### 12.1 수정한 핵심 파일

```text
FE/app/src/main/java/com/ssafy/e102/eumgil/app/navigation/MainNavGraph.kt
FE/app/src/main/java/com/ssafy/e102/eumgil/app/navigation/AppNavHost.kt
FE/app/src/test/java/com/ssafy/e102/eumgil/app/navigation/MainNavGraphTopLevelNavigationPolicyTest.kt
FE/app/src/test/java/com/ssafy/e102/eumgil/feature/search/SearchViewModelTest.kt
```

`SearchRoute.kt`와 `SearchViewModel.kt`는 흐름 확인 대상이었지만, 전역 overlay 전환을 위해 직접 구조를 바꾸지는 않았다.

### 12.2 변경 전 흐름

기존 검색 화면 mic 버튼 흐름은 아래와 같았다.

```text
SearchScreen mic click
-> SearchUiAction.VoiceInputClicked
-> SearchViewModel emits SearchUiEvent.NavigateToVoiceInput
-> SearchRouteContent calls onNavigateToVoiceInput()
-> MainNavGraph navigates SearchRoute.VoiceInput.createRoute(initialEditingTarget)
-> SearchVoiceInputRoute / SearchVoiceInputExperience 진입
-> SearchViewModel의 VoiceRouteEntered, VoiceTranscriptReceived 흐름 사용
-> 2초 preview delay 후 SearchRoute.Results 이동
```

이 흐름은 검색 전용 voice bottom sheet와 검색 전용 자동 결과 이동 정책에 묶여 있다.

### 12.3 변경 후 흐름

현재 검색 화면 mic 버튼 흐름은 아래와 같다.

```text
SearchScreen mic click
-> SearchUiAction.VoiceInputClicked
-> SearchViewModel emits SearchUiEvent.NavigateToVoiceInput
-> SearchRouteContent calls onNavigateToVoiceInput()
-> MainNavGraph calls onOpenVoiceAssistant(initialEditingTarget)
-> AppNavHost opens global VoiceAssistantOverlay with VoiceAssistantContext(editingTarget)
```

중요한 경계:

- `SearchViewModel`은 여전히 `NavigateToVoiceInput` 이벤트만 발행한다.
- `SearchViewModel`은 전역 overlay 존재를 알지 않는다.
- 실제로 route를 이동할지 overlay를 열지는 상위 navigation 계층이 결정한다.
- 신규 전역 흐름에서는 `SearchVoiceInputExperience`에 들어가지 않으므로 2초 자동 검색 이동 로직을 사용하지 않는다.

### 12.4 MainNavGraph 변경

`mainNavGraph`가 아래 콜백을 받도록 확장됐다.

```kotlin
onOpenVoiceAssistant: (RouteEditingTarget) -> Unit
```

`SearchEntryRoute`와 `SearchResultsRoute`의 `onNavigateToVoiceInput` 연결은 더 이상 `SearchRoute.VoiceInput.createRoute(...)`를 호출하지 않는다.

현재 정책:

```kotlin
onNavigateToVoiceInput = {
    onOpenVoiceAssistant(initialEditingTarget)
}
```

`initialEditingTarget`을 그대로 넘기는 이유:

- 출발지 검색 화면에서 mic를 눌렀으면 전역 음성 검색 결과도 출발지 편집 대상으로 반영해야 한다.
- 도착지 검색 화면에서 mic를 눌렀으면 기본 도착지 편집 대상으로 유지해야 한다.
- 이 값을 잃으면 `VoiceAssistantAction.SearchPlace`가 `SearchRoute.Results.createRoute(query, editingTarget)`로 변환될 때 잘못된 편집 대상에 반영된다.

### 12.5 AppNavHost 연결

`AppNavHost`는 `mainNavGraph`에 아래처럼 callback을 넘긴다.

```kotlin
mainNavGraph(
    navController = navController,
    onOpenVoiceAssistant = { editingTarget ->
        openVoiceAssistant(currentVoiceAssistantSourceContext.copy(editingTarget = editingTarget))
    },
)
```

의도:

- 현재 route, top-level route, user type은 `currentVoiceAssistantSourceContext`에서 유지한다.
- 검색 화면에서 들어온 `RouteEditingTarget`만 덮어쓴다.
- permission 요청 중에도 보류된 `voiceAssistantSourceContext`가 같은 editing target을 유지해야 한다.

후속 작업자는 검색 화면에서 별도 overlay state를 만들지 말고 이 callback 경로를 유지한다.

### 12.6 SearchRoute.VoiceInput 호환 유지

기존 `SearchRoute.VoiceInput` route 등록은 유지했다.

유지 이유:

- 기존 deep link나 테스트, 내부 route 호출이 남아 있을 수 있다.
- `SearchVoiceInputRoute`는 검색 전용 음성 입력 경험과 2초 preview delay 정책을 포함한다.
- 전역 overlay 전환과 legacy route 제거는 별도 단계로 분리해야 한다.

주의:

- 신규 mic 진입점에서 `SearchRoute.VoiceInput.createRoute(...)`를 다시 호출하지 않는다.
- legacy route를 제거하려면 `SearchVoiceInputRoute`, `SearchVoiceInputExperience`, 관련 copy/test, KWS pause 호환 조건까지 함께 정리한다.

### 12.7 SearchViewModel 테스트 관점

`SearchViewModel`의 기존 mic 테스트는 "검색 route로 이동한다"가 아니라 "상위 voice input callback을 요청한다"는 관점으로 유지한다.

테스트명도 아래처럼 정리했다.

```text
voice input click requests parent voice input callback
```

이 테스트가 확인하는 것은 아래 하나다.

```text
SearchUiAction.VoiceInputClicked
-> SearchUiEvent.NavigateToVoiceInput
```

이 이벤트 이름은 아직 `NavigateToVoiceInput`이지만, 실제 의미는 "상위 계층이 voice input 의도를 처리하라"에 가깝다. 이름을 바꾸려면 `SearchContract.kt`, `SearchRoute.kt`, low vision search route의 무시 처리까지 함께 정리해야 하므로 이번 작업에서는 변경하지 않았다.

### 12.8 2초 자동 검색 이동 로직 관련 주의

`SearchViewModel`에는 아래 상수가 남아 있다.

```kotlin
VOICE_INPUT_RESULT_PREVIEW_DELAY_MILLIS = 2_000L
```

이 지연 검색 이동은 `SearchVoiceInputExperience`가 `VoiceTranscriptReceived`를 ViewModel에 보낼 때만 동작한다.

전역 overlay 흐름에서는 아래 action들을 사용하지 않는다.

```text

SearchUiAction.VoiceTranscriptReceived
SearchUiAction.VoiceCaptureButtonClicked
SearchUiAction.VoiceInputDismissed
```

전역 overlay에서 transcript를 해석한 결과는 `VoiceAssistantAction.SearchPlace`로 변환되고, AppNavHost의 adapter가 `SearchRoute.Results.createRoute(query, editingTarget)`로 이동시키는 방식이 현재 방향이다.

후속 작업자가 전역 overlay에서도 "말한 문장 preview 후 2초 뒤 이동" UX를 원한다면, `SearchViewModel`의 legacy 로직을 재사용하지 말고 `VoiceAssistantViewModel` 또는 overlay 전용 state machine에서 별도 정책으로 설계해야 한다.

## 14. 추가 완료 작업: 전역 overlay와 지도 상세/지도 voiceSearch 충돌 방지

작성일: 2026-05-17
작업 범위: FE

이 섹션은 `VoiceAssistantOverlay`가 전역으로 열릴 때 지도 화면 안쪽의 기존 상태와 충돌하지 않도록 정리한 작업 기록이다. 지도 화면에는 시설 상세 bottom sheet와 기존 지도 내부 `voiceSearch` bottom sheet가 따로 있고, 전역 overlay가 이 상태들 위에 열리면 bottom tab 노출 조건, back 처리, KWS pause 조건, sheet z-order가 서로 충돌할 수 있다.

이번 작업의 방향은 AppNavHost가 MapViewModel을 직접 만지지 않고, Map route의 `SavedStateHandle`에 one-shot 정리 신호만 남기는 것이다. 실제 지도 상태 정리는 `MapRoute`가 신호를 consume한 뒤 기존 `MapUiAction`을 호출해서 처리한다.

### 14.1 수정한 주요 파일

```text
FE/app/src/main/java/com/ssafy/e102/eumgil/app/navigation/AppNavHost.kt
FE/app/src/main/java/com/ssafy/e102/eumgil/app/navigation/MainNavGraph.kt
FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/MapRoute.kt
FE/app/src/test/java/com/ssafy/e102/eumgil/app/navigation/AppNavHostPolicyTest.kt
FE/app/src/test/java/com/ssafy/e102/eumgil/app/navigation/MainNavGraphTopLevelNavigationPolicyTest.kt
FE/app/src/test/java/com/ssafy/e102/eumgil/feature/map/MapRouteLifecycleTest.kt
```

`MapContract.kt`와 `MapViewModel.kt`는 흐름 확인 대상이었지만, 새 action을 추가하지 않고 기존 `FacilityDetailDismissed`, `VoiceSearchDismissed`를 재사용했다.

### 14.2 AppNavHost 변경

`AppNavHost`에서 전역 overlay를 실제로 여는 `showVoiceAssistant(...)` 진입점에 지도 정리 요청을 추가했다. 현재 route가 정확히 `TopLevelRoute.Map.route`일 때만 현재 Map back stack entry의 `SavedStateHandle`에 dismiss request를 기록한다.

```kotlin
if (shouldRequestMapFacilityDetailDismissOnGlobalVoiceAssistantOpen(currentRoute)) {
    navController.currentBackStackEntry?.savedStateHandle?.requestMapFacilityDetailDismiss()
}
```

분기 함수:

```kotlin
internal fun shouldRequestMapFacilityDetailDismissOnGlobalVoiceAssistantOpen(currentRoute: String?): Boolean =
    currentRoute == TopLevelRoute.Map.route
```

주의할 점:

- 검색 route, report route, saved route 등 map이 아닌 route에서는 이 신호를 보내지 않는다.
- 전역 overlay 경로에서 `MapViewModel.openVoiceSearch()`를 직접 호출하지 않는다.
- AppNavHost는 MapViewModel을 알지 않고, `SavedStateHandle` 신호만 쓴다.

### 14.3 SavedStateHandle one-shot event 설계

boolean flag는 재구성, 재수집, 빠른 연속 요청에서 중복 소비 문제가 생길 수 있다. 그래서 request id/count 형태로 구현했다.

추가된 key:

```text
mapFacilityDetailDismissRequestId
mapFacilityDetailDismissConsumedId
```

helper:

```kotlin
internal fun SavedStateHandle.requestMapFacilityDetailDismiss(): Long
internal fun SavedStateHandle.consumeMapFacilityDetailDismissRequest(requestId: Long): Boolean
```

동작 규칙:

```text
requestMapFacilityDetailDismiss()
-> 현재 request id + 1 저장
-> 증가된 id 반환

consumeMapFacilityDetailDismissRequest(id)
-> id가 0 이하이면 false
-> 현재 request id와 다르면 false
-> 이미 consumed id 이하이면 false
-> consumed id를 현재 id로 저장하고 true
```

이 방식으로 같은 request id는 한 번만 소비되고, 새 request 이후 오래된 id가 뒤늦게 consume되는 경우는 무시된다.

### 14.4 MainNavGraph와 MapRoute 연결

`MainNavGraph`의 Map destination은 `SavedStateHandle`의 dismiss request id를 StateFlow로 수집해서 `MapRoute`에 넘긴다.

```kotlin
val facilityDetailDismissRequestId by
    backStackEntry.savedStateHandle
        .getStateFlow(
            MAP_FACILITY_DETAIL_DISMISS_REQUEST_ID_KEY,
            MAP_FACILITY_DETAIL_DISMISS_REQUEST_INITIAL_ID,
        )
        .collectAsStateWithLifecycle()
```

`MapRoute` 전달 값:

```kotlin
facilityDetailDismissRequestId = facilityDetailDismissRequestId,
onFacilityDetailDismissRequestConsumed = { requestId ->
    backStackEntry.savedStateHandle.consumeMapFacilityDetailDismissRequest(requestId)
}
```

`MapRoute`는 request id가 들어오면 먼저 consume을 시도하고, 성공했을 때만 ViewModel action을 보낸다.

```kotlin
if (facilityDetailDismissRequestId <= 0L) return@LaunchedEffect
if (!onFacilityDetailDismissRequestConsumed(facilityDetailDismissRequestId)) return@LaunchedEffect

viewModel.onAction(MapUiAction.FacilityDetailDismissed)
viewModel.onAction(MapUiAction.VoiceSearchDismissed)
```

중요한 점:

- 요구사항대로 신호를 받으면 `MapUiAction.FacilityDetailDismissed`를 호출한다.
- 전역 overlay가 열리는 동안 기존 지도 내부 voiceSearch가 같이 남지 않도록 `MapUiAction.VoiceSearchDismissed`도 같이 호출한다.
- `VoiceSearchDismissed`는 이미 no-op guard가 있는 기존 action이라 voiceSearch가 열려 있지 않아도 안전하다.

### 14.5 상태 흐름 요약

전역 overlay 열기 요청 흐름:

```text
KWS 또는 검색 mic 등에서 openVoiceAssistant(...)
-> AppNavHost.showVoiceAssistant(...)
-> currentRoute == map 이면 savedStateHandle.requestMapFacilityDetailDismiss()
-> voiceAssistantVisible = true
-> VoiceAssistantOverlay 표시
```

지도 정리 흐름:

```text
MainNavGraph map destination이 request id 수집
-> MapRoute로 facilityDetailDismissRequestId 전달
-> MapRoute가 consume callback 호출
-> consume 성공 시 FacilityDetailDismissed 호출
-> consume 성공 시 VoiceSearchDismissed 호출
```

결과:

```text
전역 VoiceAssistantOverlay visible
지도 시설 상세 sheet hidden
```

## 13. 추가 완료 작업: DispatchAction 실제 navigation 연결

작성일: 2026-05-17
작업 범위: FE

이 섹션은 `VoiceAssistantViewModel`이 발행하는 `DispatchAction`을 `AppNavHost`가 받아 실제 앱 navigation으로 실행하도록 연결한 작업 기록이다.

섹션 11.5가 action adapter의 방향을 설명했다면, 이 섹션은 실제 `AppNavHost` 이벤트 수신, overlay close 정책, route mapping, 테스트 관점을 후속 작업자가 바로 이어받을 수 있도록 정리한다.

### 13.1 수정한 핵심 파일

```text
FE/app/src/main/java/com/ssafy/e102/eumgil/app/navigation/AppNavHost.kt
FE/app/src/test/java/com/ssafy/e102/eumgil/app/navigation/AppNavHostRoutingTest.kt
```

이번 작업은 FE navigation 계층 안에서만 처리했다. `VoiceAssistantContract.kt`, `VoiceAssistantViewModel.kt`, `VoiceAssistantOverlay.kt`에는 `NavController` 의존성을 추가하지 않았다.

### 13.2 AppNavHost overlay 렌더링 구조

`AppNavHost`의 `Scaffold/NavHost` 바깥을 `Box`로 감싸고, `VoiceAssistantOverlay`를 마지막 child로 렌더링한다.

현재 구조:

```text
Box
-> Scaffold
   -> NavHost
-> VoiceAssistantOverlay
```

의도:

- overlay는 각 feature route가 아니라 `AppNavHost`가 소유한다.
- overlay 표시 여부는 route navigation이 아니라 `voiceAssistantVisible` 상태로 결정한다.
- overlay dismiss/back은 `popBackStack()`을 호출하지 않고 overlay close만 수행한다.
- feature 화면이 전역 overlay state를 중복으로 만들지 않는다.

후속 작업자가 overlay 렌더링 위치를 바꿀 때는 bottom tab, map sheet, search route와의 z-order를 함께 확인해야 한다.

### 13.3 DispatchAction 처리 흐름

`VoiceAssistantViewModel.uiEvent` 수신부에서 `DispatchAction`을 실제 navigation으로 연결했다.

현재 흐름:

```text
VoiceAssistantViewModel emits DispatchAction(action)
-> AppNavHost receives event
-> voiceAssistantVisible = false
-> navController.navigateByVoiceAssistantAction(action)
```

중요한 정책:

- action dispatch 전에 overlay를 먼저 닫는다.
- overlay close는 `CloseOverlay`/`Dismissed`와 동일하게 local visible state로 처리한다.
- navigation 실행 책임은 `AppNavHost`에만 둔다.
- ViewModel은 여전히 `NavController`, `TopLevelDestination`, route string을 모른다.

`CloseOverlay` 이벤트는 navigation 없이 overlay close만 수행한다.

### 13.4 VoiceAssistantAction mapping

`AppNavHost.kt`에 아래 adapter를 추가했다.

```text
VoiceAssistantNavigationRequest
VoiceAssistantAction.toNavigationRequest()
navigateByVoiceAssistantAction(...)
```

현재 mapping:

```text
OpenReport -> navController.navigateToTopLevel(TopLevelDestination.Report)
OpenSavedRoutes -> navController.navigateToTopLevel(TopLevelDestination.SavedRoute)
OpenMyPage -> navController.navigateToTopLevel(TopLevelDestination.MyPage)
OpenMap -> navController.navigateToTopLevelMapForHomeEntry()
SearchPlace -> navController.navigate(SearchRoute.Results.createRoute(query, editingTarget))
```

top-level 이동은 기존 `navigateToTopLevel` 정책을 그대로 사용한다. `OpenMap`은 일반 top-level map 이동이 아니라 home reentry reset 정책이 있는 `navigateToTopLevelMapForHomeEntry()`를 사용한다.

`SearchPlace`는 `editingTarget`을 보존해서 `SearchRoute.Results.createRoute(query, editingTarget)`로 이동한다. 검색 화면 mic 전환에서 넘긴 `RouteEditingTarget`이 여기까지 이어져야 출발지/도착지 편집 대상이 틀어지지 않는다.

### 13.5 아직 navigation으로 연결하지 않은 action

아래 action은 이번 작업 범위에서 실제 navigation으로 연결하지 않았다.

```text
StopNavigation
ResumeNavigationGuidance
UnknownCommand
```

현재 adapter는 위 action들에 대해 navigation request를 만들지 않고 log만 남긴다.

후속 작업 기준:

- `StopNavigation`은 navigation feature의 중단 정책, confirmation UX, 도착/지도 복귀 정책을 먼저 확정해야 한다.
- `ResumeNavigationGuidance`는 현재 안내 세션 보존 위치와 TTS 재개 정책을 먼저 확인해야 한다.
- `UnknownCommand`는 navigation이 아니라 overlay 안내 메시지나 재시도 UX로 처리하는 편이 맞다.

이 action들을 연결할 때도 contract/ViewModel에 `NavController`를 넣지 말고 app/navigation adapter에서 처리한다.

### 13.6 테스트 추가

`AppNavHostRoutingTest`에 action-to-navigation mapping 테스트를 추가했다.

테스트명:

```text

```

검증하는 내용:

- `OpenMyPage`가 `TopLevelDestination.MyPage`로 변환된다.
- `OpenMap`이 `MapHomeEntry` request로 변환된다.
- `SearchPlace("부산역", ORIGIN)`이 `SearchRoute.Results.createRoute("부산역", ORIGIN)` route request로 변환된다.
- `UnknownCommand`는 navigation request를 만들지 않는다.

테스트는 `NavController`를 직접 mock하지 않고 순수 mapping 함수를 검증한다. 실제 `NavController` 호출은 기존 `navigateToTopLevel`, `navigateToTopLevelMapForHomeEntry`, `navigate(route)` helper 정책을 따른다.

- `VoiceAssistantViewModel`에 `NavController`를 주입하지 않는다.

- 시설 상세 CTA 문자열은 현재 리소스인 `map_facility_detail_set_destination_action` 기준으로 맞췄다.
- Kakao map viewport 설정 테스트는 현재 bearing, camera tracking, facility label layer 설정에 맞췄다.

후속 AI 팀원은 실패한 테스트를 단순히 relaxed assertion으로 낮추지 말고, 먼저 현재 UI contract/resource/route helper가 실제로 무엇을 보장하는지 확인해야 한다.

```text
route registration과 policy test는 남긴다.
```

남겨둔 이유:

- 기존 route 문자열을 통해 들어오는 navigation state나 deep link 호환성을 즉시 깨지 않기 위해서다.
- `MainNavGraph`에는 legacy route registration이 남아 있다.
- `Route.kt`에는 호환용 route라는 KDoc을 추가했다.
- policy test는 이 route가 top-level 판단에서 map context로 취급되는지 계속 검증한다.

금지할 것:

- 신규 mic 버튼에서 `SearchRoute.VoiceInput.createRoute(...)`를 호출하지 않는다.
- 전역 overlay를 열 수 있는 위치에서 feature-local voice route를 새로 만들지 않는다.
- compatibility route를 제거하면서 테스트만 삭제하지 않는다. 제거하려면 사용자 진입 경로, 저장된 back stack, 딥링크 가능성을 먼저 확인한다.

### 15.7 전체 단위 테스트 중 같이 정리한 stale 테스트

전체 단위 테스트를 실행하면서 전역 음성 어시스턴트와 직접 관련은 약하지만 현재 FE 코드와 어긋난 테스트를 함께 정리했다.

대표 수정 범위:

- onboarding start destination과 external detail intent spec
- app info 외부 링크 intent spec
- DataStore test scope를 `backgroundScope`로 변경
- social token, profile, bookmark, route bookmark repository stale expectation 수정
- route geometry parser의 unsupported type 검사 순서 수정
- top-level tab selected icon expectation 수정
- route setting 화면의 현재 UI 문자열, segment count, waypoint swap metadata 기대값 수정
- report/mypage event collector가 실제 event를 놓치지 않도록 `CoroutineStart.UNDISPATCHED` 또는 `backgroundScope` 사용

이 수정들은 테스트를 억지로 통과시키기 위한 변경이 아니라, 현재 FE 구현의 실제 contract에 테스트 기대값을 맞춘 것이다.
