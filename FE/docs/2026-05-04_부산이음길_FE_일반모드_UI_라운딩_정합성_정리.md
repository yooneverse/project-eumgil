## 1. 결론

일반모드에서 우선 통일해야 할 지점은 `카드가 전반적으로 너무 둥글게 보이는 문제`와 `버튼 섀도우 기준이 역할별로 분리되지 않은 문제`다. 라운딩 문제의 원인은 Figma px를 dp로 옮긴 것 자체가 아니라, 코드의 `EumRadius` 토큰 이름과 값이 디자인 컨벤션의 radius scale과 맞지 않고, 기본 카드 성격의 surface에 `EumRadius.large`가 반복 사용된 것이다. 버튼 섀도우는 버튼 자체와 컨테이너 섀도우를 분리해서, 일반 버튼은 띄우지 않고 지도 위 플로팅 조작만 띄운다.

Figma 모바일 디자인의 `px`는 Android Compose에서 보통 `dp`로 1:1 대응한다. 따라서 `12px -> 12.dp`, `16px -> 16.dp` 적용은 타당하다. 현재 문제는 `large = 20.dp`처럼 디자인 토큰에 없는 값이 존재하고, 그 값이 기본 카드에 넓게 쓰이고 있다는 점이다.

## 2. 기준 라운딩 토큰

### 2.1 디자인 컨벤션 기준

| 디자인 토큰 | 값 | 일반모드 사용 기준 |
| --- | ---: | --- |
| `radius-xs` | 4px | 얇은 배지, 작은 보조 요소 |
| `radius-s` | 8px | 필터 chip, 작은 tag, 작은 보조 surface |
| `radius-m` | 12px | 버튼, 입력, 기본 카드, 목록 카드, 상태 카드 |
| `radius-l` | 16px | 선택 카드, 섹션 컨테이너, 시트, 지도 preview, hero surface |
| `radius-xl` | 24px | 지도 플로팅 컨트롤, 특수 강조 surface |
| `radius-full` | 999px | pill, FAB, 원형에 가까운 배지 |

일반모드에서는 값 선택을 다음처럼 고정한다. 기본 정보 카드와 반복 목록 카드는 `12dp`, 선택 카드와 큰 섹션 컨테이너는 `16dp`다. 일반모드에서 `20dp`는 디자인 컨벤션에 없는 중간값이므로 쓰지 않는다.

### 2.2 현재 코드 기준

현재 `EumRadius`는 다음 값으로 되어 있다.

```kotlin
object EumRadius {
    val small = 12.dp
    val medium = 16.dp
    val large = 20.dp
    val full = 999.dp
}
```

현재 이름 기준으로는 `small`이 디자인의 `M`, `medium`이 디자인의 `L` 역할을 하고 있고, `large`는 디자인 시스템에 없는 `20dp`다. 이 상태에서 `EumRadius.large`를 기본 카드에 쓰면 카드가 전반적으로 과하게 둥글어진다.

### 2.3 Elevation/Shadow 기준

섀도우 값은 디자인 컨벤션의 4단계만 쓴다. Compose 구현에서는 디자인 문서의 blur 값을 그대로 표현하기 어렵기 때문에, 프로젝트 컴포넌트 토큰에는 CSS 기준값과 Compose 근사값을 함께 둔다.

| 토큰 | 디자인 기준값 | Compose 기준 | 사용 기준 |
| --- | --- | ---: | --- |
| `elevation-0` | shadow 없음 | 0dp | 일반 버튼, 입력, chip, 리스트 내부 조작 |
| `elevation-1` | `0px 1px 2px rgba(0,0,0,0.06)` | 2dp | 지도 플로팅 버튼 pressed 상태, 작은 보조 overlay |
| `elevation-2` | `0px 4px 12px rgba(0,0,0,0.08)` | 6dp | 지도 플로팅 버튼, 바텀시트, 지도 위 검색/필터 overlay |
| `elevation-3` | `0px 8px 24px rgba(0,0,0,0.12)` | 10dp | dialog, modal, 큰 오버레이 컨테이너 |

버튼은 기본적으로 `elevation-0`을 쓴다. 화면 위계가 필요할 때 버튼을 띄우지 않고, 버튼을 담는 sheet, dialog, map overlay 컨테이너가 elevation을 가진다.

## 3. 토큰 정합성 개선 원칙

### 3.1 목표 토큰

코드 토큰은 디자인 컨벤션과 같은 scale을 가져야 한다.

```kotlin
object EumRadius {
    val xSmall = 4.dp
    val small = 8.dp
    val medium = 12.dp
    val large = 16.dp
    val xLarge = 24.dp
    val full = 999.dp
}
```

다만 현재 코드에서 `small`, `medium`, `large`는 이미 다른 의미로 쓰이고 있으므로 값을 먼저 바꾸면 기존 화면이 한 번에 틀어진다. 따라서 `값 변경`보다 `사용처 재분류`를 먼저 해야 한다.

### 3.2 마이그레이션 순서

1. `EumRadius`에 디자인 scale 전체를 추가한다.
2. 일반모드 surface를 역할별로 분류한다.
3. 기본 카드, 입력, CTA, chip, sheet, floating control에 맞는 새 토큰을 지정한다.
4. 화면별 `EumRadius.large` 사용처를 역할에 따라 단일값으로 낮춘다. 기본/목록/상태/입력/CTA는 `12dp`, 선택/섹션/시트/지도 preview/hero는 `16dp`다.
5. `20.dp`, `35.dp`, `MaterialTheme.shapes.large` 같은 비토큰 사용을 제거한다.
6. 마지막에 기존 토큰 값을 디자인 scale에 맞게 확정한다.

## 5. 버튼별 섀도우 통일 기준

버튼 섀도우는 버튼의 시각적 중요도 기준이 아니라 `버튼이 어떤 표면 위에 놓이는지` 기준으로 정한다. 일반 화면의 버튼은 색, 테두리, 텍스트 weight, 배치로 위계를 만들고 섀도우는 쓰지 않는다. 지도처럼 배경 정보가 계속 움직이고 버튼이 독립적으로 떠 있는 경우에만 버튼 자체 elevation을 쓴다.

### 5.1 버튼 자체 섀도우

| 버튼 역할 | Default | Pressed | Disabled/Loading | 기준 |
| --- | --- | --- | --- | --- |
| PrimaryButton / 전체폭 CTA | `elevation-0` | `elevation-0` | `elevation-0` | CTA는 fill과 위치로 강조한다. |
| SecondaryButton | `elevation-0` | `elevation-0` | `elevation-0` | outline/tonal 스타일은 shadow를 쓰지 않는다. |
| TertiaryTextButton | `elevation-0` | `elevation-0` | `elevation-0` | 텍스트 버튼은 표면을 만들지 않는다. |
| GhostButton | `elevation-0` | `elevation-0` | `elevation-0` | 배경 없는 버튼은 shadow를 쓰지 않는다. |
| DestructiveButton | `elevation-0` | `elevation-0` | `elevation-0` | 위험도는 색상과 문구로만 표현한다. |
| SocialLoginButton | `elevation-0` | `elevation-0` | `elevation-0` | 흰 배경 버튼은 1px border로 구분한다. |
| AppBarIconButton | `elevation-0` | `elevation-0` | `elevation-0` | 상단 바 표면 안의 아이콘 버튼은 띄우지 않는다. |
| ListIconButton / trailing action | `elevation-0` | `elevation-0` | `elevation-0` | 리스트 카드 내부 조작은 카드 elevation을 따르지 않는다. |
| SheetPrimaryButton / SheetSecondaryButton | `elevation-0` | `elevation-0` | `elevation-0` | sheet 컨테이너가 elevation을 가진다. |
| DialogButton | `elevation-0` | `elevation-0` | `elevation-0` | dialog 컨테이너가 elevation을 가진다. |
| NavigationControlTrayButton | `elevation-0` | `elevation-0` | `elevation-0` | 조작 tray 내부 버튼은 tray 표면에 붙인다. |
| MapFloatingControlButton | `elevation-2` | `elevation-1` | `elevation-0` | 지도 위 단독 플로팅 조작은 버튼 자체를 띄운다. |
| CurrentLocationFab / MapActionFab | `elevation-2` | `elevation-1` | `elevation-0` | 지도 위 FAB는 플로팅 컨트롤과 같은 규칙을 쓴다. |

### 5.2 컨테이너 섀도우

버튼과 컨테이너가 동시에 shadow를 가지면 화면이 탁해진다. 다음 컨테이너가 elevation을 가질 때 내부 버튼은 항상 `elevation-0`으로 둔다.

| 컨테이너 | Default | 기준 |
| --- | --- | --- |
| 일반 카드 / 목록 카드 / 상태 카드 | `elevation-0` | 일반모드 반복 surface는 border와 배경색으로 구분한다. |
| 지도 위 검색/필터 overlay | `elevation-2` | 지도 배경에서 분리되어야 한다. |
| NavigationControlTray | `elevation-2` | tray가 떠 있고, 내부 버튼은 떠 있지 않다. |
| BottomSheet | `elevation-2` | sheet 자체가 상위 표면이다. |
| Dialog / Modal | `elevation-3` | 최상위 overlay 컨테이너다. |

## 7. 저시력자 전용으로 분리해야 할 후보

다음 파일은 이번 일반모드 정리 대상에서 제외한다.

- `feature/lowvision/LowVisionHomeScreen.kt`
- `feature/lowvision/LowVisionNavigationScreen.kt`
- `feature/lowvision/LowVisionBookmarkScreen.kt`
- `feature/lowvision/LowVisionMyPageScreen.kt`
- `feature/lowvision/LowVisionVoiceInputScreen.kt`

또한 `feature/terms/TermsGuideScreen.kt`는 파일 위치상 `lowvision` 패키지는 아니지만, 검은 배경, 노란 35dp 카드, 40sp 대형 텍스트 등 저시력/고대비 흐름에 가까운 시각 언어를 사용한다. 일반모드 디자인 통일 작업에서는 즉시 수정 대상에 넣지 말고, 이 화면이 일반 약관 화면인지 저시력자 보조 가이드인지 먼저 분류해야 한다.

## 8. 우선순위

### P0: 토큰과 공통 컴포넌트

1. `EumRadius`를 디자인 scale과 맞추되, 기존 call site를 함께 마이그레이션한다.
2. `MaterialTheme.shapes.large` 의존을 제거하거나 프로젝트 shape를 주입한다.
3. `EumCard`, `EumButton`, `EumTextField`, `MapBottomSheetSurface`, `MapFloatingControl`의 radius와 elevation 기준을 먼저 문서와 맞춘다.
4. screen 코드에서 raw radius와 raw shadow/elevation을 직접 쓰지 않도록 한다.

### P1: 일반모드 주요 화면

1. 온보딩 선택 카드와 CTA
2. 검색 결과/저장 목록/마이페이지 목록 카드
3. 제보 폼 카드와 사진/설명 섹션
4. 경로 설정/경로 상세/길안내 카드
5. 지도 상태 카드와 필터/시트

### P2: 예외와 정리

1. debug section의 사용자 노출 여부 확인
2. `TermsGuideScreen`의 일반모드/저시력자 모드 소속 결정
3. 모든 `RoundedCornerShape(12.dp)`, `RoundedCornerShape(16.dp)`, `RoundedCornerShape(20.dp)`, `RoundedCornerShape(35.dp)`를 토큰 사용으로 정리

## 9. 구현 시 주의점

- `small = 12.dp`를 곧바로 `small = 8.dp`로 바꾸면 현재 입력 필드, 최근 검색 항목, 작은 카드가 의도보다 각져질 수 있다.
- `medium = 16.dp`를 곧바로 `medium = 12.dp`로 바꾸면 현재 버튼과 프로필 카드 계열이 함께 작아진다.
- 따라서 토큰 값을 먼저 바꾸지 말고, 컴포넌트 역할에 맞는 새 토큰 이름으로 call site를 바꾼 뒤 값 변경을 해야 한다.
- 일반모드에서 큰 터치 영역이 필요할 때는 corner radius가 아니라 높이, padding, min touch target으로 해결한다.
- 강조가 필요할 때도 radius를 키우기보다 border, badge, title weight, elevation, 좌측 accent bar를 먼저 사용한다.

## 10. 수용 기준

일반모드 라운딩 정합성 작업은 다음 상태가 되어야 완료로 본다.

1. 디자인 컨벤션의 `4 / 8 / 12 / 16 / 24 / full` scale과 코드 토큰이 일치한다.
2. 일반 정보 카드, 목록 카드, 상태 카드는 `12dp`를 쓴다.
3. CTA와 입력 필드는 `12dp` 기준으로 통일된다.
4. 일반 목록 카드와 상태 카드는 `20dp`를 쓰지 않는다.
5. 일반모드 선택 카드, 시트, 지도 preview, hero surface는 `16dp`를 쓴다.
6. 필터 chip과 작은 tag는 `8dp`를 쓴다.
7. pill badge만 `full`을 쓴다.
8. 플로팅 컨트롤은 `24dp`를 쓴다.
9. 일반 버튼은 `elevation-0`을 쓴다.
10. 지도 위 단독 플로팅 버튼은 default `elevation-2`, pressed `elevation-1`, disabled/loading `elevation-0`을 쓴다.
11. sheet, dialog, navigation tray 내부 버튼은 `elevation-0`을 쓰고 컨테이너만 elevation을 가진다.
12. screen 코드의 raw radius와 raw shadow/elevation 사용이 사라진다.
13. 저시력자 전용 화면은 일반모드 토큰 정리에서 분리되어 유지된다.
