## 2. 레퍼런스 비교 분석

### 2.1 온보딩 코드_성현 폴더의 장점

- 단일 모듈에서 `ui/data/di`를 분리해 초반 진입 장벽이 낮다.
- `StateFlow + collectAsState()` 흐름이 명확하고, `Intent/State/ViewModel` 패턴이 빠르게 복제 가능하다.
- `Repository`가 온라인/오프라인 분기를 숨기고 있어 발표용 fallback 구조를 설명하기 쉽다.
- `ui/component`, `ui/theme`를 먼저 만든 뒤 화면에 붙이는 흐름은 팀 초반 생산성에 유리하다.

### 2.2 온보딩 코드_성현 폴더의 한계

- `frontend/ui/MainActivity.kt`에 라우팅, 권한 요청, 초기화, 화면 등록이 과도하게 집중돼 있다.
- `frontend/data/repository/GestureRepository.kt`처럼 리포지토리 하나가 너무 많은 책임을 가지는 구조는 현재 프로젝트에서도 그대로 가져오면 빠르게 비대해진다.
- `ui/component/TemplateLayouts.kt` 같은 공통 템플릿은 유용하지만, 화면 특성이 다른 지도형 앱에 그대로 적용하면 오히려 제약이 된다.
- 패키지/파일 규칙이 문서상 템플릿과 실제 코드에서 완전히 일치하지 않는다. 그대로 복사하면 "문서상 구조"와 "실제 구조"가 다시 벌어진다.

### 2.3 특화 프로젝트 S14P21E106 FE 구조의 장점

- `app/core/data/di/feature` 구조가 현재 프로젝트에 가장 가깝다.
- `app/navigation/AppNavHost.kt`, `AuthNavGraph.kt`, `HomeNavGraph.kt`, `SettingsNavGraph.kt`처럼 NavGraph를 분리해 `MainActivity`를 얇게 유지한 점은 바로 채택할 가치가 있다.
- `feature/*Contract.kt`에 `UiState`, `UiEffect`, `Intent`를 모으는 방식은 파일 수를 과도하게 늘리지 않으면서도 상태 패턴을 고정하기 좋다.
- `core/ui/component/*`, `core/ui/theme/*`처럼 토큰과 공통 컴포넌트를 기능 코드와 분리한 점은 피그마 병행 작업에 유리하다.
- `docs/maintenance/2026-03-28_FE_디자인시스템_구현기준_및_문서정합성.md`처럼 "현재 구현 기준"과 "목표 기준"을 분리해 기록한 점은 실무적으로 좋다.

### 2.4 특화 프로젝트 S14P21E106 FE 구조의 한계

- 컴포넌트 계층이 깊고 범위가 넓다. 지금 프로젝트 일정에서 처음부터 같은 깊이로 시작하면 문서만 많고 화면은 늦어진다.
- 실제 문서에서 이미 "문서와 구현의 불일치"를 별도 문서로 관리하고 있다. 이 자체가 나쁜 것은 아니지만, 지금 프로젝트에서 같은 수준의 문서 체계를 초반부터 만들면 운영비가 크다.
- 공통 컴포넌트가 많아질수록 예외 스타일이 feature 내부에서 다시 생겼다. 지도형 서비스도 같은 실수를 반복할 가능성이 높다.
- 현재 서비스 색상/브랜드 규칙은 구독 관리 앱에 맞춰져 있다. 부산이음길에 그대로 가져오면 서비스 성격이 맞지 않는다.

### 2.5 두 레퍼런스에서 공통으로 가져갈 패턴

- 단일 `app` 모듈로 시작한다.
- `app / core / data / di / feature` 패키지 구조를 쓴다.
- `MainActivity`는 앱 진입과 `AppNavHost` 호출만 담당한다.
- `Route -> Screen -> ViewModel -> Repository` 흐름을 고정한다.
- `UiState + UiAction + UiEvent` 단방향 상태 패턴을 쓴다.
- 공통 UI는 primitive부터 만들고, feature 특화 컴포넌트는 재사용 근거가 있을 때만 올린다.

### 2.6 현재 프로젝트 일정상 버려야 할 과설계

- 멀티모듈 분리
- `domain/usecase` 계층을 feature마다 전부 만드는 구조
- 모든 리포지토리에 인터페이스/구현체 쌍을 강제하는 구조
- 지도 SDK를 완전히 추상화한 범용 지도 엔진 계층
- 초기부터 30개 이상 공통 컴포넌트를 만드는 디자인 시스템 선구축
- 다크 모드 동시 지원
- 화면 하나에서만 쓰는 UI를 `core`로 끌어올리는 습관

### 2.7 지금 가장 먼저 고정해야 할 뼈대

1. 패키지 구조
2. NavGraph 분리 방식
3. `Contract/ViewModel/Route/Screen` 파일 템플릿
4. DataStore + Room + Retrofit 기본 레이어
5. 인증 세션 gate와 서버/로컬 저장 역할 분리
6. 디자인 토큰 이름과 Figma 이름 규칙
