### 작업 전제

- 현재 지도 화면은 실제 지도 SDK가 아니라 fallback surface 기반이다.
- 문서 방향은 `MapViewport` 내부 렌더링만 실제 SDK로 교체하는 것이다.
- `MapUiState`, `cameraTarget`, `markerOverlayState`, `facilityDetailSheetState` 같은 상태 계약은 최대한 유지한다.
- 장소 데이터는 지금 seed/mock 기반이지만, 목표 구조는 실제 지도 SDK + 실제 장소 API 소비 구조다.
- 경로 API는 문서와 실제 구현 차이가 있을 수 있으므로, 이번 계획의 중심은 `MAP-01/FAC-01`이다.

## 1. 현재 상태 요약

### 1-1. 지도 렌더링 상태

- `MapViewport`는 아직 실제 SDK 지도를 그리지 않는다.
- `MapIntegrationState.Unbound`는 fallback surface를 그린다.
- `MapIntegrationState.Bound`여도 실제 SDK가 아니라 fallback surface를 재사용한다.

### 1-2. 화면 구조 상태

- `MapScreen`은 이미 `MapShellScaffold` 기반으로 상단 검색, 필터, 지도 영역, 하단 상세 시트를 조합한다.
- `FAC-01`은 별도 화면이 아니라 `MAP-01` 위의 bottom sheet overlay 구조다.
- 따라서 화면 레이아웃 자체를 다시 짜기보다 지도 렌더링 레이어만 바꾸는 방향이 맞다.

### 1-3. 상태 계약 상태

- `MapUiState`에는 이미 다음 계약이 있다.
- `cameraTarget`
- `selectedDestination`
- `selectedMarkerId`
- `locationStatus`
- `recenterButtonState`
- `markerOverlayState`
- `markerFilterState`
- `shortcutFilterState`
- `recentDestinations`
- `facilityDetailSheetState`

### 1-4. 데이터 소스 상태

- `MapViewModel`의 browse/detail 표시는 `FacilitySeedRepository`에 기대고 있다.
- `PlacesRepository`, `SearchRepository`는 존재하지만 원격 연결은 완성되지 않았다.
- `PlacesRemoteDataSource`는 `/places`, `/places/{placeId}`가 TODO 상태다.
- `SearchRemoteDataSource`는 `/places/search`가 TODO 상태다.

### 1-5. 카카오 관련 현재 코드 근거

- `FE/app.local.properties`에 `app.debug.kakaoNativeAppKey`, `app.release.kakaoNativeAppKey`가 있다.
- `FE/app/build.gradle.kts`에서 `KAKAO_NATIVE_APP_KEY`를 `BuildConfig`와 manifest placeholder로 넣고 있다.
- `FE/app/src/main/AndroidManifest.xml`에 Kakao auth scheme 설정이 있다.
- `FE/app/src/main/java/com/ssafy/e102/eumgil/app/BusanEumgilApp.kt`에서 `KakaoSdk.init(...)`를 수행한다.
- 즉, 현재 프로젝트의 공급사 맥락은 카카오 쪽이 맞다.
- 다만 지도 SDK 의존성 자체는 아직 `build.gradle.kts`에 추가되어 있지 않으므로, 지도 SDK 연동은 별도 작업이다.

## 2. 문서 기준 target contract 요약

### 2-1. 화면 계약

- `MAP-01`은 실제 지도 위에 검색, 필터, 현재 위치, 마커, 상세 overlay가 올라가는 구조여야 한다.
- `FAC-01`은 별도 route가 아니라 `MAP-01` 내부 bottom sheet overlay로 유지한다.

### 2-2. SDK 교체 계약

- 실제 지도 SDK로 바꿀 때 `MapViewport` 내부 렌더링만 교체한다.
- `MapScreen`, `MapShellScaffold`, overlay slot 구조는 유지한다.
- ViewModel과 상위 상태 계약은 SDK 타입을 몰라야 한다.

### 2-3. 데이터 계약

- 지도 browse 데이터는 `GET /places`
- 상세 sheet 데이터는 `GET /places/{placeId}`
- 검색 결과는 `GET /places/search`
- 음성 검색 진입은 `POST /voice/analyze`

### 2-4. 분리 원칙

- “지도 SDK 자체 연동”과 “지도 위에 올릴 데이터 교체”는 분리해서 본다.
- 1단계에서는 둘 다 다루되, 순서를 명확히 분리한다.
- 경로 preview/navigation map 확장은 2단계 후속 계획으로 남긴다.

## 3. SDK 연동 시 유지해야 할 상태/컴포넌트 경계

### 유지해야 할 상태 계약

- `MapUiState` 필드 구조
- `cameraTarget.requestId` 기반 카메라 이동 트리거
- `selectedMarkerId`와 상세 sheet 선택 상태 연동
- `facilityDetailSheetState`의 상위 제어 책임
- `PlaceDestination` handoff 계약

### 유지해야 할 컴포넌트 경계

- `MapScreen`은 화면 조립만 담당한다.
- `MapViewModel`은 상태 계산과 이벤트 처리만 담당한다.
- `MapViewport`는 지도 provider adapter의 진입점만 담당한다.
- SDK 타입과 marker/camera 객체는 `feature/map/component` 내부로 숨긴다.

### 유지해야 할 fallback 경계

- Preview
- Test
- SDK 초기화 실패 시 개발용 fallback

6. Preview/Test에서 SDK 비활성 fallback을 유지할 최소 조건 정리

## 6. SDK 후보와 tradeoff

| 후보 | 장점 | 단점 | 판단 |
| --- | --- | --- | --- |
| Google Maps SDK | 레퍼런스가 많고 Compose 연계 자료가 풍부함 | 국내 POI 맥락과 현재 백엔드 검색 공급사 설명력이 약함 | 우선순위 낮음 |
| Kakao Map SDK | 현재 프로젝트의 Kakao key/init 흐름과 공급사 맥락이 맞고, 국내 지도/POI 친화성이 높음 | Compose 전용 API가 아니어서 `AndroidView` 브리지가 필요하고, 지도 SDK 의존성은 별도 추가해야 함 | 추천 |
| Naver Map SDK | 안정적인 대체안이며 marker/camera/polyline 기능은 충분함 | 현재 프로젝트의 Kakao 환경설정, 장소 검색 맥락과 직접 이어지지 않음 | 후보 가능하지만 기준안은 아님 |

### 추천

- 추천 SDK: `Kakao Map SDK`
- 추천 이유 1: 현재 저장소의 앱 키, manifest, 앱 초기화 흐름이 이미 Kakao 축으로 정리되어 있다.
- 추천 이유 2: 장소 검색 결과의 공급사 맥락과 맞춰 운영 설명을 단순화할 수 있다.
- 추천 이유 3: 이번 작업의 핵심은 상위 상태를 유지한 채 `MapViewport` 내부 엔진만 교체하는 것이므로, 공급사 일관성을 우선하는 편이 낫다.

## 7. 단계별 작업계획

### 7-1. 1단계: MAP-01/FAC-01 실제 지도 SDK + 장소 마커/상세 연결

#### Step 1. SDK skeleton 정리

- `FE/app/build.gradle.kts`에 Kakao Map SDK 의존성을 추가할 위치를 확정한다.
- 기존 `KAKAO_NATIVE_APP_KEY`를 지도 SDK 설정에 재사용 가능한지 확인한다.
- 필요 시 `AndroidManifest.xml`, `BusanEumgilApp.kt`의 지도 SDK 초기화 지점을 정리한다.
- Preview/Test에서 지도 SDK 없이도 fallback surface가 동작하도록 guard 기준을 정한다.

#### Step 2. provider adapter 경계 정의

- `MapViewport`는 상위 API를 유지하고 내부에서 provider-backed renderer를 호출하는 진입점으로 고정한다.
- 신규 provider 전용 파일을 `feature/map/component` 아래에 둔다.
- SDK 객체와 lifecycle, marker diffing, camera sync는 provider adapter 내부 책임으로 묶는다.
- `MapViewModel`, `MapContract`, `MapScreen`에는 SDK 타입을 노출하지 않는다.

#### Step 3. `MapViewport` 실제 SDK 교체

- `MapIntegrationState.Bound`일 때만 실제 카카오 지도 renderer를 붙인다.
- `cameraTarget.requestId`를 카메라 이동 트리거로 유지한다.
- marker click은 기존 `onMarkerClick(markerId)` 계약을 그대로 유지한다.
- Unbound/preview/test 경로는 fallback surface를 유지한다.

#### Step 4. `GET /places` 기반 browse 마커 연결

- `PlacesRemoteDataSource`의 `/places` 호출 TODO를 실제 구현 대상으로 전환한다.
- DTO, mapper, repository를 통해 map browse 모델로 변환하는 흐름을 만든다.
- `MapViewModel`의 seed browse 의존을 repository 기반 browse로 치환한다.
- `FacilitySeedRepository`는 production browse source가 아니라 preview/demo/test fallback 용도로 축소한다.

#### Step 5. marker/filter 매핑 규칙 정리

- 현재 FE quick filter와 API `featureType`을 일대일 또는 우선순위 규칙으로 매핑한다.
- 현재 seed 기반 marker category와 API `category`/`featureType` 차이를 adapter 규칙으로 흡수한다.
- marker icon/visual rule은 ViewModel이 아니라 renderer 입력 모델 또는 전용 mapper에서 결정한다.

#### Step 6. `GET /places/{placeId}` 기반 FAC-01 상세 연결

- marker tap 시 `placeId`를 기준으로 상세 fetch를 수행한다.
- `facilityDetailSheetState`는 기존 상위 제어 구조를 유지한 채 실제 응답으로 채운다.
- `PlaceDestination` handoff는 그대로 유지한다.
- 상세 응답의 bookmark/accessibility/설명 필드가 현재 sheet UI와 어떻게 대응되는지 정리한다.

#### Step 7. `GET /places/search`와 검색 진입 정리

- `SearchRemoteDataSource`의 `/places/search` TODO를 실제 구현 대상으로 전환한다.
- `SearchRepository`와 `SearchViewModel`이 실제 검색 결과를 소비하도록 정리한다.
- `matched=false`, `placeId=null` 결과는 1단계에서 정책만 확정하고, UI는 최소 안전 동작 기준으로 처리한다.
- `POST /voice/analyze`는 검색어 생성 진입점으로만 보고, 지도 SDK 연동 계획과는 분리한다.

#### Step 8. 회귀 방지 테스트 추가

- `PlacesRemoteDataSource` 테스트
- `SearchRemoteDataSource` 테스트
- `PlacesRepository`/`SearchRepository` 매핑 테스트
- `MapViewModel` browse/detail/search 상태 전환 테스트
- manual QA 시나리오 문서화

### 7-2. 2단계: route preview/navigation map overlay 확장용 후속 계획

#### Step 1. 공용 map renderer 패턴 고정

- 1단계에서 만든 provider adapter 구조를 route preview/navigation에도 재사용 가능한 형태로 정리한다.
- `MapViewport`와 별개로 overlay 입력 모델을 재사용하는 방향을 명확히 한다.

#### Step 2. overlay 모델 확장

- polyline
- current location puck
- destination pin
- step highlight

#### Step 3. ROUTE-01 preview 적용 계획

- route preview는 지도 SDK 연동 자체보다 overlay 확장 문제로 분리한다.
- 경로 API 확정 전까지는 지도 renderer 확장 포인트만 정의한다.

#### Step 4. NAV-01 적용 계획

- navigation map은 현재 위치 추적, reroute, maneuver emphasis가 필요하므로 1단계와 분리한다.
- route API 문서와 실제 구현 차이가 정리된 뒤 별도 구현 계획으로 이어간다.

### 수동 검증

- `MAP-01` 진입 시 실제 지도 렌더링 확인
- 현재 위치 recenter 동작 확인
- quick filter 선택 시 marker 반영 확인
- marker tap 시 `FAC-01` sheet 열림과 상세 데이터 반영 확인
- 검색 결과에서 장소 선택 후 지도/상세/목적지 handoff 연결 확인

### fallback 검증

- Preview에서 SDK 없이도 화면이 깨지지 않는지 확인
- Test 환경에서 fallback surface로 상태 테스트가 가능한지 확인

## 10. 리스크

1. Kakao login 설정과 Kakao map 설정 요구사항이 다를 수 있다.
2. `/places`의 `category`와 `featureType`이 현재 FE marker/filter 모델과 정확히 일치하지 않을 수 있다.
3. `/places/search`의 `matched=false`, `placeId=null` 결과가 현재 FE 검색/목적지 계약과 충돌할 수 있다.
4. Compose `AndroidView` 기반 SDK 연동에서 lifecycle 누수나 blank map 문제가 날 수 있다.
5. 지도 SDK 렌더링은 단위 테스트만으로 다 검증하기 어려워 수동 QA 의존도가 높다.

## 11. 백엔드 확인 필요 항목 handoff

1. `/places`에서 지도 browse용 query parameter, 페이징, 기본 정렬 기준이 무엇인지 확인 필요
2. `/places` 응답의 `category`와 accessibility `featureType` enum이 FE가 바로 매핑 가능한 수준으로 고정되어 있는지 확인 필요
3. `/places/{placeId}`에서 `description`, `isBookmarked`, 접근성 필드가 항상 내려오는지 확인 필요
4. `/places/search`의 `matched=false`, `placeId=null`, `providerPlaceId` 조합에서 FE가 목적지 handoff를 어떻게 해야 하는지 계약 확인 필요
5. `POST /voice/analyze` 결과가 바로 `/places/search` 입력으로 연결되는 단일 query인지, 추가 정규화가 필요한지 확인 필요

## 결론

- 1단계는 `MapViewport`를 `Kakao Map SDK` 기준으로 실제 지도 엔진으로 교체하고, `GET /places`와 `GET /places/{placeId}`로 `MAP-01/FAC-01`의 마커와 상세를 실제 데이터로 바꾸는 작업이다.
- 2단계는 같은 provider adapter와 overlay 입력 모델을 route preview/navigation map으로 확장하기 위한 후속 계획이다.
- 전체 방향은 “상위 상태 계약 유지, SDK 타입은 viewport 내부로 격리, seed/mock은 preview fallback으로 축소”로 고정한다.
