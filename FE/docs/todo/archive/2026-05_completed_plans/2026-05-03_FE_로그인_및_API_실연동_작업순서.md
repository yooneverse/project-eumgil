# FE 로그인 및 API 실연동 작업순서

> **작성일** 2026-05-03
> **작성자** 이재호
> **문서 역할** 현재 FE의 로컬 mock 로그인 구조를 실제 서버 인증/API 연동 구조로 전환하기 위한 작업 순서 문서
> **상위 기준 문서** `Docs/기획/2026-04-09_화면명세서.md`, `Docs/PRD/2026-04-14_기능명세서.md`, `Docs/API/*`
> **비역할** 이 문서는 제품 정책 자체를 새로 결정하지 않는다. 정책 변경은 상위 문서를 먼저 수정한다.

---

## 1. 현재 상태 요약

- 로그인은 필수 정책이지만, FE는 아직 실제 소셜 인증과 서버 로그인 API를 붙이지 못했다.
- 현재 로그인 버튼은 `LocalOnlyAuthLoginRepository`를 통해 로컬 mock 세션만 저장한다.
- 신규 사용자 `signupToken` 분기, `POST /auth/signup` 완료 시 세션 생성, `/users/me` 기반 사용자 타입 동기화가 구현되어 있지 않다.
- 장소, 검색, 경로, 제보, 북마크도 일부는 mock/local 중심이거나 원격 DataSource가 미구현 상태다.

---

## 2. 선행 확인 사항

실연동 작업을 시작하기 전에 아래 항목을 먼저 확정한다.

1. 백엔드 dev base URL, 인증 헤더 규칙, 공통 응답 래퍼 형식
2. `POST /auth/social-login`의 기존 회원/신규 사용자 응답 예시
3. `POST /auth/signup`, `POST /auth/reissue`, `POST /auth/logout`, `GET /users/me`의 실제 스펙과 에러 코드
4. 카카오/네이버/구글 중 실제 FE가 먼저 붙일 소셜 제공자 우선순위
5. 제보 이미지 업로드 정책
6. mock fallback을 실연동 이후에도 debug에서 유지할지 여부

---

## 3. 목표 전환 구조

현재 구조:

```text
AUTH-001 클릭
-> 로컬 mock accessToken 저장
-> 로컬 DataStore 기준으로 다음 화면 결정
```

목표 구조:

```text
AUTH-001 소셜 인증
-> POST /auth/social-login
-> 기존 회원: accessToken/refreshToken 저장 -> 사용자 유형 기준 홈 분기
-> 신규 사용자: signupToken 저장 -> ONB-001~ONB-005
-> POST /auth/signup 성공 시 accessToken/refreshToken 저장
-> 이후 /users/me 기준으로 사용자 정보 동기화
```

---

## 4. 권장 작업 순서

### 0. 네트워크 기반 준비

- Retrofit/OkHttp 또는 팀에서 사용할 HTTP 클라이언트를 앱에 추가한다.
- `AuthApi`, 공통 응답 DTO, 인증 헤더 주입 구조를 먼저 만든다.
- 로컬 mock 전용 구현과 실연동 구현을 분리할 수 있게 DI 구조를 정리한다.

이 단계가 먼저 필요한 이유:
- 현재 앱에는 로그인 API를 실제로 호출할 네트워크 스택이 없다.

### 1. 로그인 도메인 모델 정리

- `AuthSession`, `AuthGateState`, 관련 저장소 모델에서 문서 기준과 맞지 않는 `isProfileCompleted` 중심 분기를 제거한다.
- `signupToken`, `selectedPrimaryUserType`, `selectedMobilitySubtype`를 실연동 기준으로 다시 정의한다.
- 앱 시작 분기 기준을 `세션 없음 / signupToken 있음 / 기존 세션 있음` 구조로 단순화한다.

수정 대상:
- `FE/app/src/main/java/com/ssafy/e102/eumgil/data/repository/AuthLoginRepository.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/data/repository/AuthSessionRepository.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/data/local/datasource/AuthSessionLocalDataSource.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/core/model/AuthModels.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/app/navigation/AppStartDestination.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/app/navigation/AuthNavGraph.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/app/navigation/AppNavHost.kt`

### 2. AUTH-001 실연동

- 소셜 제공자 SDK 또는 인증 토큰 획득 경로를 붙인다.
- `providerKey` UI 값과 서버 요청값 `socialProvider`를 분리한다.
- `socialAccessToken`을 받아 `POST /auth/social-login`을 호출한다.
- 기존 회원과 신규 사용자를 응답값으로 분기한다.

완료 기준:
- 기존 회원은 서버 토큰 저장 후 홈 분기로 이동한다.
- 신규 사용자는 `signupToken`만 저장하고 `ONB-001`로 이동한다.

### 3. 온보딩 완료와 회원가입 API 연결

- 현재 로컬 저장만 하던 온보딩 값을 `POST /auth/signup` 요청값에 매핑한다.
- `ONB-005` 완료 시 `signupToken + selectedPrimaryUserType + selectedMobilitySubtype + requiredTermsAccepted`를 보내 가입을 완료한다.
- 성공하면 그때 `accessToken/refreshToken`을 저장한다.

수정 대상:
- `FE/app/src/main/java/com/ssafy/e102/eumgil/app/navigation/OnboardingNavGraph.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/data/local/datasource/InitSettingsLocalDataSource.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/data/repository/SettingsRepository.kt`

### 4. `/users/me` 동기화와 앱 시작 분기 안정화

- 앱 시작 시 로컬 토큰만 믿지 말고 `/users/me` 응답과 사용자 유형을 동기화할 구조를 만든다.
- `selectedPrimaryUserType=LOW_VISION`이면 `LV-HOM-01`, `MOBILITY_IMPAIRED`이면 `MAP-01`로 분기한다.
- 로그아웃, 재실행, 토큰 만료 시나리오를 이 단계에서 같이 정리한다.

왜 이 단계가 중요한가:
- 이후 북마크, 제보, 자주 가는 길은 모두 인증 세션이 안정적으로 잡혀 있어야 한다.

### 5. 장소/검색 API 연동

- `GET /places/search`, `GET /places`, `GET /places/{placeId}`를 우선 연결한다.
- 검색 결과 `placeId` nullable 처리와 장소 카테고리/접근성 feature 매핑을 맞춘다.

왜 인증 다음에 붙이나:
- 경로, 북마크, 제보보다 공용 의존성이 크고, `placeId` 모델이 이후 기능의 기준값이 된다.

### 6. 경로 API 연동

- `POST /routes/search`를 연결한다.
- 응답의 `warningLabels`, `usedConditionalStairs`, 세그먼트 위험 정보 매핑을 맞춘다.
- 필요 시 `POST /route-ratings`, `POST /routes/reroute`를 이어서 붙인다.

왜 장소/검색 다음인가:
- 목적지 선택과 내부 `placeId` 모델이 정리된 뒤 붙이는 편이 충돌이 적다.

### 7. 북마크/자주 가는 길 연동

- 장소 북마크와 자주 가는 길을 서버 API 기준으로 분리해 연결한다.
- 기존 Room/local-only 흐름은 캐시 역할로 축소한다.

왜 경로 다음인가:
- 자주 가는 길 저장 구조가 경로/장소 모델 확정에 의존한다.

### 8. 제보 API 연동

- `POST /hazard-reports`와 내 제보 조회 API를 붙인다.
- `reportType`, `reportPoint`, `imageUrls` 기준으로 FE 상태를 정리한다.

왜 뒤로 두나:
- 인증 의존성은 높지만 장소/경로 모델처럼 다른 기능의 기준 모델은 아니어서 독립 처리 가능하다.

### 9. 로그아웃/토큰 재발급/에러 처리 정리

- `POST /auth/logout`, `POST /auth/reissue`를 붙인다.
- 401 응답 처리, 세션 만료 후 `AUTH-001` 복귀, 로컬 세션 정리 방식을 확정한다.

### 10. 문서와 테스트 정리

- FE 문서에서 로컬 mock 로그인 전제를 제거하거나 debug-only로 명시한다.
- 로그인, 시작 분기, 온보딩, 장소/검색, 경로, 제보, 북마크 단위 테스트를 갱신한다.
- 마지막에 전체 회귀 테스트와 compile 검증을 수행한다.

---

## 5. 선후관계 요약

1. 인증 모델 정리와 AUTH-001 실연동이 가장 먼저다.
2. 장소/검색은 경로와 북마크보다 먼저 붙인다.
3. 경로는 장소 모델 정리 후 붙인다.
4. 북마크는 장소/경로 연동 이후 붙인다.
5. 제보는 인증 이후 독립적으로 병행 가능하지만, 우선순위는 장소/경로보다 낮다.
6. 로그아웃/재발급과 전체 문서 정리는 마지막 마감 단계에서 묶는다.

---

## 6. 1차 완료 기준

- `AUTH-001`이 실제 소셜 인증과 `POST /auth/social-login`을 사용한다.
- 기존 회원과 신규 사용자가 `accessToken`과 `signupToken`으로 올바르게 분기된다.
- `ONB-005` 완료 시 `POST /auth/signup`이 호출되고, 성공 후 세션이 생성된다.
- 앱 시작 분기가 `profile_completed`가 아니라 인증 세션과 사용자 유형 기준으로 동작한다.
- 장소 검색, 장소 상세, 경로 검색, 북마크, 제보가 최소 1회 이상 dev 서버와 실제 연동된다.
- FE 문서에서 “로컬 mock 로그인 전용 구조”를 현재 기준선처럼 읽히지 않게 정리한다.

---

## 7. 확인 필요 메모

- 소셜 인증을 FE SDK에서 먼저 처리할지, 웹뷰/백엔드 위임 방식인지 확정이 필요하다.
- 신규 사용자의 `signupToken` 보존 위치를 `AuthSession`과 별도 상태로 둘지 결정이 필요하다.
- 이미지 업로드 API가 분리되면 제보 API 작업 순서를 앞뒤로 다시 조정할 수 있다.
- debug mock 로그인은 유지하더라도 기본 주입 구현과 명확히 분리해야 한다.
