# FE 리플 효과 및 클릭 host 표준화 작업계획

## 1. 작업 배경

- 이번 작업의 핵심 목적은 앱 전반의 ripple 색상을 한 번에 바꾸는 것이 아니라, 클릭을 받는 host 컴포넌트의 기준을 통일해 화면별 구현 편차를 줄이는 것이다.
- 현재 FE 코드에는 ripple/press indication을 전역에서 일괄 제어하는 공통 레이어가 거의 없고, 각 화면이 `Button`, `Surface`, `Modifier.clickable`, `selectable`, `rememberRipple`, `indication = null`을 개별적으로 조합하고 있다.
- 이 상태에서는 같은 역할의 UI라도 눌림 범위, shape clipping, 접근성 role, disabled 처리, 중첩 클릭 동작이 화면마다 달라질 가능성이 높다.
- 특히 카드 전체 클릭, 탭/칩 선택, 검색바 클릭, 보조 액션 버튼이 섞인 화면에서 구현 방식 차이가 눈에 띄며, 이후 화면 추가 시 같은 문제가 반복될 가능성이 높다.
- 본 문서는 FE 코드만을 대상으로 하는 점진적 리팩터링 계획서이며, BE 수정 없이 진행 가능한 범위만 다룬다.

## 2. 현재 구현 현황

### 2.1 현재 구현 사실

- 전역 ripple 정책을 제공하는 공통 Provider 또는 wrapper는 현재 확인되지 않았다.
- `rememberRipple()` 직접 사용은 코드 검색 기준 아래 위치에서 확인됐다.
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/core/designsystem/component/place/PlaceListCard.kt`
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/lowvision/LowVisionSearchScreen.kt`
- `indication = null` 사용은 현재 확인 범위에서 아래 2곳이다.
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/FacilityDetailBottomSheetShell.kt`
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/MapTopSearchBar.kt`
- `Surface(onClick = ...)` 사용 사례는 이미 일부 존재한다.
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/core/designsystem/component/map/EumMapFloatingControls.kt`
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/MapShortcutFilterRow.kt`
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/search/SearchScreen.kt`
- 반면 같은 “전체를 누르는 UI”가 아래처럼 여러 방식으로 구현돼 있다.
  - `Surface + Modifier.clickable(...)`
  - `Column/Row/Box + Modifier.clickable(...)`
  - `selectable(...)`
  - Material Button 계열 직접 사용
- 탭/선택형 UI의 기준도 분산돼 있다.
  - `EumTopLevelTabBar`는 `selectableGroup + selectable(role = Role.Tab)`를 사용한다.
  - `OnboardingSelectionCard`는 `selectable(role = Role.RadioButton)`를 사용한다.
  - `SavedRouteScreen`의 탭 버튼은 `clickable(role = Role.Tab)` 기반이다.
  - `LowVisionHomeBottomNav`는 `clickable(role = Role.Button)` 기반이다.
  - `RouteSettingScreen`의 교통수단 탭과 경로 옵션 카드는 `clickable`과 수동 semantics 조합을 사용한다.
- 문서상 `EumButton`은 언급돼 있으나, 현재 코드 검색 기준으로 공용 구현 파일은 확인되지 않았다.
  - 문서 언급: `FE/docs/2026-04-13_부산이음길_FE_코드_컨벤션.md`
  - 추가 문서 언급: `FE/docs/2026-04-13_부산이음길_FE_컴포넌트_가이드.md`

### 2.2 제안하는 목표 구조

- `CTA 버튼`은 `Button`, `IconButton`, `TextButton`, `OutlinedButton` 중심으로 정리한다.
- `카드/칩/탭/검색바`처럼 전체가 눌리는 UI는 `Surface(onClick)` 또는 `selectable` 중심으로 정리한다.
- `scrim/overlay`만 `indication = null` 예외로 유지한다.
- `rememberRipple()`는 브랜드 색상을 꼭 고정해야 하는 일부 예외만 남기고 제거 방향으로 간다.
- 표준화의 우선순위는 `ripple 색상 통일`보다 `클릭 host 통일`, `선택 semantics 통일`, `shape 안에서 눌림 범위가 일관되게 보이도록 만드는 것`에 둔다.

## 3. 문제 유형 정리

| 문제 유형 | 현재 구현 사실 | 정리 방향 |
| --- | --- | --- |
| 같은 역할 UI의 host 불일치 | 탭/칩/카드/검색바가 `clickable`, `Surface(onClick)`, `selectable`로 혼재 | 역할별 host를 고정한다. CTA는 Material Button 계열, 선택형은 `selectable`, 전체 클릭 카드는 `Surface(onClick)` 중심으로 정리한다. |
| 시각 컨테이너와 클릭 host 분리 | `MapTopSearchBar`, `SavedBookmarkTabButton`, `RouteTravelModeTab`, `RouteOptionCard` 등에서 `Surface`는 장식만 맡고 클릭은 `Modifier.clickable`이 담당 | shape, ripple, border, enabled 상태를 같은 host가 관리하도록 이동한다. |
| ripple 예외 규칙 부재 | `MapTopSearchBar`도 `indication = null`을 사용하고 있어 scrim 예외와 일반 클릭 요소가 섞여 있음 | `indication = null`은 scrim/overlay로만 제한한다. 일반 검색바/카드/탭은 기본 indication을 복원한다. |
| 직접 ripple 커스텀 산재 | `PlaceListCard`, `LowVisionSearchScreen`에서 `rememberRipple()`을 직접 사용 | 공통 host 도입 후 기본 indication으로 치환하고, 브랜드 색 강제 이유가 명확한 경우만 예외 문서화한다. |
| 선택 semantics 불일치 | 일부 탭은 `Role.Tab`, 일부는 `Role.Button`, 일부는 `Role.RadioButton`을 사용 | 탭, 라디오형 카드, 경로 옵션 선택형 UI를 역할별로 구분해 `selected/stateDescription` 기준을 통일한다. |
| 중첩 클릭 충돌 가능성 | 전체 카드 클릭과 내부 보조 버튼이 같이 있는 카드가 여럿 존재 | 부모 클릭은 “상세 이동/대표 액션”만, 자식 버튼은 “보조 액션”만 담당하도록 규칙을 명확히 한다. |

## 4. 표준화 원칙

1. `CTA 버튼`은 기본적으로 `Button`, `IconButton`, `TextButton`, `OutlinedButton`을 사용한다.
2. 버튼의 시각 규격이 반복되면 wrapper를 만들되, Button 계열 자체를 우회하는 custom clickable 버튼부터 늘리지 않는다.
3. 카드, 검색바, 리스트 아이템, 플로팅 액션처럼 “컨테이너 전체가 눌리는 요소”는 `Surface(onClick)`을 기본값으로 본다.
4. 탭, 라디오 카드, 선택 칩처럼 선택 상태가 핵심인 UI는 `selectable`과 `selectableGroup`을 기본값으로 본다.
5. `indication = null`은 scrim, overlay dismiss 영역처럼 눌림 피드백이 없어야 하는 경우에만 허용한다.
6. `rememberRipple()`는 브랜드 색상 또는 특수 contrast 대응이 꼭 필요한 경우만 유지하고, 유지 이유를 코드 주석 또는 문서로 남긴다.
7. 클릭 host는 가능하면 shape, border, background, enabled 상태를 함께 관리해야 한다.
8. 시맨틱 정보는 host와 일치해야 한다. 버튼은 `Role.Button`, 탭은 `Role.Tab`, 라디오형 선택 카드는 `Role.RadioButton` 또는 이에 준하는 selectable semantics를 사용한다.
9. 최소 터치 영역은 48dp 이상을 유지하고, 저시력 모드에서는 현재보다 작아지지 않도록 한다.
10. 부모와 자식이 둘 다 clickable인 구조에서는 이벤트 우선순위와 눌림 범위를 의도적으로 설계한다.

## 5. 공통 컴포넌트 설계 방향

### 5.1 `EumClickableSurface`

- 제안 위치: `FE/app/src/main/java/com/ssafy/e102/eumgil/core/designsystem/component/surface/EumClickableSurface.kt`
- 목적: `Surface(onClick)`를 공통 host로 삼아 카드/검색바/리스트 아이템의 클릭 처리, shape clipping, border, enabled 상태, semantics를 한 곳에서 맞춘다.
- 1차 책임 범위
  - `onClick`, `enabled`, `role`, `onClickLabel`
  - `shape`, `color`, `border`, `shadowElevation`, `tonalElevation`
  - `mergeDescendants`, `contentDescription`, `stateDescription` 같은 기본 semantics 훅
- 1차 적용 후보
  - `MapTopSearchBar`
  - `SavedRouteScreen`의 탭 버튼 또는 카드형 리스트 아이템
  - `RouteSettingScreen`의 경로 옵션 카드
  - `PlaceListCard`, `LowVisionSearchResultCard`

### 5.2 `EumSelectableCard` 또는 `EumSelectableTab`

- 제안 위치
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/core/designsystem/component/selection/EumSelectableCard.kt`
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/core/designsystem/component/selection/EumSelectableTab.kt`
- 목적: 선택형 카드/탭의 `selected`, `stateDescription`, `role`, `selectableGroup` 사용 규칙을 공통화한다.
- 1차 책임 범위
  - `selected`, `enabled`, `onClick`
  - `role = Role.Tab` 또는 `Role.RadioButton`
  - 선택/비선택 색상 토큰 주입
  - 선택 상태 semantics 일관화
- 1차 적용 후보
  - `EumTopLevelTabBar`
  - `OnboardingSelectionCard`
  - `SavedRouteScreen`의 `SavedBookmarkTabButton`
  - `RouteSettingScreen`의 `RouteTravelModeTab`
  - 필요 시 `RouteOptionCard`도 selectable 카드로 확장 검토

### 5.3 `EumPrimaryButton`

- 제안 위치: `FE/app/src/main/java/com/ssafy/e102/eumgil/core/designsystem/component/button/EumPrimaryButton.kt`
- 목적: 공용 `EumButton` 구현이 아직 불명확한 상태에서, 실제 화면에 가장 많이 필요한 “기본 CTA”부터 우선 정리한다.
- 원칙
  - 내부 구현은 `Button` 기반으로 유지한다.
  - loading/disabled/leading icon 같은 공통 패턴만 먼저 묶는다.
  - `Outlined`나 `Text` 변형은 필요 시 `EumSecondaryButton`, `EumTextActionButton`으로 분리하거나 이후 `EumButton` 계열로 확장한다.
- 1차 적용 후보
  - `OnboardingStepScaffold`
  - `SavedBookmarkStateActions`
  - `RouteSettingBottomBar`
  - `RecentDestinationBottomSheetShell`

## 7. 단계별 작업 계획

### 1차. 공통 컴포넌트 도입

1. `core/designsystem/component` 아래에 `surface`, `selection`, `button` 패키지 추가 여부를 결정한다.
2. `EumClickableSurface` 초안을 작성하고 preview 또는 샘플 호출로 shape, border, disabled 상태를 먼저 검증한다.
3. `EumSelectableCard` 또는 `EumSelectableTab` 초안을 작성하고 `selected/stateDescription/role` 기준을 정한다.
4. `EumPrimaryButton` 초안을 작성하되, full `EumButton` 체계 전체를 한 번에 구현하려 하지 않는다.
5. 공통 컴포넌트 도입 단계에서 “scrim만 `indication = null` 허용” 규칙을 문서와 코드 주석에 함께 반영한다.

### 2차. 대표 화면 이관

1. `MapTopSearchBar`를 먼저 이관해 일반 인터랙션에서 `indication = null`을 제거한다.
2. `PlaceListCard`와 `LowVisionSearchResultCard`를 묶어 카드 host와 내부 액션 버튼 host를 분리 정리한다.
3. `SavedRouteScreen`의 `SavedBookmarkTabButton`을 selectable 패턴으로 이관하고, 상태 카드 CTA는 Button 계열 기준으로 정리한다.
4. `RouteSettingScreen`에서 아래 순서로 이관한다.
   - `RouteTravelModeTab`
   - `RouteWaypointRow`
   - `RouteWaypointSwapButton`
   - `RouteOptionCard`
   - `RouteOptionDetailArrowButton`
   - 하단 CTA
5. 이 단계에서 `clickable(role = Role.Button)`로 구현된 선택형 UI가 실제로 탭/라디오 의미를 갖는지 다시 분류한다.

### 3차. 잔여 화면 정리

1. `LowVisionHomeBottomNav`, `SearchScreen`, `RecentDestinationBottomSheetShell`, `OnboardingStepScaffold`를 대상으로 잔여 규칙을 정리한다.
2. `rg` 기준으로 남아 있는 `rememberRipple()`과 비-scrim `indication = null` 사용처를 다시 점검한다.
3. 역할이 같은 UI끼리 아직 host 방식이 다른 케이스를 정리한다.
4. 공통 컴포넌트 적용이 오히려 복잡도를 높이는 단발성 화면은 억지로 이관하지 않고 예외 목록에 남긴다.

## 8. 테스트 및 검증 계획

### 8.1 수동 검증

1. 클릭 영역이 의도한 컨테이너 전체를 덮는지 확인한다.
2. ripple 범위가 shape 밖으로 새지 않는지 확인한다.
3. border, background, ripple이 같은 radius를 기준으로 clipping되는지 확인한다.
4. 탭/카드/버튼의 접근성 role이 실제 의미와 맞는지 TalkBack으로 확인한다.
5. `selected`, `stateDescription`, `contentDescription`이 중복되거나 빠지지 않는지 확인한다.
6. disabled 상태에서 시각 상태와 실제 클릭 차단이 일치하는지 확인한다.
7. 카드 전체 클릭과 내부 보조 버튼 클릭이 동시에 발화하지 않는지 확인한다.
8. 저시력 화면에서 터치 타깃이 축소되지 않았는지 확인한다.
9. scrim 영역은 계속 ripple 없이 dismiss만 동작하는지 확인한다.

## 9. 리스크 및 롤백 기준

### 9.1 주요 리스크

- `Surface(onClick)`로 옮기면서 기존 clickable 순서에 의존하던 semantics가 달라질 수 있다.
- 카드 전체 클릭과 내부 버튼 클릭이 있는 UI에서 이벤트 충돌이 발생할 수 있다.
- 저시력 모드에서 기존 커스텀 ripple 제거가 시인성 저하로 느껴질 수 있다.
- 선택형 UI를 selectable로 바꾸는 과정에서 기존 `Role.Button` 기반 접근성 문구와 충돌할 수 있다.
- 공통 컴포넌트를 너무 크게 설계하면 초기 이관 범위보다 설계 작업이 커질 수 있다.

### 9.2 롤백 기준

1. parent와 child 클릭이 한 번의 탭에서 중복 발화하면 해당 화면 이관을 즉시 되돌린다.
2. ripple이 shape 밖으로 번지거나 clipping이 깨지면 공통 host 적용을 보류하고 화면 단위로 롤백한다.
3. disabled인데도 클릭이 동작하면 해당 컴포넌트 도입을 롤백한다.
4. TalkBack role 또는 selected 상태가 이전보다 불명확해지면 selectable 추상화를 재검토한다.
5. 저시력 모드에서 시각적 피드백이 약해졌다는 QA 피드백이 있으면 해당 화면만 예외 규칙으로 분리하고 전체 정책은 유지한다.

### 9.3 롤백 단위

- 공통 컴포넌트 도입과 대표 화면 이관은 같은 커밋에 몰지 않는다.
- 화면 단위 롤백이 가능하도록 `공통 컴포넌트 추가`와 `화면 적용`을 분리한다.
- `FacilityDetailBottomSheetShell`의 scrim 예외는 일반 인터랙션 정리 작업과 분리해 유지한다.
