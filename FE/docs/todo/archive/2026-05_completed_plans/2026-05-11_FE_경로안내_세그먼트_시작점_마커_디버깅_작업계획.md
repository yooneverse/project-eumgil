# 경로안내 세그먼트 시작점 마커 디버깅 작업계획

> **작성일** 2026-05-11
> **작성자** 이재호
> **최종 수정일** 2026-05-11

**Goal:** 길안내/내비게이션 지도에서 세그먼트 시작점 마커가 실제 화면에 보이지 않는 원인을 런타임 증거로 특정하고, FE 범위에서 수정 가능한 지점을 확정한다.

**Work Lane:** FE

## 문제 정의

- 현재 코드에는 세그먼트 시작점 마커 생성 로직이 존재하지만, 실제 사용자 화면에서는 마커가 전혀 보이지 않는다.
- 이미 "첫 세그먼트 숨김", "세그먼트 시작점 사용", "walk/transit 색 분기", "1점 polyline 허용"까지 반영했지만 증상은 유지된다.
- 따라서 다음 작업은 추측 기반 추가 수정이 아니라, 데이터가 어느 단계에서 사라지는지 계측하는 디버깅 중심으로 진행해야 한다.

## 범위

- FE 코드와 FE 문서만 다룬다.
- Kakao SDK 렌더링 경로와 fallback overlay 경로를 모두 확인한다.
- BE/API 스펙 변경은 하지 않는다.

## 실행 계획

### 1. 런타임 계측 추가

- `NavigationViewModel.kt`
  - `toMapOverlayUiState()` 직후 `routeSegments.size`, 각 세그먼트 `sequence`, `travelKind`, `polyline.size`를 구조화 로그로 남긴다.
- `MapViewportOverlay.kt`
  - `toSegmentMarkerOverlays()` 결과 개수와 생성된 `overlayId`, 시작 좌표를 기록한다.
- `KakaoMapViewportBindings.kt`
  - `SEGMENT_JUNCTION` projected marker 변환 개수와 색/tone 매핑 결과를 기록한다.
- `KakaoMapViewport.kt`
  - 최종 렌더링되는 `ROUTE_SEGMENT_JUNCTION` marker 수와 주요 좌표를 기록한다.

완료 기준:
- 한 번의 실제 재현에서 "어느 단계에서 0개가 되는지" 로그로 확인할 수 있어야 한다.

### 2. 재현 경로 고정

- 길안내 진입 후 실제 세그먼트가 2개 이상인 케이스를 하나 고정한다.
- focused mode와 active mode에서 각각 1회씩 재현한다.
- 가능하면 fallback overlay 경로도 preview/debug 상태에서 한 번 확인한다.

완료 기준:
- 같은 입력에서 동일 증상을 반복 재현할 수 있어야 한다.

### 3. 원인별 수정 분기

- `routeSegments` 단계에서 이미 비어 있으면:
  - `NavigationViewModel` 또는 상위 state 조립 경로를 점검한다.
- `overlayState.points`까지는 살아 있는데 projected marker가 0개면:
  - `KakaoMapViewportBindings.kt` 분기와 좌표 투영 경로를 수정한다.
- projected marker까지 살아 있는데 화면에서만 안 보이면:
  - `KakaoMapViewport.kt`의 점 크기, z-index, clipping, 화면 좌표 보정 순서를 점검한다.

완료 기준:
- 원인 위치에 맞는 최소 수정안을 1개로 좁힌다.

### 4. 회귀 테스트와 검증 정리

- 유지해야 할 테스트
  - 시작점 좌표가 `firstOrNull()` 기준인지
  - 첫 세그먼트 마커가 제외되는지
  - 1점 polyline 세그먼트가 유지되는지
  - walk/transit 색 분기가 유지되는지
  - active/focused projection 제외 규칙이 깨지지 않는지
- 실행 우선순위
  1. `MapViewportOverlayBindingsTest`
  2. `KakaoMapViewportBindingsTest`
  3. 필요 시 `NavigationViewModelTest`

완료 기준:
- 실제 수정 포인트를 반영한 테스트가 통과하거나, 환경 이슈로 막히면 정확한 blocker를 다시 문서화한다.

## 리스크

- Gradle/KSP/resource merge 불안정으로 수정 후 즉시 재검증이 막힐 수 있다.
- 실제 경로 데이터가 예상보다 더 빈약하면, 시작점 마커 생성 조건을 다시 조정해야 할 수 있다.
- Kakao 렌더링 경로와 fallback 경로가 다르게 보이면 동일 증상을 두 군데에서 따로 잡아야 한다.

## Cross-Lane Handoff

- 현재 기준 없음
- 런타임 로그로도 `routeSegments` 입력 자체가 비정상이라는 증거가 나오기 전까지는 BE handoff를 만들지 않는다
