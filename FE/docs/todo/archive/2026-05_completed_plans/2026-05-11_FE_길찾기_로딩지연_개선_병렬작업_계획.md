# FE 길찾기 로딩지연 개선 병렬작업 계획
> 문서일: 2026-05-11
> Work Lane: FE
> 상태: A 완료, B 완료, 다음 작업 C
> 기준 문서: `FE/docs/debug-report/2026-05-11_FE_길찾기_초기_로딩_지연_구조_분석_디버그_리포트.md`

## 1. 목적

- 길찾기 초기 로딩 지연 이슈를 한 번에 크게 건드리지 않고, 충돌이 적은 단위로 나눠 처리한다.
- FE만 수정한다.
- 병렬 작업은 write scope가 겹치지 않는 범위로 제한한다.
- 병렬 작업 이후에는 상태/UI 충돌 가능성이 높은 항목을 순차로 이어서 처리한다.

## 2. 현재 판단

- `A. 초기 로딩 직렬 구조 개선`: 완료
- `B. route parsing 백그라운드 오프로딩`: 완료
- `C. 현재 위치 캐시 키 완화 검토`: 다음 작업
- `D. 위치 대기 grace window`, `E. 재조회 시 이전 결과 유지`는 `C` 이후 순차 진행이 안전하다.

이유:

- `A`, `B`는 이미 마무리되었고 현재 남은 병렬 후보 중 즉시 진행할 수 있는 것은 `C`다.
- `D`, `E`는 `RouteSettingViewModel.kt`, `RouteSettingScreen.kt`에 동시에 손댈 가능성이 높다.
- 상태 모델, 로딩 UI, 재조회 UX를 함께 건드리면 merge cost와 회귀 위험이 커진다.

## 7. 각 작업의 완료 기준

### A 완료 기준

- transit 기본 케이스에서 첫 usable 결과가 기존보다 빨리 보인다.
- 기존 route option 선택, CTA, 에러 처리 흐름이 크게 깨지지 않는다.
- compile 또는 test 검증 결과가 남아 있다.

### B 완료 기준

- route domain mapping/geometry parsing 비용을 줄이는 방향의 변경이 반영된다.
- cancellation handling이 깨지지 않는다.
- compile 또는 test 검증 결과가 남아 있다.

### C 완료 기준

- 캐시 키 완화가 실제 구현됐거나, 위험성 때문에 보류됐다는 판단이 코드 기준으로 정리된다.
- compile 또는 test 검증 결과가 남아 있다.

### D 완료 기준

- 현재 위치 미확보 시 fallback origin 선조회가 불필요하게 먼저 나가지 않도록 조정된다.
- 위치 대기 상태가 영원히 지속되지 않는다.

### E 완료 기준

- 후속 reload 시 기존 성공 결과를 유지한 채 새 경로 확인 UX가 제공된다.
- 화면이 과도하게 비워지지 않는다.

## 8. 작업 시 공통 주의사항

- FE만 수정한다.
- BE 작업이 필요하면 마지막에 별도 handoff로 정리한다.
- 디버그용 정보 surface는 함부로 확장하지 않는다.
- 가능하면 기존 테스트를 보강하고, 테스트 실패가 구현 문제인지 기존 환경 문제인지 분리해서 기록한다.
- route loading 체감 개선이 목표이므로, 단순 리팩터링보다 사용자 체감 효과가 큰 변경을 우선한다.

## 9. 백엔드 개발자에게 전달할 작업

- `/routes/search/walk`와 `/routes/search/transit`를 한 번에 요약해 주는 통합 응답이 가능하면 초기 로딩 체감을 크게 줄일 수 있다.
- 대중교통 상세 응답이 느릴 때 카드 표시용 최소 요약을 먼저 주고 상세를 후속 조회하는 API 분리도 검토 가치가 있다.

## 10. 후속 메모

- 다른 스레드에 작업을 넘길 때는 각 스레드가 수정하면 안 되는 파일을 명시해야 merge 충돌을 줄일 수 있다.
- `D`, `E`는 별개 작업처럼 보이지만 실제로는 로딩 UX와 상태 표면을 공유하므로, 최종적으로는 한 사람이 이어서 보는 편이 안전하다.
