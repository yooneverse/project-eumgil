### 요약

- [현재 사실] 오전 API 문서는 경로 조회를 `POST /routes/search/walk`, `POST /routes/search/transit`로 분리하고, 안내 시작 확정은 `POST /routes/{routeId}/select`로 따로 정의한다.
- [현재 사실] 검색 응답은 `searchId`와 `routes[]`를 반환하며, 각 route는 `routeId`, `transportMode`, `routeOption`, `title`, `distanceMeter`, `estimatedTimeMinute`, `badges`, `geometry`, `legs[]`를 가진다.
- [현재 사실] WALK leg는 `steps[]`를 포함할 수 있고, 각 step에는 `instruction`, `distanceMeter`, `geometry`, `badges`, `alerts`, `slopePercent`, `widthState`가 정의되어 있다.
- [현재 사실] `select` 성공 응답은 `data: null`이며, 의미는 “선택한 route를 안내 세션으로 확정”하는 것이다. FE는 `select` 이후에만 `transit-refresh`, `reroute`, `route-ratings`를 호출해야 한다.
- [현재 사실] `transit-refresh`는 도착예정 정보만 갱신한다. 기존 route/leg/step/geometry는 덮어쓰지 않는다.
- [현재 사실] `reroute`는 `routeId`와 `currentPoint`만 받으며, `currentLegSequence`는 보내지 않는다. 성공 시 새 `route`와 새 `routeId`를 돌려줄 수 있다.
- [현재 사실] `route-ratings`는 안내 종료 후 마지막 안내 route ID 기준으로 평가를 저장한다.
- [현재 사실] ERD는 `route_sessions`를 2026-05-06 기준으로 반영했고, “검색 후보 전체”가 아니라 “실제 선택해 안내 시작한 route만” 영속 저장한다고 명시한다.

### FE 관점 핵심 해석

- [현재 사실] 오전 문서 기준 타깃 계약은 현재 FE의 `segments[]` 중심 모델보다 더 크다.
- [현재 사실] 오전 문서는 FE가 `searchId`, `routeId`, `route session`, `legs[]`, `steps[]`, `transit-refresh`, `reroute`를 모두 다루는 구조를 전제한다.
- [오픈 이슈] 현재 FE의 route 모델과 runtime navigation 구조는 아직 이 계약을 수용하지 못한다.

## 6. 현재 경로안내 흐름

### 6.1 경로 조회

- [현재 사실] `RouteSettingViewModel`이 목적지 해석 후 `routeRepository.getRouteSearchData(...)`를 호출한다.
- [현재 사실] `DefaultRouteRepository`는 로컬 캐시를 먼저 확인하고, 없으면 mock fixture로 route search data를 만든다.
- [현재 사실] 현재 FE는 실제 BE route API를 호출하지 않는다.

### 6.2 route 선택

- [현재 사실] 사용자가 SAFE/SHORTEST 옵션을 고르면 `latestSearchData` 안에서 선택 route만 바꾼다.

### 6.3 navigation 진입

- [현재 사실] WALK 모드에서만 `StartNavigationRequested`가 발생한다.
- [현재 사실] 이때 FE는 `RouteNavigationRequest`를 만들어 navigation screen으로 전달한다.
- [현재 사실] low vision 경로는 `LowVisionNavigationRoute`가 별도 route search를 실행해 바로 `RouteNavigationRequest`를 만든다.

### 6.4 안내문 생성

- [현재 사실] `supportingText + distanceLabel + instruction` 조합으로 `briefingText`를 만든다.
- [현재 사실] progress 표시는 `"1 / 전체 segment 수"`로 고정된다.

### 6.5 남은 거리/시간 갱신

- [현재 사실] 현재 위치 갱신이 들어오면 preview polyline 전체를 기준으로 남은 거리를 다시 계산한다.
- [현재 사실] 계산은 “가장 가까운 polyline point + 남은 tail distance” 방식이다.
- [현재 사실] step, segment, leg 전환은 이 계산에 반영되지 않는다.

### 6.6 음성안내

- [현재 사실] FE가 안내 문자열을 만든다.
- [현재 사실] `NavigationViewModel`이 `SpeakBriefing` 이벤트를 보낸다.
- [현재 사실] `NavigationRoute`가 `AndroidTextToSpeechController`로 Android TTS를 호출한다.
- [현재 사실] 서버가 음성을 주는 구조가 아니다.

## 7. 현재 한계와 문제점

- [현재 사실] 실제 BE route API 연동이 전혀 없다. 현재 route source는 mock fixture다.
- [현재 사실] FE route DTO는 오전 API 문서의 `searchId`, `routeId`, `transportMode`, `legs[]`, `steps[]`, `badges`, `alerts`, `transferCount`, `arrivalStatus`를 담지 못한다.
- [현재 사실] 길안내 시작 시 `select` API를 호출하지 않으므로 route session 개념이 FE runtime에 없다.
- [현재 사실] reroute, transit-refresh, route-ratings를 호출할 식별자(`routeId`, `searchId`)가 FE navigation request 안에 없다.
- [현재 사실] 현재 navigation runtime은 preview polyline 기반 남은 거리 계산만 있고, 현재 몇 번째 segment인지 판정하지 않는다.
- [현재 사실] 자동 step 전환, 다음 안내 preview, hysteresis, 오프루트 판정, 자동 도착 판정이 없다.
- [현재 사실] 안내문은 첫 번째 segment의 `guidanceMessage`를 계속 재사용하므로 위치 변화에 따라 동적으로 바뀌지 않는다.
- [현재 사실] detail UI 의미는 일부 `guidanceMessage` 키워드와 safety flag에 의존해 FE가 추론한다. 서버 자연어 표현이 바뀌면 UI 분류 품질이 흔들릴 수 있다.
- [현재 사실] 대중교통 travel mode UI는 존재하지만 실제 navigation start는 WALK-only다.
- [현재 사실] 저시력 브리핑에는 임시 하드코딩이 섞여 있어 메인 runtime 로직과 동기화되지 않는다.
- [오픈 이슈] summary ETA 60m/min, runtime ETA 80m/min 차이를 제품적으로 유지할지 통일할지 결정이 필요하다.

## 8. BE 연동 이후 목표 구조

- [제안] 검색 결과 단계에서는 `searchId + routes[]`를 보관하고, 각 route는 `routeId`, `transportMode`, `geometry`, `legs[]`, `steps[]`, `badges`, `alerts`를 포함한 “runtime 진입 가능한 구조”로 FE 메모리에 유지한다.
- [제안] 사용자가 길안내 시작을 누르면 FE는 `POST /routes/{routeId}/select`를 호출해 route session을 확정하고, 이후 runtime state를 초기화한다.
- [제안] navigation runtime은 “선택 route 원본 payload + FE runtime state” 2축으로 분리한다.
- [제안] WALK leg에서는 `steps[]` 중심으로, BUS/SUBWAY leg에서는 `boarding/alighting/arrivalStatus` 중심으로 안내한다.
- [제안] reroute 성공 시 현재 runtime payload를 새 route payload로 교체하고, 진행 상태는 새 route 기준으로 재초기화한다.
- [제안] TTS 문자열은 여전히 FE가 만들되, 장기적으로는 `guidanceMessage` 단일 자연어보다 구조화된 maneuver 메타데이터를 우선 사용한다.

## 9. 경로안내 FE 구현계획

### Phase 1: 실제 route API 연동

- [목표] mock fixture 기반 route search를 실제 API 계약으로 교체한다.
- [현재 사실] 현재 FE는 `segments[]`만 다루고 `searchId`, `routeId`, `legs[]`, `steps[]`가 없다.
- [제안]
  - search contract를 `walk search`, `transit search`, `select`, `transit-refresh`, `reroute`, `route-ratings`까지 나눠 FE 데이터 계층을 설계한다.
  - `RouteNavigationRequest`와 FE route domain model에 `searchId`, `routeId`, `transportMode`, `geometry`, `legs[]`, `steps[]`를 추가한다.
  - 초기 연결 단계에서는 mock fallback을 완전히 제거하기보다 dev 전환 플래그나 repository 분기 정책을 두고 단계적으로 전환한다.
  - 오전 API 문서 기준으로 “walk 전용 route”와 “public transit route”를 같은 상위 인터페이스 아래 다루되, runtime 분기 처리를 명확히 둔다.
- [오픈 이슈] `segments[]`를 유지하면서 `legs[]/steps[]`를 어댑터로 변환할지, 도메인 모델을 재정의할지 결정이 필요하다.

### Phase 2: runtime navigation state 도입

- [목표] route payload와 별개로 실시간 안내 상태를 FE 내부에 둔다.
- [제안] 아래 상태를 `NavigationViewModel` 중심 runtime state로 도입한다.
  - `distanceAlongRoute`
  - `currentSegmentIndex`
  - `distanceToSegmentEnd`
  - `previewSegmentIndex`
  - `lastAnnouncedSegmentIndex`
  - `isTransitionCooldown`
  - `isOffRoute`
- [제안] 이 상태는 “현재 route payload를 어떻게 읽고 있는가”를 나타내며, reroute 시 전체 초기화된다.
- [제안] progress label도 이 상태를 기준으로 계산해 `"현재 step / 전체 step"`이 실제 이동에 따라 바뀌도록 한다.

### Phase 3: current segment 판정

- [목표] 현재 위치가 route 어디에 있는지 FE가 안정적으로 판정한다.
- [현재 사실] 지금은 preview polyline 전체 거리만 계산하고 current segment는 없다.
- [제안]
  - WALK leg에서는 step geometry 또는 segment geometry를 기준으로 snap한다.
  - snap 결과는 “가장 가까운 점”만 보지 말고, 이전 `distanceAlongRoute`보다 뒤로 크게 튀지 않는 단조 진행 조건을 같이 본다.
  - `currentSegmentIndex`는 위치 정확도와 route 진행 방향을 반영해 판정한다.
- [오픈 이슈] geometry 품질이 낮거나 step geometry가 빈 배열인 경우 fallback 판정 규칙이 필요하다.

### Phase 4: 다음 안내 예고 / step 전환

- [목표] 현재 step과 다음 step을 구분하고, 경계 근처에서 자연스럽게 전환한다.
- [제안]
  - segment 경계 근처에서 preview를 먼저 보여준다.
  - 실제 진입이 확정되면 `currentSegmentIndex`를 전환한다.
  - GPS 튐 방지를 위한 hysteresis를 둔다.
  - accuracy 기반 threshold 조정을 한다.
  - 오프루트 판정 기준을 명시한다.
- [제안] 예시 규칙
  - `distanceToSegmentEnd < previewThreshold`이면 `previewSegmentIndex = current + 1`
  - 다음 geometry에 일정 시간/횟수 이상 안정적으로 snap되면 current 전환
  - accuracy가 나쁘면 preview와 transition threshold를 더 크게 둔다
  - route geometry에서 일정 거리 이상 벗어나고, 동일 조건이 연속 확인되면 `isOffRoute = true`
- [오픈 이슈] threshold는 실내/실외, 도심 협곡, 정류장 주변에서 달라질 수 있어 QA 보정이 필요하다.

### Phase 5: 음성안내 고도화

- [목표] 음성안내를 runtime state와 연결하고 중복/누락을 줄인다.
- [현재 사실] 지금은 첫 segment guidance를 기반으로 재생 요청만 보낸다.
- [제안]
  - `lastAnnouncedSegmentIndex`와 `isTransitionCooldown`으로 중복 안내를 막는다.
  - preview 단계와 current step 확정 단계를 구분해 다른 문구를 낼 수 있게 한다.
  - route payload가 자연어 `guidanceMessage`만 주는 경우에는 이를 fallback으로 쓰되, 향후 BE가 구조화된 maneuver 정보를 주면 우선 사용한다.
  - `LowVisionRouteBriefingViewModel`의 하드코딩 문구는 shared runtime formatter로 흡수한다.
- [오픈 이슈] 저시력 모드에서 “출발 전 전체 브리핑”과 “runtime turn-by-turn”의 문구 톤을 분리할지 결정이 필요하다.

### Phase 6: QA / 예외 처리

- [목표] route session, GPS, geometry 품질, API 오류를 실제 사용자 흐름 기준으로 검증한다.
- [제안]
  - search 성공 후 select 실패
  - select 성공 후 route session 만료
  - reroute 응답 `NO_REROUTE_NEEDED`
  - reroute 응답 새 route 교체
  - transit-refresh `NO_CURRENT_ARRIVAL`
  - geometry 없음 / step 없음 / guidanceMessage 없음
  - GPS accuracy 악화 / 점프 / 멈춤
  - 도착 근처에서 자동 도착 판정
  - 앱 백그라운드/복귀 시 runtime state 복원
- [제안] ETA는 최소 “summary ETA”, “runtime ETA”, “transit arrival ETA”를 서로 다른 지표로 구분해 표기 정책을 명확히 둔다.

## 10. 백엔드 개발자에게 전달할 작업사항

- `search/walk`, `search/transit` 응답에 FE runtime이 바로 쓸 수 있는 `searchId`, `routeId`, `transportMode`, `geometry`, `legs[]`, `steps[]`, `badges`, `alerts`, `transferCount`, 정류장/역 정보가 안정적으로 포함돼야 한다.
- `select`는 FE가 route session을 확정하는 시작점이므로 `searchId + routeId` 기반 idempotency와 만료/소유권 오류 코드를 명확히 유지해야 한다.
- `reroute`는 `routeId + currentPoint`만으로 현재 leg를 서버가 추정해야 하므로, route snapshot 기준 현재 leg 추정 규칙과 `RT4091` 반환 기준이 필요하다.
- `transit-refresh`는 geometry를 바꾸지 않고 도착예정 정보만 갱신해야 하며, `arrivalStatus`와 `transits[]` 필드는 FE 문구 생성에 바로 쓸 수 있어야 한다.
- `guidanceMessage` 자연어만 주지 말고, 장기적으로는 아래 구조화 필드를 함께 주는 방안을 검토해야 한다.
  - `maneuverType`
  - `modifier`
  - `announcementText`
  - `anchorCoordinate`
  - segment 또는 step별 duration

## 11. 리스크 / 오픈 이슈

- [리스크] 현재 FE route domain이 `segments[]` 중심이라, 오전 API 문서의 `legs[]/steps[]` 계약으로 전환할 때 route setting, route detail, navigation, low vision이 동시에 영향을 받는다.
- [리스크] 대중교통 travel mode UI는 있으나 실제 runtime은 WALK-only이므로, API만 붙여도 바로 동작하지 않는다.
- [리스크] `guidanceMessage` 키워드 기반 detail 분류는 서버 문구 변화에 취약하다.
- [리스크] summary ETA 60m/min, runtime ETA 80m/min 불일치는 사용자 신뢰를 해칠 수 있다.
- [오픈 이슈] FE 도메인에서 `segments[]`를 유지할지, `legs[]/steps[]`로 재정의할지 결정이 필요하다.
- [오픈 이슈] 자동 도착 판정 threshold를 거리 기준, 속도 기준, 목적지 anchor 기준 중 무엇으로 잡을지 미정이다.
- [오픈 이슈] reroute 후에도 기존 progress/history 일부를 유지할지, 완전 초기화할지 UX 결정이 필요하다.
- [오픈 이슈] low vision 출발 전 브리핑과 main navigation runtime을 하나의 formatter로 통합할지 별도 유지할지 결정이 필요하다.

## 12. 추천 구현 순서

1. route domain 모델을 오전 API 문서 기준으로 재정의하고, `searchId`, `routeId`, `transportMode`, `legs[]`, `steps[]`를 FE 상태에 올린다.
2. `select` API를 붙여 route session 기준 navigation 진입 구조로 바꾼다.
3. WALK 경로 기준 runtime navigation state를 먼저 도입하고 `currentSegmentIndex`, `distanceAlongRoute`를 계산한다.
4. 다음 안내 preview, step 전환, hysteresis, 오프루트 판정을 붙인다.
5. reroute, rating, transit-refresh를 순서대로 연결한다.
6. low vision 브리핑 임시 하드코딩을 shared runtime formatter로 정리한다.
7. ETA 정책, 도착 판정, session 만료 예외를 QA 시나리오로 고정한다.

## 결론

현재 FE의 경로안내는 mock fixture 기반 보행 경로 MVP이며, “첫 segment 안내문 + preview polyline 거리 계산 + Android TTS” 수준까지는 정리돼 있다. 다만 2026-05-06 오전 경로 API 문서가 전제하는 `searchId`, `routeId`, `select`, `route session`, `legs[]`, `steps[]`, `reroute`, `transit-refresh` 구조를 수용하려면 FE route domain과 navigation runtime state를 함께 재설계해야 한다.
