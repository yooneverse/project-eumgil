# FE Frontend Refactoring Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox syntax for tracking.

**Goal:** Reduce FE maintenance cost and regression risk by refactoring the highest-complexity features first without changing route flow, design contract, or API behavior.

**Architecture:** Keep the current single-`app` module and `Contract/Route/Screen/ViewModel` pattern. Refactor in a `route-first -> map-second -> shared screen cleanup` order, extracting pure presentation logic and feature-local UI components before touching broader shared abstractions.

**Tech Stack:** Kotlin, Jetpack Compose, ViewModel, StateFlow, Hilt, JUnit4, Android unit tests, Gradle

## Work Lane

- Explicit non-goals:

## Problem List

- `route` feature complexity is concentrated in too few files.
  - `RouteSettingScreen.kt` is about 2,487 lines.
  - `RouteSettingViewModel.kt` is about 1,940 lines.
  - The feature has only 5 source files, so responsibility boundaries are too coarse.
- `map` feature is large even after partial decomposition.
  - `MapViewModel.kt` is about 1,568 lines.
  - `MapScreen.kt` is about 1,283 lines.
  - `KakaoMapViewport.kt` is about 1,248 lines.
- Existing route and map tests are partly coupled to source-text layout details rather than stable public behavior.
  - This makes safe extraction harder because moving code between files can fail tests even when behavior stays correct.
- Shared UI duplication exists across `search`, `report`, `route`, and `map`, but a shared abstraction should be introduced only after route/map boundaries are stabilized.
- The codebase already has explicit FE conventions for `Contract/Route/Screen/ViewModel`, token usage, and accessibility labels, but large files are drifting away from those conventions.

## Architecture And Data Flow

- Refactor order:
  1. Stabilize tests around behavior and policy instead of file-text coupling where possible.
  2. Split `route` screen into feature-local components and move pure presentation logic out of the ViewModel.
  3. Split `map` state handling into focused internal units without changing external screen contracts.
  4. Clean up duplicated UI state assembly in `search` and `report` only after high-risk features are smaller.
- State ownership rules:
  - `Route` files keep wiring and side effects.
  - `Screen` files render composed UI and delegate callbacks only.
  - `ViewModel` files own state transitions, async calls, and event emission only.
  - Feature-local formatter/mapper/policy files own string assembly, badge selection, tone mapping, and section projection.
- Data flow rules:
  - Repository access remains in ViewModels only.
  - DTO/Entity objects remain outside `UiState`.
  - New extracted files should accept domain or UI models, not repositories or `NavController`.
- Design rules:
  - Preserve existing FE token usage, route labels, accessibility wording, and top-level route flow.
  - New shared components should be introduced only when at least two screens need the same behavior and naming contract.

## Risk Register

- Risk statement: Route tests currently depend on source-file text structure.
  - Why it matters: Component extraction can fail tests even when behavior is preserved.
  - What can be mitigated now: Convert high-value assertions to policy/projection tests first.
  - What remains open: A few implementation-detail assertions may still need to remain.
  - Owner or next action: FE, Unit 1.
- Risk statement: `MapViewModel` mixes multiple responsibilities with runtime state.
  - Why it matters: Refactoring can accidentally break recenter, marker selection, or destination preview behavior.
  - What can be mitigated now: Keep SDK-facing viewport code out of the first map pass and refactor only internal coordination.
  - What remains open: Later viewport-specific cleanup may still be needed.
  - Owner or next action: FE, Unit 4.
- Risk statement: Premature shared abstraction can create vague `Common`/`Helper` style files.
  - Why it matters: The codebase already warns against generic utility drift.
  - What can be mitigated now: Share only after route/map extraction reveals real duplication.
  - What remains open: Search/report cleanup should be delayed until extraction patterns are proven.
  - Owner or next action: FE, Unit 5.
- Risk statement: Large refactor scope can hide regressions behind many file moves.
  - Why it matters: Review quality drops when structure and behavior changes are mixed.
  - What can be mitigated now: Keep units small and merge each one with passing tests before starting the next.
  - What remains open: Sequence discipline must be maintained during implementation.
  - Owner or next action: FE, implementation order enforcement.
