# FE 워치 연동 길안내 작업 방향

## Work Lane

FE

## 목표

- 목표는 `워치 단독형`이 아니라 `폰 주도형 Wear OS companion`이다.
- 폰 앱이 길안내 상태를 계산하고, 워치는 폰에서 전달받은 현재 분기 안내를 표시하거나 진동으로 알려준다.
- 현재 범위에서는 백엔드 작업 없이 FE만으로 PoC와 1차 구현을 진행한다.

## 결론 요약

- 현재 방향에서 가장 현실적인 출발점은 `폰 로컬 알림 -> 워치 미러링`이다.
- 그 다음 단계로 `Wear OS companion 앱/모듈`을 붙여 분기 이벤트를 워치에 직접 전송하는 구조로 확장한다.
- 현재 목표가 `폰과 블루투스 기반 연동`이므로 백엔드 의존성은 필수가 아니다.
- 다만 실서비스 품질까지 끌어올리려면 `중복 알림 방지`, `백그라운드 안정성`, `경로 단계 정확도`를 FE에서 먼저 정리해야 한다.

## 현재 구현 기준 사실

- 길안내 화면은 이미 `RouteNavigationRequest`를 받아서 선택된 경로와 segment 목록을 보유한다.
- `NavigationViewModel`은 현재 위치 업데이트를 받아 `activeSegmentIndex`를 계산하고 현재 안내 상태를 갱신한다.
- 분기 방향은 `guidanceMessage + safetyFlags`를 기준으로 `TURN_LEFT`, `TURN_RIGHT`, `CROSSWALK`, `ELEVATOR` 같은 액션으로 해석하고 있다.
- 현재는 TTS 음성 안내만 연결되어 있고, 알림 채널, 워치 전송, foreground service, wearable dependency는 아직 없다.
- 현재 route repository는 실 API가 아니라 mock fixture 기반이다. 따라서 워치 연동은 가능하지만, 실데이터 연결 전까지는 정확도 검증 한계가 있다.

## 권장 방향

### 1안. 1차 PoC

- 방식: 폰에서 분기 변경 시 로컬 알림 생성, 워치는 기본 notification mirroring으로 수신
- 장점: 구현 속도가 가장 빠르고 구조 변경이 적다
- 단점: 워치 UI와 진동 패턴을 세밀하게 제어하기 어렵다
- 권장도: 높음

### 2안. 2차 구현

- 방식: Wear OS companion 앱 또는 wear 모듈을 추가하고, 폰이 `MessageClient` 또는 `DataClient`로 분기 이벤트를 워치에 직접 전송
- 장점: 워치에서 직접 카드 UI, 남은 거리, 방향 아이콘, 진동 패턴 제어 가능
- 단점: 안드로이드 앱 외에 wear 쪽 구조와 상태 동기화를 새로 설계해야 한다
- 권장도: 높음

### 비권장 방향

- 워치 단독 로그인, 워치 단독 경로 조회, 서버 푸시 기반 워치 알림
- 현재 목표 대비 범위가 과도하고 BE/API/인프라 작업이 따라온다

## 구현 난이도 평가

| 단계 | 내용 | 난이도 | 비고 |
| --- | --- | --- | --- |
| Phase 1 | 폰 로컬 알림 + 워치 미러링 | 중 | FE 단독 가능, 가장 빠른 PoC |
| Phase 2 | Wear OS companion 직접 연동 | 중상 | wearable 통신, 워치 UI/상태 설계 필요 |
| Phase 3 | 백그라운드 안정화 및 실사용 품질 보강 | 상 | foreground service, 중복 방지, 배터리/재연결 이슈 대응 |

## FE 작업 범위

### Phase 1. 알림 미러링 PoC

- `NotificationCompat`, 알림 채널, `POST_NOTIFICATIONS` 권한 추가
- 내비게이션 진행 중 분기 변경을 감지하는 이벤트 레이어 추가
- `activeSegmentIndex` 변경 시 안내 문구, 방향 아이콘, 남은 거리로 알림 발행
- 같은 segment에서 알림이 반복 발행되지 않도록 dedupe 상태 추가
- 알림을 누르면 길안내 화면으로 복귀하도록 pending intent 연결

### Phase 2. Wear OS companion 직접 연동

- app 모듈에 wearable dependency 추가
- wear 앱 또는 wear 모듈 구조 결정
- 폰에서 워치로 보내는 payload 정의
- 예시 payload
  - `routeId`
  - `segmentIndex`
  - `guidanceAction`
  - `guidanceMessage`
  - `remainingDistanceMeters`
  - `estimatedMinutes`
  - `isReroute`
- 워치 수신 화면에서 다음 안내 카드, 아이콘, 진동 처리
- 연결 끊김/재연결 시 마지막 안내 상태 재동기화

### Phase 3. 실사용 안정화

- 백그라운드 길안내 유지 정책 정리
- 필요 시 foreground service 도입 검토
- 위치 흔들림으로 인한 segment oscillation 방지
- 재탐색 시 이전 route와 신규 route를 구분해 중복 알림 차단
- 워치 미연결 상태 fallback 정책 정의
  - 폰 알림만 유지
  - 워치 재연결 시 마지막 안내 재전송

## 예상 주요 리스크

### 1. 분기 판정 흔들림

- 현재 active segment 계산은 `현재 좌표와 가장 가까운 segment` 방식이다.
- 코너 구간이나 평행 보도에서는 segment가 왔다 갔다 바뀌며 알림이 중복될 수 있다.
- 해결 방향
  - 최근 알림 segment 기억
  - 최소 거리/시간 threshold 추가
  - 이전 segment에서 다음 segment로만 이동하는 순방향 조건 추가

### 2. 현재 route 데이터 한계

- 지금은 mock fixture 기반이라 실제 API 연결 전까지는 안내 문구 다양성 검증이 부족하다.
- 실 API 연결 후 `guidanceMessage` 품질과 `step/segment` 일관성을 다시 점검해야 한다.

### 3. 백그라운드 제약

- 화면이 꺼진 상태나 앱이 백그라운드로 간 상태에서 위치 업데이트와 알림 전송이 약해질 수 있다.
- Phase 1 PoC에서는 우선 foreground 전환 없이 확인하고, 누락이 크면 Phase 3에서 foreground service를 검토한다.

### 4. 워치 UX 과설계 위험

- 처음부터 워치 전용 상세 지도, 다단계 리스트, 설정 화면까지 넣으면 범위가 급격히 커진다.
- 1차 목표는 `다음 분기 안내 카드 + 진동 + 간단한 남은 거리` 수준으로 제한하는 것이 맞다.

## 권장 구현 순서

1. 폰 알림 기반 PoC부터 만든다.
2. `segment 변경 감지 + dedupe`를 먼저 안정화한다.
3. 워치 미러링만으로 사용성이 충분한지 확인한다.
4. 부족하면 Wear OS companion 직접 연동으로 확장한다.
5. 마지막에 foreground service와 실사용 안정성을 보강한다.

## 권장 산출물

- FE 코드
  - navigation guidance event 추출
  - notification helper
  - wearable message sender
- FE 문서
  - 워치 payload 정의
  - 알림 발행 조건
  - 중복 방지 규칙
  - 백그라운드 동작 정책

## 이번 범위에서 BE 작업 필요 여부

- 없음

## 추후 필요할 수 있는 Cross-Lane Handoff

- 현재 목표에서는 없음
- 단, 아래 조건이 생기면 BE handoff 필요
  - 실시간 재탐색을 워치에서도 독립 제어해야 하는 경우
  - 워치 상태를 서버에 저장해야 하는 경우
  - route step 식별자와 maneuver 타입을 서버 계약으로 고정해야 하는 경우

## 최종 권장안

- 이번 작업은 `Phase 1 -> Phase 2` 순서로 진행한다.
- 즉시 착수 범위는 `폰 로컬 알림 + 워치 미러링 + 분기 dedupe`다.
- 그 다음 스프린트 또는 후속 태스크에서 `Wear OS companion 직접 연동`을 붙이는 것이 비용 대비 효과가 가장 좋다.
