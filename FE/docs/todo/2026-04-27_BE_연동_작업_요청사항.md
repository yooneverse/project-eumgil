# BE 연동 작업 요청사항

> 작성일: 2026-04-27  
> 기준: `FE/docs/2026-04-22_부산이음길_FE_화면_인벤토리_및_라우트_맵.md`, `FE/docs/2026-04-13_부산이음길_FE_코드_컨벤션.md`  
> 목적: FE 문서 정합성 재검토 중 확인된 백엔드 확인/구현 필요 사항 정리  
> 주의: 이 문서는 API 명세를 확정하지 않는다. 실제 URL, 필드명, enum 코드, 인증 정책은 백엔드에서 확정한다.
> 문서 성격: 2차 handoff 문서다. FE 화면 계약 변경은 1차 FE 계약 문서에 먼저 반영한다.

---

## 1. 사용자/프로필

### 1.0 AUTH-001 소셜 로그인 연동 확인 필요

- 현재 FE의 `AUTH-001`은 Google, Naver, Kakao 버튼을 화면 상태 검증용 UI provider key로만 다룬다.
- 현재 FE 실험 구현에는 local-only 로그인 성공 뒤 다음 gate를 다시 판단하는 흔적이 남아 있을 수 있으나, 현재 문서 계약은 `profileCompleted` gate를 사용하지 않는다. FE 기준 gate는 기존 회원 `authSession -> selectedPrimaryUserType 홈 분기`, 신규 사용자 `signupToken -> 온보딩 -> POST /auth/signup`이며, `저시력자`는 `LV-HOM-01`, `보행약자`는 `MAP-01`로 연결한다.
- FE는 실제 OAuth provider enum, 인가 코드 전달 방식, access/refresh token 필드명, token 저장/갱신 정책을 확정하지 않는다.
- BE 확인 필요 항목:
  - MVP에서 실제 노출할 소셜 로그인 제공자 목록
  - 각 제공자의 FE 시작 방식과 BE callback/exchange 방식
  - 로그인 성공 응답의 access token, refresh token, 만료 시간, 사용자 식별자 필드
  - 프로필 미완료/완료 판단을 로그인 응답에 포함할지, 별도 프로필 조회 API로 판단할지 여부
  - 실패 응답 코드와 FE 표시용 오류 문구 매핑 기준

### 1.1 사용자 유형 데이터 매핑

- 1차 사용자 유형은 `저시력자`, `보행약자` 2개로 정리하고 FE 상태와 서버 필드는 모두 `selectedPrimaryUserType`으로 맞춘다.
- FE 표시 보행약자 세부 유형은 아래 3개로 정리한다.
  - `휠체어 사용자 (전동)`
  - `휠체어 사용자 (수동)`
  - `기타 보행약자`
- 위 3개 값은 FE 상태와 서버 필드 모두 `selectedMobilitySubtype`으로 관리하고, 보행약자일 때만 저장한다. 저시력자는 `selectedMobilitySubtype`을 받지 않는다.
- 마이페이지의 `이용권 사용자`, `보행약자 모드`가 어떤 서버 데이터에 대응하는지는 미정이다.

### 1.2 사용자 유형 변경

- 마이페이지의 `사용자 유형 변경`은 FE에서 `ONB-001`부터 다시 시작한다.
- 변경된 사용자 유형을 저장할 기존 API 재사용 여부 또는 별도 처리 방식은 BE 확인이 필요하다.

## 2. 권한/약관/설정

### 2.1 마이크/STT

- 마이크 권한은 필수가 아니며 온보딩에서 요청하지 않는다.
- 지도 홈 검색바의 마이크 아이콘은 STT 기반 음성 장소 검색 버튼이다.
- FE가 STT 텍스트를 만들고 `POST /voice/analyze`를 호출한다.
- 보행약자는 응답의 `placeName`으로 바로 `GET /places/search`를 호출한다.
- 저시력자는 `confirmationMessage`를 TTS로 확인하고 `confirmed=true`일 때만 `GET /places/search`를 호출한다.
- `GET /places/search`는 자연어 문장을 해석하지 않고 확정된 `keyword`만 검색한다.

### 2.2 약관/개인정보 웹뷰

- 약관 전문과 개인정보 화면은 노션 웹뷰로 표시한다.
- FE에서 사용할 약관/개인정보 노션 URL 제공 방식은 BE 또는 기획 확인이 필요하다.

### 2.3 알림

- 푸시/진동 알림은 MVP 범위에서 제외하거나 후순위로 정리한다.
- 서버 설정 필드가 이미 존재하더라도 FE에서는 현재 활성 설정으로 노출하지 않는다.

### 2.4 로그아웃

- 로그아웃 시 온보딩/프로필 상태 유지 여부는 미정이다.
- 확인 다이얼로그 필요 여부와 클라이언트/서버 토큰 삭제 범위는 BE/FE 추가 확인이 필요하다.

### 2.5 회원탈퇴

- 회원탈퇴 시 재인증이 필요하다.
- 탈퇴 시 북마크 데이터는 삭제한다.
- 탈퇴 시 제보/평가 데이터는 삭제하지 않는다.
- 탈퇴 처리 후 FE는 로그인 화면으로 이동한다.
- 실제 재인증 방식과 삭제/보존 처리 범위는 BE 확정이 필요하다.

## 3. 검색/지도/경로

### 3.1 목적지/검색 흐름

- 검색 결과 카드 선택 시 바로 경로 설정으로 가지 않고, 항상 지도 바텀시트 확인을 거친다.
- `DEST-01`~`DEST-04`는 FE 문서에서 별도 화면 ID로 유지한다.
- 서버 응답에는 바텀시트 확인 화면과 경로 탐색 요청에 필요한 장소 식별자/좌표/주소 정보가 일관되게 제공되어야 한다.

### 3.2 경로 옵션

- FE 경로 옵션은 `안전한 길`, `최단거리` 2개만 사용한다.
- `최소계단` 옵션은 삭제한다.
- 기존 서버/문서/응답에 `최소계단` 또는 유사 옵션이 남아 있다면 제거 또는 미노출 처리가 필요하다.

### 3.3 경로 상세 진입

- `ROUTE-01` 옵션 카드 선택은 선택 상태 변경이다.
- 상세 이동은 옵션 카드 우측 상단 화살표 버튼으로 진입한다.
- 서버 작업은 아니지만, 경로 카드와 상세 화면에서 같은 경로 식별/옵션 데이터가 유지되어야 한다.

### 3.4 대중교통 탭

- `ROUTE-01`의 대중교통 탭은 MVP에 포함한다.
- 대중교통 후보 조회, 저상버스/역 접근성 검증 범위, 외부 API 연동 범위는 BE 구현 계획 확인이 필요하다.

### 3.5 경로 이탈/재탐색

- 길안내 중 경로 이탈 시 FE 문구는 재검색/재탐색으로 정리한다.
- 현재 위치 기준 재탐색 요청/응답 계약이 실제 길안내 흐름에서 충분한지 BE 확인이 필요하다.

### 3.6 도착/평가

- 도착 완료 시 FE는 `ARR-01`로 이동한다.
- `ARR-01` 진입 즉시 평가 모달을 자동 표시한다.
- 평가는 우측 상단 X 버튼으로 건너뛸 수 있다.
- 평가 저장 API는 신규 구현 예정이다.
- 평가 저장 API의 URL, 인증, 요청 필드, 저장 스키마는 BE 확정이 필요하다.

## 4. 북마크

### 4.1 명칭

- FE 표시 명칭은 `북마크`로 통일한다.
- 기존 `저장`, `SavedRoute`, `SAVE-01` 등은 FE 문서에서 최신 디자인 기준의 `북마크`/`BMK` 체계로 정리한다.
- 서버 내부 엔드포인트나 테이블명이 기존 명칭을 유지하더라도 FE 응답/문서 노출 용어는 `북마크` 기준으로 맞춘다.

### 4.2 편집/삭제

- 북마크 편집 모드는 삭제 기능만 제공한다.
- 정렬, 전체 선택, 다중 선택은 제공하지 않는다.
- BE는 삭제 단위와 실패/권한 오류 응답을 FE가 명확히 처리할 수 있게 확인해야 한다.

### 4.3 카드 선택/길찾기

- 북마크 카드 본문 선택 시 상세 화면으로 이동한다.
- 저장된 경로의 `길찾기`는 `NAV-01`로 즉시 진입한다.
- 즉시 길안내 진입에 필요한 출발지/도착지/경로 옵션 데이터 제공 방식은 BE 확인이 필요하다.

## 5. 제보

### 5.1 인증/계정 연결

- 제보는 사용자 계정과 연결한다.
- 익명 제보로 처리하지 않는다.
- 사용자별 제보 내역 화면은 `MYP-02`로 연결한다.

### 5.2 제보 유형

- 제보 유형은 아래 6개를 API enum 후보에 포함해야 한다.

| enum | 화면 제목 | 보조 설명 |
| --- | --- | --- |
| `STAIRS_STEP` | 계단·단차 있음 | 안내와 달리 계단이나 단차가 있어요 |
| `BRAILLE_BLOCK` | 점자블록 문제 | 손상, 미설치, 잘못된 설치 |
| `SIDEWALK_MISSING` | 인도 없음 | 안내와 달리 보행 가능한 인도가 없어요 |
| `RAMP` | 경사로 문제 | 경사로 이용이 어렵거나 위험해요 |
| `SIDEWALK_WIDTH` | 인도폭 문제 | 인도가 좁아 통행이 어려워요 |
| `OTHER_OBSTACLE` | 기타 장애물 | 위 항목에 없는 보행 불편 상황 |

### 5.3 위치/사진

- FE는 제보 위치 확정 방식으로 pin drag를 제공한다.
- BE는 FE가 확정한 좌표를 저장할 수 있어야 한다.
- 사진 첨부는 갤러리와 카메라를 모두 허용한다.
- FE는 presigned upload URL로 사진 binary를 먼저 업로드하고, 제보 저장 요청에는 업로드 결과 `imageObjectKeys`를 전달한다.
- 이미지 업로드 API는 `POST /hazard-reports/images/presigned-upload`를 기준으로 하며, 용량/만료/삭제 정책의 운영 세부값만 추가 확인한다.

### 5.4 처리 상태

- 내 제보 내역(`MYP-02`)에서는 사용자 본인의 제보 처리 상태를 표시한다.
- 공개 지도에는 모든 제보가 아니라 승인된 공개 제보(`APPROVED`)만 마커로 노출한다.

## 6. 마이페이지/활동

### 6.1 제보 내역

- 제보 완료 화면의 `제보 내역 확인하기` CTA는 `MYP-02` 목록으로 연결한다.
- `MYP-02`에 필요한 제보 목록 조회 방식은 BE 확인이 필요하다.
- 목록과 상세에는 처리 상태를 표시한다.

### 6.2 활동 배지

- `제보왕`, `도움왕`, `참여왕`의 달성 조건과 계산 방식은 미정이다.
- 배지를 서버에서 계산할지, FE에서 표시만 할지 추가 확인이 필요하다.

### 6.3 앱 정보/도움말

- 이용 가이드는 처음 튜토리얼 화면을 앱 내부 화면으로 재사용할 수 있다.
- 고객센터, 1:1 문의는 선택 시 `준비중인 기능입니다.` 메시지를 표시한다.
- 해당 항목은 현재 별도 BE 연동이 필요하지 않다.

## 7. 추가 확인 필요

- 권한 안내 화면에서 카드 선택 시 권한을 요청할지, 하단 CTA에서 일괄 요청할지 미정
- 접근성 필터 `더보기` 포함 항목 미정
- 사용자 상태 설문 노출 조건 미정
- 길안내 중도 종료 후 이동 정책 미정
- 마이페이지 프로필 필드 `이용권 사용자`, `보행약자 모드` 데이터 매핑 미정
- 로그아웃 확인 다이얼로그, 토큰 삭제 범위, 온보딩/프로필 상태 유지 여부 미정
- 평가 저장 API 상세 미정
- 이미지 업로드 운영 세부값(용량, URL 만료, 삭제 정책) 확인 필요
