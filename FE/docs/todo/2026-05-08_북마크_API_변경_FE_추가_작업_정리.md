## 1. 배경

박세홍이 Story 1(북마크) Task들을 진행하며 BMK-02 머지(2026-05-06) 직후, **다음 날 BE에서 경로 북마크 저장(POST) 계약을 완전히 교체**하고 상세 조회 API를 신규 추가했다. 그 시점에 진행 중이던 박세홍 Task(T3 handoff, T4 QA/QC)는 **저장 흐름을 직접 건드리지 않아 영향이 없었지만**, 향후 시연 시점에 경로 북마크 저장 흐름이 동작하려면 신규 명세 반영이 필수다.

본 문서는 다음 세 가지를 분리해서 Jira 신규 Task로 등록하는 것을 목표로 한다.

## 3. 신규 Jira Task 등록안

### 3.1 Task A — 경로 북마크 저장 흐름 신규 명세 반영 + ARR-02 진입점 정상화

| 필드 | 값 |
| --- | --- |
| **Epic** | `S14P31E102-194` 제보 및 로컬 저장 기반 구축 |
| **Story** | `[FE] 북마크 장소/경로 API 실연동 및 재진입 흐름 마감` (Story 1) ← 후속 또는 신규 Story 분기 |
| **제목** | `[FE] 경로 북마크 저장 신규 명세(routeId 기반) 반영 및 ARR-02 진입점 정상화` |
| **Story Point** | 5 |
| **우선순위** | High |
| **Branch 이름 제안** | `fe/feat/route-bookmark-save-routeid-{TaskNo}` |
| **연결 이슈** | `S14P31E102-410` (BE 저장 계약 변경) |
| **의존성** | 이재호 navigation 도메인의 `NavigationCompleteClicked → NavigateToArrival` UI 연결 마감 필요 |

#### 작업 범위

- [ ] `RouteCandidate` 모델에 서버 발급 `routeId: String` 필드 추가
- [ ] `RouteSearchDtoMapper`에서 BE search 응답의 `routeId` 매핑 (장기적으로 mock-only 흐름에서 실제 search API로 전환 필요 — 별도 이슈로 분리할 수도 있음)
- [ ] `RouteBookmarkSaveRequest`에 `routeId` 필드 추가, `RouteBookmarkDraft`에서 함께 흘림
- [ ] `FavoriteRoutesRemoteDataSource.createFavoriteRoute` body 신규 명세에 맞게 변경 — `{routeId, startLabel, endLabel}`만 전송
- [ ] `RouteBookmarkRepository.saveRouteBookmark` 구현 검증 + 단위 테스트 갱신
- [ ] navigation 도메인 협업: `NavigationScreen`에 도착 완료 트리거 UI 연결 또는 자동 도착 감지 로직 연결 (이재호 영역과 협업)
- [ ] ARR-02 도착 화면에서 경로 북마크 저장 흐름 end-to-end 시연
- [ ] 신규 에러 코드 처리 (`403 FR4030`, `404 FR4041`) — 사용자 친화적 메시지 매핑

#### 완료 기준

- 안내 종료 → ARR-02 도착 화면 진입 → 경로 저장 다이얼로그 → 서버 저장 성공 → 경로 북마크 탭에서 새 카드 노출
- 다른 기기에서 같은 계정으로 로그인 시 동일 카드 노출 (서버 동기화 검증)
- 단위 테스트: 신규 페이로드 검증 + 에러 코드별 fallback 검증

#### 비고

- route search API가 mock-only인 상태에서는 실 서버에서 발급된 `routeId`가 없어 end-to-end 동작 불가. **search API 실연동이 선행되거나 같은 Task로 묶여야 함**.
- 박세홍 영역(저장 payload 정합성)과 이재호 영역(ARR-02 wiring + route search) **두 도메인이 합류**하는 작업이라 사전 정렬 필요.

- [ ] `FavoriteRoutesRemoteDataSource.getFavoriteRouteDetail(favRouteId)` 추가 — `GET /favorite-routes/{favRouteId}`
