# FE 지도 홈복귀 시 KakaoMap 무한 로딩 디버깅 리포트

> 작성일: 2026-05-11  
> Work Lane: FE  
> 이슈: 길안내/도착 화면에서 홈 지도 화면으로 복귀할 때 `지도를 불러오는 중입니다.` 오버레이가 지속되고, 사용자 입장에서는 자동 재시도 여부가 보이지 않으며 복구되지 않는 것처럼 보이는 문제

## 1. 증상 요약

- 재현 경로는 `NavigationRoute -> Map`, `ArrivalRoute -> Map` 복귀 흐름이다.
- 사용자는 홈으로 돌아온 뒤 지도 본문 대신 `지도를 불러오는 중입니다.` 카드만 계속 보게 된다.
- 기존 가설은 "카카오 렌더러 timeout 이후 재시도가 안 된다"였으나, 실제 사용자 재현 이후에는 "재시도는 일부 들어가더라도 홈 복귀 컨텍스트 자체가 정상적으로 초기화되지 않는다" 쪽으로 가설이 이동했다.

## 2. 현재 구현 기준 확인 사실

### 2.1 현재 구현 fact

- `지도를 불러오는 중입니다.` 문구는 `FE/app/src/main/res/values/strings_map_renderer.xml`의 `map_viewport_title_renderer_loading`이다.
- 이 오버레이는 `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/KakaoMapViewport.kt`에서 `rendererStatus != Ready`일 때 렌더링된다.
- Kakao renderer는 `KAKAO_RENDERER_READY_TIMEOUT_MILLIS = 4_000L` 동안 `Ready`가 되지 않으면 `markRendererTimedOut()`으로 `Error` 상태로 전환된다.
- 기존 자동 복구는 홈 복귀 자체를 다시 만드는 구조가 아니라, 동일 Compose 세션 내부에서 Kakao renderer controller만 한 번 다시 띄우는 구조였다.

### 2.2 사용자 관점 문제

- 자동 재시도가 실제로 돌더라도 화면 문구가 계속 `지도를 불러오는 중입니다.`라면 사용자는 재시도 중인지, 멈춘 것인지 구분할 수 없다.
- 따라서 기능 문제와 별개로 현재 UX는 "복구 시도 중임을 보여주지 못한다"는 결함이 같이 있다.

## 3. 원인 추적 과정

### 3.1 1차 확인

- `MapViewModel`의 위치 재시도 로직과, Kakao renderer 초기화 로딩 오버레이는 서로 다른 경로였다.
- 따라서 recenter/location retry를 고쳐도 `지도를 불러오는 중입니다.` 문제는 바로 해결되지 않는다.

### 3.2 1차 대응과 한계

- 커밋 `71a8936e`에서 `timeout`, `unexpected destroy`에 대해 Kakao renderer를 1회 자동 복구하도록 수정했다.
- 이 변경은 `KakaoMapViewport.kt`, `KakaoMapViewportBindings.kt`에 반영되었다.
- 하지만 실제 사용자 재현 결과 "여전히 같은 로딩 카드만 돈다"는 피드백이 들어왔고, 이로 인해 자동 복구 1회 추가만으로는 실제 문제를 해결하지 못한다는 사실이 확인됐다.

### 3.3 현재 가설

- 홈 복귀 시 `TopLevelRoute.Map` 백스택 엔트리와 activity-scoped `MapViewModel`이 재사용된다.
- 이 경우 renderer 내부 재시도만으로는 부족하고, 홈 복귀 자체를 별도 reset 시나리오로 처리해 `MapViewModel` 상태와 `MapViewport` 세션을 같이 새로 만들어야 한다.
- 즉, 현재 핵심 의심 지점은 "재시도가 없어서"가 아니라 "홈 복귀 시 stale map session이 재사용되어 동일 렌더러 컨텍스트 안에서 계속 맴돈다"이다.

### 3.4 추가로 확인된 실제 우회 경로

- `AppNavHost.kt`의 top-level 탭 바는 현재 route가 실제 `map`이 아니어도 `arrival`, `search/*`를 map 탭으로 간주해 홈 탭을 선택 상태로 보여준다.
- 하지만 탭 클릭 콜백은 이 경우에도 기존 `navigateToTopLevel(TopLevelDestination.Map)`만 호출하고 있었다.
- 따라서 사용자가 도착 화면이나 검색 계열 화면에서 홈 탭을 눌러 복귀하면, `navigateToTopLevelMapForHomeEntry()`로 연결한 reset signal 경로를 아예 타지 않을 수 있었다.
- 이 경로는 "수정이 들어갔는데도 여전히 동일 증상"이라는 사용자 재현과 가장 잘 맞는 우회 케이스다.

## 6. 현재 결론

- `지도를 불러오는 중입니다.` 문제는 단순 위치 재시도 문제로 보면 안 된다.
- 1차 자동 복구 커밋만으로는 실제 사용자 재현이 해소되지 않았고, 홈 복귀 시 map session 재사용까지 포함해서 봐야 한다.
- 여기에 더해, 도착/검색 계열 화면의 홈 탭 자체가 reset 경로를 우회하고 있던 실제 라우팅 구멍도 확인됐다.
- 현재 FE에서 추가한 방향은 타당하다.
  - 홈 복귀를 별도 reset 이벤트로 승격
  - renderer session key로 viewport 자체 재생성
  - 자동 재시도 UX 가시화
  - top-level 홈 탭이 alias route에서 reset 경로를 우회하지 못하게 차단
- 다만 이 방향이 실제 기기/실사용 흐름에서 증상을 끝까지 없애는지는 아직 최종 확인 전이다.

## 7. 남은 가설과 다음 확인 포인트

- `navigateToTopLevelMapForHomeEntry()`가 실제 재현 경로에서 항상 호출되는지 확인 필요
- `MapRoute`의 `LaunchedEffect(shouldResetForHomeEntry)`가 실제 홈 복귀 타이밍에 실행되는지 확인 필요
- `MapViewModel.onHomeReentered()` 실행 후 `rendererSessionKey`가 실제로 증가하고 `MapViewport` 재마운트까지 이어지는지 확인 필요
- `KakaoMapViewportController.bind() -> startMap() -> onMapReady()` 체인이 홈 복귀 시점에도 정상 재시작되는지 runtime log 확인 필요
- signal path가 정상인데도 계속 로딩이면, `MapView.start()` / lifecycle resume sequencing 또는 top-level route reuse 정책 자체가 병목일 가능성이 있다

## 8. 권장 다음 액션

- `MainNavGraph`, `MapRoute`, `MapViewModel`, `KakaoMapViewport`에 임시 구조화 로그를 추가해 홈 복귀 1회 동안의 state transition을 순서대로 수집
- 실제 재현 기기에서 아래 순서가 모두 찍히는지 확인
  - `navigateToTopLevelMapForHomeEntry`
  - `shouldResetForHomeEntry=true`
  - `onHomeReentered`
  - `rendererSessionKey increment`
  - `KakaoMapViewport bind`
  - `startMap`
  - `onMapReady` 또는 `onMapError` / `markRendererTimedOut`
- 만약 `rendererSessionKey` 증가 후에도 `onMapReady`가 오지 않으면, renderer 내부 자동 재시도보다 강한 route-level teardown 전략 검토 필요

- 현재 기준 없음

## 10. 2026-05-12 후속 디버깅 및 수정 결과

### 10.1 원인 확정

- `MapViewModel.onHomeReentered()` 자체는 이미 `selectedDestination`, destination preview, 시설 상세 상태를 비우고 현재 위치 카메라로 복귀시키는 역할을 한다.
- 하지만 이 함수는 `MAP_HOME_REENTRY_RESET_KEY`가 설정된 경우에만 실행된다.
- 2026-05-11 시점 대응은 `search/*`, `arrival` 같은 일부 map alias 경로에만 reset 진입을 연결했고, `saved_route`, `report`, `my_page`처럼 top-level bar가 그대로 보이는 다른 복귀 경로는 plain `navigateToTopLevel(TopLevelDestination.Map)`를 계속 사용하고 있었다.
- 이 경우 activity-scoped `MapViewModel`이 재사용되며 직전 `selectedDestination`이 살아 남는다.
- 현재 구현상 `selectedDestination`이 남아 있으면 위치 업데이트가 와도 홈 진입 시 카메라/선택 상태가 자동으로 현재 위치 기준으로 덮어써지지 않는다. 즉 사용자 관점에서는 "홈으로 돌아왔는데도 최근 목적지가 선택된 지도"가 복원된다.
- 따라서 이번 이슈의 실제 핵심은 Kakao renderer 재시도 부족만이 아니라, "홈 복귀 reset 경로가 일부 라우트에서 누락되어 stale destination state가 유지되는 라우팅 구멍"이었다.

### 10.2 이번에 적용한 FE 수정

- `AppNavHost.kt`
  - `shouldNavigateToTopLevelMapForHomeEntry()` 조건을 `map alias route` 한정에서 `bookmark(saved_route)`를 제외한 `visible한 non-map top-level route`로 넓혔다.
  - 즉 하단 탭이 보이는 상태에서 사용자가 `홈` 탭을 눌렀고 현재 route가 `map`이 아니면, 기본적으로 `navigateToTopLevelMapForHomeEntry()`를 타도록 변경하되, `saved_route`는 기존 plain top-level navigate를 유지한다.
- `MainNavGraph.kt`
  - `ReportRoute -> onNavigateToMap`
  - 이 경로를 plain top-level navigate가 아니라 `navigateToTopLevelMapForHomeEntry()`로 연결했다.
- `AppNavHostRoutingTest.kt`
  - `saved_route`는 제외하고, `report`, `my_page`, `search/*`에서는 home reentry reset 경로를 타야 한다는 규칙을 추가했다.
- `MainNavGraphTopLevelNavigationPolicyTest.kt`
  - top-level navigation policy 관점에서도 동일 규칙을 별도 검증으로 추가했다.

### 10.3 의도적으로 유지한 동작

- 검색 결과/검색 엔트리에서 "지도 미리보기로 복귀"하는 경로는 이번 수정으로 바꾸지 않았다.
- 이 경로는 선택한 목적지를 지도에서 계속 보여줘야 하는 handoff 성격이므로, "명시적 홈 복귀"와 동일하게 reset하면 오히려 UX가 깨진다.
- 따라서 이번 수정은 "홈으로 돌아오는 액션"과 "선택 목적지를 유지한 채 지도 preview로 돌아오는 액션"을 분리하는 방향이다.

### 10.5 현재 상태 정리

- FE 기준 홈 복귀 reset 경로 누락은 수정했다.
- 따라서 제보/마이페이지/검색 alias 상태에서 홈으로 복귀할 때 최근 목적지가 남아 있던 증상은 이번 변경으로 해소될 가능성이 높다.
- `saved_route`는 현재 코드상 reset 제외로 유지되어 있으므로, 이 경로에서도 동일 증상이 재현되는지는 실제 기기 기준으로 별도 확인이 필요하다.
- 남은 확인 포인트는 실제 기기에서 홈 복귀 시 `onHomeReentered()`가 호출되고, 그 직후 `selectedDestination cleared -> camera current location -> renderer ready` 순서가 정상적으로 이어지는지다.
