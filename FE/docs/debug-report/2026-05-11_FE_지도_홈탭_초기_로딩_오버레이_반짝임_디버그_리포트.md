# FE 지도 홈탭 초기 로딩 오버레이 반짝임 디버그 리포트
> 작성일: 2026-05-11
> Work Lane: FE
> 대상 화면: 일반사용자 모드 지도 홈탭

## 1. 증상

- 하단 탭 또는 다른 화면에서 지도 홈탭으로 복귀할 때, 화면 중앙에 무엇인가 잠깐 나타났다가 그 위로 지도가 덮이는 것처럼 보였다.
- 사용자가 체감한 표현은 `가운데 반짝`, `뭔가 나왔다가 지도에 덮임`이었다.
- BottomSheet가 아래에서 올라오는 애니메이션과는 별개의 현상이다.

## 2. 재현 관찰

- 지도 홈탭 진입 직후 매우 짧은 시간 동안 중앙 정렬 로딩 카드가 노출될 수 있다.
- 이후 Kakao renderer가 `Ready` 상태가 되면 해당 카드가 사라지고 실제 지도가 보인다.
- 일반사용자 모드의 화면 전환 리플 제거 후에도 이 현상은 남아 있었다.

## 3. 원인 분석

### 3.1 직접 원인

- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/KakaoMapViewport.kt`
  - `rendererStatus != Ready` 조건만으로 `MapRendererFallbackOverlay(...)`를 바로 그린다.
- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/MapViewport.kt`
  - `MapRendererFallbackOverlay`는 중앙 카드 + `CircularProgressIndicator()`를 포함한다.
- 따라서 renderer 준비가 아주 짧게 지연되는 정상 구간에서도 중앙 카드가 즉시 보이며, 이것이 사용자가 본 `반짝임`의 본체였다.

### 3.2 구조적 배경

- `MapView` 재생성이 많아질수록 이 overlay가 보일 기회도 늘어난다.
- 이전 시도에서 아래 두 가지를 이미 줄였다.
  - top-level 탭 `saveState/restoreState` 활성화
  - `onHomeReentered()`에서 `rendererSessionKey` 강제 증가 제거
- 이 수정은 강제 재생성 빈도는 줄였지만, `초기 로딩 overlay를 즉시 보여 주는 정책` 자체는 그대로였다.
- 그래서 renderer 준비가 빠른 케이스에서도 중앙 카드가 잠깐 보일 수 있었다.

## 4. 이번 적용 해결책

### 4.1 목표

- `짧은 정상 초기화`에서는 중앙 로딩 카드를 띄우지 않는다.
- 하지만 `실제 에러`와 `자동 재시도`는 기존처럼 즉시 보여 준다.
- BottomSheet 애니메이션은 건드리지 않는다.

### 4.2 적용 내용

- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/KakaoMapViewport.kt`
  - 초기 renderer loading 상태에만 300ms grace period를 추가했다.
  - grace period 안에 renderer가 `Ready`가 되면 fallback overlay를 아예 띄우지 않는다.
  - `renderer error`, `automatic retry`, `isRendererRestarting` 경로는 즉시 overlay를 유지한다.
- 정책 함수:
  - `shouldShowKakaoRendererFallbackOverlay(...)`
- 지연 상수:
  - `KAKAO_RENDERER_LOADING_OVERLAY_DELAY_MILLIS = 300L`

## 5. 기대 효과

- 지도 홈탭 복귀 직후 renderer가 빠르게 준비되는 대부분의 케이스에서 중앙 로딩 카드가 보이지 않는다.
- 사용자가 보던 `가운데 반짝` 체감이 줄어든다.
- 실제 renderer 오류나 자동 복구 상황에서는 안내 UI를 계속 유지한다.

  - `Ready` 상태에서는 미노출

## 7. 이번 시도로도 잔상이 남을 경우 다음 후보

- 후보 1
  - `MapRendererFallbackOverlay`가 아니라 `MapView` 자체의 빈 surface 노출이 남는지 실제 기기 로그와 화면 녹화로 분리 관찰
- 후보 2
  - top-level 탭 복귀 시 `MapView` host 자체를 더 오래 유지하는 구조로 전환
- 후보 3
  - renderer 준비 직전까지 마지막 지도 프레임 또는 브랜드 없는 neutral placeholder를 유지해 빈 surface 노출을 감춤

## 8. 결론

- 이번 현상의 직접 원인은 `rendererStatus != Ready`일 때 중앙 fallback overlay를 즉시 보여 주는 정책이었다.
- 이전 수정은 `재생성 빈도`를 줄이는 방향이었고, 이번 수정은 `노출 정책` 자체를 완화하는 방향이다.
- 즉, 이번 변경은 `원인 축소`가 아니라 `원인 UI 직접 조정`에 해당한다.

## 10. 현재 디버그 경계

### 10.1 여기까지 확인 완료

- 중앙에 잠깐 보이는 UI는 BottomSheet 애니메이션이 아니라 `MapRendererFallbackOverlay` 계열 표시다.
- 직접 원인 1차 분석은 완료했다.
  - `rendererStatus != Ready`이면 초기 loading 상태에서도 중앙 카드가 즉시 노출됐다.
- 재생성 빈도를 줄이는 선행 수정은 이미 반영돼 있다.
  - `MainNavGraph.kt`
    - `restoreState = true`
    - `saveState = true`
  - `MapViewModel.kt`
    - `onHomeReentered()`는 유지하되 renderer 강제 재생성 방향은 약화했다.
- 이번 단계에서 overlay 노출 정책까지 조정했다.
  - `KakaoMapViewport.kt`
    - `KAKAO_RENDERER_LOADING_OVERLAY_DELAY_MILLIS = 300L`
    - `shouldShowKakaoRendererFallbackOverlay(...)`
    - 초기 loading만 grace period를 거친 뒤 표시
    - `renderer error`, `automatic retry`, `isRendererRestarting`는 즉시 표시 유지

### 10.2 아직 미확인

- 실제 기기/에뮬레이터에서 이번 300ms grace period가 사용자의 체감 문제를 충분히 줄였는지 최종 확인은 아직 없다.
- 만약 여전히 번쩍임이 남는다면, 남은 후보는 overlay가 아니라 `MapView`의 빈 surface 또는 renderer attach 직후의 무채색 프레임 노출일 가능성이 높다.
- 즉, 현재 단계에서는 `중앙 카드 즉시 노출` 원인은 조정했지만, `MapView 표면 자체 노출` 여부는 아직 확정하지 못했다.
