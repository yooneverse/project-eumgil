# FE 길찾기 화살표 및 세그먼트마커 디버깅 리포트
> 작성일: 2026-05-15
> 작성자: 이재호
> 범위: FE 길찾기 지도 오버레이
> 수정 범위 제한: FE만 대상, 백엔드 미수정

---

## 1. 목적

길찾기 화면에서 보고된 아래 두 증상을 현재 FE 구현 기준으로 정리한다.

1. 화살표가 이전보다 너무 적게 보인다.
2. 확대/축소 중 화살표와 세그먼트 마커가 깜빡인다.

이 문서는 원인 후보를 코드 근거와 함께 정리하고, 다음 스레드에서 바로 이어서 디버깅할 수 있도록 관측 포인트와 프롬프트를 남긴다.

---

## 2. 증상 요약

### 2.1 사용자 체감 증상

- zoom-out 시 화살표 수가 급격히 줄어든다.
- 짧은 구간에서는 화살표가 0개 또는 1개만 보이는 경우가 생긴다.
- 확대/축소 종료 시점마다 화살표가 다시 그려지는 느낌이 강하다.
- 세그먼트 마커도 같은 타이밍에 함께 깜빡여 보인다.

### 2.2 이번 구현 이후 확인된 변화

- 화살표는 선분별 계산이 아니라 polyline 전체 누적 거리 기준으로 바뀌었다.
- 네비게이션에서는 한 레이어만 화살표를 그리도록 바뀌었다.
- Kakao/native 경로는 현재 카메라 zoom, latitude, bearing을 반영해 화살표 위치와 회전을 다시 계산한다.

- 핵심 상수:

  - `ROUTE_DIRECTION_ARROW_EDGE_PADDING_DP = 8.0`

  - `interval + edgePadding * 2` 보다 총 길이가 짧으면 화살표를 0개 반환한다.
  - 최소 1개 보장 로직은 없다.

### 3.2 Kakao/native 경로

- 파일:
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/KakaoMapViewportBindings.kt`
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/KakaoMapViewport.kt`
- 경로:
  - `createKakaoOverlayMarkerRenderStates(...)`
  - `createKakaoRouteDirectionArrowRenderStates(...)`
  - `syncMarkers(...)`
- 특징:
  - 화면 기준 `44dp` 간격을 현재 zoom, density, latitude로 meter 간격으로 환산한다.
  - bearing 변경 시 화살표 회전값도 다시 계산한다.
  - 화살표와 세그먼트 마커가 같은 overlay marker 리스트에 포함된다.

### 3.3 fallback 경로

- 파일: `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/MapViewportOverlayBackdrop.kt`
- 특징:
  - Compose `Canvas`에 polyline과 화살표를 직접 그린다.
  - 세그먼트 마커는 별도 Composable marker로 그린다.
  - Kakao native label layer clear 경로는 타지 않는다.

### 3.4 네비게이션 다중 polyline 처리

- 파일: `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/MapViewportOverlay.kt`
- 특징:
  - `navigation-focused > navigation-active > navigation-route` 우선순위로 한 레이어만 화살표를 그린다.
  - baseline segment polyline은 `showDirectionArrows = false` 이다.

## 4. 코드 근거 기반 관측

### 4.1 화살표가 적어 보이는 직접 원인

- 공통 샘플러는 최소 1개 보장을 하지 않는다.
- Kakao/native 쪽은 zoom-out 할수록 간격 meter 값이 커진다.
- 현재 테스트도 `zoom-out 시 화살표 수 감소`를 정상 동작으로 검증하고 있다.

### 4.2 zoom 기준 대략 간격 수치

lat `35.17`, density `3.0` 기준 계산값:

| Zoom | 화살표 간격(m) | 가장자리 패딩(m) | 화살표 1개라도 보이기 위한 최소 길이(m) |
| --- | ---: | ---: | ---: |
| 19 | 32.2 | 5.9 | 43.9 |
| 18 | 64.4 | 11.7 | 87.9 |
| 17 | 128.9 | 23.4 | 175.7 |
| 16 | 257.7 | 46.9 | 351.5 |

해석:

- zoom 17 이하에서 짧은 세그먼트는 화살표가 거의 사라지는 것이 현재 공식상 자연스럽다.
- 사용자가 원하는 "축소해도 최소한의 화살표 유지"와 현재 구현 방향이 다르다.

### 4.3 깜빡임 직접 원인 후보

#### 1순위

- `syncMarkers(...)` 에서 overlay marker state가 바뀌면 `labelManager.removeAllLabelLayer()` 로 전체 label layer를 삭제한다.
- 이후 facility marker layer와 overlay marker layer를 전부 다시 만든다.
- overlay marker에는 세그먼트 마커와 화살표가 함께 들어간다.

해석:

- 화살표 개수나 회전이 바뀌기만 해도 세그먼트 마커까지 같이 재생성된다.
- 사용자 입장에서는 둘이 같이 깜빡이는 것으로 보일 가능성이 높다.

#### 2순위

- Kakao route arrow는 `TransformMethod.AbsoluteRotation` 기반 map-space 모델이므로 기본 회전은 `segment heading` 자체를 사용한다. `segment heading - camera bearing`은 screen-relative transform fallback에만 해당한다.
- style cache key에 `rotationDegrees.roundToInt()` 가 포함된다.

해석:

- zoom뿐 아니라 bearing 변화도 bitmap/style 재사용을 깨뜨릴 수 있다.
- 회전값이 조금씩 바뀌면 화살표 스타일 캐시 miss와 marker 재구성이 더 자주 일어날 수 있다.

#### 3순위

- projected marker는 카메라 이동 중 추적 루프가 있다.
- native overlay marker는 이동 중 프레임 추적이 아니라 카메라 이동 종료 후 `syncMarkers(...)` 재실행 쪽에 더 가깝다.

해석:

- projected marker는 비교적 부드럽고, native overlay marker만 덜 부드럽게 갱신될 수 있다.

## 6. 현재 로그와 관측 포인트

### 6.1 이미 존재하는 로그

- 태그: `KakaoMapViewport`
- `SegmentMarkerTrace[MapViewportOverlay]`
  - navigation overlay 조합 결과
- `SegmentMarkerTrace[Projection]`
  - projected marker 추적 상태
- `SegmentMarkerTrace[RenderPath] renderPath=native-label`
  - native overlay marker render path
- `Markers synced ... overlay=<count> bearing=<deg>`
  - marker sync 시점의 overlay marker 개수와 bearing

### 6.2 현재 로그의 한계

- `arrowCount` 가 별도로 분리되어 있지 않다.
- `segmentMarkerCount` 와 `arrowCount` 가 같이 합쳐진 `overlay` 개수만 보인다.
- `zoomLevel`, `spacingMeters`, `edgePaddingMeters` 가 marker sync 로그에 없다.

## 7. 추천 디버깅 절차

### 7.1 재현 환경

- 길찾기 화면 진입
- route polyline, 세그먼트 마커가 모두 보이는 상태
- Kakao/native 지도 경로 우선 확인

### 7.2 로그 수집

아래 로그를 함께 본다.

```bash
adb logcat -s KakaoMapViewport System.out
```

### 7.3 확인 순서

1. zoom-in, zoom-out 을 각각 천천히 한 번씩 수행한다.
2. 같은 동작을 화면 녹화로 함께 남긴다.
3. `SegmentMarkerTrace[MapViewportOverlay]` 값이 고정인지 먼저 본다.
4. `SegmentMarkerTrace[RenderPath] renderPath=native-label` 이 zoom 종료마다 다시 찍히는지 본다.
5. `Markers synced ... overlay=` 값이 zoom마다 얼마나 바뀌는지 본다.
6. `SegmentMarkerTrace[Projection] cameraMoving=` 로그와 비교해 projected marker는 추적되는데 native overlay만 리셋되는지 본다.

### 7.4 다음으로 보강하면 좋은 로그

아래 값은 현재 미노출 상태라 추가 계측 후보로 적합하다.

- `createKakaoRouteDirectionArrowRenderStates(...)`
  - `zoomLevel`
  - `cameraBearingDegrees`
  - `spacingMeters`
  - `edgePaddingMeters`
  - `arrowCount`
- `syncMarkers(...)`
  - `segmentMarkerCount`
  - `arrowCount`
  - `didClearWholeLabelLayer`

## 10. 2026-05-15 적용 방향(A + C)

- A. 공통 화살표 샘플러에 `minimumPlacementCount = 1` 기반 최소 화살표 보장 로직을 추가한다.
- Kakao/native 화살표와 fallback 화살표 모두 공통 샘플러의 최소 보장 옵션을 사용한다.
- C. Kakao/native `syncMarkers()` 에서 `removeAllLabelLayer()` 를 제거하고 layer 단위 sync로 바꾼다.
- native overlay marker를 `eumgil-overlay-markers`(세그먼트/환승/정류장 등 point overlay)와 `eumgil-overlay-arrows`(direction arrow)로 분리한다.
- 화살표 개수/회전만 바뀌는 경우 세그먼트 마커 layer는 재생성하지 않도록 상태 비교를 분리한다.
- 검증 제약: 고정 Gradle cache 경로 `C:\tmp\codex-gradle-home` 의 `native-platform.dll.lock` 접근 거부 때문에 FE 단위 테스트 재실행은 이 세션에서 완료하지 못했다.

## 11. 2026-05-15 추가 업데이트(밀도 상향 + 화살표 깜빡임 재디버깅)

- 현재 상태 요약:
  - 세그먼트 마커 깜빡임은 `eumgil-overlay-markers` 분리 이후 재현 빈도가 크게 낮아졌다.
  - 화살표 깜빡임은 아직 남아 있었고, 직접 원인은 `direction arrow` layer 자체를 매 sync마다 `removeAll()`로 비우고 다시 만들던 경로였다.
  - 추가로, `startProjectedMarkerTracking()` 내부에서 이동 중 매 프레임 `syncMarkers()` 를 다시 태우면 zoom/bearing 변화 중 화살표 state가 계속 달라져 깜빡임이 커질 수 있었다.

- 이번 FE 수정:
  - 공통 화살표 간격 기준을 `44dp -> 28dp` 로 낮춰 기본 밀도를 올렸다.
  - Kakao/native 간격 정책에 `12m <= spacing <= 48m` clamp를 추가해서 zoom-out 시 간격이 과도하게 벌어지지 않도록 조정했다.
  - `minimumPlacementCount = 1` 유지로 짧은 route도 최소 1개 화살표는 보이도록 둔다.
  - `direction arrow` layer는 `removeAll()` 대신 label 단위 증분 갱신으로 변경했다.
  - 증분 갱신은 `add / moveTo / changeStyles / remove` 로 나누고, `RouteArrowLayerTrace` 로그에 각각의 개수를 남긴다.
- projected marker 추적 루프에서는 camera move 중에도 `syncMarkers(...)`와 projected overlay update를 매 프레임 함께 수행한다. 현재 구현 기준 native arrow sync는 일반 render 시점뿐 아니라 tracking loop에서도 계속 갱신된다.

- 현재 원인 우선순위 업데이트:
  1. 화살표가 적어 보이던 문제의 1순위는 meter 간격 정책이었다. 기본 간격 44dp와 zoom-out 시 무제한 spacing 증가가 합쳐져 route 길이에 비해 화살표 수가 너무 빨리 줄었다.
  2. 화살표 깜빡임의 1순위는 arrow layer의 전체 삭제/재생성이었다. 세그먼트 마커와 레이어는 분리됐지만, 화살표 레이어 자체는 여전히 매번 다시 만들어지고 있었다.
  3. 이동 중 추적 루프에서 native marker sync를 매 프레임 호출하던 경로가 2순위 증폭 요인이었다.
  4. bearing 변화에 따라 화살표 bitmap rotation style이 바뀌는 구조는 남아 있으므로, `changeStyles()` 호출 빈도는 계속 관찰이 필요하다.

- 추가 로그 포인트:
  - `RouteArrowLayerTrace count=<n> added=<n> moved=<n> styleChanged=<n> removed=<n> unchanged=<n> cameraMoving=<bool>`
  - `RouteArrowTrace count=<n> ... cameraBearingRad=<v> cameraBearingDeg=<v> finalRotation=<v>`
  - `Markers synced ... overlayPoints=<n> overlayArrows=<n> bearingRad=<v> bearingDeg=<v>`

- 이번 코드로 확인하고 싶은 것:
  - zoom 중 `RouteArrowLayerTrace` 에서 `added/removed` 가 크게 줄고, 대부분 `moved` 또는 `unchanged` 로 바뀌는지
  - zoom end 시점에만 arrow count 재계산이 일어나고, 이동 중에는 같은 label이 유지되는지
  - 여전히 깜빡임이 남으면 `styleChanged` 비중이 높은지 확인해서, 다음 단계로는 회전 quantization(예: 1도 대신 4~6도 bucket) 또는 zoom 중 rotation freeze 실험을 진행할지 판단

- 다른 디버깅 방법:
  - 실험 1. `RouteArrowLayerTrace` 와 화면 녹화를 함께 수집해서 `added/removed` 타이밍과 육안 깜빡임을 1:1로 맞춘다.
  - 실험 2. zoom gesture 동안 arrow rotation update를 잠시 고정하고, zoom end에서만 bearing 반영하도록 분기해 본다. 이때 깜빡임이 줄면 원인은 style churn 쪽이다.
  - 실험 3. arrow bitmap rotation을 1도 단위가 아니라 4~6도 bucket으로 양자화해서 `styleChanged` 횟수를 비교한다.
  - 실험 4. 현재 bitmap pre-rotation 방식 대신 label transform rotation을 사용할 수 있는지 별도 브랜치에서 검증한다. 가능하면 style 교체 없이 `rotateTo()` 만으로 갱신할 수 있다.
  - 실험 5. 화면에 보이는 route 길이 기준으로 최소 화살표 개수를 계산하는 정책(예: visible route span당 최소 2개)과 현재 polyline total length 기반 정책을 비교한다.

## 12. 2026-05-15 추가 업데이트(zoom-out 화살표 과밀)

- 증상:
  - 세그먼트 마커 깜빡임은 해결됐지만, 화살표는 zoom-out 시 화면에 기대보다 많이 남아 있었다.
  - 이 이슈는 `최소 1개 보장` 자체보다는 zoom-out 구간에서 spacing meter 상한을 너무 낮게 묶어둔 정책 영향이 컸다.

- 원인 정리:
  - 현재 화살표 샘플러는 `minimumPlacementCount = 1` 덕분에 짧은 경로에서도 `0개`로 떨어지지 않는다.
  - 그런데 Kakao/native spacing 계산을 `12m <= spacing <= 48m`로 clamp 하면, zoomLevel 이 내려가도 spacing 이 48m 이상 벌어지지 않는다.
  - 그 결과 zoom-out 시에도 화살표가 충분히 성기게 줄지 않고, 화면에 더 많은 경로가 들어오면 화살표 총량이 과밀하게 보일 수 있다.

- 이번 수정:
  - `minimumPlacementCount = 1`은 유지했다.
  - 대신 Kakao/native spacing 계산에서 `maximumValue = 48m` 상한을 제거하고, `minimumValue = 12m`만 유지했다.
  - 즉 zoom-out 할수록 spacing meter 는 다시 자연스럽게 커지고, 짧은 route 는 최소 1개만 남는다.

- 기대 효과:
  - zoom-in: 기존보다 비교적 촘촘한 화살표 유지
  - zoom-out: 화살표 수가 다시 감소
  - short route: `0개` 대신 `1개` 유지

- 검증:
  - `:app:compileDebugKotlin`, `:app:compileDebugUnitTestKotlin` 컴파일 통과
  - 테스트 기대값도 zoom 18 > zoom 17 조건 하에서 `3개 -> 1개`로 최신화
