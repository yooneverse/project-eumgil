# FE 경로안내 화살표 회전 디버깅 리포트

- 작성일: 2026-05-15
- 작성자: 이재호
- 범위: FE만 분석
- 상태: 코드 수정 없이 원인 분석 및 다음 디버깅 단계 정리

---

## 1. 문제 요약

경로안내 화면에서 사용자가 두 손가락으로 지도를 회전하면, 경로 화살표가 세그먼트의 실제 진행 방향을 유지하지 못하고 화면 회전에 끌려가는 것처럼 보인다.

지난 FE 수정에서는 Kakao/native 경로 화살표에 카메라 bearing 보정을 넣었지만, 사용자 확인 결과 체감상 문제가 해결되지 않았다. 현재 시점에서 가장 중요한 것은 "보정식 자체가 틀렸는지", "보정값 단위가 틀렸는지", "렌더 타이밍이 늦는지", "Kakao Label transform 기본 동작을 잘못 가정했는지"를 분리해서 확인하는 것이다.

## 2. 현재 FE 구현 기준 사실

### 2.1 bearing 상태 흐름

- `MapCameraTarget`에는 `center`, `source`, `requestId`, `zoomLevel`만 있고 bearing은 없다.
- `MapUiAction.ViewportCameraChanged`도 `center`, `zoomLevel`만 받는다.
- `MapViewModel.handleViewportCameraChanged(...)` 역시 center/zoom만 상태에 반영한다.
- 다만 overlay 전용 화면은 feature 상태를 거치지 않고 `MapOverlayViewport -> KakaoMapViewport` 경로로 직접 렌더링되므로, bearing을 전역 상태에 넣지 않아도 렌더 시점에 Kakao 실제 카메라 값을 읽어 보정하는 구조는 가능하다.

관련 파일:

- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/model/MapCameraTarget.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/MapContract.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/MapViewModel.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/MapOverlayViewport.kt`

### 2.2 Kakao/native 화살표 렌더링

- Kakao 경로 화살표는 polyline 자체가 아니라 `ROUTE_DIRECTION_ARROW` overlay marker bitmap으로 생성된다.
- `syncMarkers(...)`에서 `readyMap.getCameraPosition()`을 읽어 `cameraLatitude`, `cameraZoomLevel`, `cameraBearingDegrees`를 구한다.
- `createKakaoOverlayMarkerRenderStates(...)` -> `createKakaoRouteDirectionArrowRenderStates(...)`로 들어가며, 각 화살표 rotation은 `segmentStart.rotationDegreesTo(end) - cameraBearingDegrees`로 계산한다.
- 카메라 회전 종료 후 `setOnCameraMoveEndListener`에서 `syncMarkers(...)`를 다시 호출한다.

관련 파일:

- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/KakaoMapViewport.kt`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/KakaoMapViewportBindings.kt`

### 2.3 fallback 화살표 렌더링

- fallback 경로는 Compose `Canvas`에서 projected screen points를 만든 뒤 `atan2(deltaY, deltaX)`로 화살표 각도를 계산한다.
- fallback surface는 north-up 고정 투영이므로, 별도 bearing 보정 없이 projected angle이 최종 화면 각도가 된다.

관련 파일:

- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/MapViewportOverlayBackdrop.kt`

### 2.4 화살표가 붙는 polyline 레이어

- navigation overlay 기준 화살표는 `focused -> active -> route` 우선순위로 단 한 레이어에만 붙는다.
- baseline segment polyline은 `showDirectionArrows = false`로 유지된다.

관련 파일:

- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/MapViewportOverlay.kt`

## 3. 가장 유력한 원인 가설

### 1순위. `rotationAngle` 단위 해석 오류

신뢰도: 높음

근거:

- Kakao 공식 문서상 `CameraPosition.getRotationAngle()`은 radian이다.
- 현재 FE 변수명은 `cameraBearingDegrees`인데, `rotationAngle` 값을 degree 변환 없이 그대로 쓰고 있다.
- 화살표 보정식은 degree 기반 세그먼트 각도에서 `cameraBearingDegrees`를 빼는 구조다.
- 따라서 지도 90도 회전 시 기대 보정량은 `90`이어야 하는데 실제로는 약 `1.57`만 빠질 가능성이 높다.

의미:

- 보정식 방향은 맞더라도, 보정량이 거의 안 들어가서 사용자가 보기엔 "전혀 안 고쳐진 것처럼" 느껴질 수 있다.

### 2순위. 카메라 회전 중간 프레임 미추종

신뢰도: 높음

근거:

- 현재 코드는 `setOnCameraMoveStartListener` / `setOnCameraMoveEndListener`만 사용한다.
- 회전 중간에는 overlay marker를 연속 갱신하지 않고, 종료 시점에만 `syncMarkers(...)`를 다시 탄다.
- Kakao 공식 문서에는 최신 카메라 값을 지속적으로 받을 수 있는 `OnCameraPositionListener`가 별도로 있다.

의미:

- 단위를 바로잡아도 "회전하는 동안"은 계속 끌려가는 것처럼 보일 수 있다.
- 단, 회전이 끝난 뒤에도 틀리면 1순위 원인이 더 우선이다.

### 3순위. Kakao Label transform 기본값 가정 오류

신뢰도: 중간

근거:

- 현재 `LabelOptions.setTransform(...)`를 화살표 label 생성 시 명시하지 않는다.
- Kakao 공식 문서상 `LabelOptions`는 `TransformMethod`를 가질 수 있고, 기본값 `Default`는 지도 회전과 무관하게 라벨이 화면 위쪽을 유지한다.
- `AbsoluteRotation`, `AbsoluteRotation_Decal` 등 transform 종류에 따라 회전 해석이 달라질 수 있다.

의미:

- 현재 보정식을 어떤 좌표계 기준으로 적용해야 하는지는 transform 방식과 묶여 있다.
- 다만 1순위와 2순위를 먼저 확인하는 것이 더 싸고 빠르다.

### 4순위. bearing 미보유 상태 자체가 직접 원인

신뢰도: 낮음

근거:

- overlay-only 경로는 `MapOverlayViewport`에서 직접 Kakao renderer를 호출한다.
- 이미 `syncMarkers(...)` 안에서 실제 카메라 값을 직접 읽고 있어, 전역 상태 미보유가 즉시 blocker는 아니다.

의미:

- bearing을 상태로 올리는 작업은 구조 정리에는 도움이 될 수 있지만, 이번 현상 미해결의 1차 원인으로 보이지는 않는다.

## 4. 코드 기준 핵심 증거

### 4.1 camera move end에서만 marker sync

- `KakaoMapViewport.kt`
  - `setOnCameraMoveStartListener(...)`
  - `setOnCameraMoveEndListener(...)`
  - `latestState?.let { state -> syncMarkers(readyMap = readyMap, state = state) }`

해석:

- 회전 gesture 도중에는 route arrow가 연속 재계산되지 않는다.

### 4.2 bearing 값을 degree처럼 사용 중

- `KakaoMapViewport.kt`
  - `val cameraBearingDegrees = cameraPosition?.rotationAngle ?: 0.0`
- `KakaoMapViewportBindings.kt`
  - `segmentStart.rotationDegreesTo(end = segmentEnd) - cameraBearingDegrees`

해석:

- 변수명은 degree지만 Kakao SDK 값은 공식 문서상 radian이다.

### 4.3 화살표 rotation 로그 부재

- 현재 debug summary는 `bearing=` 정도만 남기고 있다.
- route arrow별 `segment heading / camera bearing(rad) / camera bearing(deg) / final marker rotation` 로그는 없다.
- `createNativeSegmentRenderPathDebugSummary(...)`는 `ROUTE_SEGMENT_JUNCTION` 위주라 route arrow rotation 진단에는 직접 도움이 부족하다.

해석:

- 실제로 어떤 값이 들어가는지 확인 가능한 관측점이 아직 부족하다.

### 4.4 fallback은 다른 좌표계

- fallback은 projected point delta 기반 `atan2(deltaY, deltaX)`다.
- 여기서는 Kakao camera bearing 자체가 개입하지 않는다.

해석:

- native 문제와 fallback 문제를 혼동하면 안 된다.

## 6. 다음 디버깅 순서

### Step 1. 단위 문제 먼저 검증

목표:

- 실제 90도 회전 후 logcat의 `bearing=` 값이 `1.57` 근처인지 확인

판정:

- `1.57` 근처면 radian/degree mismatch 확정
- `90` 근처면 다른 원인이 우선

### Step 2. route arrow 상세 로그 추가

추가할 로그:

- overlayId
- segment start/end
- segment absolute heading(deg)
- camera bearing raw(rad)
- camera bearing converted(deg)
- final arrow rotation(deg)

목표:

- 계산식이 실제로 어떤 값을 쓰는지 route arrow 단위로 확인

### Step 3. bearing degree 변환 반영

수정 방향:

- `cameraPosition.rotationAngle`을 `Math.toDegrees(...)`로 변환한 뒤 arrow rotation 계산에 사용

목표:

- 회전 종료 후 화살표 end-state가 정상인지 확인

### Step 4. 회전 중간 프레임 추종 여부 확인

확인 포인트:

- 종료 후는 맞는데 회전 중간에만 끌려가 보이는지

필요 시:

- `OnCameraPositionListener`를 사용해 회전 중에도 `syncMarkers(...)` 또는 route arrow 재계산을 태울지 검토

### Step 5. transform method A/B 확인

비교 대상:

- 기본값 유지
- `TransformMethod.AbsoluteRotation`
- 필요 시 `AbsoluteRotation_Decal`

목표:

- Kakao Label이 FE가 기대한 좌표계로 회전을 적용하는지 확인

## 8. 이번 리포트 기준 결론

현재 시점에서 가장 유력한 직접 원인은 `CameraPosition.rotationAngle`이 radian인데 FE가 degree처럼 사용하고 있다는 점이다. 이 문제를 먼저 확인하고 수정하지 않으면 bearing 보정이 들어간 것처럼 보여도 실제 보정량이 거의 0에 가까워 체감상 해결되지 않는다.

그 다음으로는 현재 구현이 `camera move end`에서만 marker를 다시 그리기 때문에, 회전 gesture 도중에는 여전히 끌려가는 것처럼 보일 가능성이 높다. 따라서 다음 작업은 "단위 수정"과 "회전 중간 프레임 추종 여부 확인" 순서로 진행하는 것이 가장 효율적이다.

## 10. 2026-05-15 추가 추적 결과

### 10.1 이번에 실제로 시도한 FE 수정

- `KakaoMapViewport.kt`
  - `CameraPosition.rotationAngle` raw 값을 `cameraBearingRadians` 로 읽고, `Math.toDegrees(...)` 로 변환한 `cameraBearingDegrees` 를 별도로 계산
  - marker sync 로그에 `bearingRad`, `bearingDeg` 를 모두 출력
  - route arrow 계산 로그를 보기 위해 arrow별 상세 로그를 추가
  - 회전 gesture 중간에도 `postOnAnimation` 루프 안에서 `syncMarkers(...)` 를 다시 돌리도록 수정
- `KakaoMapViewportBindings.kt`
  - route arrow rotation 계산식을 degree 기준으로 정리
  - `segment heading / camera bearing(rad) / camera bearing(deg) / final rotation` 디버그 상태를 분리
  - route arrow label 에는 `TransformMethod.None` 을 명시

### 10.2 사용자 확인 결과

- 사용자 확인 기준으로 문제는 해결되지 않았다.
- 즉, `rotationAngle` radian -> degree 변환과 `camera move end` 외 중간 프레임 재동기화는 "필요한 수정" 이었을 가능성은 높지만, "충분한 수정" 은 아니었다.

### 10.3 이번 추가 추적으로 바뀐 해석

- 기존 1순위 가설이었던 "`rotationAngle` 단위 mismatch" 는 여전히 실제 버그였을 가능성이 높다.
- 다만 그 수정 이후에도 사용자가 같은 현상을 봤다면, 현재 이슈의 직접 체감 원인은 bearing 단위 하나만으로 설명되지 않는다.
- 따라서 원인 우선순위는 아래처럼 다시 정리해야 한다.

## 11. 수정 후 다시 정렬한 원인 가설

### 1순위. `getCameraPosition()` 자체가 회전 중에는 최신 bearing 을 보장하지 않는다

신뢰도: 높음

근거:

- Kakao 공식 camera 문서에는 `kakaoMap.getCameraPosition()` 이 "API 호출 시점의 카메라 값" 을 가져오며, 진행 중인 카메라 이동이 있더라도 현재 진행 중인 지도의 상태 값과 차이가 있을 수 있다고 명시되어 있다.
- 현재 FE 는 회전 중간 프레임에서도 여전히 `readyMap.getCameraPosition()` 을 읽어 bearing 을 구한다.
- 즉 "프레임마다 다시 계산" 하더라도, 입력으로 쓰는 bearing 값이 stale snapshot 이면 화살표는 여전히 늦게 따라오거나 잘못된 각도로 계산될 수 있다.

의미:

- 이번 수정으로 `camera move end` 의존성은 줄였지만, 계산 입력이 stale 이면 회전 중 잘못된 값을 계속 다시 그리는 셈이 된다.
- 이 경우 체감상 "여전히 끌려간다" 또는 "끝까지 안 맞는다" 가 둘 다 가능하다.

FE-only 다음 검증:

- 회전 gesture 중 `getCameraPosition()` 값과 `getCameraPosition(OnCameraPositionListener)` 값이 실제로 얼마나 다르게 들어오는지 같은 프레임 기준으로 비교 로그를 남겨야 한다.
- 단, Kakao 공식 문서상 async getter 도 "이전 카메라 작업 수행 후" 값을 돌려주므로, 실제로는 move end 시점에만 돌아올 가능성도 있다. 이 부분은 반드시 실측 로그로 확인해야 한다.

### 2순위. 현재 route arrow 좌표계 모델 자체가 Kakao Label transform 방식과 어긋날 수 있다

신뢰도: 높음

근거:

- Kakao 공식 문서상 `TransformMethod.Default` 는 라벨의 회전 값이 항상 스크린 위쪽 방향을 유지한다.
- `TransformMethod.None` 은 `Default` 와 같은 속성을 갖는다고 명시되어 있다.
- 즉 현재 FE 가 가정한 "screen-aligned label 위에, bearing 을 빼서 screen rotation 을 직접 계산한다" 는 모델은 Kakao 기본 transform 해석 위에 얹힌 가정이다.
- 사용자 확인상 단위 수정 + 중간 프레임 재동기화 후에도 해결되지 않았다면, 현재 가정 자체가 틀렸을 가능성을 다시 봐야 한다.

의미:

- 현재 모델은 "bitmap 내부를 회전시킨 뒤, label 은 screen-upright" 라는 구조다.
- 다른 모델은 "bitmap 은 segment heading 기준으로만 회전시키고, label transform 을 `AbsoluteRotation` 또는 `AbsoluteRotation_Decal` 로 바꿔 Kakao 가 map rotation 을 처리하게 맡기는 방식" 이다.

FE-only 다음 검증:

- 아래 2가지를 A/B 로 비교해야 한다.
  - A안: 현재 방식 유지
    - screen rotation = `segmentHeading - cameraBearingDeg`
    - transform = `None` 또는 미지정
  - B안: map-space 방식
    - bitmap rotation = `segmentHeading`
    - transform = `AbsoluteRotation` 또는 `AbsoluteRotation_Decal`
- 이 비교가 필요한 이유는, 현재 문제의 남은 원인이 "bearing 입력" 이 아니라 "label transform 기준" 일 수 있기 때문이다.

### 3순위. Kakao native label layer 자체가 회전 중 frame-by-frame 재생성에 약할 수 있다

신뢰도: 중간

근거:

- 현재 route arrow 는 polyline 이 아니라 label bitmap marker 다.
- FE 는 회전 중에 arrow layer 를 remove/re-add 하며 새 bitmap style 을 계속 만들어서 동기화한다.
- 이 방식은 계산이 맞아도 Kakao native layer 반영이 gesture frame 을 충분히 따라오지 못하면 체감상 계속 늦을 수 있다.

의미:

- 원인이 계산식이 아니라 native label pipeline 자체의 반영 타이밍이면, bearing/transform 보정만으로는 끝나지 않는다.
- 이 경우 "회전 중에는 native arrow 를 잠시 포기하고 Compose fallback arrow 를 보여주는 전략" 이 더 안정적일 수 있다.

FE-only 다음 검증:

- 회전 시작 시 native route arrow layer 를 숨기고, 회전 중에는 `MapViewportOverlayBackdrop` 의 projected arrow 만 노출
- 회전 종료 후 native label arrow 를 다시 노출
- 이 A/B 로 사용자가 체감하는 "끌려감" 이 사라지면, 계산 문제보다 native render timing 문제가 더 유력하다.

### 4순위. segment heading 계산식이 지도 투영 기준과 완전히 일치하지 않을 수 있다

신뢰도: 중간 이하

근거:

- 현재 heading 은 `atan2(-(latDelta), lngDelta)` 기반 단순 계산이다.
- Busan 범위에서는 큰 오차가 나기 어렵지만, Kakao 내부 projection 이나 실제 screen transform 과 완전히 같다고 보장할 수는 없다.
- 다만 이 이슈는 보통 "일정한 각도 오프셋" 형태로 나타날 가능성이 높고, 사용자가 보고한 "회전할 때 끌려가는 느낌" 을 1차로 설명하기에는 힘이 약하다.

의미:

- 이 가설은 1~3순위가 부정된 뒤에 보는 보조 가설이다.

## 12. 현재 기준으로 추천하는 다음 해결 방법

### 해결 방법 A. `getCameraPosition()` stale 여부를 먼저 실측

목표:

- 회전 중 frame 마다 읽는 `getCameraPosition()` bearing 이 실제로 stale 인지 확인

방법:

- 회전 gesture 중 같은 순간에 아래 값을 모두 로그로 남긴다.
  - `getCameraPosition().rotationAngle`
  - async `getCameraPosition(OnCameraPositionListener)` callback 값
  - `onCameraMoveEnd` 의 최종 bearing

판정:

- 회전 중 sync 루프의 raw bearing 이 늦게 따라오면, 현재 FE 의 per-frame sync 구조는 입력값부터 틀린 구조다.

### 해결 방법 B. screen-space 모델 대신 `AbsoluteRotation` 계열로 A/B

목표:

- Kakao Label transform 기준이 현재 FE 가정과 다른지 확인

방법:

- route arrow 전용으로 아래 두 구성을 비교
  - 현재 방식: `segmentHeading - cameraBearingDeg`, transform=`None`
  - 대안 방식: `segmentHeading`, transform=`AbsoluteRotation`
- 필요 시 `AbsoluteRotation_Decal` 도 비교

판정:

- B안에서만 회전 중/회전 후 둘 다 안정적이면, 현재 root cause 는 bearing 단위가 아니라 label transform 기준 mismatch 다.

### 해결 방법 C. 회전 중에는 Compose fallback arrow 로 전환

목표:

- Kakao native label frame delay 자체를 우회

방법:

- route arrow 만은 회전 시작~종료 사이에 projected Compose overlay 로 임시 전환
- 회전 종료 후 Kakao native arrow layer 복귀

장점:

- fallback 은 이미 projected screen angle 기반이라 현재 화면 방향과 직접 맞는다.

단점:

- native / fallback 시각 차이, 전환 타이밍, 깜빡임 보정이 추가로 필요하다.

### 해결 방법 D. route arrow 자체를 label bitmap 이 아닌 다른 renderer 로 옮기기

목표:

- label transform, layer re-add, bitmap 회전 캐시 문제를 구조적으로 제거

후보:

- RouteLine pattern
- Shape / Polyline 기반 direction glyph
- 전 구간 Compose overlay

비고:

- 구현량은 커지지만, 현재 이슈가 Kakao label path 특유의 제약이라면 장기적으로 가장 안정적일 수 있다.

## 13. 검증 및 문서 상태

### 이번 추가 추적 기준 검증 결과

- FE 단위 테스트는 요청한 고정 캐시 경로로 실행했다.
- 그러나 현재 프로젝트 테스트 러너 상태 때문에 전체 `:app:testDebugUnitTest` 와 대상 테스트 클래스 실행 모두 `initializationError / ClassNotFoundException` 으로 실패했다.
- 따라서 이번 작업에서 자동화로 확인한 것은 "테스트 클래스 산출물 생성" 과 "Gradle compile 단계 통과" 수준이고, 실제 런타임 동작 검증은 사용자 수동 확인이 기준이었다.

### backend 영향

- 현재까지 backend 수정 필요 근거는 없다.
- 이 이슈는 여전히 FE 렌더링 / Kakao SDK 해석 / overlay 전략 문제로 보는 것이 타당하다.

### 현재 문서 기준 결론

- `rotationAngle` radian -> degree 수정은 필요했지만, 사용자 체감 문제의 최종 해결책은 아니었다.
- 가장 먼저 다시 확인해야 할 것은 `getCameraPosition()` stale snapshot 문제다.
- 그 다음은 route arrow 를 "screen-upright label 위 bitmap 회전" 으로 처리할지, 아니면 `AbsoluteRotation` 계열 transform 으로 Kakao 에 맡길지의 좌표계 선택 문제다.
- 만약 둘 다 만족스럽지 않으면, 회전 중 native arrow 를 포기하고 Compose fallback 또는 다른 renderer 로 바꾸는 것이 더 현실적인 FE 해결책이다.

## 14. 2026-05-15 A안 실패 확인

### 14.1 사용자 검증 결과

- 사용자 실제 검증 기준으로 A안 적용 후에도 route direction arrow가 회전 gesture 중 지도 회전에 끌려가는 현상이 유지됐다.
- 따라서 이번 실패는 "listener 기반 최신 bearing을 못 받아서만 생기는 문제"로 단정할 수 없다.

### 14.2 실패 해석

- 이번 결과는 두 가능성 중 하나다.
  1. `getCameraPosition()` stale snapshot 가설이 일부 맞더라도, 그것만 고쳐서는 현상 핵심이 해결되지 않는다.
  2. route arrow 자체를 screen-space 회전으로 계산하는 현재 좌표계 모델이 Kakao Label transform 기본 동작과 맞지 않는다.

- 특히 Kakao 공식 문서상 `TransformMethod.Default` / `None` 은 라벨이 화면 위쪽 방향을 유지하는 성격이 강하고,
  `AbsoluteRotation` 계열은 라벨이 가진 고유 회전 방향을 유지한다.
- 이번 사용자 검증 실패는 bearing 입력보다 "label이 어떤 좌표계에서 회전하느냐"가 더 핵심일 가능성을 높인다.

## 15. B안 전환: map-space absolute rotation

### 15.1 전환 이유

- A안은 "segmentHeading - cameraBearingDeg" 방식의 screen-relative 보정 모델을 유지한 채 입력 freshness를 개선한 접근이었다.
- 그런데 체감 현상이 유지됐으므로, 이제는 arrow를 screen-space에서 직접 보정하지 말고 map-space 절대 회전으로 넘겨 Kakao transform이 지도 회전을 처리하게 하는 모델을 검증해야 한다.

### 15.2 이번 FE 수정 방향

- route arrow bitmap 회전값을 `segmentHeadingDegrees` 자체로 사용한다.
- route arrow `LabelOptions.setTransform(...)` 은 `TransformMethod.AbsoluteRotation` 으로 둔다.
- 즉 계산 모델은 아래처럼 바뀐다.

```text
기존 A안:
- finalRotation = segmentHeading - cameraBearingDeg
- transform = None

이번 B안:
- finalRotation = segmentHeading
- transform = AbsoluteRotation
```

### 15.3 이번 B안에서 확인할 로그

- `RouteArrowTrace`
  - `rotationModel=map-absolute`
  - `transform=AbsoluteRotation`
  - `cameraBearingDeg`
  - `finalRotation`
- `CameraBearingTrace`
  - stale snapshot 여부는 계속 비교하되, B안에서는 bearing 값이 맞더라도 최종 회전은 camera bearing을 직접 빼지 않는 점이 핵심이다.

### 15.4 B안 실패 시 즉시 다음 포인트

1. `TransformMethod.AbsoluteRotation_Decal` 로 즉시 교체해서 지도 기울기/표면 부착 차이를 비교한다.
2. 그래도 실패하면 Kakao native label layer 반영 타이밍 이슈로 보고, 회전 중에는 native arrow를 숨기고 Compose projected arrow로 임시 전환하는 C안을 검증한다.

### 15.5 backend 요청 필요 여부

- 여전히 없음.
- 이번 단계도 FE route arrow 렌더링 모델과 Kakao SDK label transform 해석 범위다.
