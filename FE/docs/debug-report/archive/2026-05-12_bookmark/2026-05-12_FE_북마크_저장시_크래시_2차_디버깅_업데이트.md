# FE 북마크 저장 시 크래시 2차 디버깅 업데이트

> 작성일: 2026-05-12
> Work Lane: FE
> 범위: "북마크 화면이 항상 깨지는 것"이 아니라 "북마크를 등록했을 때만 터진다"는 추가 제보를 기준으로 저장 직후 경로를 재추적한 결과

## 1. 증상 재정의

- 기존 가설
  - 북마크 화면 전체 레이아웃 또는 탭 복원 정책 문제

- 사용자 추가 제보
  - 화면 자체는 항상 깨지지 않음
  - 북마크를 등록했을 때만 터짐

- 이번 턴 판단
  - `SavedRouteScreen` 일반 렌더링보다 `북마크 저장 -> 북마크 화면 진입/복귀` 경로가 더 유력

## 2. 저장 경로 비교 결과

- 예외를 이미 흡수하는 경로
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/MapViewModel.kt`
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/search/SearchViewModel.kt`
  - 공통점: `bookmarkRepository.saveBookmark()`를 `runCatching`으로 감싸고 실패를 UI 상태로 되돌림

- 예외를 그대로 올리던 경로
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/navigation/NavigationViewModel.kt`
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/lowvision/LowVisionNavigationCompleteRoute.kt`
  - 공통점: `bookmarkRepository.saveBookmark()` 성공/실패를 분기하지 않고 바로 다음 화면 이동으로 이어짐

## 3. 이번 턴 적용한 FE 패치

- `NavigationViewModel.kt`
  - `saveDestinationBookmarkAndNavigate()`를 `runCatching`으로 감쌌다
  - 저장 성공일 때만 `completeNavigation(NavigateToSavedRoute)`를 실행하도록 변경했다
  - 재현 시점 추적용 로그 추가
    - `BookmarkSaveTrace[NavigationViewModel] result=success ...`
    - `BookmarkSaveTrace[NavigationViewModel] result=failure ...`

- `LowVisionNavigationCompleteRoute.kt`
  - 저장을 `runCatching`으로 감쌌다
  - 저장 성공일 때만 `onNavigateToBookmark()`를 호출하도록 변경했다
  - 재현 시점 추적용 로그 추가
    - `BookmarkSaveTrace[LowVisionNavigationCompleteRoute] result=success ...`
    - `BookmarkSaveTrace[LowVisionNavigationCompleteRoute] result=failure ...`

## 4. 테스트 보강

- `FE/app/src/test/java/com/ssafy/e102/eumgil/feature/navigation/NavigationViewModelTest.kt`
  - 저장 성공 시 `NavigateToSavedRoute`로 종료되는 케이스 추가
  - 저장 실패 시 네비게이션이 유지되고 북마크 화면 이동이 일어나지 않는 케이스 추가

- `FE/app/src/test/java/com/ssafy/e102/eumgil/feature/lowvision/LowVisionNavigationCompleteScreenTest.kt`
  - 저시력 완료 저장 경로가 `runCatching`과 성공 시 이동 구조를 유지하는지 정적 검증 추가

## 5. 검증 결과

- 성공
  - `./gradlew.bat :app:compileDebugUnitTestKotlin`

- 한계
  - `./gradlew.bat :app:testDebugUnitTest --tests "com.ssafy.e102.eumgil.feature.navigation.NavigationViewModelTest" --tests "com.ssafy.e102.eumgil.feature.lowvision.LowVisionNavigationCompleteScreenTest"`
  - 기존 테스트 런타임 환경의 `initializationError` / `ClassNotFoundException` 때문에 실행 결과를 신뢰할 수 없었음
  - 즉, 이번 변경은 컴파일 안전성까지는 확인됐고, 테스트 런타임 블로커 때문에 실행 통과 여부까지는 닫지 못함

## 6. 다음 확인 포인트

- 사용자가 같은 경로를 다시 눌렀을 때 아래 로그가 찍히는지 확인
  - `BookmarkSaveTrace[NavigationViewModel] result=success|failure`
  - `BookmarkSaveTrace[LowVisionNavigationCompleteRoute] result=success|failure`

- `failure`가 찍히는 경우
  - `BookmarkRepository` 또는 원격 북마크 저장 응답을 추가 추적
  - 저장 실패가 특정 장소 타입, provider-only 장소, 또는 저시력 완료 경로에 한정되는지 분기

- `success`가 찍히는데도 북마크 화면 진입 시 문제가 남는 경우
  - 그때는 다시 `SavedRouteScreen`의 `EMPTY -> CONTENT` 전환과 첫 카드 렌더 경로를 추적

## 7. Cross-Lane Handoff

- 현재 기준 BE 전달사항 없음
- 아직은 FE 저장 호출부의 예외 처리 누락과 FE 화면 전환 경로가 핵심 범위

## 8. 3차 디버깅 업데이트 (리스트 카드 레이아웃 가설)

### 8.1 추가 관찰

- 사용자가 다시 "깨져 있다"고 피드백함
- 이 시점에서 저장 예외 가드만으로는 체감 증상이 해소되지 않았다고 판단
- `북마크가 0개일 때는 정상`이고 `1개 이상일 때만 깨짐`이면, 빈 상태가 아니라 리스트 카드 렌더링 경로를 의심하는 쪽이 더 타당함

### 8.2 새 가설

- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/savedroute/SavedRouteScreen.kt`
- `SavedBookmarkPrimaryActionButton()`이 현재 `NoRippleSavedRouteNavigationButton()`을 사용함
- 이 커스텀 버튼 내부는 `Row(fillMaxWidth())`를 사용하고 있었음
- 그런데 이 버튼은
  - 빈/에러 상태 CTA에서는 `Modifier.weight(1f)`로 사용됨
  - 북마크 리스트 카드 안에서는 별도 width 제한 없이 사용됨
- 결과적으로 리스트 카드 안에서는 버튼 내부 content가 불필요하게 전체 가로폭을 요구해 제목/주소 컬럼을 짓누르거나 레이아웃이 깨질 가능성이 있었음

### 8.3 이번 턴 적용 패치

- `SavedRouteScreen.kt`
  - `NoRippleSavedRouteNavigationButton()`에 `fullWidthContent` 플래그를 추가
  - 빈/에러 상태 CTA(`SavedBookmarkStateActions`)에서는 `fullWidthContent = true`
  - 리스트 카드 액션 버튼(`SavedBookmarkPrimaryActionButton`)에서는 기본값 `false`를 사용해 content가 wrap 되도록 분리

- `SavedRouteScreenPolicyTest.kt`
  - 상태 카드 CTA는 `fullWidthContent = true`를 유지하는지 검증 추가
  - 리스트 카드 액션 버튼은 `fullWidthContent = true`를 강제하지 않는지 검증 추가

### 8.4 검증 결과

- 성공
  - `./gradlew.bat :app:compileDebugUnitTestKotlin`

### 8.5 다음 확인 포인트

- 일반 북마크 화면에서
  - 장소 카드가 한 줄 이상일 때 버튼이 과도하게 넓어지지 않는지
  - 제목/주소 텍스트가 눌려 깨지지 않는지
  - 경로 북마크 카드도 동일하게 정상 배치되는지

- 만약 이 수정 후에도 동일하면 다음 축으로 넘어감
  - `SavedRouteScreen`의 리스트 아이템 자체가 아니라 데이터 값
  - 특히 `placeId`, `routeName`, `address`, `category`, `transportMode`가 특정 저장 경로에서만 비정상 포맷으로 들어오는지 확인
