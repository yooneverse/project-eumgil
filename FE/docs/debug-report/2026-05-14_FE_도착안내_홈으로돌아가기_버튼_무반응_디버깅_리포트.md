# FE 도착 안내 홈으로 돌아가기 버튼 무반응 디버깅 리포트

> 작성일: 2026-05-14
> Work Lane: FE
> 상태: Investigating / FE 완화 조치 반영
> 범위: 도착 안내 화면에서 평가 시트가 닫힌 뒤 `홈으로 돌아가기` 버튼이 눌리지 않는 것처럼 보이는 FE 이슈의 원인 추적 기록

## 1. 현재 결론

- 현재까지 확보한 근거상 원인은 OS 입력, 디바이스 터치 드라이버, BE 응답 문제가 아니다.
- 사용자 로그 기준 터치는 앱 프로세스까지 정상 전달된다.
- 코드 기준 `홈으로 돌아가기`의 ViewModel 이벤트와 네비게이션 체인은 살아 있다.
- 따라서 가장 가능성이 높은 원인은 `ArrivalScreen` 내부 레이어/오버레이/하단 액션 배치 구조로 인해 `홈으로 돌아가기` 버튼의 실제 hit target 이 가려지거나, 닫히는 시트와 같은 레이어 문맥 안에서 눌림 처리가 안정적으로 보장되지 않는 FE 화면 구조 문제다.
- FE 쪽 1차 완화로 `홈으로 돌아가기`와 `새 경로 탐색` 버튼을 완료 본문 내부에서 분리해, 평가 시트가 닫힌 뒤에만 루트 레벨 하단 액션으로 따로 렌더링하도록 변경했다.
- 다만 본 문서 시점에는 사용자 실기기 재검증까지 끝나지 않았으므로, 본 결론은 `코드 근거 기반의 강한 가설 + FE 완화 적용 상태`로 기록한다.

## 2. 증상 정리

- 이슈 화면은 `도착 안내 화면`이다.
- 사용자가 명확히 지목한 무반응 CTA 는 `홈으로 돌아가기`다.
- 초기 분석에서는 `평가 바텀시트가 홈 버튼 위를 덮고 있다`는 가설이 가장 강했다.
- 이후 사용자 관찰에 따르면, 실제 런타임에서는 `시트가 닫힌 것처럼 보이는 상태에서도 홈으로 돌아가기만 안 눌린다`고 제보되었다.
- 따라서 단순히 `시트 visible 플래그가 true 라서 당연히 막힌다` 수준의 문제로는 설명이 완전히 끝나지 않는다.

## 3. 확보된 사실

### 3.1 런타임 로그 fact

- 사용자 제공 로그에는 반복적으로 아래 흐름이 나타난다.
  - `InputDispatcher` 가 앱 윈도우 `com.ssafy.e102.eumgil` 로 `action: 0x0`, `action: 0x1` 을 전달
  - `ViewPostIme pointer 0/1` 출력
- 이 로그는 "터치 자체가 앱에 도달하지 않는다"는 가설과 맞지 않는다.
- 같은 구간에서 치명적인 예외, 크래시, Activity 전환 실패 로그는 확인되지 않았다.
- 로그에 반복해서 보이는 `MapKwsVM RECORD_AUDIO 권한 없음 - KWS 재시작 스킵` 경고는 지도 KWS 재시작 관련 경고이며, 도착 화면의 홈 버튼 무반응을 직접 설명하는 증거는 아니다.

### 3.2 이벤트 체인 fact

- `ArrivalViewModel` 은 `ArrivalUiAction.HomeClicked` 를 받으면 바로 `ArrivalUiEvent.NavigateToMap` 을 emit 한다.
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/arrival/ArrivalViewModel.kt`
- `ArrivalRoute` 는 `uiEvent` 를 collect 해서 `NavigateToMap` 이 오면 `onNavigateToMap()` 을 호출한다.
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/arrival/ArrivalRoute.kt`
- `MainNavGraph` 는 도착 화면의 `onNavigateToMap` 을 `navController.navigateToArrivalHome(selectedPrimaryUserType)` 에 연결한다.
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/app/navigation/MainNavGraph.kt`
- 따라서 코드상 `홈으로 돌아가기` 버튼의 네비게이션 경로 자체가 끊겨 있지는 않다.

### 3.3 화면 구조 fact

- `ArrivalUiState.isEvaluationSheetVisible` 의 기본값은 `true` 다.
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/arrival/ArrivalContract.kt`
- `EvaluationSheetDismissed` 와 평가 제출 성공 시점에는 `isEvaluationSheetVisible = false` 로 전환된다.
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/arrival/ArrivalViewModel.kt`
- 기존 구조에서는 완료 화면 본문과 홈 CTA 가 같은 완료 콘텐츠 내부에 있고, 평가 시트와 scrim 이 그 위에 다시 올라오는 방식이었다.
- 이 구조에서는 시트가 열린 동안 하단 CTA 가 뒤 레이어에 놓여 실제 터치 타깃으로 접근되지 않을 가능성이 높다.

## 4. 원인 가설의 변화

### 4.1 1차 가설

- 가장 먼저 세운 가설은 `평가 바텀시트가 여전히 앞에 떠 있어서 홈 버튼을 막는다` 는 것이었다.
- 이 가설은 코드 구조와 잘 맞았다.
  - 완료 콘텐츠 먼저 렌더링
  - scrim 오버레이 추가
  - 평가 바텀시트 추가
- 다만 사용자가 `실제로는 시트가 닫혀 보인다`고 명확히 말했기 때문에, 이 설명만으로는 런타임 체감과 완전히 일치하지 않았다.

### 4.2 2차 가설

- 시트가 시각적으로 닫힌 뒤에도,
  - 닫히는 애니메이션 구간의 레이어
  - 같은 루트 내부의 오버레이 surface
  - 하단 액션이 완료 본문 내부에 묶여 있는 배치 구조
  중 하나가 `홈으로 돌아가기`의 실제 hit target 을 계속 불안정하게 만들고 있을 가능성이 더 높다고 판단했다.
- 즉 문제의 핵심은 `네비게이션 로직 불량`보다 `버튼이 존재하는 레이어 문맥`에 있다고 정리했다.

## 5. 적용한 FE 대응

### 5.1 구조 변경

- `홈으로 돌아가기`와 `새 경로 탐색` 버튼을 완료 본문 내부에서 제거했다.
- 별도 `ArrivalCompletionActions()` 컴포저블로 분리했다.
- 평가 시트가 닫힌 상태에서만 이 하단 액션을 루트 레벨에서 렌더링하도록 바꿨다.
- 루트 레벨 하단 액션에는 `Modifier.align(Alignment.BottomCenter).zIndex(1f)` 를 적용했다.
- 관련 파일
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/arrival/ArrivalScreen.kt`

### 5.2 의도

- 저장/평가 중에는 기존 시트 내부의 액션을 그대로 사용하게 둔다.
- 시트가 닫힌 뒤에 보여야 하는 `홈으로 돌아가기`와 `새 경로 탐색`은 더 이상 시트와 동일한 레이어 문맥 안에 두지 않는다.
- 이렇게 해서 "시트가 닫힌 것처럼 보이지만 실제 hit target 은 여전히 취약한 상태"를 구조적으로 회피하려는 목적이다.

## 7. 다음 확인 포인트

- 사용자 단말에서 `시트가 닫힌 뒤 홈으로 돌아가기 탭` 재검증
- 여전히 재현된다면 아래 순서로 FE 추가 확인
  - `HomeClicked` 진입 로그를 임시로 추가해 실제 onClick 진입 여부 확인
  - `AnimatedVisibility` 종료 타이밍과 하단 액션 hit target 겹침 재점검
  - `navigationBarsPadding`, 하단 inset, z-index 우선순위, 전체 화면 Box 내 겹침 영역 재확인
