# FE 주소검색 일반주택 북마크 저장실패 디버그 리포트

> 작성일: 2026-05-13
> Work Lane: FE
> 범위: 검색 fallback 도로명 주소(`EXTERNAL_ADDRESS`)의 북마크 저장 실패 분석과 FE 수정 기록
> 최종 수정일: 2026-05-13

## 1. 증상 정리

- 사용자는 검색 자체는 정상 동작하지만, 일반 주택처럼 건물명이 없는 도로명 주소 결과를 북마크 저장할 때 실패한다고 제보했다.
- 같은 검색 결과라도 `preview` 진입은 가능하지만, 북마크 저장 단계에서 실패 메시지가 보인다는 맥락으로 관찰됐다.
- 문제 범위는 내부 장소(`INTERNAL_PLACE`)가 아니라 검색 fallback으로 생성된 외부 주소 결과(`EXTERNAL_ADDRESS`)로 좁혀졌다.

## 2. 1차 가설과 반례

- 1차 가설:
  - 일반 주택 주소 결과는 건물명이 비어 있을 수 있고, 이 빈 이름이 그대로 `SearchResult.title`과 `BookmarkData.placeName`으로 흘러가 북마크 저장 요청에서 `name` 필드가 빠질 수 있다고 판단했다.
- 1차 조치:
  - `SearchDtoMapper`에서 외부 결과의 `name`이 blank면 `address`를 `displayTitle`로 승격하도록 보정했다.
- 반례:
  - 사용자 재확인에서 “또 실패”가 나왔고, 빈 이름 보정만으로는 실제 실패 원인이 제거되지 않았음이 확인됐다.

## 3. 2차 조사 근거

### 3.1 북마크 API 계약 재확인

- 문서:
  - `Docs/API/사용자_도메인/2026-04-12_북마크_API_명세.md`
- 핵심 규칙:
  - `placeId`가 없으면 외부 snapshot 북마크로 처리한다.
  - 외부 snapshot은 `name`, `point`가 필수다.
  - `providerPlaceId`, `providerCategory`, `address`는 선택값이다.
  - 현재 외부 snapshot `provider`는 `KAKAO`를 기준으로 설계되어 있다.

### 3.2 FE 데이터 흐름 재확인

- 주소 fallback 결과 생성:
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/data/repository/SearchRepository.kt`
  - fallback 결과는 source 식별용으로 `provider = ANDROID_GEOCODER_PROVIDER`를 가진다.
- 북마크 저장 payload 생성:
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/search/SearchViewModel.kt`
  - 기존 구현은 `SearchResult.provider`를 거의 그대로 북마크/preview 경로에 흘려보냈다.
- 결론:
  - 검색 source 식별자(`ANDROID_GEOCODER`)가 북마크 저장 계약의 external provider 값으로 그대로 새어 나가고 있었다.

## 4. 최종 원인

- 실제 원인은 “건물명이 없는 것” 자체가 아니라, 주소 fallback 검색 결과가 북마크 저장 시 `provider=ANDROID_GEOCODER`로 전달되던 점이다.
- 북마크 API 계약상 외부 snapshot 저장은 현재 `KAKAO` 기준으로 설계되어 있어, FE가 `ANDROID_GEOCODER`를 전달하면 서버에서 요청을 거절할 수 있다.
- 즉, 이번 이슈는 검색 source provider와 북마크 저장 provider 계약이 섞인 FE 매핑 문제다.

## 5. FE 수정 내용

### 5.1 표시명 보정 유지

- 파일:
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/data/remote/mapper/SearchDtoMapper.kt`
- 수정:
  - 외부 검색 결과의 `name`이 blank면 `address`를 `displayTitle`로 사용하도록 유지했다.
- 목적:
  - 건물명이 없는 일반 주택 주소도 검색 결과 타이틀, handoff, 북마크 표시명에서 비어 있지 않게 만든다.

### 5.2 주소 fallback 북마크 provider 정규화

- 파일:
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/search/SearchViewModel.kt`
- 수정:
  - 주소 fallback 결과(`provider == ANDROID_GEOCODER_PROVIDER`)는 저장용 provider를 `KAKAO`로 정규화한다.
  - 주소 fallback 결과는 저장용 `providerPlaceId`를 보내지 않는다.
  - 이 규칙을 아래 3경로에 공통 적용했다.
    - 검색 결과 `preview`
    - 일반 북마크 토글
    - 저시력 북마크 저장

## 6. 테스트 보강

- 파일:
  - `FE/app/src/test/java/com/ssafy/e102/eumgil/data/remote/mapper/SearchDtoMapperTest.kt`
  - `FE/app/src/test/java/com/ssafy/e102/eumgil/feature/search/SearchViewModelTest.kt`

### 6.1 mapper 회귀 테스트

- 외부 결과 `name`이 blank면 `title`이 `address`로 대체되는지 검증 추가

### 6.2 SearchViewModel 회귀 테스트

- 주소 fallback preview는 `detailType=EXTERNAL_ADDRESS`를 유지하면서 저장용 provider는 `KAKAO`로 정규화되는지 검증
- 주소 fallback 일반 북마크 저장 시 `BookmarkData.provider == "KAKAO"`이고 `providerPlaceId == null`인지 검증
- 주소 fallback 저시력 북마크 저장 시에도 동일한 payload 규칙을 사용하는지 검증

## 8. 현재 판단

- FE 기준 최종 원인은 “주소 fallback source provider를 북마크 저장 provider로 오용한 매핑 문제”로 정리했다.
- 이번 수정으로 일반 주택 주소 결과는
  - 표시명 비어 있음 문제
  - unsupported provider 전송 문제
  를 모두 FE에서 방어하도록 변경됐다.

## 10. Cross-Lane Handoff

- 현재 FE 변경만으로 우선 대응했다.
- 다만 FE 수정 후에도 동일 현상이 계속 재현되면 BE 확인이 필요하다.

### BE 확인 요청 사항

- `POST /bookmarks`가 아래 payload를 정상 external address snapshot으로 허용하는지 확인
  - `provider = "KAKAO"`
  - `providerPlaceId = null`
  - `name = 도로명 주소`
  - `point = {lat, lng}`
  - `address = 도로명 주소`
- 위 payload가 거절된다면 API 문서와 실제 validation 규칙 차이를 정리해 FE에 다시 전달 필요
