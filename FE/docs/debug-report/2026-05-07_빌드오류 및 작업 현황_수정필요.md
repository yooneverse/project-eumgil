## 2. 에러 원인 분석

`DestinationSelectionRepository.selectionRequests`의 emit 타입이 변경되었지만(`Flow<PlaceDestination?>` → `Flow<RouteSelectionRequest>`), 일부 consumer가 같이 수정되지 않았다. 또한 `SearchUiEvent.NavigateToResults`에 `editingTarget` 파라미터가 추가되었지만 테스트에는 반영되지 않았다.

### 2.1 selectionRequests 타입 변경 사유 (파악)

`DestinationSelectionRepository.kt:30-33`에 새 데이터 클래스 정의:

```kotlin
data class RouteSelectionRequest(
    val state: RouteSelectionState,
    val reason: RouteSelectionRequestReason,
)
```

기존에는 `selectionRequests: Flow<PlaceDestination?>`였으나, origin/destination/swap/clear 등 다양한 reason을 함께 전달하기 위해 `RouteSelectionRequest`로 확장된 것으로 추정된다.

### 2.2 NavigateToResults editingTarget 추가 사유 (파악)

`SearchContract.kt:83-86`에 `editingTarget: RouteEditingTarget` 필드 추가. 검색 결과 → 경로 설정으로 navigation 시 origin/destination 중 어디를 편집 중인지 전달 목적으로 추정.

## 3. 발견된 컴파일 에러 (3건)

### Error 1 — RouteSettingViewModel.kt:82

```text
Type mismatch: inferred type is RouteSelectionRequest but PlaceDestination? was expected
```

**위치:** `app/src/main/java/com/ssafy/e102/eumgil/feature/route/RouteSettingViewModel.kt:79-86`

**문제 코드:**
```kotlin
destinationSelectionRepository.selectionRequests.collectLatest { selectedDestination ->
    hasLoadedInitialDestination = true
    loadRouteShell(
        destinationResolution = resolveDestination(selectedDestination),  // ❌ 타입 불일치

}
```

**적용 가능한 fix:**
```kotlin
destinationSelectionRepository.selectionRequests.collectLatest { request ->
    hasLoadedInitialDestination = true
    loadRouteShell(
        destinationResolution = resolveDestination(request.state.selectedDestination),
        resetSelectedOption = true,
    )
}
```

### Error 2 — MapViewModel.kt:326

```text
Type mismatch: inferred type is RouteSelectionRequest but PlaceDestination was expected
```

**위치:** `app/src/main/java/com/ssafy/e102/eumgil/feature/map/MapViewModel.kt:318-332`

**문제 코드:**
```kotlin
private fun observeSelectionRequests() {
    viewModelScope.launch {
        destinationSelectionRepository.selectionRequests.collectLatest { destination ->
            if (clearSelectedFacilitySelection()) {
                renderSelectedFacilityState()
            }
            syncCameraToSelectedDestination(
                destination = destination,  // ❌ 타입 불일치
                incrementRequestId = true,
            )
            renderUiState()
        }
    }
}
```

**적용 가능한 fix:**
```kotlin
private fun observeSelectionRequests() {
    viewModelScope.launch {
        destinationSelectionRepository.selectionRequests.collectLatest { request ->
            if (clearSelectedFacilitySelection()) {
                renderSelectedFacilityState()
            }
            val destination = request.state.selectedDestination ?: return@collectLatest
            syncCameraToSelectedDestination(
                destination = destination,
                incrementRequestId = true,
            )
            renderUiState()
        }
    }
}
```

> **검토 필요:** `selectedDestination`이 `null`인 경우(예: `DESTINATION_CLEARED` reason) 카메라 동기화를 skip하는 처리이다. 원래 의도가 destination이 항상 있는 케이스만 처리하는 것이라면 이 fix가 맞고, 다른 reason(SWAPPED 등)에서도 별도 처리가 필요하다면 추가 로직이 필요할 수 있다.

## 10. 종합 권장 사항

### 10.1 데이터 클래스/시그니처 변경 시 grep 점검 습관

§1~7의 빌드 break는 **데이터 클래스 시그니처 변경 후 모든 consumer를 grep으로 점검하지 않은 것**이 원인이다. 향후 다음 패턴의 변경 시 grep 필수:

- `data class` 필드 추가/변경/제거
- `Flow<T>`의 emit 타입 변경
- `enum` 값 추가/변경/제거
- `sealed interface/class`의 variant 추가/제거

### 10.2 UI 표기와 영속 계층 정합성 점검

§8 같은 케이스(UI는 5장, 저장은 1장)를 방지하기 위해, **UI 제약(MAX_COUNT 등)과 영속 계층/API 제약이 일치하는지 PR 시점에 명시적으로 점검**.

### 10.3 시연 시나리오 회귀 점검 강화

§9 같은 navigation 잔존 이슈는 **단위 테스트로 잡기 어려운 사용자 동선 이슈**라, 박세홍 doc [§4 T17](FE/docs/sprint_backlog/2026-05-04_FE_박세홍_업무_정리.md#L75) "북마크/제보 시연 회귀 QA/QC"에서 명시적으로 점검 항목으로 다루는 것이 안전하다.
