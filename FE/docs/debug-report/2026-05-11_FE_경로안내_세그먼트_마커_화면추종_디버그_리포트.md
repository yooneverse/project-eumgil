# FE 경로안내 세그먼트 마커 화면추종 디버그 리포트
> 작성일: 2026-05-11  
> Work Lane: FE  
> 이슈: 경로안내 화면에서 세그먼트 시작점 마커가 생성되더라도 지도를 드래그하거나 카메라가 움직일 때 지도 좌표에 고정되지 않고 화면을 따라오는 것처럼 보이는 문제

## 1. 현재 증상

- 이전 FE 수정으로 세그먼트 시작점 마커 자체는 다시 보이기 시작했다.
- 하지만 사용자가 지도를 이동하면 마커가 지도 위 좌표에 붙어 있지 않고 화면 레이어에 떠 있는 것처럼 움직인다.
- 사용자 피드백 기준으로 현재 상태는 "미노출" 단계는 넘겼지만 "지도 좌표 고정"은 아직 해결되지 않았다.

## 2. 현재 FE 코드 기준 사실

### 2.1 세그먼트 시작점 좌표 fallback은 이미 들어갔다

- `NavigationViewModel`은 세그먼트 polyline이 비어도 `segmentStartCoordinate`를 계산해서 UI state에 내려준다.
- 관련 파일:
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/navigation/NavigationContract.kt`
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/navigation/NavigationViewModel.kt`
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/MapViewportOverlay.kt`

즉 "마커 후보 좌표가 없다" 단계는 일부 케이스에서 이미 넘긴 상태다.

### 2.2 세그먼트 junction를 projected marker에서 native overlay marker로 빼는 시도를 했다

- `KakaoMapViewportBindings.kt`에서 `MapViewportPointKind.SEGMENT_JUNCTION`은 더 이상 projected marker로 내려가지 않게 바꿨다.
- 대신 `createKakaoOverlayMarkerRenderStates(...)`로 native overlay marker state를 따로 만든다.
- 관련 파일:
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/KakaoMapViewportBindings.kt`

### 2.3 Kakao label layer에 junction marker를 같이 그리도록 시도했다

- `KakaoMapViewport.kt`의 `syncMarkers(...)` 안에서 facility marker와 함께 overlay marker도 같은 `LabelManager`로 다시 그리도록 바꿨다.
- `KakaoOverlayMarkerStyleCache`를 추가해서 junction marker용 원형 비트맵을 생성하도록 했다.
- 관련 파일:
  - `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/map/component/KakaoMapViewport.kt`

## 3. 왜 아직 미해결이라고 보는가

현재 사용자 체감은 "아직도 따라 움직인다"이므로, 코드 변경만으로는 해결 여부를 증명하지 못했다. 가능한 경우는 아래 셋 중 하나다.

1. 사용자가 보고 있는 마커가 여전히 projected Compose overlay다.
   - 이 경우 native label path가 실제로는 안 붙었거나, 다른 projected path가 계속 살아 있다.

2. native label path는 붙었지만 화면에서 보이는 것은 다른 debug/fallback overlay다.
   - 예: 같은 좌표를 두 경로가 동시에 그리고 있는데, 눈에 띄는 쪽이 screen-space overlay일 수 있다.

3. marker는 실제로 지도 좌표에 붙어 있지만, 카메라 fit / focus 이동 / projection 보정 때문에 사용자는 "화면을 따라온다"고 느끼고 있다.
   - 이 경우 문제는 marker 자체가 아니라 카메라 이동 입력 또는 overlay 구성 순서일 수 있다.

즉 지금 단계의 핵심 부족분은 "사용자가 본 그 점이 어느 렌더 경로에서 나왔는지"를 증명하는 로그가 없다는 점이다.

## 4. 지금까지 확인한 한계

### 4.1 projected marker 디버그 로그만으로는 부족하다

- 기존 `SegmentMarkerTrace[Projection]`, `SegmentMarkerTrace[Compose]`는 projected marker 기준 관측이다.
- native label marker는 Kakao 내부 label layer에서 그려지므로, 현재 로그만으로는 "지금 보이는 점이 native인지 projected인지" 바로 구분되지 않는다.

### 4.2 테스트는 코드 컴파일 단계까지만 신뢰 가능하다

- `:app:compileDebugUnitTestKotlin`은 현재 변경 기준으로 통과했다.
- 다만 `:app:testDebugUnitTest --tests "com.ssafy.e102.eumgil.feature.map.component.KakaoMapViewportBindingsTest"`는 이번 변경이 아니라 기존 테스트 컴파일 이슈인 `RouteRepositoryTest.kt:79 Cannot find a parameter with this name: routeMappingDispatcher`에서 멈춘다.
- 따라서 현재는 "코드가 컴파일되고 런타임에서 어떤 경로가 보이는지"를 중심으로 검증해야 한다.

## 5. 다음 디버깅 목표

다음 단계의 목표는 단순하다.

- "지금 화면에서 보이는 세그먼트 마커가 projected Compose overlay인지, Kakao native label인지"를 구분한다.
- "사용자가 말한 따라 움직임이 marker 문제인지, 카메라 문제인지"를 분리한다.

## 6. 다음 실행 계획

### Step 1. 렌더 경로 식별 로그를 추가한다

상태: 완료

- projected Compose overlay 경로
  - `KakaoMapViewport.kt`
  - `SegmentMarkerTrace[RenderPath] renderPath=projected ...` 형식으로 로그가 찍히도록 반영했다.
- native Kakao label 경로
  - `KakaoMapViewport.kt`
  - `SegmentMarkerTrace[RenderPath] renderPath=native-label ...` 형식으로 로그가 찍히도록 반영했다.
- 요약 포맷 helper
  - `KakaoMapViewportBindings.kt`
  - projected/native 경로를 서로 다른 summary 함수로 분리했다.
- 관련 테스트
  - `KakaoMapViewportBindingsTest.kt`
  - 두 경로 summary 문자열을 고정하는 단위 테스트를 추가했다.

다음 재현 때는 Logcat에서 아래 문자열 하나만 필터링하면 된다.

- `SegmentMarkerTrace[RenderPath]`

재현 시 판단 기준:

- `renderPath=projected segmentCount>0`
  - 아직 screen-space overlay 경로가 살아 있다.
- `renderPath=native-label segmentCount>0`만 보인다
  - 지도 좌표 기반 label 경로는 살아 있다.
- 두 경로가 동시에 보인다
  - 중복 렌더 또는 fallback/debug overlay 겹침 가능성이 높다.

### Step 2. junction marker만 강한 디버그 스타일로 분리한다

다음 작업자가 바로 시작할 첫 작업이다.

- native junction marker 크기만 임시로 1.5배~2배 키운다.
- fill/stroke 대비를 더 강하게 만든다.
- 가능하면 marker id 또는 sequence를 색/크기 차이로 구분한다.

목표:

- 사용자가 보고 있는 점이 native label 하나인지 시각적으로 바로 식별
- "고정은 돼 있는데 너무 작거나 다른 점과 겹쳐서 따라오는 것처럼 보이는 경우" 배제

### Step 3. projected와 native를 동시에 켜고 한쪽씩 끈다

다음 작업자가 Step 2 다음으로 바로 할 일이다.

1. projected만 켠다
2. native만 켠다
3. 둘 다 켠다

관찰 포인트:
- 어떤 모드에서만 "따라 움직임"이 재현되는지
- 어떤 모드에서만 사용자가 마커를 인지하는지

이 실험이 끝나면 문제는 거의 확정된다.

### Step 4. 카메라 이동 원인과 marker 이동 원인을 분리한다

다음 로그를 같이 본다.

- `SegmentMarkerTrace[NavigationViewModel]`
- `SegmentMarkerTrace[MapViewportOverlay]`
- `SegmentMarkerTrace[KakaoProjectedMarkers]`
- `SegmentMarkerTrace[Projection]`
- 새로 넣을 `renderPath=native-label`
- 카메라 fit/request id 로그

판단 기준:
- marker 경로는 고정인데 카메라 request가 계속 바뀌면 카메라 문제
- 카메라는 안정적인데 projected screen 좌표가 계속 새로 찍히면 overlay 경로 문제
- native label만 살아 있는데도 사용자가 따라온다고 하면 시각적 기준점이 marker가 아닐 가능성 검토

## 7. 현재 우선순위

1. Logcat에서 `SegmentMarkerTrace[RenderPath]`만 필터링한 실제 재현 로그 확보
2. native junction marker 디버그 스타일 확대
3. projected-only / native-only 분리 실험
4. 그 다음에야 카메라 fit 또는 overlay 구조 수정 여부 결정

지금은 추가 구조 수정부터 들어가면 다시 추측성 패치가 된다. 먼저 "무슨 경로가 보이는가"를 증명해야 한다.
