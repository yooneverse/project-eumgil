# FE 길찾기 초기 로딩 지연 구조 분석 디버그 리포트

> 문서일: 2026-05-11
> Work Lane: FE
> 상태: 코드 기준 원인 분석 완료, FE 우선 개선안 정리

## 1. 문제 정의

- 길찾기 화면에서 첫 경로 카드가 보이기까지 체감 시간이 길다.
- 특히 대중교통이 기본 모드로 선택되는 케이스에서 대기 시간이 더 길게 느껴진다.
- 이 이슈가 단순 백엔드 응답 지연인지, FE 호출 순서와 상태 관리 구조까지 포함한 문제인지 분리해서 볼 필요가 있다.

## 2. 이번 분석의 결론

- 현재 구현은 첫 화면 로딩을 빠르게 보여주는 구조가 아니라, `도보 결과를 먼저 기다린 뒤` 필요하면 `대중교통 결과를 추가로 기다리는` 직렬 구조다.
- 따라서 대중교통이 기본 모드로 결정되는 케이스에서는 첫 usable 상태까지의 체감 시간이 `walk API + transit API + 응답 파싱`으로 길어진다.
- 현재 위치가 아직 없을 때는 fallback origin으로 한 번 조회하고, 이후 GPS가 들어오면 다시 조회할 수 있어 초기 진입 체감이 더 나빠질 수 있다.
- 경로 API 자체도 route 전용 timeout이 `connect 5초 / read 7초`로 설정돼 있어, transit timeout이 발생하면 FE에서는 꽤 오래 기다린 뒤 실패를 보여주게 된다.
- 결론적으로 이 이슈는 `백엔드만의 문제`가 아니라, FE의 `초기 로딩 순서`, `재조회 조건`, `캐시 키 구조`, `파싱 비용`이 함께 만든 체감 지연 문제다.

## 3. 현재 구현 흐름

### 3.1 화면 진입 시 바로 경로 재조회가 시작됨

- `RouteSettingViewModel`은 `init`에서 `refreshLatestLocation()`을 호출한 뒤 곧바로 `requestRouteReload(force = true)`를 수행한다.
- 동시에 `RouteSettingEntryRoute`는 `DisposableEffect`에서 `startLocationUpdates()`를 시작한다.

근거:

- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/route/RouteSettingViewModel.kt`
  - `init { ... requestRouteReload(...) }`
  - `startLocationUpdates()`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/route/RouteSettingEntryRoute.kt`
  - `DisposableEffect(viewModel) { viewModel.startLocationUpdates() ... }`

해석:

- 화면 진입 직후 경로 조회와 위치 확보가 거의 동시에 시작된다.
- 사용자가 수동 출발지를 고르지 않은 상태라면, 위치 스냅샷 유무에 따라 초기 조회와 후속 재조회가 모두 발생할 수 있다.

### 3.2 초기 로딩은 항상 WALK를 먼저 조회함

- `loadRouteShell()`은 먼저 `fetchSearchData(mode = WALK)`를 호출한다.
- 그 다음 `determineDefaultTravelMode(walkSearchData)`로 SAFE 도보 거리 기준을 계산한다.
- SAFE 도보 거리가 `750m`를 초과하면 그제야 `fetchSearchData(mode = TRANSIT)`를 추가 호출한다.

근거:

- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/route/RouteSettingViewModel.kt`
  - `loadRouteShell(...)`
  - `fetchSearchData(mode = RouteTravelMode.WALK, ...)`
  - `determineDefaultTravelMode(walkSearchData)`
  - `fetchSearchData(mode = RouteTravelMode.TRANSIT, ...)`
  - `WALK_TO_TRANSIT_THRESHOLD_METERS = 750`

해석:

- 대중교통이 기본 모드인 케이스도 transit를 바로 치지 않는다.
- 반드시 walk를 먼저 받아야만 transit가 필요한지 판단할 수 있다.
- 즉 초기 대기 시간은 병렬이 아니라 직렬이다.

### 3.3 경로 API 캐시는 있지만 현재 위치에는 잘 안 맞을 수 있음

- `DefaultRouteRepository.getSearchData()`는 먼저 `RouteLocalDataSource.getCachedSearchData(query)`를 확인한다.
- 캐시 키는 origin, destination, requestedOptions 조합이며, 좌표는 소수점 6자리까지 문자열로 포함된다.

근거:

- `FE/app/src/main/java/com/ssafy/e102/eumgil/data/repository/RouteRepository.kt`
  - `localDataSource.getCachedSearchData(query)`
  - `localDataSource.updateCachedSearchData(query = query, searchData = searchData)`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/data/local/datasource/RouteLocalDataSource.kt`
  - `RouteSearchQuery.cacheKey()`
  - `RouteWaypoint.cacheKey()`
  - `String.format(Locale.US, "%.6f", this)`

해석:

- 정적인 출발지/도착지 조합에는 캐시가 유효하다.
- 하지만 현재 위치 기반 origin은 좌표가 조금만 달라도 완전히 다른 키가 된다.
- 따라서 실제 사용자 체감이 가장 중요한 “현재 위치 출발” 케이스에서는 캐시 적중률이 낮을 수 있다.

### 3.4 위치가 들어오면 자동 재조회가 가능함

- `latestLocation.collectLatest`에서 origin이 자동 위치 기반일 때, 이전 좌표와 현재 좌표를 비교해 의미 있는 변화면 다시 `requestRouteReload()`를 호출한다.
- 재조회 임계치는 `25m` 이상 또는 GPS accuracy보다 큰 변화다.

근거:

- `FE/app/src/main/java/com/ssafy/e102/eumgil/feature/route/RouteSettingViewModel.kt`
  - `observeLocationUpdates()`
  - `shouldReloadForAutomaticOriginUpdate(...)`
  - `AUTOMATIC_ORIGIN_RELOAD_THRESHOLD_METERS = 25.0`

해석:

- 무한 재조회까지는 아니고 억제 장치가 있다.
- 다만 초기 진입 시점에 location snapshot이 늦게 들어오면 “fallback origin으로 1회 조회 -> GPS 확보 후 다시 조회” 흐름은 여전히 가능하다.

### 3.5 현재 위치 매니저는 즉시 GPS fix를 보장하지 않음

- `AndroidCurrentLocationManager.refreshLatestLocation()`은 `lastKnownLocation`만 읽고, 2분 이내의 fresh location만 인정한다.
- live update는 `requestLocationUpdates`로 시작되며 간격은 2초, 최소 이동 거리는 5m다.

근거:

- `FE/app/src/main/java/com/ssafy/e102/eumgil/core/location/AndroidCurrentLocationManager.kt`
  - `refreshLatestLocation()`
  - `startLocationUpdates()`
  - `requestLocationUpdates(...)`
  - `LOCATION_UPDATE_INTERVAL_MILLIS = 2_000L`
  - `LOCATION_UPDATE_MIN_DISTANCE_METERS = 5f`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/core/location/LocationModels.kt`
  - `isFreshCurrentLocation(...)`
  - `MAX_CURRENT_LOCATION_AGE_MILLIS = 2 * 60 * 1_000L`

해석:

- 앱이 화면에 들어왔다고 해서 즉시 신뢰 가능한 현재 위치를 얻는 구조는 아니다.
- 최근 위치가 없으면 첫 경로 로딩은 fallback origin에 의존할 수 있고, 그 뒤 실제 위치가 들어오며 재조회가 붙는다.

## 4. 체감 지연이 커지는 주요 원인

### 4.1 가장 큰 원인: WALK -> TRANSIT 직렬 호출

- 현재 구조는 transit 기본 케이스도 walk 응답이 먼저 와야 한다.
- 따라서 첫 usable 결과를 보여주는 시점이 transit API 단독 대기보다 더 늦어진다.

실제 영향:

- walk가 1.5초, transit가 2초 걸리면 사용자는 약 3.5초 뒤에야 최종 기본 모드를 본다.
- transit timeout이면 walk 성공 후에도 추가로 timeout까지 기다린 뒤 실패 상태를 본다.

### 4.2 timeout 상한이 길다

- route API timeout은 기본 10초가 아니라 route 전용으로 `connect 5초 / read 7초`다.
- transit timeout이 발생하면 사용자는 의미 있는 결과 없이 꽤 오래 기다린 뒤 실패 화면을 보게 된다.

근거:

- `FE/app/src/main/java/com/ssafy/e102/eumgil/app/AppContainer.kt`
  - `ROUTE_CONNECT_TIMEOUT_MILLIS = 5_000`
  - `ROUTE_READ_TIMEOUT_MILLIS = 7_000`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/data/remote/HttpJsonClient.kt`
  - `connectTimeout = timeoutConfig.connectTimeoutMillis`
  - `readTimeout = timeoutConfig.readTimeoutMillis`

### 4.3 응답 이후 파싱 비용도 적지 않음

- route 응답은 단순 summary만 만들지 않는다.
- route, leg, step, segment 단위로 geometry를 파싱하고, preview polyline도 별도로 조립한다.
- `RouteGeometryParser`는 `LINESTRING`을 split하고 모든 좌표를 순회해 `RoutePolyline`으로 만든다.

근거:

- `FE/app/src/main/java/com/ssafy/e102/eumgil/data/route/RouteSearchDtoMapper.kt`
  - `RouteSearchResponseDto.toDomain(...)`
  - `RouteDto.toDomain(...)`
  - `RouteLegDto.toDomain(...)`
  - `RouteStepDto.toDomain(...)`
  - `RouteSegmentDto.toLegacyDomain(...)`
  - `toPreviewPolyline()`
  - `toStepPreviewPolyline()`
- `FE/app/src/main/java/com/ssafy/e102/eumgil/data/route/RouteGeometryParser.kt`
  - `DefaultRouteGeometryParser.parse(...)`
  - `rawCoordinates = coordinatePayload.split(",")`
  - `RoutePolyline(points = coordinates)`

해석:

- 백엔드 응답이 빠르더라도 FE가 응답을 도메인 모델과 지도 preview용 데이터로 변환하는 데 추가 비용이 든다.
- 응답 payload가 큰 route일수록 비용이 커질 수 있다.

### 4.4 현재 위치 기반 캐시 효율이 낮을 수 있음

- 캐시가 메모리 기반이고 키가 좌표에 민감하다.
- 현재 위치 출발은 좌표가 약간만 달라도 다른 query로 처리된다.

해석:

- “같은 장소에서 다시 열었는데 왜 또 느리지?”라는 체감이 생길 수 있다.
- 정적 목적지와 달리 동적 origin에서는 캐시 재사용 폭이 좁다.

## 5. 지금 구조에서 우선순위 높은 FE 개선안

### 5.1 1순위: 초기 로딩을 직렬에서 병렬 또는 단계적 노출로 변경

권장 방향:

- `WALK`와 `TRANSIT`를 동시에 호출하고, 먼저 도착한 결과부터 화면에 붙인다.
- 또는 `WALK`를 먼저 보여주되 `TRANSIT`은 백그라운드로 로드하고, 첫 화면 표시를 transit 완료까지 막지 않는다.

기대 효과:

- 첫 usable 결과 노출 시간이 줄어든다.
- transit가 느려도 화면 전체가 오래 잠기지 않는다.

주의:

- 현재는 기본 모드 결정이 walk 거리 `750m` 기준에 묶여 있으므로, 병렬 호출 시 모드 결정 정책도 함께 정리해야 한다.

### 5.2 2순위: 위치 대기용 짧은 grace window 추가

권장 방향:

- 수동 origin이 없고 latest location이 아직 null이면, 아주 짧은 시간 동안은 fallback origin으로 바로 route search를 보내지 말고 `현재 위치 확인 중`만 보여준다.
- 예: 300ms~700ms 내 위치가 들어오면 실제 위치로 첫 조회를 바로 시작.

기대 효과:

- 초기 fallback 조회 후 GPS 조회로 다시 갈아끼우는 이중 로딩 빈도를 줄일 수 있다.

### 5.3 3순위: 이전 성공 결과 유지 + 상단 재조회 상태만 갱신

권장 방향:

- 위치 변동이나 모드 전환으로 재조회가 시작될 때 전체 route shell을 비우지 않는다.
- 직전 성공 결과를 유지한 채 상단에 `새 경로 확인 중` 상태만 노출한다.

기대 효과:

- 실제 네트워크 시간이 줄지 않아도 사용자 체감은 크게 개선된다.
- 지도와 카드가 사라졌다가 다시 나오는 공백 시간을 줄일 수 있다.

### 5.4 4순위: route parsing을 background CPU 작업으로 분리

권장 방향:

- `response.toDomain(...)`과 geometry parsing을 repository 또는 mapper 계층에서 `Dispatchers.Default`로 넘긴다.

기대 효과:

- 네트워크는 끝났는데 UI가 늦게 붙는 문제를 줄일 수 있다.

### 5.5 5순위: 현재 위치 origin용 캐시 키 완화 검토

권장 방향:

- 현재 위치 origin에 한해서는 좌표를 더 거칠게 normalize하거나 grid key를 따로 쓰는 방식 검토.

기대 효과:

- 재진입, 근거리 이동, 동일 장소 재탐색에서 캐시 적중률이 올라갈 수 있다.

주의:

- 너무 거칠면 “이미 다른 지점인데 예전 경로가 재사용되는” 부작용이 생길 수 있다.
- 이 부분은 UX 기대치와 정확도 기준을 같이 정해야 한다.

## 6. FE 관점에서 가장 현실적인 정리

- 지금 느린 이유를 한 줄로 요약하면 `대중교통 기본 화면이 사실상 walk 선행 조회에 종속돼 있고, 현재 위치 확보/재조회/파싱 비용이 그 뒤에 덧붙는 구조`다.
- 따라서 첫 개선은 백엔드보다 먼저 FE의 초기 로딩 순서를 바꾸는 쪽이 효과가 크다.
- 특히 `병렬 조회 또는 단계적 노출`과 `이전 성공 결과 유지`는 FE만으로도 체감 개선 폭이 큰 편이다.

## 7. 다음 작업 제안

1. 초기 route loading 구조를 `직렬 -> 병렬 또는 단계적 노출`로 전환한다.
2. 위치 미확보 시 짧은 grace window를 도입해 fallback origin 선조회 빈도를 줄인다.
3. 재조회 시 전체 shell reset 대신 stale-while-revalidate 형태로 바꾼다.
4. route response domain mapping을 background thread로 옮긴다.
5. 이후에도 느리면 route API별 실제 duration log를 수집해 FE/BE 비중을 다시 분리한다.

## 8. 백엔드 개발자에게 전달할 작업

- `/routes/search/walk`와 `/routes/search/transit`를 따로 기다리지 않도록, 두 모드의 요약 응답을 한 번에 주는 통합 API가 가능하면 초기 체감 속도를 크게 줄일 수 있다.
- 대중교통이 느릴 때 전체 상세 대신 카드 렌더에 필요한 최소 요약만 먼저 주고 상세를 후속 조회하는 API 분리도 검토 가치가 있다.
