# FE 북마크 장소 최근목적지 미노출 디버깅 리포트
> 작성일: 2026-05-12
> Work Lane: FE
> 이슈: 북마크에 저장된 장소를 눌러 지도/길찾기로 진입해도 지도 홈의 최근 목적지 목록에 바로 보이지 않는 문제

## 1. 증상 요약

- 사용자 제보 기준 증상은 `북마크 장소 선택 -> 지도 홈 진입` 이후 최근 목적지 시트에 방금 선택한 장소가 보이지 않는 것이다.
- 기존 코드상 검색 결과에서 목적지를 선택한 경우에는 최근 목적지 저장이 있었지만, 북마크 장소 선택 경로에는 같은 저장 호출이 빠져 있었다.
- 2026-05-12 현재 FE 수정으로 북마크 경로에도 저장 호출은 연결했지만, 사용자 체감상 여전히 "안 보인다"는 피드백이 있어 저장 이후 표시 단계까지 다시 분리해서 봐야 한다.

## 4. 현재까지 확인된 사실

### 4.1 저장 호출 자체는 이제 북마크 경로에도 존재한다

- 코드상 누락되어 있던 저장 호출은 보완됐다.
- 저장소 하위 구현도 `placeId` 기반으로 정상 dedupe/저장을 수행한다.
- 따라서 "북마크 경로는 아예 저장을 안 한다"는 1차 원인은 해소됐다.

### 4.2 지도 홈의 최근 목적지 목록은 반응형 구독이 아니라 스냅샷 재조회다

- `MapViewModel`은 최근 목적지를 Flow로 구독하지 않는다.
- `refreshRecentDestinations()`가 호출되는 시점에만 목록이 갱신된다.
- 즉 저장이 조금 늦거나, 지도 홈이 재조회 없이 기존 상태를 유지하면 사용자는 방금 저장한 항목을 못 볼 수 있다.

### 4.3 최근 목적지가 저장돼도 시트가 가려질 수 있다

- 지도 홈 시트는 `selectedMapPinCoordinate == null`일 때만 보인다.
- 시설 상세 시트가 살아 있으면 최근 목적지 시트는 숨겨진다.
- 따라서 데이터는 들어갔더라도 화면에서는 "안 뜬다"로 보일 수 있다.

## 5. 남아 있는 유력 가설

### 5.1 저장과 화면 이동의 비동기 순서 경쟁

- `SavedRouteViewModel.handoffPlace(...)`는 `persistRecentDestination(destination)`와 `emitUiEvent(event)`를 모두 `viewModelScope.launch` 기반 비동기로 처리한다.
- 이 구조에서는 최근 목적지 저장이 끝나기 전에 `NavigateToMap` 또는 `NavigateToRouteSetting` 이벤트가 먼저 처리될 수 있다.
- 이후 지도 홈이 `onRouteStarted()`에서 최근 목적지를 재조회해도, 그 시점에는 아직 DataStore 저장이 끝나지 않았을 가능성이 있다.
- 현재 지도 홈 목록이 반응형 구독이 아니므로, 이 타이밍을 놓치면 이번 진입에서는 안 보이고 다음 재진입 때만 보일 수 있다.

### 5.2 지도 홈 재조회 트리거가 실제 사용자 동선에서 항상 보장되지 않을 가능성

- 최근 목적지 갱신은 사실상 `init`과 `onRouteStarted()`에 묶여 있다.
- 북마크 -> 지도 전환 시 `MapRoute`의 라이프사이클이 기대한 순서로 다시 시작되지 않으면 재조회가 누락될 수 있다.
- 특히 activity-scoped `MapViewModel` 재사용 구조에서는 "화면 전환은 됐는데 목록 새로고침은 안 된" 상태가 생길 여지가 있다.

### 5.3 저장은 되었지만 선택/상세 상태 때문에 최근 목적지 시트가 숨겨질 가능성

- 북마크 장소를 handoff하면 동시에 `destinationSelectionRepository.updateSelectedDestination(destination)`도 호출한다.
- 이 선택 상태가 지도 홈에서 destination preview 또는 선택 상태를 유지하게 만들면, 최근 목적지 시트 노출 조건을 막을 수 있다.
- 즉 사용자 입장에서는 "최근 목적지 저장 안 됨"처럼 보이지만 실제로는 "선택 상태가 시트를 가림"일 수 있다.

## 11. 2026-05-12 viewport 기반 추가 수정

### 11.1 문제 재정의

- 최근 목적지는 지도 홈의 메인 컨텐츠인데, 기존 구현은 `selectedMapPinCoordinate != null`이면 무조건 최근 목적지 시트를 숨겼다.
- 이 방식은 핀이 이미 화면 밖으로 이탈했더라도 선택 상태만 남아 있으면 recent를 계속 가린다.
- 사용자 기대에 맞는 규칙은 `핀 선택 상태 존재`가 아니라 `핀이 현재 viewport 안에서 실제로 보이는가`다.

### 11.2 이번 수정 방향

- Kakao 지도에서 카메라 이동이 끝날 때, 선택 핀 좌표를 실제 screen point로 투영한다.
- 투영된 screen point가 현재 `MapView`의 width/height 범위 안에 있으면 visible, 아니면 offscreen으로 판단한다.
- offscreen이면 `selectedMapPinCoordinate`, `destinationPreview`, `mapTapDetail` 등 map pin 관련 선택 상태를 정리한다.
- recent 시트 계산식 자체는 유지하되, offscreen 상태에서는 선택 핀 값이 해제되므로 홈 메인 컨텐츠가 다시 드러나게 한다.

### 11.3 적용한 FE 변경

- `feature/map/component/KakaoMapViewportBindings.kt`
  - `resolveSelectedMapPinViewportVisibility(...)` 추가
  - Kakao 투영 screen point와 viewport 크기를 받아 핀 가시 여부를 계산
- `feature/map/component/KakaoMapViewport.kt`
  - `setOnCameraMoveEndListener`에서 선택 핀의 실제 viewport 가시 여부를 계산
  - 카메라 이동 종료 콜백에 `selectedMapPinVisibleInViewport` 값을 함께 전달
