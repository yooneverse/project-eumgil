# FE 경로안내 세그먼트 시작점 마커 미노출 디버그 리포트

> 작성일: 2026-05-11  
> Work Lane: FE  
> 이슈: 길찾기/내비게이션 지도에서 세그먼트 시작점 원형 마커가 실제 화면에 전혀 보이지 않는 문제

## 1. 증상 요약

- 사용자가 확인한 실제 증상은 "세그먼트 마커가 하나도 안 뜬다"이다.
- 대상은 길안내/내비게이션 지도이며, 기존 출발지/도착지/현재 위치 마커와 활성/포커스 구간 로직은 별도로 계속 동작한다.
- 즉 문제 범위는 `SEGMENT_JUNCTION` 계열 오버레이에 집중된다.

## 2. 이번 이슈에서 이미 적용한 FE 변경

- 세그먼트 마커 좌표를 대표 중간점이 아니라 각 세그먼트 `polyline.firstOrNull()`로 변경했다.
- 첫 세그먼트는 출발지 핀 중복을 피하기 위해 숨기도록 유지했다.
- 도보/대중교통 세그먼트에 서로 다른 tone과 색을 부여하도록 `SEGMENT_JUNCTION` 렌더링 경로를 확장했다.
- fallback overlay와 Kakao projected marker가 같은 색 규칙을 사용하도록 맞췄다.
- 이후 실제 화면에서 아무 것도 뜨지 않는 피드백을 받고, `segment.isRenderable` 필터를 제거해 `polyline`이 1점만 있어도 마커를 만들도록 완화했다.
- 추가로 Kakao projected marker가 초기 `toScreenPoint()` 실패로 빈 overlay 리스트가 되는 경우를 의심해, projected marker는 존재하지만 screen point 투영 결과가 0개이면 최대 6프레임까지 재투영을 재시도하도록 FE 로직을 보강했다.

## 3. 현재 코드 기준 확인된 사실

### 3.1 데이터/렌더링 경로

- `NavigationViewModel`은 `routeSegments`와 `travelKind`를 계산한다.
- `MapViewportOverlay.kt`의 `createNavigationViewportOverlayState()`는 `routeSegments`를 `SEGMENT_JUNCTION` 점 오버레이로 바꾼다.
- `KakaoMapViewportBindings.kt`는 이 점 오버레이를 projected marker로 변환한다.
- `KakaoMapViewport.kt`는 `ROUTE_SEGMENT_JUNCTION`를 실제 원형 점으로 그린다.
- `MapViewportOverlayBackdrop.kt`도 fallback 경로에서 동일한 점을 그린다.

### 3.2 단위 테스트 기준 확인된 사실

- 오버레이 생성 테스트는 "첫 세그먼트 제외", "시작점 좌표 사용", "walk/transit 색 분기"를 커버하도록 갱신했다.
- 이후 "1개 좌표만 있는 세그먼트도 시작점 마커를 만든다"는 테스트를 추가했다.
- projected marker 변환 테스트에는 "projection 결과가 0개면 retry 필요", "하나라도 투영되면 retry 중단" 시나리오를 추가했다.
- 즉 현재 단위 테스트는 "오버레이 모델을 어떻게 만들어야 하는가"와 "retry 판단을 어떻게 해야 하는가"는 일부 검증하지만, "실제 런타임 지도에 왜 안 보이는가"까지는 증명하지 못한다.

### 3.3 현재 워크트리 기준 주의점

- 같은 map component 영역에 이번 이슈와 직접 무관한 renderer/session 관련 워크트리 변경도 함께 존재한다.
- 따라서 다음 수정은 "세그먼트 마커 디버깅용 임시 로그/수정"과 다른 map 이슈 변경을 섞지 않도록 분리할 필요가 있다.

## 4. 이번 디버깅에서 얻은 핵심 판단

### 4.1 1차 가설과 한계

- 1차 가설은 "실제 세그먼트 polyline이 1점만 들어와서 `isRenderable` 필터에 모두 걸린다"였다.
- 이 가설은 코드상 충분히 가능했고, 그래서 필터를 제거해 `firstOrNull()`만 있으면 마커를 만들도록 바꿨다.
- 하지만 그 뒤에도 사용자 재현에서는 "여전히 하나도 안 뜬다"는 결과가 나왔다.
- 따라서 `isRenderable` 필터는 일부 원인이었을 수 있어도, 현재 미노출의 단일 원인이라고 결론내릴 수는 없다.

### 4.2 2차 가설과 이번 추가 시도

- 2차 가설은 "Kakao map ready 직후 `createKakaoProjectedMarkerRenderStates()`까지는 마커가 살아 있지만, 초기 `toScreenPoint()`가 모두 `null`을 반환해 `projectedMarkerOverlays`가 빈 리스트로 굳는다"였다.
- 이 가설에 따라 projected marker 목록과 실제 screen point 투영 결과를 분리하는 helper를 추가했고, "projected marker는 존재하지만 overlay는 0개"인 경우 다음 animation frame에서 다시 투영하도록 보강했다.
- 이 변경은 `KakaoMapViewport`의 projected marker 경로에만 국한했고, 기존 출발지/도착지/현재 위치/시설 마커 흐름은 직접 건드리지 않았다.
- 하지만 사용자 재확인 결과는 여전히 "실제 화면에 마커가 안 보인다"였다.
- 따라서 초기 projection miss는 가능한 원인이지만, 현재 증상 전체를 설명하는 단일 원인이라고 볼 수 없다.

### 4.3 지금 더 의심해야 하는 지점

- 실제 런타임에서 `routeSegments` 자체가 비어 있거나, polyline이 생각보다 더 빈번하게 비어 있을 수 있다.
- `MapViewportOverlayState.points`에는 마커가 들어가지만, Kakao projected marker 변환 이후 화면 좌표 계산이나 가시 영역 판단에서 누락될 수 있다.
- retry 로직이 들어갔어도 실제 런타임에서는 `projectedMarkers.isNotEmpty() && overlays.isEmpty()` 조건 자체가 성립하지 않거나, retry 이전에 다른 경로에서 state가 덮일 수 있다.
- 마커는 생성되지만 다른 레이어에 가리거나, 기대보다 작은 점으로 그려져 체감상 "없는 것처럼" 보일 수 있다.
- focused/active mode의 projection 제외 규칙은 카메라 계산용이라 직접적인 미노출 원인일 가능성은 낮지만, 실제 오버레이 상태와 함께 런타임 증거로 다시 확인해야 한다.

## 5. 아직 없는 증거

- 실제 사용자 재현 순간의 `routeSegments.size`
- 세그먼트별 `polyline.firstOrNull()` 존재 여부
- `createNavigationViewportOverlayState()` 결과의 `SEGMENT_JUNCTION` 개수
- `createKakaoProjectedMarkerRenderStates()` 이후 `ROUTE_SEGMENT_JUNCTION` 개수
- `updateProjectedMarkerOverlays()`에서 `projectedMarkers.size`, `projectionResult.overlays.size`, retry 예약 여부
- `KakaoMapViewport`에서 최종 렌더링된 marker overlay 개수와 화면 좌표

현재는 이 여섯 단계의 수치 로그가 없다. 그래서 "어디서 0개가 되는지"가 아니라 "어딘가에서 안 보인다" 수준의 가설 상태에 머물러 있다.

## 6. 검증 이슈

- 이전에는 ASCII 드라이브(`subst X:`) 경로에서 관련 단위 테스트 일부를 통과시켰다.
- 이번 추가 시도 후 `cmd /c ".\\gradlew.bat :app:compileDebugUnitTestKotlin"`는 성공했다.
- 하지만 `cmd /c ".\\gradlew.bat :app:testDebugUnitTest --tests com.ssafy.e102.eumgil.feature.map.component.KakaoMapViewportBindingsTest"`는 test 본문 실행 전 `KakaoMapViewportBindingsTest > initializationError`와 `ClassNotFoundException`으로 실패했다.
- 같은 워크트리에서 broader filter로 `:app:testDebugUnitTest --tests *KakaoMapViewportBindingsTest`를 실행해도 동일한 `ClassNotFoundException`이 반복됐다.
- 즉 현재 blocker는 "새 테스트 assertion 실패"가 아니라 "이 환경의 Gradle test worker가 test class를 정상 로드하지 못함"이다.
- 이 문제는 "현재 코드가 틀렸다"는 직접 증거는 아니지만, 수정 후 즉시 신뢰도 높은 재검증을 막고 있다.
- 결과적으로 지금 필요한 것은 추가 추측이 아니라 런타임 증거 수집과 테스트 실행 환경 정상화다.

## 7. 현재 결론

- 세그먼트 마커 미노출은 아직 미해결이다.
- 오버레이 생성 규칙을 시작점 기준으로 바꾸고 1점 세그먼트 허용까지 했지만, 실제 화면 미노출은 계속 재현되고 있다.
- 추가로 "초기 screen point projection miss" 가설에 따라 projected marker 재투영 retry도 넣었지만, 사용자 재현 기준으로는 여전히 해결되지 않았다.
- 따라서 현재 상태의 결론은 "오버레이 모델 수정만으로는 부족하며, 실제 런타임에서 어느 단계가 0개 또는 비가시 상태가 되는지 로그로 잡아야 한다"이다.
- 다음 단계는 로직을 더 바꾸는 것이 아니라, `routeSegments -> overlay points -> projected markers -> projected overlays -> composed markers` 각 단계의 개수를 로그로 고정해 실제 소실 지점을 찾는 것이다.

## 10. 2026-05-11 추가 런타임 계측

- 이번 단계에서는 세그먼트 마커 표시 로직 자체를 더 바꾸지 않고, `routeSegments -> overlay points -> projected markers -> projected overlays -> composed markers`의 각 단계가 몇 개를 유지하는지 로그로 고정했다.
- 추가한 로그 prefix는 아래 다섯 가지다.
  - `SegmentMarkerTrace[NavigationViewModel]`
  - `SegmentMarkerTrace[MapViewportOverlay]`
  - `SegmentMarkerTrace[KakaoProjectedMarkers]`
  - `SegmentMarkerTrace[Projection]`
  - `SegmentMarkerTrace[Compose]`

### 10.1 `NavigationViewModel`

- `createNavigationSegmentMarkerDebugSummary(mapOverlay)`를 추가했다.
- 여기서 아래 값을 한 줄로 남긴다.
  - `focusMode`
  - `routeSegments count`
  - active/focused index
  - 각 segment의 `idx`, `sequence`, `travelKind`, `polyline.size`, `polyline.firstOrNull()`
- 따라서 실제 재현 시점에 세그먼트 데이터 자체가 이미 비어 있는지 가장 먼저 확인할 수 있다.

### 10.2 `MapViewportOverlay`

- `createSegmentJunctionOverlayDebugSummary(mapOverlay, overlayState)`를 추가했다.
- 여기서 아래 값을 남긴다.
  - 생성된 전체 `points` / `polylines`
  - `SEGMENT_JUNCTION` 개수
  - 각 junction의 `overlayId`, `coordinate`, `tone`, `includeInProjection`
  - focused mode 여부를 간접 확인할 수 있는 `projectionPoints`, `projectionPolylines`
- 이 로그로 "세그먼트는 있었지만 junction overlay로 안 만들어졌는지"를 바로 분리할 수 있다.

### 10.3 `KakaoMapViewportBindings`

- `createProjectedSegmentMarkerDebugSummary(projectedMarkers)`를 추가했다.
- 여기서 아래 값을 남긴다.
  - 전체 projected marker 수
  - `ROUTE_SEGMENT_JUNCTION` 수
  - 각 junction projected marker의 lat/lng, `sizeDp`, `zIndex`, fill/stroke color
- 즉 overlay point가 Kakao projected marker로 바뀌는 단계에서 누락되는지 확인할 수 있다.

### 10.4 `KakaoMapViewport.updateProjectedMarkerOverlays()`

- `createProjectedSegmentMarkerPipelineDebugSummary(...)`를 추가했다.
- 여기서 아래 값을 남긴다.
  - `projectedMarkers.size`
  - `segmentProjected`
  - `projectionResult.overlays.size`
  - `segmentRendered`
  - `shouldRetry`
  - 실제 `retryScheduled`
  - `retryCount`
  - `cameraMoving`
  - 최종 screen point 기준 `screen=x,y`, `sizeDp`, `zIndex`
- 이 단계 로그가 있으면 "marker는 있는데 screen point projection이 비어 있는지", "retry 조건은 맞는데 실제 예약이 안 되는지"를 확인할 수 있다.

### 10.5 `KakaoMapViewport` Compose 렌더 단계

- renderer ready 시점에 `createRenderedSegmentJunctionOverlayDebugSummary(projectedMarkerOverlays)`를 한 번 더 기록하도록 넣었다.
- 여기서 실제 Compose가 그리려는 `ROUTE_SEGMENT_JUNCTION` 목록의
  - 개수
  - `screen=x,y`
  - `sizeDp`
  - `zIndex`
  - fill/stroke color
  를 그대로 남긴다.
- 따라서 projection 단계까지 살아 있는 마커가 Compose 직전에도 그대로 유지되는지 확인할 수 있다.

## 12. 다음 런타임 확인 포인트

1. `SegmentMarkerTrace[NavigationViewModel]`에서 `count>1`인데 2번 세그먼트 이후 `first=null`이 반복되면, 현재 가장 유력한 원인은 "실제 route segment polyline 자체가 비어 있다"는 쪽이다.

3. 2단계에서 `junctions>0`인데 `SegmentMarkerTrace[KakaoProjectedMarkers]`에서 `segmentProjected=0`이면 Kakao projected marker 바인딩 경로가 누락 지점이다.
4. 3단계에서 `segmentProjected>0`인데 `SegmentMarkerTrace[Projection]`에서 `segmentRendered=0`이면 screen point projection 또는 retry 조건이 누락 지점이다.
5. 4단계에서 `segmentRendered>0`인데 `SegmentMarkerTrace[Compose] count=0`이면 Compose 렌더 경로 또는 renderer ready 타이밍이 누락 지점이다.
