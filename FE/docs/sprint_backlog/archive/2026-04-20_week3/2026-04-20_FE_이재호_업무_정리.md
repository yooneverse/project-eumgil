## 3. 작업별 상세 내용

### 3.1 `S14P31E102-107` 로컬 저장소 및 mock 데이터 전환 기반 구축

- `107`은 `117`, `118`, `119`를 묶는 상위 Story로 관리한다.
- 실제 구현은 `117`에서 seed/model과 repository 반환 구조를 먼저 정리하고, 그 위에 `118`, `119`를 올리는 순서로 본다.
- `107` 자체를 별도 구현 MR로 보지 않고, 하위 작업 3개가 닫히면 함께 완료 처리하는 기준으로 본다.

**완료 기준**
- `117`, `118`, `119`의 범위와 연결 관계가 한 묶음으로 정리된다.
- 하위 작업 3개 완료 후 `107` Story를 닫을 수 있다.

### 3.2 `S14P31E102-117` 시설 및 점자블록 mock 시드 데이터 모델 구성

- 시설 카테고리 enum, 점자블록 타입, 좌표, 접근성 태그 필드를 먼저 확정한다.
- seed fixture를 화면 렌더링용과 상세 바텀시트용으로 동시에 쓸 수 있게 만든다.
- marker icon mapping과 상세 화면 표시값이 서로 어긋나지 않도록 label 기준을 통일한다.

**완료 기준**
- 지도 layer가 읽을 수 있는 seed 데이터와 model이 준비된다.
- `118`, `119`가 같은 facility model을 바로 재사용할 수 있다.

### 3.3 `S14P31E102-118` 지도 마커 오버레이 및 카테고리 필터 구현

- 지도 위에서 시설/점자블록 마커가 동시에 렌더링되더라도 카메라 이동과 겹침이 과하게 깨지지 않게 처리한다.
- 카테고리 필터는 다중 선택 기준으로 보고, 전체 보기 상태와 개별 토글 상태를 분리해 관리한다.
- 필터 전환 시 바텀시트 선택 상태가 어떻게 유지되거나 해제되는지 규칙을 같이 정한다.

**완료 기준**
- 카테고리 선택에 따라 지도 마커가 일관되게 토글된다.
- 필터 상태 변경 시 지도와 선택 상태가 서로 충돌하지 않는다.

### 3.4 `S14P31E102-119` 시설 상세 바텀시트 shell 및 길안내 진입 액션 구현

- 바텀시트에는 시설명, 분류, 거리, 접근성 태그, 주요 안내 문구, CTA가 들어가야 한다.
- route setting으로 넘길 목적지 모델을 이 시점에서 확정해서 `200`과 바로 연결되게 한다.
- 이후 `212` 북마크 토글이 들어올 수 있도록 버튼 slot 또는 상태 영역을 미리 구조화한다.

**완료 기준**
- 마커 탭 시 상세 바텀시트가 열리고 길안내 CTA가 동작한다.
- 북마크 토글이 나중에 붙어도 구조를 다시 바꾸지 않아도 된다.

### 3.5 `S14P31E102-198` route fixture 및 geometry parser 정리

- fixture 원본 포맷과 앱 내부에서 쓰는 route model을 분리해 관리한다.
- geometry parser는 정상 케이스뿐 아니라 좌표 누락, segment 비정상 값 같은 실패 케이스 fallback도 정의한다.
- `199`, `200`, `201`, `202`가 동일 route model을 쓰도록 parser 결과 shape를 먼저 고정한다.

**완료 기준**
- fixture route 1건 이상을 앱에서 안정적으로 파싱할 수 있다.
- parser 결과를 route setting, navigation, TTS에서 공통으로 쓸 수 있다.

### 3.6 `S14P31E102-199` `ROUTE-01` 경로 설정 화면 shell 및 2옵션 카드 구현

- 출발지/도착지 영역, `안전한 길`/`최단거리` 2옵션 카드, 경로 요약 카드, 시작 CTA 순서로 화면 구조를 잡는다.
- 서버 응답이 없어도 fixture만으로 preview와 summary가 동시에 보여야 한다.
- 최신 기준 2옵션 외 추가 옵션을 일정에 넣지 않고, 카드 구조는 `안전한 길`/`최단거리`만 기준으로 점검한다.

**완료 기준**
- 경로 설정 화면이 독립 진입 가능하고 summary가 보인다.
- fixture 기반 preview와 CTA 영역이 동시에 동작한다.

### 3.7 `S14P31E102-200` 검색 목적지 handoff와 경로 미리보기 연결

- 검색 화면에서 넘기는 목적지 최소 데이터셋을 정의하고 route setting 진입 시 누락 없이 받는다.
- 목적지 반영 후 preview polyline, summary 정보, CTA enable 상태가 함께 갱신되도록 연결한다.
- handoff 실패 시 빈 화면이 아니라 fallback 문구나 재선택 상태를 보여주도록 처리한다.

**완료 기준**
- 검색 결과 선택 후 route preview가 즉시 노출된다.
- 목적지 반영과 preview 렌더링이 한 흐름으로 이어진다.

### 3.8 `S14P31E102-201` 내비게이션 진행 화면 shell 및 step 카드 구현

- 현재 안내 step, 남은 거리, 예상 도착 시간, 종료 CTA를 기준으로 화면 우선순위를 잡는다.
- TTS가 꺼져 있어도 화면만으로 다음 행동을 이해할 수 있게 핵심 문구를 남긴다.
- 장시간 상태값 없이도 demo 기준으로 안정적으로 유지되는 화면 구조를 우선 구현한다.

**완료 기준**
- route 시작 후 진행 화면이 안정적으로 노출된다.
- step 카드와 종료 CTA만으로도 기본 demo 흐름이 가능하다.

### 3.9 `S14P31E102-202` TTS on/off 및 scripted briefing 제어 구현

- navigation 진입 시 시작 브리핑, 중간 step 안내, 중지 상태를 구분해서 제어한다.
- TTS off 상태에서도 UI 상태와 토글 상태가 명확히 보이게 해야 한다.
- 음성 엔진 실패 시 fallback 텍스트만으로도 흐름이 깨지지 않도록 처리한다.

**완료 기준**
- 음성 안내 on/off가 동작하고 fixture 기준 브리핑이 재생된다.
- 음성 기능 실패 시에도 navigation 화면 자체는 정상 동작한다.

### 3.10 `S14P31E102-203` 경로 미리보기 지도 레이어 연동 및 내비게이션 진입 플로우 정리

- route setting의 시작 CTA 조건과 navigation 진입 파라미터를 명확히 고정한다.
- route preview map state를 기준으로 선택 경로 polyline, 출발지 marker, 도착지 marker를 fallback map surface 위에 표시한다.
- 실제 지도 SDK 레이어 직접 연동은 아직 후속이며, 현재 구현은 fixture/서버 응답 polyline을 Compose fallback surface에 projection하는 방식이다.
- route option 변경, polyline 없음, 목적지 없음, route search 실패 fallback을 unit test로 검증한다.

**완료 기준**
- 경로 설정 화면에서 fixture 또는 서버 응답 기반 polyline이 fallback map surface 위에 표시된다.
- route option 변경 시 선택 route 기준으로 preview overlay가 갱신된다.
- polyline 없음, 목적지 없음, route search 실패 시 fallback UI와 CTA 비활성 상태가 유지된다.
- route setting -> navigation 진입 이벤트가 기존 흐름대로 유지된다.
