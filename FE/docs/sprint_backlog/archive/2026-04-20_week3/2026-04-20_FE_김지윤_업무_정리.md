## 3. 작업별 상세 내용

### 3.1 `S14P31E102-209` 지도 메인/시설 상세 흐름 및 목업 작성

- 지도 메인, 필터 적용, 마커 선택, 상세 바텀시트 열림까지의 상태 전이를 한 흐름으로 정리한다.
- 바텀시트 안에서 보여줄 정보 우선순위와 CTA 위치를 명확히 잡는다.
- 개발에서 바로 참고할 수 있도록 컴포넌트 이름과 상태 차이를 메모로 남긴다.

**완료 기준**
- map/detail 핵심 흐름 mockup이 리뷰 가능한 상태로 정리된다.
- 이재호가 `119`, `118` 구현 시 바로 참고할 수 있는 화면 기준이 생긴다.

### 3.2 `S14P31E102-210` 경로 설정/내비게이션/제보 화면 mockup 및 상태 시안 작성

- route setting, navigation, report 세 화면은 각각 정상 상태와 오류/빈 상태를 같이 정리한다.
- 발표용 시연 화면과 실제 구현 최소 화면이 다를 경우, 어떤 요소가 필수인지 구분해서 적는다.
- 상태별 CTA 문구와 우선 정보 영역이 무엇인지 명확히 표시한다.

**완료 기준**
- route/nav/report 주요 화면과 상태별 시안이 준비된다.
- 구현 최소 화면과 발표용 화면의 차이가 문서상에서 구분된다.

### 3.3 `S14P31E102-211` 공통 컴포넌트, 접근성 문안, Figma handoff 정리

- button, bottom sheet, input, status card 기준을 공통 컴포넌트 단위로 정리한다.
- TalkBack 또는 접근성 라벨에 가까운 문안을 버튼, 필터, 상태 메시지 기준으로 정리한다.
- 개발 handoff에는 spacing, typography, 상태별 문구 차이, 예외 케이스 메모를 같이 남긴다.

**완료 기준**
- Figma 링크와 handoff 메모가 개발 기준으로 바로 쓸 수 있는 상태가 된다.
- 화면별 CTA 문구와 접근성 문구가 빠지지 않는다.

### 3.4 `S14P31E102-204` 제보 작성 폼 spec 및 입력 상태 모델 정리

- 제보 유형, 위치 선택, 사진 첨부, 설명 입력의 필수/선택 여부를 먼저 확정한다.
- 입력 전, 입력 중, 오류, 제출 가능, 제출 완료 상태를 표로 정리한다.
- `205`, `206`, `207`이 같은 상태 정의를 쓰도록 상태명을 먼저 고정한다.

**완료 기준**
- validation 규칙과 상태표가 구현 기준으로 바로 사용 가능하다.
- 제보 화면에서 필요한 상태명이 문서와 코드에서 일관되게 쓰인다.

### 3.5 `S14P31E102-205` 제보 작성 화면 shell 구현

- 입력 영역 우선순위를 `유형 -> 위치 -> 사진 -> 설명 -> 제출` 흐름으로 잡는다.
- 빈 상태와 기본 placeholder를 한 번에 확인할 수 있게 layout을 구성한다.
- `206` validation이 바로 얹힐 수 있게 상태별 문구 영역과 오류 표시 위치를 미리 확보한다.

**완료 기준**
- 제보 작성 화면의 기본 입력 동선이 동작한다.
- validation과 상태 문구가 붙을 자리가 명확히 확보된다.

### 3.6 `S14P31E102-206` 제보 작성 validation/state 및 submit 가능 조건 구현

- 필수값 누락, 형식 오류, 사진 미첨부 허용 여부, 위치 미선택 상태를 명확히 구분한다.
- submit 버튼 활성/비활성 조건과 오류 메시지 노출 규칙을 일관되게 적용한다.
- draft 저장 전 상태와 실제 submit 가능 상태가 충돌하지 않도록 `207`과 같이 본다.

**완료 기준**
- 필수값 미입력 시 제출 불가, 입력 완료 시 제출 가능 상태가 정확히 동작한다.
- 오류 메시지와 버튼 상태가 서로 모순되지 않는다.

### 3.7 `S14P31E102-207` 제보 draft/outbox 로컬 저장 및 submit feedback 구현

- 작성 중 draft 자동 저장 또는 명시 저장 기준을 정하고 재진입 복원을 지원한다.
- submit 후에는 서버 전송이 아니라 outbox 등록 상태로 처리하고, 사용자에게 피드백을 준다.
- 작성 완료, 임시 저장, 재편집 복귀 세 가지 흐름이 모두 동작하도록 마무리한다.

**완료 기준**
- 네트워크 없이도 draft 복원과 임시 제출 상태 확인이 가능하다.
- 제보 기능이 BE 연동 전 임시 저장 기준으로 한 번 닫힌다.

### 3.8 `S14P31E102-212` 시설 상세에서 북마크 toggle 연결

- `119`에서 내려오는 시설 식별값을 이용해 북마크 on/off를 BE 연동 전 임시 저장소와 연결한다.
- 저장 여부에 따라 버튼 상태와 문구가 즉시 바뀌도록 처리한다.
- 저장 실패나 중복 저장 같은 예외 상태도 최소한의 사용자 피드백을 준다.

**완료 기준**
- 시설 단위 북마크 저장/해제가 가능하다.
- 상세 화면의 북마크 상태와 실제 임시 저장 데이터가 일치한다.

### 3.9 `S14P31E102-213` `BMK-01` 북마크 장소 목록 및 empty state 구현

- 북마크 장소 목록은 이름, 카테고리, 최근 북마크 여부를 빠르게 볼 수 있게 정리한다.
- 데이터가 없을 때는 empty state 문구와 다시 탐색으로 가는 CTA를 함께 제공한다.
- 삭제 action 이후 목록 갱신과 empty 전환이 자연스럽게 이어지도록 처리한다.

**완료 기준**
- placeholder가 아닌 실제 목록 화면이 동작한다.
- empty state와 삭제 후 갱신 흐름이 자연스럽다.

### 3.10 `S14P31E102-214` `BMK-01` 북마크 장소에서 지도/경로 handoff 연결

- 북마크 목록에서 선택한 장소를 지도 중심 이동 또는 경로 설정 진입으로 넘길 수 있어야 한다.
- handoff 시 필요한 최소 데이터셋을 `119`, `200`과 충돌 없이 맞춘다.
- 사용자가 목록에서 길안내 흐름으로 다시 들어갈 때 중간 단계가 끊기지 않게 연결한다.

**완료 기준**
- 북마크 목록에서 다시 길안내 흐름으로 진입 가능하다.
- 지도 복귀와 경로 설정 진입 중 어느 경로로 가는지 사용자가 이해할 수 있다.
