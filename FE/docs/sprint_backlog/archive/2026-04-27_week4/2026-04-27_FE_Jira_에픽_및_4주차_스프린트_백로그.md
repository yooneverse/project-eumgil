## 0. 문서 역할과 2~3주차 대비 4주차 재분류

2주차와 3주차 백로그는 당시 구현/이월 이력으로 유지한다. 4주차 백로그는 MVP 마감 전 마지막 기능 마감 주간의 실행 문서이며, 핵심 기준은 `미완성 MVP 기능을 닫으면서 이미 확정된 디자인 문서와 현재 구현 화면의 정합성 갭을 누가 어떤 Task로 해결하는가`다.

| 문서 구분 | 역할 | 4주차에서 읽는 방식 |
| --- | --- | --- |
| 2주차 백로그/업무정리 | 초기 기반 구축과 이월 기록 | 과거 구현 기준으로만 참고하고 최신 화면 ID와 디자인 기준으로 재해석 |
| 3주차 백로그/업무정리 | mock/fallback 구현 완료와 carry-over 근거 | 완료된 프레임을 4주차 디자인 정합성 수정 대상의 입력값으로 사용 |
| 4주차 백로그/업무정리 | MVP 마감 전 실행 계획 | 새로 닫아야 할 MVP 기능과 디자인 문서 기준의 화면/문구/컴포넌트/상태/접근성 정합성 갭을 함께 Task화 |

## 1. 4주차 스프린트 목표

4주차 목표는 3주차까지 잡힌 프레임에 남은 MVP 기능을 붙이고, 동시에 확정 디자인과 맞게 고쳐 5주차 통합/QA/버그수정/시연 안정화로 넘기는 것이다.

1. `AUTH-001`, `ONB-001`~`ONB-005`를 로그인 필수 앱 진입 gate와 확정 디자인 기준으로 맞춘다.
2. `MAP-01 -> DEST-01~DEST-04 -> ROUTE-01 -> ROUTE-02 -> NAV-01 -> ARR-01/ARR-02` 핵심 이동 흐름의 MVP 동작과 디자인 정합성 갭을 닫는다.
3. 하단 탭은 `홈/북마크/제보/마이페이지` 4탭 기준으로 유지한다.
4. 북마크는 사용자 노출명과 화면 ID 모두 `북마크`/`BMK` 체계로만 정리한다.
5. `REPORT-01`~`REPORT-04`, `BMK-01`~`BMK-02`, `MYP-01`~`MYP-03`의 상태, CTA, 문구, 접근성 라벨을 5주차 QA 기준으로 맞춘다.
6. BE 연동이 필요한 항목은 API를 임의 확정하지 않고 `BE 확인 필요`, `추후 구현`, `추가 확인 필요`로 분리한다.

## 2. Jira 및 브랜치 컨벤션 반영 원칙

| 항목 | 4주차 적용 |
| --- | --- |
| Jira 계층 | Epic 아래 Story를 두고, Story와 Task는 `연결된 업무 항목(Linked Issues)`으로 연결한다. 서브태스크는 사용하지 않는다. |
| Story 본문 | Story 본문은 비워두고 상세 내용은 연결된 Task에 작성한다. |
| Story 제목 | `[FE] 기능 명칭` 형식으로 작성한다. 예: `[FE] 경로/길안내 MVP 기능 완성 및 디자인 정합성 수정` |
| Task 제목 | `[FE] 상세 업무 내용` 형식으로 작성한다. 예: `[FE] ROUTE-01 대중교통 탭 및 2옵션 카드 디자인 정합성 수정` |
| Story Point | 피보나치 기준 `1, 2, 3, 5, 8`만 사용한다. `6`은 사용하지 않는다. |
| Task Point | 각 Task에도 개별 예상 Point를 배정한다. 4주차 Task는 `1, 2, 3` 중심으로 쪼갠다. |
| MR 기준 | 개발 완료와 MR은 Task 단위로 진행한다. `Task 1개 = branch 1개 = MR 1개`다. |
| Branch | `fe/feat/{작업-slug}-{TaskNo}` 형식을 사용한다. `{TaskNo}`에는 Jira 숫자만 넣고, 브랜치명 끝은 반드시 이슈 번호로 끝낸다. |
| MR 제목/본문 | MR 제목에 Jira 이슈 키를 포함하고, 본문에 `Closes S14P31E102-{TaskNo}` 또는 `Resolves S14P31E102-{TaskNo}`를 남긴다. |
