# 기능명세서 재구성안

> 2026-04-28 기준 최신 구현 정책은 `Docs/PRD/2026-04-14_기능명세서.md`를 따른다. 이 문서는 발표 흐름을 재구성하던 초안이며, 아래 항목 중 온보딩/카테고리/로그인/북마크 정책은 최신 기능명세서로 대체한다.

## 1. 재구성 기준

> 참고: 이 문서는 재구성 초안이다. 최신 기준선과 충돌할 경우 `Docs/기획/2026-04-09_화면명세서.md`, `Docs/PRD/2026-04-09_부산이음길_PRD.md`, `Docs/PRD/2026-04-14_기능명세서.md`를 우선한다.

- 핵심 타깃은 보행약자와 저시력자로 설정한다.
- 청각장애인 관련 기능은 핵심 MVP에서 제외하고 후순위 확장 기능으로 분리한다.
- 발표 피드백을 반영해 “기능 나열”보다 “사용자 특성 파악 -> 맞춤 지도/경로 -> 위험 안내 -> 데이터 개선” 흐름이 보이도록 구성한다.
- 온보딩에서는 사용자가 먼저 `저시력자` 또는 `보행약자`를 선택한다. `저시력자`는 별도 `LV-*` 화면군으로, `보행약자`는 공통 `MAP/DEST/ROUTE/NAV` 화면군으로 진입한다.
- 제보 기능은 핵심 차별점으로 과도하게 강조하지 않고, 사용자가 길 상태를 가볍게 표시하는 기능으로 배치한다.
- AI 기능은 단순 챗봇이 아니라 목적지 탐색, 경로 확인, 위험 안내를 자연스럽게 연결하는 기능으로 정의한다.

| 우선순위 | 의미 | 포함 기준 |
| --- | --- | --- |
| P0 | MVP 필수 | 서비스 이용을 위해 반드시 필요한 기능 |
| P1 | 핵심 차별화 | 배리어프리 서비스 정체성을 강화하는 기능 |

| P3-09 | 추천 확장 | 주변 접근 가능 장소 추천 | 전체 사용자 | 현재 위치와 사용자 특성을 기반으로 주변 접근성 좋은 장소를 추천한다. | 추천 품질을 담보할 데이터가 충분히 쌓인 뒤 확장 기능으로 검토한다. |

## 7. 발표/문서 구성 시 강조 순서

1. 문제 정의: 보행약자와 저시력자는 같은 경로라도 이동 시간, 위험 요소, 안내 방식이 다르다.
2. 핵심 해결: 온보딩에서 사용자 유형을 먼저 선택하게 하고, 보행약자는 공통 지도 중심 화면군, 저시력자는 별도 `LV-*` 음성 중심 화면군을 제공한다.
3. 데이터 흐름: 기존 미활용 데이터 -> 지도 매핑 포인트 생성 -> 중복 병합 -> 경로 레이어 반영.
4. 차별점: 단순 지도 서비스가 아니라 배리어프리 조건과 사용자별 이동 특성을 반영한 경로 안내.
5. 확장성: 길 상태 표시, 사용자 경로 기록, AI 안내는 더 편한 길 안내로 이어질 수 있는 보조 축이다.
6. 후순위: 청각장애인 기능은 마이크 권한과 알림 UX 검증이 필요하므로 MVP 이후 확장으로 둔다.

## 8. 기존 기능 조정 요약

| 기존 항목 | 조정 방향 |
| --- | --- |
| 제보/신고 | 발표 핵심에서 보조 기능으로 이동. “신고서 작성”이 아니라 지도 위에 길 상태를 가볍게 남기는 기능으로 설명한다. |
| AI 챗봇 | 맛집/관광 추천보다 목적지 탐색, 경로 확인, 위험 안내 연결 기능을 우선한다. |
| 저시력자 전용 UI | 별도 `LV-*` 화면군을 기준으로 보고, P1/P2에서는 음성 검색, TTS, 사전 브리핑 같은 고도화 기능 중심으로 정리한다. |
| 청각장애인 기능 | P3 후순위 확장으로 분리한다. 발표와 MVP 범위에서는 핵심 기능으로 내세우지 않는다. |
| 웨어러블 | 핵심 기능 완성 후 확장 기능으로 둔다. |
