# 관리자 병목 지도 실데이터 검증

## 결론

현재 DB만으로는 `실제 이동속도 기반 병목 지도`를 바로 만들 수 없다. `route_sessions`에는 선택된 경로 스냅샷 geometry와 시작/종료 시각이 있지만, 네비게이션 중 주기적으로 수집한 위치 샘플, 샘플별 속도, 세그먼트 매칭 결과가 없다.

따라서 1차 관리자 홈에서는 아래처럼 구분한다.

- 가능: 사용자 수, 신규 가입, 경로 선택/완료 수, 평균 세션 시간, 신고 수, road segment/시설 데이터 품질, 경로 스냅샷 geometry 정합성 검증
- 제한적 가능: 경로 스냅샷과 `road_segments`를 이용한 `이용 경로 밀도`, 접근성 속성/신고를 결합한 `병목 후보`
- 불가: 실제 이동속도, 정지/저속 구간, 세그먼트별 평균 속도, 실측 기반 병목 TOP 5

## 공식 문서 기준

- Kakao Maps JavaScript API는 `Polyline`, `Circle`, `CustomOverlay`, `MarkerClusterer`를 제공한다. 별도 heatmap 레이어는 문서에서 확인되지 않았다. 병목 지도는 프론트가 원본 경로 수만 개를 직접 그리는 방식보다, 서버에서 집계한 세그먼트/격자 결과를 `Polyline`/`Circle`/`CustomOverlay`로 얹는 방식이 적합하다.
- Kakao `MarkerClusterer`는 점 제보 밀집도에 적합하고, 병목 자체는 도로 선형 데이터이므로 marker cluster만으로 표현하면 신고 위치 지도처럼 보일 위험이 크다.
- PostGIS `ST_LineLocatePoint`, `ST_DWithin`, `ST_HexagonGrid`는 GPS 샘플을 도로 세그먼트 또는 격자에 붙이고 집계하는 데 적합하다.
- GraphHopper Map Matching은 GPX/측정 GPS를 도로망에 스냅하는 기능을 제공한다. 다만 현재 서비스가 이미 내부 `road_segments` 기반 GraphHopper를 사용하므로, 운영 집계는 DB에서 `road_segments.edge_id` 기준으로 정규화하는 편이 더 단순하다.
- OpenSearch `geotile_grid`는 포인트/도형을 지도 타일 격자로 묶는 데 적합하다. 검색/대시보드 집계가 커지면 후보지만, 도로 세그먼트 정합성은 PostGIS 전처리가 먼저다.

참고:
- https://apis.map.kakao.com/web/documentation/
- https://postgis.net/docs/manual-3.3/ST_LineLocatePoint.html
- https://postgis.net/docs/ST_HexagonGrid.html
- https://docs.graphhopper.com/openapi/map-matching/postgpx

## 2026-05-15 dev DB 연동 검증

로컬 Admin(`http://localhost:3001`)과 Docker 로컬 WAS(`http://localhost:8080`)를 S1/dev DB, Redis, MinIO, GraphHopper 터널에 연결해 관리자 홈 API를 검증했다.

| 항목 | dev DB 실측 |
|---|---:|
| 전체 사용자 | 60 |
| 전체 경로 세션 | 283 |
| 최근 7일 경로 세션 | 283 |
| 활성 경로 | 126 |
| 완료 경로 | 157 |
| 완료율 | 55.5% |
| 최근 7일 평균 경로 속도 | 3.3021 m/s |
| 불편 신고 | 8 |
| 대기 중 신고 | 8 |
| 최근 7일 WALK geometry 후보 | 97 |

검증 중 확인한 수정 사항은 아래와 같다.

- `route_snapshot_json ? 'geometry'` 형태의 PostgreSQL JSONB key 체크는 JDBC prepared statement에서 `?`를 파라미터로 오인해 실패했다. 네이티브 쿼리는 `jsonb_exists(route_snapshot_json, 'geometry')`로 변경했다.
- 관리자 홈 기본 기간은 오늘 하루가 아니라 최근 7일로 잡았다. dev DB에 오늘 데이터가 없더라도 운영 홈 첫 화면에서 실제 누적 경로와 신고 흐름을 바로 볼 수 있게 하기 위함이다.
- dev DB에 `admin_audit_logs` 테이블이 없어 관리자 API가 감사 로그 저장 단계에서 실패했다. `schema.sql`에 `CREATE TABLE IF NOT EXISTS admin_audit_logs`와 조회 인덱스를 추가했고, dev DB에도 동일 DDL을 적용했다.
- dev DB에는 영구 `ADMIN` 사용자가 없어서, 검증 시에만 기존 사용자 1명을 임시 `ADMIN`으로 올려 토큰을 발급한 뒤 즉시 `USER`로 되돌렸다. 자연 로그인 검증을 하려면 실제 관리자 계정을 명시적으로 승격해야 한다.

현재 Admin 홈 화면은 `/admin/dashboard/summary`, `/admin/dashboard/bottlenecks` 응답을 사용해 실제 dev DB 수치를 표시한다. 다만 불편 신고 좌표가 병목 후보 경로와 가까운 데이터가 아직 없어, 병목 TOP 5의 신고 건수는 0으로 나올 수 있다. 이는 화면 오류가 아니라 현재 dev DB 데이터 분포의 결과다.

## 경로 geometry 정합성

`route_sessions.route_snapshot_json.geometry`는 S1/dev 260건, S2/prod 1건 모두 유효한 WKT LineString이었다.

최근 S1/dev 12개 세션을 25m 간격 샘플로 쪼개 `road_segments`와 30m 이내 매칭 여부를 확인했다.

- 도보 경로는 대부분 평균 0.01m 수준으로 road segment 위에 있었다.
- 대중교통 경로 일부는 버스/지하철 이동선이 포함되어 road segment와 30m 이상 떨어진 샘플이 많았다.
- 즉 병목 집계는 `transportMode = WALK` 또는 대중교통 중 도보 leg만 분리해서 수행해야 한다.

## 관리자 홈 프리뷰 지도 검증

2026-05-14 관리자 홈에 임시로 표시하는 병목 프리뷰 좌표도 검증했다.

- 기존 임의 좌표는 24개 중 15개만 `road_segments` 30m 이내였고, 최대 122.71m까지 벗어난 점이 있었다.
- 현재 프리뷰 좌표는 S1 GraphHopper `pedestrian_safe` 경로 응답에서 추출한 선형 좌표만 사용한다.
- GraphHopper 연결이 실패한 부산항/일부 단절 포인트는 프리뷰 지도 데이터에서 제외했다.
- 변경 후 프리뷰 좌표 108개를 S1/dev `road_segments` 최근접 거리로 다시 검증한 결과, 108개 모두 5m 이내였고 최대 오차는 0.06m였다.

따라서 현재 화면의 히트맵은 아직 실측 속도 기반은 아니지만, 적어도 표시 좌표 자체는 운영 보행 네트워크/GraphHopper 경로 위에 놓인 검증된 프리뷰다.

## 권장 집계 구조

1. 프론트는 네비게이션 중 원본 위치를 계속 보내지 않고, 우선 종료 시점 batch 업로드를 기본값으로 둔다.
2. 클라이언트는 동의 기반으로 `sessionId`, `routeId`, `timestamp`, `lat`, `lng`, `accuracy`, `speedMps`, `heading`, `appVersion`, `transportMode`, `batterySaving 여부` 정도만 보낸다.
3. 서버는 원본 샘플을 오래 보관하지 않고, 배치 작업에서 `road_segments.edge_id`로 스냅한 뒤 세그먼트 집계 테이블에 누적한다.
4. 관리자 지도 API는 원본 샘플이 아니라 기간/줌/bbox 기준으로 이미 집계된 `edge_id`, `avg_speed_mps`, `p50_speed_mps`, `sample_count`, `report_count`, `confidence`만 반환한다.
5. 지도는 줌 레벨별로 다르게 반환한다.
   - 낮은 줌: hex/grid 또는 tile bucket
   - 중간 줌: top N 병목 세그먼트
   - 높은 줌: road segment polyline과 상세 수치

## UI 적용 기준

- 실측 속도 수집 전에는 평균 속도/병목 수를 실제값처럼 표시하지 않는다.
- 점 신고는 `MarkerClusterer` 또는 작은 점 레이어로 별도 표시한다.
- 병목은 도로 선형을 따라 이어지는 리본/열 지도 형태로 표현한다.
- 초기 지도는 병목 후보를 클러스터 단위로 묶어 보여주고, 확대 단계에 따라 `클러스터 -> 주요 구간 리본 -> 세부 히트셀/핫스팟` 순서로 정보를 점진 공개한다.
- 히트맵은 운영 대시보드의 바탕 지도와 도로명을 가리지 않도록 낮은 opacity와 작은 blur를 기본값으로 둔다.
- TOP 5 테이블은 실측 수집 전에는 `병목 후보` 또는 `검수 우선순위`로 표시해야 한다.
