# 운영도구 Secret 관리 기준

이 문서는 S1 운영도구의 secret 위치와 관리 책임을 정리한다. secret 값과 env 템플릿은 Git에 올리지 않고, 실제 키 목록은 노션의 운영 env 기준을 따른다.

## 원칙

- 앱 런타임 secret과 운영도구 secret은 분리한다.
- `.env.prod`는 prod 애플리케이션 런타임 전용으로 사용한다.
- Jenkins, Mattermost, Grafana, SonarQube 운영도구 secret은 로컬 루트 `.env.ops`에 작성한 뒤 S1 서버 로컬 환경 파일로 배포한다.
- Git에는 실제 `.env` 파일과 env example 파일을 올리지 않는다.
- secret을 변경하면 서버 로컬 파일, Jenkins credential, 관련 문서를 함께 확인한다.
- `JWT_SECRET`, `JPA_DDL_AUTO` 같은 애플리케이션 런타임 값은 `.env.prod`에서 관리한다. 현재 bootstrap 단계에서는 `JPA_DDL_AUTO=update`를 허용하지만, 운영 모드 전환 전에는 `validate`로 되돌린다.

## 파일 위치

| 위치 | 용도 | Git 관리 |
|---|---|---|
| `/home/ubuntu/e102/INF/jenkins/s1/.env.jenkins` | Jenkins 자체 설정, GitLab OAuth, Mattermost webhook, 복구 관리자 계정 | 제외 |
| `/home/ubuntu/e102/ops/.env` | Grafana/SonarQube OAuth, 운영도구 관리자 계정, SonarQube DB | 제외 |
| `/home/ubuntu/e102/.env.prod` | prod backend/AI 런타임 설정 | 제외 |
| `.env.ops` | 운영도구 secret 원본 파일 | 제외 |

## 작성 위치

운영도구 OAuth 값은 루트 `.env.ops`에 작성한다.

`.env.ops`는 Git ignore 대상이다. 필요한 key 목록은 노션의 운영 env 기준을 확인한다. 이 파일을 기준으로 Jenkins용 값은 `/home/ubuntu/e102/INF/jenkins/s1/.env.jenkins`, Grafana/SonarQube용 값은 `/home/ubuntu/e102/ops/.env`에 반영한다.

서버 로컬 env 파일을 갱신할 때는 기존 파일을 직접 편집하거나 덮어쓰지 말고, 임시 파일을 만든 뒤 rename하는 방식으로 반영한다. 파일 소유권이 root로 고정돼 있는 환경에서도 권한 문제를 줄이기 위함이다.

## Jenkins

GitLab Application 설정:

```text
Name: E102 Jenkins
Redirect URI: https://jenkins.busaneumgil.com/securityRealm/finishLogin
Scopes: api, read_user
Confidential: checked
```

루트 `.env.ops`에 작성하고, S1 서버 `/home/ubuntu/e102/INF/jenkins/s1/.env.jenkins`에 반영한다.

```bash
JENKINS_GITLAB_OAUTH_CLIENT_ID=
JENKINS_GITLAB_OAUTH_CLIENT_SECRET=
JENKINS_GITLAB_DEV_GROUP=s14-final
JENKINS_GITLAB_ADMIN_USER=
MATTERMOST_WEBHOOK_URL=
```

## Grafana

GitLab Application 설정:

```text
Name: E102 Grafana
Redirect URI: https://grafana.busaneumgil.com/login/gitlab
Scopes: api, read_user, openid, profile, email
Confidential: checked
```

루트 `.env.ops`에 작성하고, S1 서버 `/home/ubuntu/e102/ops/.env`에 반영한다.

```bash
GRAFANA_GITLAB_OAUTH_CLIENT_ID=
GRAFANA_GITLAB_OAUTH_CLIENT_SECRET=
GITLAB_ALLOWED_GROUPS=ssafy_14th
GITLAB_BASE_URL=<INTERNAL_GIT_BASE_URL>
```

## SonarQube

GitLab Application 설정:

```text
Name: E102 SonarQube
Redirect URI: https://sonarqube.busaneumgil.com/oauth2/callback/gitlab
Scopes: api, read_user
Confidential: checked
```

루트 `.env.ops`에 작성하고, S1 서버 `/home/ubuntu/e102/ops/.env`에 반영한다.

```bash
SONAR_GITLAB_OAUTH_CLIENT_ID=
SONAR_GITLAB_OAUTH_CLIENT_SECRET=
SONAR_PUBLIC_URL=https://sonarqube.busaneumgil.com
SONAR_ADMIN_USER=
SONAR_ADMIN_PASSWORD=
```

SonarQube OAuth는 compose 환경변수만으로 관리하지 않고, Settings API 적용 스크립트로 반영한다.

```bash
cd /home/ubuntu/e102/ops
./scripts/apply-sonarqube-oauth.sh
```

## Portainer

Portainer는 GitLab OAuth 대상에서 제외한다. Docker socket 권한이 크기 때문에 외부 공개하지 않고 SSH 터널로만 접근한다.

```bash
make portainer-tunnel
```

접속 주소:

```text
http://localhost:19000
```

## GitLab group 제한 기준

Grafana와 SonarQube는 GitLab OAuth 응답의 group claim을 기준으로 접근을 제한한다.
현재 SSAFY GitLab OAuth에서 확인된 group path는 `ssafy_14th`이므로 운영도구 공통 제한값은 아래처럼 둔다.

```bash
GITLAB_ALLOWED_GROUPS=ssafy_14th
```

`s14-final`은 화면상 프로젝트/그룹 목록에는 보일 수 있지만 OAuth group claim에 항상 포함된다는 보장이 없어 Grafana/SonarQube 제한값으로 사용하지 않는다.
