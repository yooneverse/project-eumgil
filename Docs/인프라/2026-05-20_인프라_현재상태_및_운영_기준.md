> 작성일: 2026-05-20

> 목적: 기존 4월 인프라 설계 문서와 5월 운영 자산을 현재 런타임 상태 기준으로 정렬한다.

## 1. 결론

현재 운영 구조는 **S1 dev/Jenkins/운영도구 + S2 primary prod**다.

```text
사용자/관리자
  -> Route53 / DNS
  -> S2 Nginx 80/443
  -> prod backend / AI / ADMIN / GraphHopper blue-green
  -> RDS PostgreSQL / ElastiCache Redis / S3

개발/운영자
  -> S1 Nginx 80/443
  -> dev backend / dev AI / Jenkins / Grafana(PLG) / SonarQube
  -> Portainer는 SSH tunnel 전용
```

S1과 S2를 서로 다른 prod blue/green 서버로 쓰지 않는다. Prod blue/green은 **S2 내부 GraphHopper runtime slot**인 `graphhopper-blue`, `graphhopper-green`에만 적용한다.

## 5. 배포와 GraphHopper

### Prod app deploy

Jenkins `e102-prod-deploy`가 S2 `/home/ubuntu/e102/prod`에 archive와 `e102-prod-env-file` credential을 업로드하고 `scripts/deploy/prod-deploy.sh`를 실행한다.

기본 배포 대상:

- `backend`
- `ai`
- `admin`
- `graphhopper-blue`
- `graphhopper-green` (`DEPLOY_GRAPHHOPPER=true` 기본값)

`BUILD_GRAPHHOPPER=true`일 때는 일반 app deploy 중에도 `scripts/graphhopper/prod-bluegreen-refresh.sh`를 호출한다. 일반 배포에서는 graph-cache rebuild를 매번 수행하지 않는다.

### GraphHopper refresh

Jenkins `e102-graphhopper-refresh`는 3시간마다 실행된다.

최근 확인된 refresh report:

| buildId | status | 전환 |
| --- | --- | --- |
| `jenkins-52-fde21192499d` | `SUCCESS` | green -> blue |
| `jenkins-53-fde21192499d` | `SUCCESS` | blue -> green |
| `jenkins-54-fde21192499d` | `SUCCESS` | green -> blue |

GraphHopper endpoint 선택은 Redis key를 기준으로 한다.

| Key | 의미 |
| --- | --- |
| `graphhopper:active-slot` | 현재 active slot |
| `graphhopper:previous-slot` | active 실패 시 fallback slot |
| `graphhopper:blue:url` | blue service URL |
| `graphhopper:green:url` | green service URL |
| `graphhopper:active-build-id` | 마지막 성공 refresh build id |

GraphHopper admin reload endpoint `/ieum/admin/overrides/reload`는 backend 내부 호출 전용이다. S1/S2 Nginx 공개 경로에서는 차단한다.

## 7. 후속 정리 항목

- S1 dev backend의 `/health/graphhopper`가 직접 GraphHopper health와 일치하도록 `GRAPHHOPPER_HEALTH_URL` 또는 dev Redis slot 초기화 정책을 정한다.
- S1의 수동 잔여 컨테이너로 보이는 `vigilant_ptolemy` Prometheus의 유지/삭제 여부를 운영자가 결정한다.
- S2의 `e102-prod-blackbox-exporter`, `e102-portainer-agent`가 장기 운영 자산이면 `INF/monitoring/s2`에 기준 파일을 승격한다.
- S2 prod 디렉터리는 Jenkins archive 배포 산출물이라 Git worktree가 아니다. 배포 버전 확인은 `.deploy-state/current-app-image`와 Jenkins release manifest를 기준으로 한다.
