### 4.1 보행 네트워크 수동 편집 보정 정책

현재 DB 반영 기준:

- 관리자 페이지에서 새 segment를 추가할 때 양 끝점 기준으로 `1.0m` 이내 기존 `road_nodes`가 있으면 해당 node를 재사용한다.
- `1.0m` 이내 기존 node가 없으면 신규 `road_nodes`를 생성한다.
- 이 기준은 `AdminRoadNetworkEditService.SNAP_DISTANCE_METER = 1.0`과 ERD v4의 `road_nodes` 설명을 따른다.

현재 UI/BE 보정 기준:

- 보행 네트워크 탭에서 `CROSS_WALK`를 추가할 때 클릭 지점 주변 `1.5m` 이내 기존 `SIDE_LINE` segment가 있으면, UI draft 좌표를 해당 segment 선 위의 가장 가까운 점으로 보정한다.
- 사용자는 지도 하단 상태 문구에서 보정 여부와 보정 거리를 확인할 수 있다.
- `SIDE_LINE` 일반 추가는 현재 `1.0m node snap` 기준을 유지하고, 선 자체 보정은 우선 `CROSS_WALK`에만 적용한다.
- `POST /admin/road-network/edits/apply`에서도 `CROSS_WALK` 추가 시 같은 기준을 적용한다. 양 끝점이 `1.0m` 이내 기존 node에 붙지 않으면, `1.5m` 이내 기존 `SIDE_LINE` 선 위로 projection 보정한 뒤 `road_nodes`, `road_segments`를 생성한다.
- projection으로 새 node가 기존 segment 중간에 생성되는 경우, 해당 기존 segment를 split해서 새 node를 공유하도록 저장한다. 이렇게 해야 화면상으로만 맞닿은 선이 아니라 라우팅 graph 기준으로도 연결된다.

현재 한계:

- UI에서 보정된 좌표가 DB 반영 요청으로 전달되므로 관리자 화면을 통한 저장은 보정 좌표 기준으로 반영된다.
- DB apply API를 UI 없이 직접 호출해도 `CROSS_WALK` 선 projection 보정과 기존 segment split을 서버에서 한 번 더 적용한다.
- split 대상은 이번 `CROSS_WALK` 추가로 새로 생성된 node가 닿는 `SIDE_LINE`에 한정한다. 이미 `1.0m` 이내 기존 node에 snap된 경우에는 기존 graph node를 재사용하므로 split하지 않는다.

후속 보정 정책:

1. `CROSS_WALK` 추가 시 양 끝점 주변 `1.0m` 이내 기존 node가 있으면 기존처럼 node에 snap한다.
2. 기존 node가 없으면 양 끝점 주변 `1.0m~1.5m` 이내 기존 `SIDE_LINE` segment를 찾는다.
3. 양 끝이 각각 연결 가능한 기존 segment를 찾을 수 있으면, 사용자가 그린 `CROSS_WALK` 선을 기존 segment 선에 맞닿도록 보정한다.
4. 보정 지점이 기존 segment 중간이면 기존 segment를 split하고, `CROSS_WALK`와 split segment가 같은 `road_nodes.vertex_id`를 공유하도록 한다.
5. 이때 보정은 화면에서만 하지 않고, 최종 DB 반영 API에서도 같은 기준으로 적용한다.
6. 보정 전후 차이가 너무 크거나 한쪽 끝만 연결 가능한 경우에는 자동 보정하지 않고 사용자에게 경고하거나 수동 재입력을 요구한다.

주의:

- 프론트에서만 선을 예쁘게 보정하고 DB에는 원래 클릭 좌표를 저장하면 데이터 정합성이 깨진다.
- 따라서 `CROSS_WALK` 선 보정은 UI preview와 BE apply 로직의 기준을 반드시 동일하게 유지해야 한다.
- 현재 UI와 BE apply의 `CROSS_WALK` projection 기준은 `1.5m`로 맞춘 상태다.
- split으로 생성된 기존 segment 조각은 원래 segment의 접근성 집계 컬럼을 우선 보존하고, `segment_features`는 `source_features` 공간 매칭으로 다시 생성한다. 신규 `CROSS_WALK` 자체도 기존 `source_features` 공간 매칭 로직으로 feature와 집계 컬럼을 보강한다.

## 5. 보행 네트워크 조회 범위 이슈

기존 운영 배포 기준:

- `admin_areas.geom` 기준 `500m buffer`의 envelope와 교차하는 segment를 조회한다.

변경 목표 기준:

- 기존 과다 조회를 줄이기 위해 `100m` 기준 검증을 원한다.
- 로컬 코드와 관리자 API 명세는 `100m buffer` 기준으로 수정했다.
- 운영 배포 전까지 실제 운영 화면은 기존 배포본 기준으로 동작할 수 있다.

운영 DB에서 강서구/명지동 계열 기준 단순 비교:

| 기준 | segment count |
| --- | ---: |
| 100m buffer envelope | 7,043 |
| 500m buffer envelope | 8,023 |

주의:

- 실제 API 응답은 현재 6,861건이었다.
- API 쿼리의 synthetic dong 조건과 단순 SQL 비교 조건이 완전히 같지 않을 수 있으므로, 정확한 변경 전후 비교는 동일 쿼리 기준으로 다시 측정해야 한다.
- `500m -> 100m` 변경은 조회 성능에는 유리하지만, 경계 근처 필요한 segment가 사라질 수 있으므로 배포 후 작업자 검수 화면에서 누락 여부를 확인해야 한다.

접근성 feature 교체 실패 원인:

- 운영 로그에서 `uk_place_accessibility_features_place_type` unique constraint 위반 확인

- 운영 배포 후 `PUT /admin/places/{placeId}/accessibility-features` 재검증 필요

로컬 UI 보강 완료:

- 편의시설 선택 시 `GET /admin/places/{placeId}`로 상세를 조회한다.
- 장소명, 카테고리, 주소, providerPlaceId, 좌표 수정 폼을 추가했다.
- `PATCH /admin/places/{placeId}` 저장 버튼을 추가했다.
- 접근성 feature 체크박스를 추가했다.
- `PUT /admin/places/{placeId}/accessibility-features` 저장 버튼을 추가했다.

주의:

- 접근성 feature 교체 API는 로컬 코드에서 500 원인을 수정했지만, 운영 배포 후 재검증이 필요하다.

## 8. DB 반영 테스트 원칙

운영 DB에서 편집 반영 테스트를 할 경우 아래 원칙을 지킨다.

1. 기존 운영 segment는 삭제하지 않는다.
2. 테스트로 새로 추가한 segment만 삭제한다.
3. 추가 전후 `road_nodes`, `road_segments`, `segment_features` count를 기록한다.
4. apply 응답의 `addedSegments`, `deletedSegments`, `createdNodes`, `snappedNodes`, `createdSegmentFeatures`, `updatedSegmentAttributes`를 기록한다.
5. 실패 시 rollback 방법이 없는 편집은 수행하지 않는다.

## 9. 후속 보강 제안

팀 운영 편의를 위해 아래 기능을 후속 과제로 둔다.

| 기능 | 목적 |
| --- | --- |
| 사용자 관리 탭 | ADMIN 승격/권한 확인 |
| 구/동 담당자 지정 | 작업자별 검수 범위 관리 |
| 담당 구/동 수정 제한 | 실수로 다른 구/동을 수정하는 것 방지 |
| 작업 상태 기록 | 미시작/진행중/완료/보류 관리 |

현재 팀원끼리만 사용하는 전제라면 복잡한 권한 체계는 1차에서 과하다.
다만 서버에서 최소한 `ADMIN` 여부와 담당 구/동 제한을 검증해야 화면 우회 요청을 막을 수 있다.

현재 운영 DB 확인 결과:

- `assign`, `admin`, `edit` 계열 테이블은 `admin_areas`만 존재한다.
- 구/동 담당자 지정 테이블이나 편집 작업 이력 테이블은 아직 없다.
- 따라서 현재 실사용 기준은 모든 `ADMIN` 사용자가 모든 구/동을 조회/편집할 수 있는 구조로 봐야 한다.

담당자 제한을 넣으려면 최소한 아래 설계가 필요하다.

| 항목 | 필요 이유 |
