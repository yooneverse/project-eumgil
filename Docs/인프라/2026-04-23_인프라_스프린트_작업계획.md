### GraphHopper 운영 반영 범위

GraphHopper는 현재 `local/dev` runtime과 S2 prod blue/green runtime까지 포함한다. 운영 배포 완성 여부는 단일 EC2 blue/green이 아니라 S2 내부 GraphHopper slot 전환과 refresh report 성공 여부로 본다.

- PostgreSQL LineString 원천 데이터에서 GraphHopper graph-cache를 생성하는 build job 자산을 추가했다.
- 생성된 graph-cache는 Docker volume의 runtime cache 위치로 배포한다.
- `docker-compose.prod.yml`의 GraphHopper service는 `graphhopper` profile의 `graphhopper-blue`/`graphhopper-green` runtime으로 분리되어 있다.
- S2에서 `backend + AI + admin + GraphHopper`가 운영 중이며, 일반 app 배포는 `BUILD_GRAPHHOPPER=false`, `DEPLOY_GRAPHHOPPER=true`를 기본 운영값으로 둔다.
- `e102-graphhopper-refresh`는 3시간마다 inactive slot을 갱신하고, 2026-05-20 확인 기준 최근 `jenkins-52`, `jenkins-53`, `jenkins-54` report가 모두 `SUCCESS`다.

- AI server는 외부 port를 열지 않고 backend 내부 호출로만 접근 가능
- GraphHopper graph-cache build job이 PostgreSQL edge/LineString 데이터를 입력으로 사용하도록 정리됨
- GraphHopper runtime은 build된 graph-cache를 serve only로 사용하도록 정리됨

- `api`와 `ai`는 `S2`, `jenkins/grafana/portainer/sonarqube/plg`는 `S1` host routing 기준이 문서화됨
- S1/S2가 같은 VPC가 아니므로 ALB target group 기반 Blue/Green을 현재 범위에서 제외함
- S2 prod primary와 S1 fallback 조건이 문서화됨
