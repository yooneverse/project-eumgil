# Docker 운영 기준

> 작성일: 2026-04-23
> 최종 수정일: 2026-05-20
> 적용 범위: 루트 compose, `.env.*`, `Makefile`, 서비스별 Dockerfile

---

## 1. 결론

이 프로젝트는 **루트 디렉토리에서 환경별 Docker Compose를 통합 관리**한다.

2026-05-20 실서버 확인 기준 S2 prod compose는 `backend + ai + admin + GraphHopper blue/green runtime`을 실행한다. 일반 app 배포는 `BUILD_GRAPHHOPPER=false`, `DEPLOY_GRAPHHOPPER=true`를 기본 운영값으로 두고, GraphHopper cache 갱신은 3시간 주기 `e102-graphhopper-refresh`가 inactive slot에서 수행한다.

```text
S14P31E102/
|-- docker-compose.shared.yml
|-- docker-compose.local.yml
|-- docker-compose.dev.yml
|-- docker-compose.prod.yml
|-- .env.local
|-- .env.dev
|-- .env.prod
|-- Makefile
|-- BE/
|   `-- Dockerfile
|-- AI/
|   `-- Dockerfile 또는 작업별 Dockerfile
`-- INF/
    |-- aws/
    |-- jenkins/
    `-- monitoring/
```

`BE/`, `AI/`, `INF/` 아래에 환경별 compose를 각각 두지 않는다.
대신 루트 compose에서 서비스와 profile을 나누고, `Makefile`로 사람이 쓰는 명령을 감싼다.

---

## 2. 왜 루트 compose인가

부산이음길은 백엔드만 단독으로 실행되는 구조가 아니다.

- Spring Boot API
- PostgreSQL/PostGIS
- Redis
- MinIO/S3
- GraphHopper runtime
- AI/ETL 작업
- Jenkins
- SonarQube
- PLG/Grafana

이 서비스들은 네트워크, 포트, env, volume을 공유한다.
따라서 영역별로 compose를 분산하면 같은 설정이 여러 곳에 반복되고, 전체 개발 환경을 한 번에 올리는 기준이 흐려진다.

루트 compose를 기준으로 하면 다음 장점이 있다.

- 전체 실행 진입점이 하나다.
- `.env.local`, `.env.dev`, `.env.prod` 계약을 한 곳에서 관리할 수 있다.
- 서비스 간 네트워크와 volume 이름이 일관된다.
- Jenkins나 운영 배포에서 같은 compose 파일을 참조하기 쉽다.
- 백엔드/AI/운영자가 필요한 일부 서비스만 profile 또는 service name으로 실행할 수 있다.

## 3. 책임 분리

| 위치 | 책임 |
|---|---|
| 루트 compose | 환경별 전체 실행 orchestration |
| 루트 shared compose | local/dev에서 재사용하는 공통 dependency definition |
| 루트 `.env.*` | 환경별 변수 계약 |

## 4. 환경별 파일 기준

### `docker-compose.shared.yml`

local/dev에서 재사용하는 공통 의존성 정의 파일이다.

현재 범위:

- PostGIS
- Redis
- MinIO
- MinIO bucket init

이 파일은 단독 실행 진입점이 아니라 `local`, `dev` compose가 재사용하는 base 역할을 한다.

### `docker-compose.local.yml`

개인 PC에서 **backend + graphhopper + 공유 의존성**을 함께 올리는 환경이다.

중요:

- 의도된 데이터 원천은 `road_segments` 같은 edge/LineString 기반 PostgreSQL 테이블이다.
- GraphHopper runtime이 PostgreSQL을 요청마다 직접 조회하지 않는다.
- PostgreSQL LineString 데이터를 GraphHopper import/build job의 입력으로 사용하고, 결과 graph-cache 아티팩트를 runtime에 배포한다.
- `local/dev` 통합 환경에는 GraphHopper runtime을 포함한다. 단, graph-cache build job과 runtime serve는 분리한다.

예상 범위:

- backend WAS
- GraphHopper runtime
- PostGIS
- Redis
- MinIO
- 개인 실험용 override

### `docker-compose.dev.yml`

dev 서버에서 실행하는 통합 개발 환경이다.

예상 범위:

- backend WAS
- GraphHopper runtime
- PostGIS
- Redis
- MinIO
- MinIO bucket init
- 필요 시 AI/ETL runtime
- 필요 시 dev 편의용 부가 서비스

원칙:

- `docker-compose.dev.yml`은 dev 서버에서 DB, 캐시, 스토리지, 경로 엔진을 함께 올리는 기준 파일이다.
- dev 서버의 backend는 compose 내부 service name인 `postgres`, `redis`, `minio`, `graphhopper`를 바라본다.
- dev 서버의 `.env.dev`는 container-network 기준 값을 채운다.
- 개발자 PC에서 dev 서버 자원만 바라보며 WAS를 띄우는 용도는 `make be-dev-up`을 사용한다.
- `make be-dev-up`은 별도 compose 파일을 만들지 않고 `scripts/make/lib/be-dev.sh`에서 backend image를 직접 build/run 한다.
- 로컬 PC의 `.env.dev`는 dev 서버 계정 정보만 채우고, endpoint는 `make be-dev-up`이 SSH tunnel과 `host.docker.internal` 기준으로 주입한다.
- 자세한 DB 연결 절차는 `Docs/인프라/2026-04-25_dev_DB_접근_가이드.md`를 따른다.
- dev 서버의 `.env.dev`는 `DB_URL=jdbc:postgresql://postgres:5432/<DB>`, `REDIS_HOST=redis`, `MINIO_ENDPOINT=http://minio:9000`처럼 compose network 기준 endpoint를 명시한다.
- dev에서는 AWS S3 대신 MinIO를 사용한다. `.env.dev`는 `MINIO_*` 변수로 관리하고, compose/script가 backend 실행 시점에 애플리케이션용 `S3_*` 변수로 매핑한다.
- GraphHopper runtime은 `GRAPHHOPPER_GRAPH_LOCATION`의 graph-cache만 읽는다.
- `GRAPHHOPPER_DATA_FILE`, `GRAPHHOPPER_DATA_DIR` 같은 raw 파일 변수는 운영 필수값으로 사용하지 않는다.
- PostgreSQL LineString 원천 데이터에서 graph-cache를 만드는 import/build job은 runtime compose와 분리한다.

2026-04-25 기준 S1 반영 상태:

- `docker-compose.dev.yml`과 S1 전용 override로 `PostGIS`, `Redis`, `MinIO`, `backend` 실행 검증 완료
- S1에서 외부 직접 노출을 피하기 위해 dev stack 포트는 `127.0.0.1`에 바인딩
- dev backend는 S1 nginx 443 proxy의 `api.dev.busaneumgil.com` host로만 외부 접근 허용
- dev AI는 S1 nginx 443 proxy의 `ai.dev.busaneumgil.com` host로만 외부 접근 허용
- PostgreSQL은 외부 공개하지 않고 SSH/SSM tunnel로만 접근
- 로컬 `make be-dev-up`은 SSH tunnel을 통해 S1 dev `PostgreSQL`, `Redis`, `MinIO`에 연결
- Jenkins `e102-dev-deploy`는 `e102-dev-env-file` credential로 `.env.dev`를 주입
- Jenkins `e102-dev-deploy` 잡에서 `develop` checkout, compose config 검증, backend build/deploy, `/v3/api-docs` smoke test 성공
- GraphHopper는 S1 dev stack에 포함한다. S1에서는 기존 `8989` 포트 충돌을 피하기 위해 host `8998/8999`로 바인딩한다.

### `docker-compose.prod.yml`

EC2 운영 환경에서 실행할 서비스만 둔다.

포함 대상:

- Spring Boot WAS
- AI Flask intent server
- 관리자 웹
- GraphHopper runtime은 `graphhopper` profile의 `graphhopper-blue`/`graphhopper-green` slot으로 실행한다.

별도 운영도구:

- Jenkins는 현재 `S1`의 별도 compose stack으로 운영한다.
- `Grafana`, `Portainer`, `SonarQube`, `PLG`는 `S1`에서 별도 운영 stack으로 구성한다.
- `Portainer`는 외부 Nginx routing 대상에서 제외하고, S1 localhost bind와 SSH 터널로만 접근한다.
- `S2`는 primary prod runtime 전용으로 유지한다.
- prod backend와 AI 포트는 host loopback(`127.0.0.1`)에만 bind하고, 외부 접근은 S2 Nginx의 `api`, `ai` host routing으로만 받는다.
- dev/prod backend의 actuator는 앱 포트와 분리된 내부 관리 포트(`MANAGEMENT_SERVER_PORT`, 기본 `18080`)를 host loopback에 bind하고, 외부 Nginx 경로에는 연결하지 않는다.
- `S1` monitoring stack은 `dev backend`, `dev redis`, `host/container` 지표를 직접 수집한다.
- `S2`는 `INF/monitoring/s2` 템플릿으로 promtail을 배치해 `prod` Docker 로그를 `S1 Loki`로 보낸다.
- `prod` 상태 카드는 S1 `blackbox-exporter`가 `api/ai/admin`과 backend가 공개하는 `db/redis` health endpoint를 점검한다.
- `prod`의 상세 JVM/Hikari metric은 현재 S1에서 직접 scrape 하지 않는다.
- prod Redis는 ElastiCache AUTH token을 사용하므로 `.env.prod`의 `REDIS_PASSWORD`, `REDIS_SSL=true`를 관리한다.
- prod backend는 현재 `JPA_DDL_AUTO=update`를 사용한다. `BE/src/main/resources/schema.sql`이 `postgis` extension 생성과 `bookmarks.place_id` nullable 보정을 함께 수행한다.
- 따라서 prod DB 최초 bootstrap 계정은 `CREATE EXTENSION IF NOT EXISTS postgis;`를 실행할 수 있어야 한다. 권한이 없다면 운영 DB에서 extension을 사전 활성화해야 한다.
- prod compose의 일반 app 실행 대상은 `backend + ai + admin`이다.
- GraphHopper runtime은 `--profile graphhopper`로 `blue/green` slot을 함께 실행하며, Jenkins prod deploy의 운영 기본값은 `DEPLOY_GRAPHHOPPER=true`다.
- GraphHopper cache를 매 app 배포마다 새로 만들지는 않는다. 일반 app 배포는 `BUILD_GRAPHHOPPER=false`를 기본값으로 두고, `e102-graphhopper-refresh`가 inactive slot cache를 주기적으로 갱신한다.
- `S1` 자원이 부족하면 운영도구는 중지하거나 자원 제한을 걸 수 있어야 한다.

제외 대상:

- RDS: 운영에서는 AWS managed RDS 사용
- ElastiCache: 운영에서는 AWS managed ElastiCache 사용
- S3: 운영에서는 AWS S3 사용
- GraphHopper import/build heavy job: 운영 active runtime slot에서 직접 수행하지 않음

#### Prod GraphHopper Blue/Green

운영 GraphHopper는 단일 `graphhopper` service가 아니라 아래 두 runtime으로 분리한다.

```text
graphhopper-blue      -> graphhopper-prod-blue-data
graphhopper-green     -> graphhopper-prod-green-data
graphhopper-candidate -> graphhopper-prod-candidate-data
```

`graphhopper-candidate`는 refresh 중 임시 검증용 runtime이다. 긴 import 단계는 candidate volume에서 수행하고, blue/green 기존 runtime은 계속 살아 있게 둔다. candidate profile smoke가 통과한 뒤에만 대상 blue/green slot으로 cache를 복사한다.

backend는 고정 service name을 직접 보지 않고 Redis slot key를 매 요청 조회한다.

| Key | 의미 |
|---|---|
| `graphhopper:active-slot` | 현재 backend가 우선 호출할 slot |
| `graphhopper:previous-slot` | active 실패 시 1회 fallback할 slot |
| `graphhopper:active-build-id` | 현재 active graph-cache build id |
| `graphhopper:blue:url` | compose 내부 blue URL |
| `graphhopper:green:url` | compose 내부 green URL |

Jenkins `e102-graphhopper-refresh`가 3시간마다 inactive slot을 갱신한다. refresh 시작 시 active slot이 꺼져 있거나 healthcheck에 실패하면 start/restart로 먼저 self-heal한다. active self-heal이 실패하고 previous slot이 건강하면 Redis active slot을 previous로 failover한 뒤 candidate rebuild를 계속 진행한다.

candidate runtime의 import, healthcheck, profile smoke가 모두 성공해야 대상 blue/green slot publish를 진행한다. 최종 publish 동안에는 Redis previous slot URL을 임시 candidate runtime으로 돌려 active 장애 시 fallback 구멍을 줄인다. publish 전 target slot cache를 snapshot하고, target slot 검증 전 publish 단계가 실패하면 snapshot restore 후 Redis previous fallback을 원복한다. target restore 또는 Redis 원복이 실패할 때만 임시 candidate runtime을 previous fallback으로 남겨둔 채 실패 처리하고 Jenkins Mattermost 알림으로 운영자 확인을 요청한다. 대상 slot health/profile smoke와 Redis active-slot 검증이 모두 성공해야 active slot을 전환한다. switch 이후 실패는 기존 active slot으로 rollback 검증을 수행한다. 전환 후 old slot은 fallback을 위해 drain window 동안 유지한다.

외부 공개 `https://api.busaneumgil.com/graphhopper/healthcheck`는 backend의 `/health/graphhopper`로 연결하며 Redis active slot 기준 GraphHopper 상태를 본다. 슬롯별 원시 상태는 `https://api.busaneumgil.com/graphhopper-blue/healthcheck`, `https://api.busaneumgil.com/graphhopper-green/healthcheck`로 분리한다.

prod shell smoke와 GraphHopper refresh script는 `.env.prod`에 `REDIS_SSL`이 없으면 backend compose 기본값과 동일하게 TLS Redis 접속을 기본값으로 사용한다. 로컬/비 TLS Redis만 예외적으로 `REDIS_SSL=false`를 명시한다.

## 5. Profile 기준

compose 파일은 service name 직접 실행과 profile 실행을 모두 허용한다.

공유 의존성 서비스는 `docker-compose.shared.yml`에 둔다.
현재는 `docker-compose.local.yml`과 `docker-compose.dev.yml`이 이를 `extends`로 재사용한다.

공유 의존성 서비스는 profile을 붙이지 않는다.
그래야 `docker compose up -d` 또는 `make dev-up`만으로 dev 서버의 기본 개발 의존성 스택이 올라간다.

profile은 backend, AI/ETL, monitoring처럼 선택적으로 추가되는 애플리케이션/운영 서비스에만 붙인다.
현재 적용 파일에서는 profile보다 파일 분리(`local`, `dev`, `prod`)와 service name 실행이 더 중요한 기준이다.
고정 `container_name`은 사용하지 않는다. Compose project name과 service name이 컨테이너 이름을 만들도록 두어 `local`, `dev`, 임시 검증 실행이 서로 충돌하지 않게 한다.

## 6. 실행 명령 기준

직접 실행도 가능하지만, 팀원은 가능하면 `make`를 사용한다.

직접 실행 예시:

```powershell
docker compose --env-file .env.dev -f docker-compose.dev.yml up -d
make be-dev-up
docker compose --env-file .env.local -f docker-compose.local.yml up -d
docker compose --env-file .env.local -f docker-compose.local.yml up -d postgres redis minio minio-init graphhopper backend
```

Makefile 기준:

```bash
make local-config
make local-up
make local-down
make local-logs
make dev-config
make dev-up
make dev-down
make be-dev-config
make be-dev-up
make be-dev-down
make be-dev-logs
make be-local-up
make ai-local-up
make dev-logs
```

## 7. 운영 원칙

- compose 파일은 실행 기준이고, 상세 설명은 `Docs/인프라`에 둔다.
- 비밀 값은 compose에 직접 쓰지 않는다.
- `.env.dev`, `.env.prod`, `.env.local`은 git에 올리지 않는다.
- 공유 가능한 env key 목록은 노션에서 관리하며, 저장소에는 env example 파일을 두지 않는다.
- 운영용 compose에는 managed service인 RDS, ElastiCache, S3를 컨테이너로 올리지 않는다.
- profile은 편의를 위한 선택 실행 단위이고, 실제 서비스 의존성은 명시적으로 관리한다.
- 경로 엔진의 import/build 작업은 runtime compose와 분리한다.
- PostgreSQL LineString은 graph-cache build의 원천 데이터이며, GraphHopper runtime은 생성된 cache를 serve only로 사용한다.

## 8. 현재 적용 상태

현재 적용된 초안:

- `docker-compose.local.yml`
  - `docker-compose.shared.yml`의 공유 의존성을 `extends`로 재사용
  - `backend`
  - `graphhopper`
  - `graphhopper-build` profile
  - `postgres`, `redis`, `minio`, `minio-init`
- `docker-compose.dev.yml`
  - `docker-compose.shared.yml`의 공유 의존성을 `extends`로 재사용
  - `backend`
  - `ai`
  - `graphhopper`
  - `graphhopper-build` profile
  - `postgres`, `redis`, `minio`, `minio-init`
- `scripts/make/lib/be-dev.sh`
  - 개발자 PC에서 backend image를 직접 build/run
  - remote dev DB/Redis/MinIO/GraphHopper endpoint 참조 전제
  - `make be-dev-up`은 WAS 로그를 follow하고, `make be-dev-down`은 자동으로 연 SSH tunnel을 함께 종료
- `docker-compose.shared.yml`
  - `postgres`
  - `redis`
  - `minio`
  - `minio-init`
- `docker-compose.prod.yml`
  - prod EC2-hosted runtime only
  - 일반 app 실행: `backend`, `ai`, `admin`
  - 운영 GraphHopper 실행: `graphhopper` profile의 `graphhopper-blue`, `graphhopper-green`
  - graph-cache 생성: `graphhopper-build` profile
  - RDS/ElastiCache/S3 같은 managed resource와 S1 운영도구는 포함하지 않음
- `.env.dev`, `.env.prod`
  - 루트에 존재하지만 `.gitignore`의 `.env.*` 규칙으로 무시됨
- `.env.local`
  - 루트에 존재하지만 `.gitignore`의 `.env.*` 규칙으로 무시됨
  - 기본값은 `backend -> postgres/redis/minio/graphhopper` 컨테이너 네트워크 기준으로 맞춘다.
  - GraphHopper는 PostgreSQL LineString 원천 데이터로부터 생성된 graph-cache를 참조하는 방향으로 관리한다.
- `S1` dev 서버
  - `PostGIS`, `Redis`, `MinIO`, `backend`, `ai`, `GraphHopper` 실행 확인
