## 📌 핵심 원칙 (꼭 기억해주세요!)

| 원칙 | 설명 |
| :--- | :--- |
| ❌ **서브태스크 사용 금지** | 서브태스크 대신 **연결된 업무 항목(Linked Issues)** 으로 관계를 맺습니다. |
| 📝 **스토리 내용 생략** | 스토리 본문은 비워두고, 상세 내용은 각 작업(Task)에 작성합니다. |

---

## 1️⃣ 이슈 계층 구조

```
Epic (에픽)
 └── Story (스토리)  ──연결──  Task (작업)
                      연결──  Task (작업)
                      연결──  Task (작업)
```

| 유형 | 역할 | 담당자 |
| :--- | :--- | :--- |
| **에픽 (Epic)** | 프로젝트의 큰 기능 범주 (로드맵 관리용) | - |
| **스토리 (Story)** | 사용자에게 제공되는 독립적인 기능 단위 | 대표 담당자 1인 |
| **작업 (Task)** | 스토리를 완료하기 위한 세부 업무 | 실무 담당자 |

> **💡 Tip:** 스토리와 작업은 부모-자식이 아니라 **`연결된 업무 항목`** 으로 관계를 맺습니다.

---

## 2️⃣ 이슈 유형별 작성 규칙

### 🟣 에픽 (Epic)

- **제목 형식:** `[기능 명칭]`
- **예시:** `결제 시스템`, `회원 인증`

---

### 🟢 스토리 (Story)

| 항목 | 규칙 |
| :--- | :--- |
| **제목** | `[파트] 기능 명칭` (예: `[FE] 로그인 페이지 구현`) |
| **내용** | ⚠️ **생략합니다.** 상세 내용은 연결된 Task에서 관리합니다. |
| **담당자** | 해당 기능을 총괄하는 대표 담당자 1인 |
| **스토리 포인트** | 피보나치 기준(1, 2, 3, 5, 8)으로 산정 |
| **연결** | 관련 Task 티켓을 모두 `연결된 업무 항목`으로 연결 |

---

### 🔵 작업 (Task)

| 항목 | 규칙 |
| :--- | :--- |
| **제목** | `[파트] 상세 업무 내용` (예: `[BE] JWT 토큰 발급 로직 구현`) |
| **내용** | MR(Merge Request)과 연동됩니다. 작업 완료 후 MR 링크를 기재합니다. |
| **담당자** | 실제 업무를 수행하는 실무 담당자 |
| **스토리 포인트** | 해당 개별 작업의 예상 포인트 |

---

## 3️⃣ 우선순위 기준

스토리와 작업 **모두** 필수로 지정합니다.

| 우선순위 | 의미 |
| :--- | :--- |
| 🔴 **Highest** | 프로젝트의 핵심 MVP — 이거 없으면 안 됨 |
| 🟠 **High** | 중요한 기능이지만 MVP 다음 순위 |
| 🟡 **Medium** | 일반적인 개발 업무 (기본값) |
| 🟢 **Low** | 급하지 않은 개선 사항, 리팩토링 |
| ⚪ **Lowest** | 안 구현해도 됨 |

---

## 4️⃣ 스토리 포인트 산정 기준

피보나치 수열 **`1, 2, 3, 5, 8`** 을 기준으로 산정합니다.

| 포인트 | 난이도 / 소요 시간 |
| :---: | :--- |
| **1~2** | 단순 수정, 반나절 내 완료 가능 |
| **3** | 하루 정도 집중 필요 |
| **5** | 난이도 높음, 2~3일 이상 소요 |
| **8** | 일주일 규모 → ⚠️ **스토리를 더 잘게 쪼개는 것을 권장** |

## 5️⃣ 실전 예시

> 아래 예시로 실제 지라에서 어떻게 구성하는지 참고해주세요.

| 유형 | 제목 | 담당자 | 포인트 | 우선순위 | 연결 관계 |
| :--- | :--- | :--- | :---: | :--- | :--- |
| **Epic** | 결제 시스템 구축 | 기획자 | - | Medium | - |
| **Story** | `[BE] 카드 결제 API 개발` | **BE 리드** | **3** | High | Task 1, 2와 연결 |
| **Task 1** | `[BE] PG사 연동 모듈 개발` | BE 팀원 A | 2 | High | Story와 연결 |
| **Task 2** | `[BE] 결제 내역 DB 스키마 설계` | BE 팀원 B | 2 | Medium | Story와 연결 |

## 6️⃣ 운영 규칙 요약

1. **스토리 내용은 비워둡니다** → `연결된 이슈`로 전체 진행 상황을 파악합니다.
2. **개발 완료 및 MR(Merge Request)** 은 **작업(Task) 단위**로 진행합니다.
3. 연결된 **모든 Task가 완료**되면 → 스토리를 `Done`으로 변경합니다.
4. 스토리 포인트 합산이 **13pt 이상**이면 → 스토리를 분리하여 관리합니다.
