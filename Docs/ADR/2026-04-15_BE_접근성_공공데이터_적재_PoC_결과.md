## 3. 이슈별 결론

### 3.1 `S14P31E102-141`

부산 접근성 공공데이터 적재 가능성 검증 자체는 성공했다.

- 장소성 데이터 4종은 `places`/`place_accessibility_features` 방향으로 검토 가능하다.
- GIS 데이터 2종은 `road_segments` 보강 방향으로 검토 가능하다.
- CSV와 API 모두 실제 확보 경로를 확인했다.

### 3.2 `S14P31E102-149`

`places`, `place_accessibility_features` 적재는 **가능(제약 있음)** 이다.

가능한 범위:

- 좌표가 있는 장소성 데이터는 `places`로 적재할 수 있다.
- 접근성 여부는 `featureType`, `isAvailable`로 적재할 수 있다.
- 전동휠체어 급속충전기는 `places.category=ETC`와 `chargingStation=true` feature로 deep load할 수 있다.
- 한국관광공사 무장애 여행 정보는 `TOURIST_SPOT`, `FOOD_CAFE`, `ACCOMMODATION` 카테고리로 바로 분기 가능하며, 산출물도 카테고리별로 분리할 수 있다.

선결 조건:

- 좌표가 없거나 불량인 row 처리 정책
- 원천 편의시설 항목의 내부 `featureType` 정규화
- 중복 장소 판단 기준
- 상세 속성 보존 방식 결정
- 관광 API 원문 문장(`route`, `publictransport`, `exit`)의 rule 기반 해석 기준 확정

### 3.3 `S14P31E102-150`

GIS 데이터의 PostGIS 적재는 **가능**하다.

검증한 내용:

- `geometry(Point, 4326)` 생성 가능
- GIST index 생성 가능
- `ST_DWithin` 기반 공간 join 가능
- `stg_crosswalks`, `stg_audio_signals` staging 구조로 재현 가능
- 최신 ERD v2 기준으로 상태 컬럼 계산 및 `segment_features` 적재 방향으로 연결 가능

핵심 수치:

- 횡단보도 3,815건 적재, point 3,815건 생성
- 음향신호기 3,060건 적재, point 3,060건 생성
- 횡단보도-음향신호기 공간 join은 15m 기준 pair 179건, 10m 기준 pair 49건 확인

즉, `road_segments.geom`이 준비되면 같은 방식으로 `crossingState`, `audioSignalState` 계산과 `segment_features` 적재 실험을 이어갈 수 있다.

### 3.4 `S14P31E102-151`

이번 PoC의 결과 정리는 완료했다.

- 데이터셋별 채택/보류/킵 구분 완료
- deep load 우선순위 정리 완료
- 운영 반영 전 필요한 정책 항목 정리 완료

## 5. 현재 ERD 기준 판단

### 5.1 `places`

현재 ERD 기준으로 아래 값은 안정적으로 적재할 수 있다.

- 장소명
- 카테고리
- 주소
- 좌표
- 원천 식별자

즉, 좌표가 확보되는 데이터셋은 `places` 적재가 가능하다.

### 5.2 `place_accessibility_features`

현재 ERD 기준으로 아래 값은 안정적으로 적재할 수 있다.

- `placeId`
- `featureType`
- `isAvailable`

즉, “해당 장소에 특정 접근성 feature가 있는지”는 저장할 수 있다.

### 5.3 현재 ERD에서 바로 안 담기는 것

아래 정보는 현재 ERD에서 손실된다.

- 출입구번호
- 상세위치
- 층 정보
- 수량
- 장치 정원/중량
- 게이트 내외
- 남/여/공용 구분
- 원천별 상세 라벨 원문

따라서 도시철도 접근성 상세 정보처럼 상세도가 높은 데이터는 아래 둘 중 하나가 필요하다.

1. 별도 detail table 추가
2. 원천별 extension table 추가

> 2026-04-28 최종 정책으로 아래 선택지는 폐기하고, `Docs/ADR/2026-04-28_BE_장소_카테고리_및_접근성_필터_정책.md`를 따른다. 최종 서비스 카테고리는 `FOOD_CAFE`, `TOURIST_SPOT`, `ACCOMMODATION`, `HEALTHCARE`, `WELFARE`, `PUBLIC_OFFICE`, `ETC` 7개다. `TOILET`, `CHARGING_STATION`은 장소 카테고리가 아니라 접근성 속성으로 관리한다.

| 버전 | 파일 | 기준 | 카테고리 | 장점 | 한계 |
| --- | --- | --- | --- | --- | --- |

## 7. 결론

이번 PoC의 결론은 아래 한 줄로 정리할 수 있다.

> 부산 접근성 공공데이터는 서비스에 활용 가능하지만, 운영 적재 전에는 좌표 보정, 라벨 정규화, 중복 기준, 상세정보 저장 정책을 먼저 확정해야 한다.

핵심은 다음과 같다.

- `places`와 `place_accessibility_features` 적재는 가능하되 일부 데이터셋은 전처리가 선행되어야 한다.
- 한국관광공사 무장애 여행 정보는 부산 251건 전량 수집이 가능했고, 관광지/음식·카페/숙박 장소를 카테고리별 산출물로 분리한 뒤 접근성 feature를 함께 적재할 수 있다.
- 장소 최종 산출물은 2026-04-28 정책에 따라 서비스 카테고리 7개와 접근성 속성으로 재정리한다.
- GIS 데이터는 PostGIS 적재와 공간 join이 가능하므로 `road_segments` 상태값 계산과 `segment_features` 적재 후보로 충분하다.
- `점자블록 유무`는 원본 공공데이터만으로 직접 확인할 수 없으므로, 후속 단계에서는 횡단보도 원천 row에 `점자블록 유무` 보강 필드를 추가한 팀 가공 데이터셋 입력을 검토한다.
- 가장 먼저 구현할 데이터는 전동휠체어 급속충전기와 부산 횡단보도/음향신호기다.
