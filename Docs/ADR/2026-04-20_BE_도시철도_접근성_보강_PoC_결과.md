## 1. 결과 요약

이번 PoC는 외부 API로 조회한 부산 도시철도 역 검색 결과에 엘리베이터, 장애인화장실 접근성 정보를 내부 데이터로 보강할 수 있는지 검증하는 것을 목표로 한다.

ERD v3에 반영된 `subway_station_elevators`, 계단 관련 공공데이터 확보 가능성도 함께 정리한다. 저상버스 노선 정적 테이블은 대중교통 길 안내에서 이번 범위에 사용하지 않는 것으로 결론 내린다.

2026-04-28 최신 장소 정책 기준으로 도시철도 역은 `places.category`의 서비스 카테고리 7개에 포함하지 않는다. 따라서 이 PoC의 결론은 `places.category` 추가가 아니라, 외부 역 검색 결과 또는 대중교통 도메인 데이터에 `elevator`, `accessibleToilet` 같은 접근성 정보를 보강할 수 있다는 의미로 해석한다.

최종 판단은 아래와 같다.

| 항목 | 결론 | 요약 |

## 3. 역명 매칭 결과

### 3.1 정규화 규칙

1차 정규화 규칙은 아래와 같다.

- 역명 뒤 `역`, `정거장` 접미어 제거
- 괄호 안 노선/부가 표기 제거
- 공백, 특수문자 차이 정규화
- 외부 API 결과와 원천 데이터 간 대표 alias 수동 매핑

### 3.2 alias 필요 역

기존 검토에서 alias가 필요했던 대표 역은 아래와 같다.

| 현재/원천 표기 | 매칭 기준 | 비고 |
| --- | --- | --- |
| 벡스코 | 시립미술관 | 2021 역사정보 원천의 옛 역명으로 `stationId`를 매칭하고, 산출물 `stationName`은 벡스코로 유지한다. |
| 국제금융센터 | 국제금융센터부산은행 | 현행 역명 기준으로 보정한다. |
| 미남광장 | 미남 | 출입구/역 좌표 매칭용 alias로만 사용한다. |
| 반여 | 반여농산물시장 | 출입구/역 좌표 매칭용 alias로만 사용한다. |

### 3.3 매칭 결론

역명 alias를 적용하면 역 단위 집계는 가능하다.

- 역 place 후보 114건 기준으로 집계 가능성을 확인했다.
- 역명 매칭 실패는 0건이었다.
- 따라서 외부 API 역 검색 결과와도 동일한 alias 규칙만 있으면 매핑 가능성이 높다.

## 4. 접근성 집계 결과

### 4.1 엘리베이터

엘리베이터 2종 데이터는 모두 역 단위 `elevator` feature 후보로 집계할 수 있다.

| 항목 | 값 |
| --- | ---: |
| 부산교통공사 엘리베이터 원천 row | 448 |
| 부산교통공사 엘리베이터 unique detail | 438 |
| 국가철도공단 엘리베이터 원천 row | 447 |
| 국가철도공단 엘리베이터 unique detail | 437 |
| 엘리베이터 feature가 붙는 역 | 113 |
| 부산교통공사 엘리베이터 원천 point 제공 row | 0 |
| 단일 출입구 geocoding 성공 row | 231 |
| `subway_station_elevators` 초기 적재 row | 231 |
| 복수 출입구/대표역 fallback 제외 row | 217 |

집계 결론:

- 역 단위로 `elevator=true` 보강 가능
- 수량도 역별 집계 가능
- `subway_station_elevators`에는 단일 출입구 좌표를 확인한 231건만 초기 적재한다.
- 복수 출입구 평균 좌표, 역 대표 좌표 fallback, 상세위치만 있는 레코드는 운영 적재에서 제외한다.
- 다만 시작층/종료층, 정원/중량은 현재 ERD에 직접 담기 어렵다

### 4.2 장애인화장실

장애인화장실 데이터는 역 단위 `accessibleToilet` feature 후보로 집계할 수 있다.

| 항목 | 값 |
| --- | ---: |
| 장애인화장실 원천 row | 238 |
| 장애인화장실 unique location | 120 |
| 장애인화장실 feature가 붙는 역 | 114 |

집계 결론:

- 역 단위로 `accessibleToilet=true` 보강 가능
- 게이트 내외, 층, 출구번호, 상세위치 같은 상세 정보는 별도 보존 전략이 필요하다

### 4.3 중복 및 보완 관계

엘리베이터 2종은 단순 중복이라기보다 보완 관계로 보는 것이 맞다.

| 비교 항목 | 결과 | 해석 |
| --- | --- | --- |

## 5. 외부 API 매핑 판단

외부 API 역 검색 결과에 내부 접근성 데이터를 붙일 수 있는지에 대한 판단을 정리한다.

결론은 **가능**이다.

가능한 이유:

- 외부 API는 역 기본 정보와 좌표를 제공할 수 있다.
- 내부 원천 데이터는 역명 기준으로 엘리베이터/장애인화장실 여부를 집계할 수 있다.
- 소수 alias 규칙만 정리하면 역 검색 결과에 접근성 정보를 붙일 수 있다.

전제:

- 역명 정규화 규칙을 공통으로 써야 한다.
- 환승역을 역 단위로 묶을지, 노선별로 유지할지 정책이 필요하다.
- 상세 정보는 boolean 보강과 별도로 다뤄야 한다.

## 6. ERD v3 추가 항목 판단

### 6.1 `subway_station_elevators`

결론은 **가능(제약 있음)** 이다.

- 공식 원천은 `부산교통공사_엘리베이터 정보`, `국가철도공단_부산교통공사_엘리베이터`로 확보 가능하다.
- `stationName`, `lineName`, `entranceNo`, `point`를 채울 수 있는 초기 적재 대상은 231건이다.
- 원천 파일의 `point`는 전부 비어 있으므로 Kakao Local 출입구 geocoding 결과를 보조 좌표로 사용한다.
- 좌표 정책은 단일 출입구 geocoding 성공 건만 채택하는 방식으로 고정한다.

즉, 테이블 도입과 초기 적재는 가능하지만 적재 범위는 제한한다.

- `stationId`, `stationName`, `lineName`, `entranceNo`, `point`만 산출물에 포함한다.
- 복수 출입구 평균 좌표는 실제 엘리베이터 위치라고 보기 어려우므로 제외한다.
- 역 대표 좌표 fallback도 개별 엘리베이터 위치가 아니므로 제외한다.
- `isOperating`은 원천 근거가 부족하고 ERD에서도 제거한다.

### 6.2 저상버스 노선 정적 테이블

결론은 **이번 ERD에서 제외**다.

- 부산 공식 시스템과 OpenAPI는 존재한다.
- 저상버스 운영 자체에 대한 공식 근거도 있다.
- 다만 PoC 결과 대중교통 길 안내에서 부산 노선별 정적 카탈로그를 직접 사용하지 않기로 했으므로 별도 테이블을 두지 않는다.

현재 남은 쟁점은 아래와 같다.

- 부산 BIMS 실시간/노선 API에서 저상 여부 필드를 어디서 읽을지 확정이 필요하다.
- 부산시버스정보시스템 교통약자 기능은 앱 외부 연결 또는 실시간 조회 보조 수단으로만 검토한다.
- 즉, 이번 ERD에는 별도 테이블을 두지 않고 외부 API 또는 외부 화면 연결로 처리한다.

### 6.3 계단 관련 공공데이터

결론은 **가능**이다.

- 에스컬레이터, 휠체어리프트, 역사 편의시설현황, 대체 이동 경로처럼 계단 대체/보조 수직이동 관련 공식 소스는 확보 가능하다.
- 보행 네트워크나 역사 내부의 `계단 위치 그 자체`를 직접 제공하는 공식 데이터는 현재 확보하지 못했다.

즉, 이번 단계에서 가능한 것은 아래다.

- 도시철도 내부 수직이동 접근성 보강
- 계단 대체 경로/보조 설비 확인

이번 단계에서 아직 어려운 것은 아래다.

- 계단 위치를 정밀 geometry로 적재하는 것

### 6.4 동해선/부산김해경전철 커버리지

현재 PoC에서 사용한 부산교통공사 원천은 1~4호선 중심이다.

- 동해선/부산김해경전철 엘리베이터 정보는 현재 231건 산출물에 포함하지 않는다.
- 동해선은 한국철도공사 원천에서 역별 승강설비 현황을 확인할 수 있다.

- 다만 두 노선 모두 현재 확인한 공개 원천만으로는 개별 엘리베이터 `point`를 바로 채우기 어렵다.
- 추가 반영 시 확인 대상은 역 목록, 역별 엘리베이터 수, 출입구번호, 좌표 제공 여부, 환승역 alias 정책이다.

## 7. 현재 ERD v3 기준 판단

### 7.1 현재 ERD로 가능한 범위

- `featureType=elevator`
- `featureType=accessibleToilet`
- `isAvailable=true/false`
- `subway_station_elevators.stationId`
- `subway_station_elevators.stationName`
- `subway_station_elevators.lineName`
- `subway_station_elevators.entranceNo`
- `subway_station_elevators.point`

### 7.2 현재 ERD에서 손실되는 정보

- 상세위치
- 층 정보
- 게이트 내외
- 정원/중량
- 수량

### 7.3 결론

현재 ERD 기준으로는 **가능(제약 있음)** 이다.

- `featureType=elevator`, `featureType=accessibleToilet`는 바로 표현 가능하다.
- `isAvailable=true` 형태의 역 단위 보강도 가능하다.
- `subway_station_elevators`는 단일 출입구 좌표를 확인한 231건만 초기 적재 가능하다.
- 그러나 층, 게이트 내외, 수량, 정원/중량은 손실된다.

따라서 도시철도 접근성 데이터를 제대로 살리려면 아래 중 하나가 필요하다.

1. 도시철도 접근성 detail table 추가
2. 원천별 extension table 추가

## 8. 최종 판단

이번 PoC의 최종 판단은 **가능(제약 일부 존재)** 다.

핵심은 아래와 같다.

- 지하철역 자체를 내부 기준 데이터로 관리하는 것도 기술적으로 가능하지만, MVP 장소 카테고리에는 포함하지 않는다. 기본 방향은 외부 API 역 검색 결과에 접근성 정보를 보강하는 것이다.
- 엘리베이터/장애인화장실 접근성 정보는 내부 데이터로 충분히 보강 가능하다.
- 에스컬레이터, 휠체어리프트, 역사 편의시설현황도 추가 접근성 보강 후보로 활용 가능하다.
- 역명 alias 규칙만 정리하면 외부 API 검색 결과에 매핑할 수 있다.
- 현재 ERD로도 `elevator`, `accessibleToilet` boolean 보강은 가능하다.
- 다만 층, 상세위치, 게이트 내외, 수량, 정원/중량까지 살리려면 detail 구조가 추가로 필요하다.
- `subway_station_elevators`는 좌표 정책을 단일 출입구 geocoding으로 고정해 231건부터 적재를 시작할 수 있다.
- 복수 출입구 평균 좌표, 역 대표 좌표 fallback, 상세위치만 있는 레코드는 이번 운영 적재에서 제외한다.
- 동해선/부산김해경전철은 현재 부산교통공사 원천 산출물에 포함하지 않는다. 역별 승강설비/편의시설 원천은 확인했지만 개별 엘리베이터 좌표 원천은 추가 확인이 필요하다.
- 저상버스 노선 정적 테이블은 ERD v3에서 제외한다. 필요한 경우 부산버스정보시스템 OpenAPI 또는 외부 화면 연결로 처리한다.

즉, 이번 PoC는 아래 방향을 뒷받침한다.

> 지하철역 접근성은 장소 카테고리가 아니라 대중교통/도시철도 보강 데이터로 다루며, 핵심은 외부 API 역 검색 결과와 내부 접근성 데이터를 같은 역명 정규화 규칙으로 연결하는 것이다.
