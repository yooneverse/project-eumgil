## 1. 목적

이 문서는 보행 네트워크를 보강하기 위해 연속수치지형도 핵심 레이어와 부산 공공데이터 가공본을 함께 검토하고, 어떤 속성을 우리 ERD v3에 연결할 수 있는지 정리한 결과 문서다.

이번 PoC의 판단 기준은 아래와 같다.

- 보행 경로 탐색과 접근성 판단에 직접 쓰이는가
- 원천 필드를 우리 ERD 필드로 해석할 수 있는가
- 지금 단계에서 바로 적재 가능한가, 아니면 계산/가공이 필요한가

## 1.1 한눈에 보는 결론

이번 검토 기준으로 보면, 팀원이 원하는 핵심 요소 중 대부분은 현재 조합으로 충족 가능하다.

### 지금 바로 가능

| 항목 | 근거 데이터 | 판단 |
|---|---|---|
| 경사도(%) | 연속수치지형도 `등고선`, `표고점` | 가능 |
| 인도 폭(m) | 연속수치지형도 `인도(보도)` | 가능 |
| 바닥 재질 타입 | 연속수치지형도 `인도(보도)` | 가능 |
| 횡단보도 여부 | `stg_crosswalks_ready.csv` | 가능 |
| 신호기 여부 | `stg_crosswalks_ready.csv`, 연속수치지형도 `신호등` | 가능 |
| 음향신호기 여부 | `stg_audio_signals_ready.csv` | 가능 |

### 가능 예정

| 항목 | 근거 데이터 | 판단 |
|---|---|---|
| 점자 블록 여부 | 팀원 추가 예정 횡단보도 데이터 | 횡단보도 쪽 데이터에 점자 블록 여부 필드를 추가해 활용 예정 |

### 부분 가능 또는 추가 데이터 필요

| 항목 | 근거 데이터 | 판단 |
|---|---|---|
| 단차 | 연속수치지형도 `성/절토`, `옹벽`, 고도 변화 | 직접 필드는 없고 위험 구간 추정 수준 |
| 엘리베이터 여부 | 현재 조합엔 없음 | `road_segments`에서는 관리하지 않고 도시철도/장소 접근성 데이터에서 별도 관리 |

팀원이 바로 기억해야 할 핵심은 아래 한 줄이다.

> **현재 조합으로 경사도, 인도 폭, 바닥 재질, 횡단보도, 신호기, 음향신호기는 가능하고, 점자 블록은 추가 예정 데이터로 보강하며, 단차는 별도 판단이 필요하다. 엘리베이터는 보행 segment가 아니라 도시철도/장소 접근성 데이터로 분리한다.**

## 5. ERD v3 매핑 검토

### 5.1 요약표

| 레이어 | 1차 적재 대상 | 핵심 매핑 | 판단 |
|---|---|---|---|
| 인도(보도) | `road_segments` | `widthMeter`, `widthState`, `surfaceState`, `walkAccess` 보강 | 가능 |
| 횡단보도 | `segment_features` + `road_segments` | `featureType=CROSSWALK`, `crossingState` 판단 근거, 폭/길이/면적 보강 | 가능 |
| 신호등/음향신호기 | `segment_features` + `road_segments` | 음향신호기는 `featureType=AUDIO_SIGNAL`, 신호등은 `crossingState=SIGNALIZED`, `audioSignalState` 판단 근거 | 가능 |
| 계단 | `segment_features` + `road_segments` | `featureType=STAIRS`, `stairsState=YES` 판단 근거 | 가능 |
| 육교 | staging/후속 가공 | 최종 `segment_features.featureType` 후보에는 포함하지 않고 접근 가능 여부를 후속 detail 구조에서 검토 | 가능(제약 있음) |
| 등고선 | staging/계산용 | `avgSlopePercent` 계산 원천 | 가능(계산 필요) |
| 표고점 | staging/계산용 | `avgSlopePercent` 계산 보정 | 가능(계산 필요) |

### 5.1.1 팀용 빠른 해석

- **직접 매칭 가능:** 인도, 횡단보도, 신호기, 음향신호기, 계단
- **가공 후 매칭:** 육교, 등고선, 표고점
- **추가 데이터 예정:** 점자 블록
- **별도 데이터 필요:** 엘리베이터

리뷰할 때는 먼저 이 구분이 맞는지만 보면 된다.

### 5.2 인도(보도)

**주 대상:** `road_segments`

| ERD 필드 | 매핑 방식 |
|---|---|
| `geom` | 인도 geometry 분할 후 사용 |
| `widthMeter` | `WIDT` 직접 사용 |
| `widthState` | `WIDT` 기준으로 threshold 분류 |
| `surfaceState` | `QUAL`을 `PAVED/UNPAVED/UNKNOWN` 계열로 단순 매핑 |
| `walkAccess` | `KIND=인도`면 보행 가능 구간으로 우선 판단 |

판단:

- 서비스 관점에서 가장 직접적인 보행 데이터다.
- `road_segments`의 base network 또는 보행 우선 구간 판단에 가장 중요하다.

### 5.3 횡단보도

**주 대상:** `segment_features`, `road_segments`

| ERD 필드 | 매핑 방식 |
|---|---|
| `segment_features.featureType` | `CROSSWALK` |
| `segment_features.geom` | 횡단보도 polygon 또는 centroid/derived line |
| 원천 상세값 | `sourceId`, `widthMeter`, `lengthMeter`, `areaSquareMeter`는 staging 또는 후속 detail 구조에 보존 |
| `road_segments.crossingState` | `stg_crosswalks_ready.csv`의 `crossingState` 사용 |

판단:

- 개별 객체 저장과 상태 계산 둘 다 가능하다.
- 이번 PoC에서는 `stg_crosswalks_ready.csv`를 우선 사용해 횡단보도 여부와 신호기 여부를 같이 가져가는 구성이 더 현실적이다.

### 5.4 신호등 / 음향신호기

**주 대상:** `segment_features`, `road_segments`

| ERD 필드 | 매핑 방식 |
|---|---|
| `segment_features.featureType` | `AUDIO_SIGNAL` |
| `segment_features.geom` | 신호등/음향신호기 point |
| 원천 상세값 | `serv`, `sourceId`, `confirmDate`는 staging 또는 후속 detail 구조에 보존 |
| `road_segments.crossingState` | 횡단보도와 공간 매칭 시 `SIGNALIZED` 판단 근거 |
| `road_segments.audioSignalState` | 음향신호기와 공간 매칭 시 `YES` 판단 근거 |

판단:

- 연속수치지형도 신호등은 공공데이터 보조 레이어로 보는 것이 맞다.
- 실제 `crossingState`는 `stg_crosswalks_ready.csv`를 우선 사용하고, 필요 시 GIS 신호등 레이어로 공간 보강한다.
- `stg_audio_signals_ready.csv`는 횡단보도 또는 도로 세그먼트와 공간 매칭한 뒤 `audioSignalState=YES` 판단 근거로 사용할 수 있다.

### 5.5 계단

**주 대상:** `segment_features`, `road_segments`

| ERD 필드 | 매핑 방식 |
|---|---|
| `segment_features.featureType` | `STAIRS` |
| `segment_features.geom` | 계단 polygon |
| 원천 상세값 | `width`, `stru`, `ufid`는 staging 또는 후속 detail 구조에 보존 |
| `road_segments.stairsState` | 계단 매칭 시 `YES` |
| `road_segments.widthMeter` | 필요 시 `WIDT` 보조 사용 |

판단:

- 교통약자 사용자 회피 규칙에 직접 연결된다.
- 현재 ERD와의 연결성이 높다.

### 5.6 육교

**주 대상:** `segment_features`

| ERD 필드 | 매핑 방식 |
|---|---|
| 후속 detail 또는 staging | 육교 polygon |
| 원천 상세값 | `type`, `length`, `width`, `height`는 staging 또는 후속 detail 구조에 보존 |
| `road_segments.walkAccess` | 선형 구간으로 풀어낸 경우에만 직접 반영 검토 |

판단:

- `TYPE`에 장애인 이용 가능/불가가 있어 서비스 관점에 유의미하다.
- 다만 원천 geometry가 면형이라서, 바로 `road_segments` 선형으로 쓰려면 추가 가공이 필요하다.
- 따라서 1차는 staging으로 보존하고, 최종 `segment_features` 후보에는 넣지 않는다.

### 5.7 등고선

**주 대상:** staging/계산용

| ERD 필드 | 매핑 방식 |
|---|---|
| `road_segments.avgSlopePercent` | 등고선 geometry와 `CONT`를 이용한 계산 |
| `segment_features.featureType` | 필요 시 `CONTOUR_REFERENCE` |

판단:

- 직접 적재보다는 경사도 계산용 원천이다.
- ETL에서 계산 후 `avgSlopePercent`만 `road_segments`에 반영하는 구조가 맞다.

### 5.8 표고점

**주 대상:** staging/계산용

| ERD 필드 | 매핑 방식 |
|---|---|
| `road_segments.avgSlopePercent` | 표고점 `NUME`로 경사도 계산 보정 |
| `segment_features.featureType` | 필요 시 `ELEVATION_POINT` |

판단:

- 등고선만으로 부족한 구간의 고도 보정용으로 적합하다.
- 직접 서비스 객체로 쓰기보다는 계산 보조 데이터다.

### 5.9 팀원 요구 항목 기준 최종 판단

| 항목 | 판단 | 근거 |
|---|---|---|
| 경사도(%) | 가능 | 등고선 + 표고점 |
| 인도 폭(m) | 가능 | 인도(보도) `WIDT` |
| 바닥 재질 타입 | 가능 | 인도(보도) `QUAL` |
| 점자 블록 여부 | 가능 예정 | 팀원이 횡단보도 데이터에 필드 추가 예정 |
| 음향신호기 여부 | 가능 | `stg_audio_signals_ready.csv` |
| 신호기 여부 | 가능 | `stg_crosswalks_ready.csv`, 신호등 레이어 |
| 횡단보도 여부 | 가능 | `stg_crosswalks_ready.csv`, 횡단보도 레이어 |
| 단차 | 부분 가능 | 성/절토, 옹벽, 고도 변화 |
| 엘리베이터 여부 | 불가 | 현재 조합에 없음 |

## 7. 서비스 핵심 1차 매핑표

이번 단계에서는 아래 데이터를 서비스 핵심 대상으로 본다.

| 레이어 | 우리 ERD 대상 | 1차 반영 필드 | 반영 방식 |
|---|---|---|---|
| 인도(보도) | `road_segments` | `walkAccess`, `widthMeter`, `widthState`, `surfaceState` | 인도 선형을 보행 가능 구간으로 해석하고 폭/재질을 상태값으로 변환 |
| 횡단보도 가공본 | `segment_features`, `road_segments` | `featureType=CROSSWALK`, `crossingState`, `widthMeter`, `lengthMeter`, `areaSquareMeter` | 횡단보도 객체를 edge에 귀속하고 신호 유무와 크기 정보를 같이 사용 |
| 음향신호기 가공본 | `segment_features`, `road_segments` | `featureType=AUDIO_SIGNAL`, `audioSignalState=YES` | 음향신호기 포인트를 횡단보도 또는 edge와 공간 매칭 |
| 계단 | `segment_features`, `road_segments` | `featureType=STAIRS`, `stairsState=YES` | 계단 객체를 edge에 귀속하고 교통약자 회피 구간으로 표시 |
| 육교 | staging/후속 detail | 최종 `segment_features` 후보에는 포함하지 않음 | 육교 객체와 장애인 이용 가능 여부 같은 상세값은 staging 또는 후속 detail 구조에 보존 |
| 등고선 | `road_segments` | `avgSlopePercent` | 등고선 수치(`CONT`)를 사용해 구간 경사도 계산 |
| 표고점 | `road_segments` | `avgSlopePercent` | 표고점 수치(`NUME`)로 등고선 기반 경사도 계산을 보정 |

### 7.1 서비스 핵심 판단

- **직접 매칭:** 인도, 횡단보도 가공본, 음향신호기 가공본, 계단
- **가공 후 매칭:** 육교, 등고선, 표고점
- **추가 보강 예정:** 점자 블록
- **별도 데이터 필요:** 엘리베이터
