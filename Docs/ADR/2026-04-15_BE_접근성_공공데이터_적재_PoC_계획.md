## 1. 목적

이번 PoC의 목적은 부산이음길 백엔드에서 접근성 공공데이터를 아래 세 계열 테이블에 적재하거나 보강할 수 있는지 검증하는 것이다.

1. `places`
2. `place_accessibility_features`
3. `road_segments`

특히 팀원 OSM 조사 결과를 기준으로, OSM만으로 채울 수 없는 접근성 필드를 공공데이터로 보강할 수 있는지 확인한다. `road_segments` 관련 판단은 최신 ERD v3 기준으로 `state enum`과 `segment_features` 적재 방향에 맞춰 해석한다.

## 3. 팀원 OSM 조사 인사이트

OSM 샘플 조사 결과, `road_segments`의 일부 필드는 OSM 태그만으로 매핑할 수 있지만 일부 필드는 공공데이터 보강이 필요하다. 아래 표는 기존 boolean 중심 판단을 최신 ERD v3의 상태 컬럼과 연결해서 다시 해석한 것이다.

| 최신 해석 대상 | OSM 기반 판단 | 공공데이터 PoC에서의 의미 |
| --- | --- | --- |
| `edgeId`, `from_node_id`, `to_node_id`, `geom`, `length_meter` | OSM 기반 생성 가능 | 공공데이터 보강 대상 아님 |
| `avgSlopePercent` | OSM만으로 계산 불가 | 경사도 PoC 범위. 이번 문서에서는 직접 검증하지 않음 |
| `width_meter`, `width_state` | `width` 태그가 있으면 일부 파싱 가능 | P0에서는 직접 보강하지 않음 |
| `stairs_state` | `highway=steps`이면 직접 판단 가능 | OSM 기반 우선 |
| `slopeState` | OSM 샘플만으로 직접 판단 불가 | 경사도 PoC 산출물과 제보/현장 데이터 후속 후보 |
| 엘리베이터 | `elevator=yes` 또는 `highway=elevator`일 때 일부 판단 가능 | `road_segments` 상태값에서는 제외. 도시철도 엘리베이터는 `subway_station_elevators`, 장소 내부 엘리베이터는 `place_accessibility_features`에서 관리 |
| `crossingState` | `crossing=*`, `traffic_signals=*`로 일부 판단 가능 | 횡단보도 공공데이터로 상태 보강 가능성 검증 |
| `audioSignalState` | OSM 샘플만으로 직접 판단 불가 | 음향신호기 공공데이터로 상태 보강 가능성 검증 |
| `braille_block_state` | `tactile_paving=yes` 또는 `contrasted` 계열 태그가 있을 때 일부 판단 가능 | 현재 P0 공공데이터로 직접 해결 어렵다. 후속 단계에서는 횡단보도 원천 row에 `점자블록 유무` 보강 필드를 추가한 팀 가공 데이터셋 입력을 검토 |
| `surface_state` | `surface` 태그를 내부 enum으로 변환 가능 | OSM 기반 우선 |
| `segment_features` | OSM/공공데이터 개별 객체를 별도 저장 가능 | 횡단보도, 음향신호기 같은 원천 객체를 edge에 귀속시키는 구조로 사용 |

## 6. places 적재 검증 기준

| ERD 컬럼 | 필수 여부 | 검증 기준 |
| --- | --- | --- |
| `placeId` | 필수 | DB에서 순차 PK 생성 가능 |
| `name` | 필수 | 원천 데이터에 시설명/업소명/화장실명 등 장소명이 있는지 확인 |
| `category` | 필수 | 내부 enum으로 분류 가능한지 확인 |

| `point` | 필수 | 위도/경도 직접 제공 또는 주소 기반 지오코딩 필요 여부 확인 |
| `providerPlaceId` | 선택 | 원천 시스템 ID 또는 `provider:normalized_key` 생성 가능 여부 확인 |

### 6.1 P0 장소성 데이터 매핑 후보

| 데이터셋 | `places.category` 후보 | 주요 리스크 |
| --- | --- | --- |

## 7. place_accessibility_features 적재 검증 기준

| ERD 컬럼 | 필수 여부 | 검증 기준 |
| --- | --- | --- |
| `id` | 필수 | DB에서 순차 PK 생성 가능 |
| `placeId` | 필수 | 적재된 `places.placeId` 참조 가능 |
| `featureType` | 필수 | 원천 편의시설 항목을 내부 `featureType`으로 매핑 가능 |

### 7.1 featureType 1차 매핑 후보

| 원천 표현 후보 | 내부 `featureType` | 비고 |
| --- | --- | --- |
| 경사로, 주출입구 접근로, 단차 없음, 무단차, 평탄 | `accessibleEntrance` | 장소 진입 접근성을 하나의 속성으로 통합 |
| 엘리베이터, 승강기 | `elevator` | 장소 내부 접근성 속성 |
| 장애인 화장실, 장애인용 대변기 | `accessibleToilet` | 공중화장실 데이터는 개수 기준으로 boolean 계산 |
| 전동휠체어 충전기, 전동보장구 충전기 | `chargingStation` | 충전소 장소 자체에도 사용 |
| 장애인 전용 주차구역 | `accessibleParking` | 장소 진입 전 접근성 속성 |
| 점자안내, 촉지안내, 음성안내, 유도안내설비 | `guidanceFacility` | 시각장애인 유도/안내 시설을 묶어서 표현 |
| 장애인 객실 | `accessibleRoom` | 숙박 계열 장소에 한해 사용 |

### 7.2 한국관광공사 무장애 여행 정보 매핑 메모

| 원천 필드 | 내부 `featureType` | 메모 |
| --- | --- | --- |
| `elevator` | `elevator` | 원문이 비어 있지 않으면 true 후보 |
| `restroom` | `accessibleToilet` | 장애인 화장실/접근 가능 화장실 문구 |
| `parking` | `accessibleParking` | 장애인 주차 가능 여부 |
| `room` | `accessibleRoom` | 숙박 계열에 한해 사용 |
| `guidesystem`, `braileblock`, `guidehuman`, `audioguide`, `bigprint`, `brailepromotion` | `guidanceFacility` | 유도/안내/점자 계열을 묶어서 사용 |
| `route`, `publictransport`, `exit` | `accessibleEntrance` | 경사로/무단차 문구를 장소 진입 접근성으로 통합 해석 |

## 8. road_segments 보강 검증 기준

`road_segments`는 원천 데이터를 바로 넣는 테이블이 아니라, OSM/GraphHopper 기반 경로 segment에 공공데이터 속성을 보강하는 대상이다.

| 공공데이터 | staging table 후보 | 보강 후보 컬럼 | 검증 방식 |
| --- | --- | --- | --- |
| 부산광역시 신호등 설치 횡단보도 위치정보 | `stg_crosswalks` | `crossingState`, `segment_features(CROSSWALK)` | `stg_crosswalks.point`와 `road_segments.geom` 공간 join |
| 부산광역시_음향신호기 현황 | `stg_audio_signals` | `audioSignalState`, `segment_features(AUDIO_SIGNAL)` | `stg_audio_signals.point`와 `road_segments.geom` 공간 join |

### 8.1 API 응답과의 연결

현재 route API segment 응답에는 아래 legacy 필드가 존재하지만 값이 `null`일 수 있다.

```json
{
  "hasStairs": null,
  "hasCurbGap": null,
  "hasCrosswalk": null,
  "hasSignal": null,
  "hasAudioSignal": null,
  "hasBrailleBlock": null
}
```

이번 PoC에서 직접 보강 가능성을 확인할 필드는 아래 세 가지다. 다만 최신 ERD v3 저장 기준은 boolean 자체가 아니라 상태 컬럼과 `segment_features`다.

| API 응답 필드 | 최신 ERD v2 해석 | 보강 데이터 |
| --- | --- | --- |
| `hasCrosswalk` | `crossingState` 계산 및 `segment_features(CROSSWALK)` 적재 | 부산광역시 신호등 설치 횡단보도 위치정보 |
| `hasSignal` | `crossingState=SIGNALIZED` 판단 근거 | 부산광역시 신호등 설치 횡단보도 위치정보 |
| `hasAudioSignal` | `audioSignalState` 계산 및 `segment_features(AUDIO_SIGNAL)` 적재 | 부산광역시_음향신호기 현황 |

`hasCurbGap`, `hasBrailleBlock`, `avgSlopePercent`는 이번 P0 공공데이터만으로 직접 해결하기 어렵다. 특히 `hasBrailleBlock`은 최신 ERD 기준으로 `brailleBlockState` 보강 문제이며, 횡단보도 원천 row에 `점자블록 유무` 보강 필드를 추가한 팀 가공 데이터셋을 후속 입력원으로 검토한다.

## 10. 결과 판단 기준

| 판단 | 기준 | 후속 조치 |
| --- | --- | --- |
| 가능 | 좌표와 핵심 식별 필드가 있고, 현재 ERD 또는 staging table로 적재 가능 | deep load 또는 다음 구현 대상으로 승격 |
