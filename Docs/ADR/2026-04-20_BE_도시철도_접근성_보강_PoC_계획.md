## 1. 목적

이번 PoC의 목적은 부산 도시철도 역 정보를 외부 API 검색 결과에 보강하거나, 후속 단계에서 내부 기준 데이터로 관리하는 두 경우 모두에서 엘리베이터와 장애인화장실 같은 접근성 정보를 안정적으로 연결할 수 있는지 검증하는 것이다.

이번 PoC의 핵심은 `역 검색 결과`와 `역 접근성 보강 정보`를 어떤 기준으로 연결할지 정리하는 것이다. 역 기본 정보는 외부 API 보강을 우선 기준으로 보고, 내부 기준 데이터 적재는 후속 정책으로 분리한다.

2026-04-28 최신 장소 정책 기준으로 도시철도 역은 `places.category`의 서비스 카테고리 7개에 포함하지 않는다. 도시철도 역 접근성은 대중교통/도시철도 보강 데이터로 관리하고, 역에 붙는 접근성 여부는 `elevator`, `accessibleToilet` 같은 접근성 속성으로 해석한다.

---

## 2. 검증 질문

이번 PoC에서 확인할 핵심 질문은 아래와 같다.

1. 도시철도 엘리베이터/장애인화장실 원천 데이터를 역 단위로 집계할 수 있는가
2. 외부 API 검색 결과의 역명과 내부 원천 데이터의 역명을 안정적으로 매칭할 수 있는가
3. 현재 ERD 기준으로 `place_accessibility_features`에 필요한 접근성 정보를 담을 수 있는가
4. 현재 ERD에서 손실되는 상세 정보는 무엇이며, 별도 detail 구조 또는 원천별 extension table이 필요한가

---

## 3. 대상 범위

이번 PoC는 아래 범위만 다룬다.

- 부산 도시철도 역 접근성 보강
- 엘리베이터 정보
- 장애인화장실 정보
- 에스컬레이터, 휠체어리프트 등 계단 대체/보조 수직이동 정보
- ERD v3 기준 `subway_station_elevators` 적재 가능성
- 저상버스 노선 정적 테이블을 이번 ERD에 둘지 여부
- 외부 API 역 검색 결과와의 매핑 가능성

이번 PoC 범위에 포함하지 않는 것은 아래와 같다.

- 도시철도 운행 정보, 도착 정보, 혼잡도 같은 실시간 운행 데이터
- 경로 탐색 엔진과의 직접 연결

## 5. 적재 방향 가설

현재 1차 가설은 아래와 같다.

1. 지하철역 자체는 외부 API 검색 결과를 우선 사용한다.
2. 내부 DB에는 역별 접근성 정보와 엘리베이터 입구 좌표를 저장하거나 계산한다.
3. 원천 데이터는 역명 기준으로 정규화 후 집계한다.
4. 서비스에서는 외부 API 결과에 내부 접근성 정보를 붙여 반환한다.

즉, 이번 PoC의 핵심은 `역 검색 결과`에 접근성 정보를 어떻게 매핑할 것인가다. 내부 `places` 적재 여부는 이번 카테고리 정책의 결정 대상이 아니다.

## 6. 검증 기준

### 6.1 역명 매칭

| 항목 | 검증 기준 |
| --- | --- |
| 역명 정규화 | 괄호, 호선 표기, 역/정거장 접미어 제거 규칙으로 통합 가능한지 확인 |
| alias 처리 | `벡스코` / `시립미술관`, `국제금융센터` / `국제금융센터부산은행` 같은 차이를 수동 alias로 정리 가능한지 확인 |
| 환승역 처리 | `서면`, `연산`, `수영`, `덕천` 등 환승역을 역 단위로 묶을지 노선별로 유지할지 결정 필요 |

### 6.2 접근성 정보 집계

| 항목 | 검증 기준 |
| --- | --- |
| 엘리베이터 | 역 단위로 `elevator=true` 집계 가능 |
| 장애인화장실 | 역 단위로 `accessibleToilet=true` 집계 가능 |
| 에스컬레이터/리프트 | 역 단위로 계단 대체/보조 수직이동 정보 집계 가능 |
| 수량 | 역별 엘리베이터 수, 장애인화장실 수를 계산 가능한지 확인 |
| 상세정보 | 출구번호, 층, 상세위치, 게이트 내외를 현재 ERD에 직접 담을 수 있는지 확인 |

### 6.3 ERD 적합성

| 대상 | 검증 기준 |
| --- | --- |
| `place_accessibility_features.featureType` | `elevator`, `accessibleToilet`로 표현 가능한지 확인 |
| `place_accessibility_features.isAvailable` | boolean으로 환원 가능한지 확인 |
| 상세 속성 | 별도 detail 구조 또는 원천별 extension table이 필요한지 확인 |
| `subway_station_elevators` | 공식 소스로 `stationId`, `stationName`, `lineName`, `entranceNo`, `point`를 채울 수 있는지 확인 |
| 저상버스 노선 정적 테이블 | 부산 공식 소스를 별도 테이블로 적재할 필요가 있는지, 아니면 외부 API/외부 화면 연결로 충분한지 확인 |

## 7. 성공 기준

이번 PoC는 아래 조건을 만족하면 성공으로 본다.

- 역명 alias 규칙이 소수의 예외로 정리된다.
- 엘리베이터/장애인화장실 데이터를 역 단위 feature로 집계할 수 있다.
- 에스컬레이터/리프트 데이터를 도시철도 접근성 보강 후보로 정리할 수 있다.
- 외부 API 역 검색 결과에 내부 접근성 정보를 붙일 수 있다는 근거가 나온다.
- 현재 ERD로 가능한 범위와 부족한 범위가 분리된다.
- `subway_station_elevators`는 적재 가능한 범위와 제약이 정리된다.
- 저상버스 노선 정적 테이블은 도입 여부와 대체 처리 방식이 정리된다.
