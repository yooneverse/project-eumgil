# 경로 API 명세

> 작성일: 2026-05-06  
> 문서 목적: 백엔드/프론트 팀 공유용 API 계약  
> 범위: 도보 경로 검색, 대중교통 경로 검색, 경로 선택, 대중교통 도착정보 갱신, 경로 이탈 재탐색, 안내 종료, 경로 평가

## 0.1 도보/대중교통 기본 분기 계약

프론트는 출발지/도착지가 확정된 뒤 초기 경로 탐색 탭과 첫 API 호출을 다음 기준으로 결정한다.

- 기준값: `750m`
- 판단 거리: GraphHopper `SAFE` 도보 경로의 `distanceMeter`
- `SAFE distanceMeter <= 750m`: 도보 탭을 기본 선택하고 `POST /routes/search/walk`를 먼저 호출한다.
- `SAFE distanceMeter > 750m`: 대중교통 탭을 기본 선택하고 `POST /routes/search/transit`을 먼저 호출한다.
- `SAFE` 도보 경로가 없으면 `SHORTEST` 도보 거리로 fallback 판단한다.
- 도보 거리 판단이 실패하거나 도보 후보가 모두 없으면 대중교통 탭을 기본 선택한다.
- 사용자가 도보/대중교통 탭을 직접 전환하면 750m 기준과 관계없이 선택한 탭의 search API를 호출한다.

백엔드는 이 기준을 일관되게 적용할 수 있도록 도보 search 응답의 `routes[].distanceMeter`를 안정적으로 내려준다. 750m 기준은 기본 선택과 선호 호출 순서만 정하며, 두 탭의 노출 여부를 바꾸지 않는다.

## 1. API 목록

| 우선순위 | API                                      | 설명                                                                                                                                                                    |
| -------- | ---------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| P1       | `POST /routes/search/walk`               | 사용자가 도보 탭을 클릭했을 때 도보 안내 후보를 조회한다. 응답 route의 geometry/guidanceEvents를 기반으로 FE가 GPS 위치를 추적하며 안내한다.                             |
| P1       | `POST /routes/search/transit`            | 사용자가 대중교통 탭을 클릭했을 때 대중교통 안내 후보를 조회한다. 응답 route의 WALK/BUS/SUBWAY leg를 기반으로 FE가 GPS 위치를 추적하며 안내한다.                        |
| P1       | `POST /routes/{routeId}/select`          | 사용자가 경로 후보 카드의 길 안내 시작 버튼을 눌렀을 때 해당 route를 안내 세션으로 확정한다.                                                                            |
| P1       | `POST /routes/{routeId}/transit-refresh` | `WALK_TO_TRANSIT` 안내 중 버스정류장 30m 이내 또는 지하철역 엘리베이터 25m 이내에 도착했을 때 곧 탑승할 BUS/SUBWAY leg의 최신 도착정보를 조회한다.                         |
| P1       | `POST /routes/reroute`                   | 안내 중 GPS 정확도 20m 이하인 현재 위치가 route geometry에서 10m 이상 이탈한 상태가 2~3회 연속 확인되거나 3초 이상 지속될 때 현재 위치 기준으로 안내 경로를 재탐색한다. |
| P1       | `POST /routes/{routeId}/end`             | 사용자가 안내를 종료하거나 목적지에 도착했을 때 선택 경로 세션을 종료한다.                                                                                              |
| P1       | `POST /route-ratings`                    | 안내 종료 후 사용자가 평가등록 버튼을 눌렀을 때 안내받은 경로의 별점을 저장한다.                                                                                        |

## 2. 공통 규칙 및 프론트 호출 흐름

### 2.1 인증

- 모든 경로 API는 로그인이 필요하다.
- 모든 요청은 `Authorization: Bearer {accessToken}` 헤더를 포함해야 한다.
- 백엔드는 요청 body에서 `userId`, `accessibilityProfile`, `destinationPlaceId`를 받지 않는다.
- 백엔드는 토큰으로 사용자를 식별하고, user table의 `selectedPrimaryUserType`, `selectedMobilitySubtype`을 조회해 경로 탐색 조건에 반영한다.
- 사용자가 생성하거나 선택한 route만 후속 API에서 접근할 수 있다.
- 다른 사용자의 `routeId`, `searchId`, `routeSession`에 접근하면 `403 A4030`을 반환한다.

#### 장소 검색과 좌표 기반 요청

- 장소명 검색, 주소 검색, 후보 목록 제공은 별도 장소 검색 API 또는 외부 장소 API의 책임이다.
- 장소 검색 응답은 `placeId`, `name`, `address`, `lat`, `lng`를 포함해야 하며, 프론트는 사용자가 선택한 정확한 장소 후보를 먼저 확정한다.
- 경로 API는 장소명 원문을 받아 geocoding하지 않는다. 프론트는 확정된 장소 후보의 좌표를 `startPoint`, `endPoint`에 넣어 경로 API를 호출한다.
- 경로 API request의 source of truth는 항상 `startPoint.lat/lng`, `endPoint.lat/lng`다. 장소명과 주소는 경로 API가 아니라 장소 검색 결과 또는 FE handoff 상태로 화면에 표시한다.

### 2.2 서비스 가능 영역

- 모든 좌표 입력값은 부산광역시 서비스 영역 안에 있어야 한다.
- 서비스 영역 밖 좌표는 외부 API까지 보내지 않고 백엔드 validation에서 차단한다.
- 서비스 영역 밖 좌표 오류는 `400 RT4003`을 반환한다.
- 프론트는 `RT4003`을 받으면 “부산광역시 안의 위치를 선택해 주세요.”를 표시한다.
- 출발지와 도착지가 20m 이하로 가까우면 `400 RT4004`를 반환한다.
- 검색 또는 재탐색 결과 추천 가능한 경로가 하나도 없으면 백엔드는 `404 RT4040`을 반환한다.
- 프론트는 `RT4040`을 받으면 추천 경로 없음 상태를 표시하고 조건 변경 재검색 안내와 두리발 호출 CTA를 함께 노출한다.
- `RT4040`의 두리발 호출 CTA는 추가 백엔드 API가 아니라 프론트 전화 연결 또는 외부 호출 흐름으로 처리한다.

### 2.3 시간값 의미

| 필드                  | 의미                                                                               |
| --------------------- | ---------------------------------------------------------------------------------- |
| `estimatedTimeMinute` | 경로, leg, 또는 `laneOptions[]` 후보를 실제 이동/탑승했을 때 걸리는 예상 소요 시간 |
| `durationSecond`      | 경로 또는 leg를 실제 이동/탑승했을 때 걸리는 예상 소요 시간의 초 단위 원본값       |
| `remainingMinute`     | 버스/지하철이 탑승 지점에 도착하기까지 남은 대기 시간                              |

`estimatedTimeMinute`는 도착예정 시간이 아니다. 프론트는 `estimatedTimeMinute`를 “소요 시간”으로 표시한다. `remainingMinute`는 대중교통이 탑승 지점에 도착하기까지 남은 “도착예정/대기 시간”이다. search 응답에서는 BUS leg의 `laneOptions[].remainingMinute`와 SUBWAY leg의 `remainingMinute`를 초기 표시값으로 내려주고, 안내 중 최신값은 `transit-refresh` 응답으로 갱신한다. search 시점의 BUS `laneOptions[].remainingMinute`는 BIMS 조회 결과가 없으면 `null`일 수 있고, SUBWAY `remainingMinute`는 시간표 조회 결과가 없으면 `null`이거나 생략될 수 있다.

`estimatedTimeMinute`는 초 단위 계산값을 분 단위로 내림 처리한다. 계산식은 `max(1, floor(durationSecond / 60))`이며, 1분 미만은 초로 표시하지 않고 1분으로 표시한다.

대중교통 route의 `durationSecond`는 항상 `legs[].durationSecond` 합계와 같아야 한다. BUS 탑승 준비 시간은 해당 BUS leg의 시작부에 포함하고, 환승 버퍼는 다음에 탑승할 BUS/SUBWAY leg의 시작부에 포함한다. 따라서 BUS/SUBWAY leg의 `durationSecond`와 `ARRIVING_POINT.durationFromLegStartSecond`는 해당 leg에 귀속된 탑승 준비 시간 또는 환승 버퍼를 포함한다. `durationFromRouteStartSecond`는 버퍼가 반영된 앞선 leg들의 `durationSecond`를 기준으로 재계산한다.

프론트가 안내 중 남은 시간을 계산할 때는 `durationSecond`를 기준으로 한다.

1. 현재 GPS 위치를 현재 route/leg geometry에 projection해 진행한 거리와 남은 거리를 계산한다.
2. 전체 route 기준 남은 시간은 `round(route.durationSecond * remainingDistanceMeter / route.distanceMeter)`로 계산한다.
3. leg 진행 상태를 알고 있으면 지나간 leg의 `durationSecond`를 제외하고, 현재 leg는 남은 거리 비율로 보정한 뒤 이후 leg의 `durationSecond`를 더해 계산한다.
4. 화면에는 `ceil(remainingSecond / 60)`을 남은 시간으로 표시하되, 이동 중 1분 미만은 1분으로 표시한다.
5. `durationSecond`는 사용자 실제 실시간 속도가 아니라 백엔드가 사용자 profile과 경로 비용 계산에 사용한 기준 소요 시간이다.

### 2.4 선택 경로

- 경로 검색 결과는 일정 시간 동안만 선택할 수 있다.
- ROUTE-01 후보 카드의 선택 상태는 FE local state다. search 응답은 어떤 후보가 현재 선택되었는지를 내려주지 않는다.
- search 결과 Redis는 단순 성능 cache가 아니라 `select` 전까지 후보 묶음을 보관하는 임시 상태 저장소다.
- 백엔드는 search 성공 응답을 반환하기 전에 `searchId`와 `routes[]` 후보 묶음을 Redis에 저장해야 한다.
- Redis 저장 실패와 연결 실패는 search API 성공 응답을 반환하지 않고 `500 I5000`으로 처리한다. 사용자가 받은 `searchId`로 후속 `select`를 진행할 수 없기 때문이다.
- Redis 저장 전 직렬화 실패와 Redis 조회 후 역직렬화 실패도 사용자 입력 오류나 외부 경로 API 실패가 아니므로 `500 I5000`으로 처리한다.
- Redis 저장/연결/직렬화/역직렬화 실패는 `EX5020`으로 분류하지 않는다. `EX5020`은 ODsay, BIMS, GraphHopper 같은 외부 경로 정보 provider 실패에만 사용한다.
- Redis miss는 장애가 아니라 TTL 만료, 재배포, 캐시 evict, 잘못된 `searchId` 등으로 인해 선택 가능한 후보 묶음이 없는 상태로 본다. 이 경우 `select`는 `404 RT4041`을 반환한다.
- `routeSearch:{searchId}`와 `routeSearchMeta:{searchId}` Redis key의 TTL은 30분이다. 프론트는 TTL 만료 가능성을 고려해 `RT4041` 수신 시 같은 출발지/도착지로 최신 검색을 다시 수행한 뒤 새 `searchId`로 select를 재시도할 수 있다.
- owner `userId`가 없는 구버전 `routeSearch:{searchId}` cache는 소유권을 검증할 수 없으므로 정상 선택 후보로 취급하지 않고 `404 RT4041`을 반환한다. 새 구조 cache에서 owner가 현재 사용자와 다르면 `403 A4030`으로 구분한다.
- owner cache 구조가 바뀌는 배포에서는 Redis 전체가 아니라 경로 검색 임시 key인 `routeSearch:*`, `routeSearchMeta:*`만 삭제해 legacy owner-less cache가 남지 않게 한다.
- 사용자가 경로를 선택하면 백엔드는 선택 경로 세션을 저장한다.
- 선택 경로 세션은 DB route session에 저장된다.
- select 처리 시 Redis 후보 묶음 조회와 route snapshot 생성은 DB transaction 밖에서 수행하고, DB transaction은 ACTIVE session 중복 확인과 저장 범위로 짧게 유지한다.
- 같은 사용자의 같은 route에 대해 ACTIVE session은 하나만 허용한다. DB는 `route_sessions.user_id + active_route_key` unique 제약으로 중복 저장을 최종 방어한다.
- 중복 select 성공 처리는 PostgreSQL unique violation `23505`이면서 제약명이 `uk_route_sessions_user_active_route`인 경우에만 적용한다. 다른 DB 제약 위반은 중복 선택으로 간주하지 않고 서버 예외로 전파한다.
- `transit-refresh`, `reroute`, `route-ratings`는 DB route session을 기준으로 route 문맥을 조회한다.
- DB route session이 없거나 저장된 snapshot을 복구할 수 없는 경우 후속 안내 API는 경로 세션 오류를 반환한다.
- 프론트는 경로 세션 오류를 받으면 사용자를 재검색 흐름으로 안내한다.
- 경로 저장은 경로 API가 아니라 북마크 API의 책임이다. 경로 API는 방금 탐색/선택/안내한 route와 rating만 다룬다.

### 2.5 도착정보 상태

`transit-refresh`는 도착분이 없어도 외부 API 호출과 정류장/노선 해석이 성공하면 성공 응답을 반환한다.

| arrivalStatus        | 의미                                                  | 프론트 표시                       |
| -------------------- | ----------------------------------------------------- | --------------------------------- |
| `REALTIME_AVAILABLE` | 실시간 도착예정분이 있음                              | “n분 뒤 도착예정”                 |
| `NO_CURRENT_ARRIVAL` | 정류장/노선은 맞지만 현재 도착 예정 차량 정보가 없음  | “현재 도착 예정 차량이 없습니다.” |
| `ARRIVAL_UNKNOWN`    | 제공기관 응답은 성공했지만 도착 여부를 판단할 수 없음 | “도착정보를 확인할 수 없습니다.”  |
| `SCHEDULE_BASED`     | 지하철 등 시간표 기반으로 남은 시간을 계산함          | “시간표 기준 n분 뒤 도착예정”     |

`NO_CURRENT_ARRIVAL`일 때 `transits`는 빈 배열이다.

### 2.6 확정 enum 및 표시 정책

이 섹션은 경로 API에서 사용하는 확정 enum 계약이다. `badges`, `guidanceEvents[].type`, `guidanceEvents[].direction`, `routeOption`, 위치 근처 기준, reroute 호출 기준은 아래 표를 기준으로 구현한다.

#### 2.6.1 badges

`badges`는 도보 접근성 상태를 요약하는 enum code 배열이다. 프론트는 code를 기준으로 한글명과 표시 색상을 매핑한다. 같은 route에 `MIDDLE_SLOPE`와 `LOW_SLOPE` 구간이 모두 있으면 두 badge를 모두 포함한다. 권장 표시 우선순위는 `STAIR > NARROW_SIDEWALK > UNPAVED > MIDDLE_SLOPE > LOW_SLOPE > CROSSWALK > ELEVATOR`다. 신호기/음향신호기 여부는 guidance event의 `features[]`로만 쓰고, route badge는 `CROSSWALK`로 요약한다.

| 영어명            | 한글명     | 표시 색상 | 의미                                  |
| ----------------- | ---------- | --------- | ------------------------------------- |
| `LOW_SLOPE`       | 낮은 경사  | 일반      | 경사가 낮아 보행 부담이 작은 구간     |
| `MIDDLE_SLOPE`    | 중간 경사  | 주의      | 경사가 있어 주의가 필요한 구간        |
| `STAIR`           | 계단       | 주의      | 계단이 포함된 구간                    |
| `CROSSWALK`       | 횡단보도   | 일반      | 횡단보도가 포함된 구간                |
| `ELEVATOR`        | 엘리베이터 | 일반      | 엘리베이터 이용 가능 구간 또는 지점   |
| `NARROW_SIDEWALK` | 좁은 보도  | 주의      | 보도 폭이 좁아 주의가 필요한 구간     |
| `UNPAVED`         | 비포장     | 주의      | 포장 상태가 좋지 않거나 비포장인 구간 |

#### 2.6.2 guidanceEvents[].type / guidanceEvents[].direction

`guidanceEvents[]`는 leg 진행 중 화면/TTS 안내가 필요한 접근성 이벤트, 시설/탑승 접근 이벤트, 대중교통 하차/도착 이벤트, 방향 보조 정보를 leg 시작점 기준 누적 거리 순서로 제공하는 배열이다. 각 이벤트는 현재 leg 기준 누적값인 `distanceFromLegStartMeter`/`durationFromLegStartSecond`와 전체 route 기준 누적값인 `distanceFromRouteStartMeter`/`durationFromRouteStartSecond`를 함께 제공한다. FE는 WALK leg에서 현재 위치를 WALK leg `geometry`에 projection해 진행 거리를 계산하고, `distanceFromLegStartMeter - progressedDistanceMeter <= 50`이 되는 이벤트를 알림 카드에 표시한다. 지나간 이벤트는 consumed 처리하고 다음 이벤트를 preview 알림으로 올린다. BUS/SUBWAY leg에서는 `ARRIVING_POINT`를 대중교통 leg 진행률 기준 하차 안내 지점으로 사용한다.

하나의 위치에 여러 접근성 이벤트가 겹치면 `STAIR > NARROW_SIDEWALK > UNPAVED > MIDDLE_SLOPE > LOW_SLOPE > CROSSWALK > ELEVATOR` 우선순위로 대표 `type`을 선택한다. 방향 전환은 `type`에 넣지 않고 `direction`에 넣는다. 방향 전환과 접근성/시설 이벤트가 같은 위치 또는 `distanceFromRouteStartMeter` 기준 3m 이내에 있으면 하나의 `guidanceEvents[]` 항목으로 병합하고, `type`은 접근성/시설 이벤트를 유지하며 `direction`은 보조 방향 정보로 함께 내려준다. `ELEVATOR`는 route `badges[]` 요약용 enum이며 `guidanceEvents[].type`에는 사용하지 않는다.

상호 배타 guidance event는 같은 trigger 지점에 동시에 내려주지 않는다. `direction`은 `STRAIGHT`, `TURN_LEFT`, `TURN_RIGHT` 중 하나만 선택한다. 횡단보도는 항상 `type=CROSSWALK`로 내려주고, 신호기/음향신호기는 `features[]`의 `SIGNAL`, `AUDIO_SIGNAL`로 표현한다. `LOW_SLOPE`와 `MIDDLE_SLOPE`도 같은 지점에 동시에 내려주지 않는다.

WALK leg에서 방향 변화가 좌/우회전 기준에 미치지 않으면 직진으로 간주한다. 직진은 기본 진행 상태이므로 보통 `guidanceEvents[]` 항목을 만들지 않는다. 다만 횡단보도를 건넌 뒤 다음 주요 이벤트 또는 목적지까지 30m 이상 남으면 `type=STRAIGHT`, `direction=STRAIGHT` 이벤트를 만들어 "직진 N m" 안내를 제공한다.

`guidanceEvents[]`는 segment 속성 전체 목록이 아니라 사용자가 새로 인지해야 하는 안내 지점 목록이다. `LOW_SLOPE`, `MIDDLE_SLOPE`, `STAIR`, `NARROW_SIDEWALK`, `UNPAVED`처럼 구간에 걸쳐 이어지는 접근성 상태는 같은 type이 연속되는 동안 첫 trigger 지점에만 이벤트를 만든다. type이 바뀌거나 일반 구간을 지난 뒤 같은 type이 다시 시작되면 새 이벤트를 만든다. 방향 전환은 `direction`으로 유지하고, 같은 위치에 type 이벤트가 있으면 별도 row가 아니라 같은 row의 보조 정보로 병합한다. `CROSSWALK`는 지점 이벤트에 가깝기 때문에 연속 상태 dedupe 대상에 넣지 않고, 같은 횡단보도가 여러 segment로 쪼개져 중복 생성되는 경우에만 하나로 합친다.

WALK leg 시작점부터 이어지는 저위험 연속 상태, 특히 `LOW_SLOPE`는 "0m 후 낮은 경사로입니다"처럼 미래 안내로 읽히면 어색하므로 `guidanceEvents[]`에서 제외하고 route `badges[]`로만 요약한다. 시작점에서 즉시 인지해야 하는 위험 상태는 별도 현재 상태 안내 계약이 생기기 전까지 타입별로 유지 여부를 검토하되, FE가 `distanceFromLegStartMeter`가 0에 가까운 이벤트를 렌더링할 때는 "0m 후"가 아니라 "현재" 문구를 사용한다.

| 영어명            | 한글명            | 의미                                 |
| ----------------- | ----------------- | ------------------------------------ |
| `CROSSWALK`       | 횡단보도 이용     | 횡단보도 접근 또는 통과 안내. 신호기/음향신호기는 `features[]`로 표현 |
| `STRAIGHT` | 직진 안내      | 횡단보도 통과 후 다음 주요 이벤트 또는 목적지까지 직진 안내 |
| `LOW_SLOPE`       | 낮은 경사        | 낮은 경사 구간 접근 안내                  |
| `MIDDLE_SLOPE`    | 중간 경사         | 경사 구간 접근 안내                  |
| `STAIR`           | 계단              | 계단 접근 안내                       |
| `NARROW_SIDEWALK` | 좁은 보도         | 좁은 보도 구간 접근 안내             |
| `UNPAVED`         | 비포장            | 비포장 또는 노면 불량 구간 접근 안내 |
| `BUS_STOP`        | 버스정류장        | 버스 탑승 정류장 접근 안내           |
| `SUBWAY_ELEVATOR` | 지하철 엘리베이터 | 지하철역 엘리베이터 접근 안내        |
| `ARRIVING_POINT`  | 하차 지점         | 대중교통 이용 시 버스/지하철 하차 지점 안내 |
| `DESTINATION`     | 최종 목적지       | 전체 경로의 최종 목적지 도착 안내    |

`guidanceEvents[].direction` enum:

| 영어명 | 한글명 | 의미 |
| --- | --- | --- |
| `STRAIGHT` | 직진 | route geometry 기준 직진 보조 정보. 기본 진행 상태이므로 일반적으로 생략한다. |
| `TURN_LEFT` | 좌회전 | route geometry 기준 좌회전 보조 정보 |
| `TURN_RIGHT` | 우회전 | route geometry 기준 우회전 보조 정보 |

`guidanceEvents[].features` enum:

| 영어명 | 한글명 | 의미 |
| --- | --- | --- |
| `SIGNAL` | 신호기 | `type=CROSSWALK` 지점에 신호기가 있음 |
| `AUDIO_SIGNAL` | 음향 신호기 | `type=CROSSWALK` 지점에 음향 신호기가 있음 |

FE 렌더링 규칙:

- `type`이 있으면 `type` 문구가 주 안내다. 예: `type=CROSSWALK`, `features=["SIGNAL"]`, `direction=TURN_RIGHT`이면 "신호기가 있는 횡단보도를 건너세요."를 주 문구로 보여주고 우회전은 방향 아이콘/보조 배지로 표시한다.
- `type`이 없고 `direction=TURN_LEFT` 또는 `TURN_RIGHT`이면 "좌회전하세요.", "우회전하세요."를 주 안내 문구로 생성한다.
- `type`이 없고 `direction=STRAIGHT`이면 기본 상태로 보고 보통 렌더링하지 않는다.

WALK leg의 `instruction`은 leg 전체 목적지 요약 문구만 제공한다. 이벤트별 화면 문구와 TTS 문구는 FE가 `guidanceEvents[].type`과 `guidanceEvents[].direction`을 기준으로 생성한다. 백엔드는 이벤트 enum, 방향 enum, 누적 거리/시간, 이벤트 지점 `POINT` geometry를 책임진다.

#### 2.6.3 leg별 guidanceEvents 책임

| leg role/type | 포함하는 guidanceEvents | 제외하는 guidanceEvents |
| ------------- | ----------------------- | ----------------------- |
| `WALK_TO_TRANSIT` / `WALK` | 도보 중 발생하는 `CROSSWALK*`, `LOW_SLOPE`, `MIDDLE_SLOPE`, `STAIR`, `NARROW_SIDEWALK`, `UNPAVED`와 탑승 접근 지점 `BUS_STOP` 또는 `SUBWAY_ELEVATOR`, 방향 보조 정보 `direction` | `ARRIVING_POINT`, `DESTINATION` |
| `TRANSIT` / `BUS`, `SUBWAY` | 대중교통 하차 지점 `ARRIVING_POINT`만 포함 | `BUS_STOP`, `SUBWAY_ELEVATOR`, 도보 접근성/회전 이벤트, `DESTINATION` |
| `TRANSIT_TO_WALK` / `WALK` | 하차 후 최종 목적지까지 도보 중 발생하는 `CROSSWALK*`, `LOW_SLOPE`, `MIDDLE_SLOPE`, `STAIR`, `NARROW_SIDEWALK`, `UNPAVED`와 최종 목적지 `DESTINATION`, 방향 보조 정보 `direction` | `BUS_STOP`, `SUBWAY_ELEVATOR`, `ARRIVING_POINT` |

`BUS_STOP`과 `SUBWAY_ELEVATOR`는 사용자가 탑승 지점까지 걸어가는 동안 "몇 m 남았는지" 알림을 제공해야 하므로 `WALK_TO_TRANSIT` leg에 둔다. `ARRIVING_POINT`는 탑승 이후 하차 지점 안내이므로 BUS/SUBWAY `TRANSIT` leg에만 둔다. `ARRIVING_POINT.distanceFromLegStartMeter`는 해당 BUS/SUBWAY leg 시작점, 즉 탑승 정류장/역 기준 하차 지점까지의 누적 거리다. `ARRIVING_POINT.durationFromLegStartSecond`는 해당 leg에 귀속된 탑승 준비 시간 또는 환승 버퍼를 포함한 하차 지점까지의 누적 시간이다. `distanceFromRouteStartMeter`와 `durationFromRouteStartSecond`는 route 시작점 기준 누적값이며, `durationFromRouteStartSecond`는 버퍼가 반영된 leg duration 축을 따른다. FE가 leg 전환 없이 전체 경로 진행률과 알림 순서를 계산할 때 사용한다. `DESTINATION`은 전체 경로의 최종 목적지 안내이므로 하차 후 `TRANSIT_TO_WALK` leg 또는 도보-only `WALK_ONLY` leg 끝 지점에 둔다.

#### 2.6.4 routeOption

| 영어명         | 한글명         | 적용 API                      | 의미                                                           |
| -------------- | -------------- | ----------------------------- | -------------------------------------------------------------- |
| `SAFE`         | 안전 경로      | `POST /routes/search/walk`    | 도보 접근성/안전성을 우선한 도보 경로                          |
| `SHORTEST`     | 최단 경로      | `POST /routes/search/walk`    | 거리를 우선한 도보 경로                                        |
| `RECOMMENDED`  | 추천 경로      | `POST /routes/search/transit` | 소요 시간, 환승, 도보 거리, 접근성을 종합한 대중교통 추천 경로 |
| `MIN_TRANSFER` | 환승 최소 경로 | `POST /routes/search/transit` | 환승 횟수를 우선 최소화한 대중교통 경로                        |
| `MIN_WALK`     | 도보 최소 경로 | `POST /routes/search/transit` | 대중교통 전후 및 환승 도보 거리를 우선 최소화한 경로           |

대중교통 후보는 최대 3개까지 제공하며, `routeOption`은 응답 내에서 중복되지 않아야 한다. 같은 실제 경로가 여러 option에 중복 선정되면 중복 route를 제거한다. 이때 `routeOption`에는 대표값 1개를 넣고, `routeOptions[]`에는 해당 경로가 선정된 모든 option을 담아 복수 라벨을 표시한다. 대표값 우선순위는 `RECOMMENDED > MIN_TRANSFER > MIN_WALK`다. 중복 병합 후 노출 가능한 대표 option이 3개보다 적으면 부족한 개수를 임의 후보로 채우지 않고, 존재하는 후보만 반환한다.

| WALK_TO_TRANSIT BUS | `{정류장명} 정류장까지 이동하세요.`                                                                                              |

| BUS leg             | `{routeNo}번 버스에 탑승하세요.`                                                                                                  |

### 2.7 위치 근처 기준

| 대상              | 기준                          |
| ----------------- | ----------------------------- |
| `BUS_STOP`        | 정류장 좌표 기준 30m 이내     |
| `SUBWAY_ELEVATOR` | 엘리베이터 좌표 기준 25m 이내 |

- `WALK_TO_TRANSIT` 안내 중 `BUS_STOP` 또는 `SUBWAY_ELEVATOR` 기준에 들어오면 프론트는 곧 탑승할 BUS/SUBWAY leg의 `sequence`로 `POST /routes/{routeId}/transit-refresh`를 호출한다.
- 같은 leg에 대해 연속 호출이 발생하지 않도록 프론트는 최소 30초 쿨다운을 둘 수 있고, 백엔드는 arrival cache TTL 안에서는 캐시된 값을 반환할 수 있다.

### 2.8 재탐색 위치 검증

- 프론트는 다음 조건을 모두 만족하면 `POST /routes/reroute`를 호출한다.
  - 현재 위치가 route geometry에서 10m 이상 벗어남
  - GPS accuracy가 20m 이하임
  - 10m 이상 이탈 상태가 2~3회 연속 확인되거나 3초 이상 지속됨
- reroute 요청에는 `currentLegSequence`를 보내지 않는다.
- 프론트는 `routeId`와 `currentPoint`만 보낸다.
- 백엔드는 저장된 route geometry와 `currentPoint`를 비교해 현재 leg를 추정하거나 full reroute 여부를 판단한다.
- 백엔드는 다음을 검증한다.
  - routeId 소유자 검증
  - currentPoint 부산 서비스 영역 검증
  - currentPoint가 기존 route 주변인지 검증
  - currentPoint가 기존 route와 지나치게 멀리 떨어져 있는지 검증
- 현재 위치가 말이 안 되게 멀거나 서비스 영역 밖이면 에러를 반환한다.
- 기존 route geometry에서 10m 이하는 새 경로가 필요 없는 상태로 보고 `route=null`을 반환한다.
- 기존 route geometry에서 10m 초과 500m 이하는 현재 위치 기준 `FULL_REROUTE`, 500m 초과는 `RT4091` 기준으로 처리한다.

### 2.9 프론트 안내 호출 흐름

프론트는 검색 API 응답만으로 바로 안내 세션을 시작하지 않는다. 검색 API는 후보를 보여주기 위한 데이터이고, 사용자가 특정 후보의 길 안내 시작 버튼을 누른 뒤 `select`가 성공해야 해당 route가 안내 세션으로 확정된다.

#### 도보 안내 흐름

1. 사용자가 출발지/도착지를 정한 뒤 도보 탭을 클릭한다.
2. 프론트는 `POST /routes/search/walk`를 호출한다.
3. 프론트는 응답의 `searchId`와 `routes[]`를 저장하고, route 후보 카드와 지도 polyline을 표시한다.
4. 사용자가 후보 카드의 길 안내 시작 버튼을 누르면 `POST /routes/{routeId}/select`를 호출한다.
5. `select`가 성공하면 프론트는 별도 route 재조회 없이 검색 응답에서 보관 중인 route payload로 안내를 시작한다.
6. 안내 중 프론트는 사용자의 GPS 이동을 실시간으로 추적하고, 현재 위치가 route/leg geometry의 어느 지점에 가까운지 판단한다. WALK leg에서는 `guidanceEvents[]`의 누적 거리와 현재 진행 거리를 비교해 화면 안내와 TTS를 갱신한다.
7. 현재 위치가 route geometry에서 10m 이상 이탈했고 GPS accuracy가 20m 이하이며, 이탈 상태가 2~3회 연속 확인되거나 3초 이상 지속되면 `POST /routes/reroute`를 호출하고, 응답의 `route`가 있으면 현재 안내 route를 새 route로 교체한다.
8. 사용자가 안내를 종료하거나 목적지에 도착하면 `POST /routes/{routeId}/end`를 호출한다.
9. 안내 종료 후 사용자가 평가등록 버튼을 누르면 `POST /route-ratings`를 호출한다.

#### 대중교통 안내 흐름

1. 사용자가 출발지/도착지를 정한 뒤 대중교통 탭을 클릭한다.
2. 프론트는 `POST /routes/search/transit`을 호출한다.
3. 프론트는 응답의 `searchId`와 `routes[]`를 저장하고, WALK/BUS/SUBWAY leg를 후보 카드와 상세 경로에 표시한다.
4. 사용자가 후보 카드의 길 안내 시작 버튼을 누르면 `POST /routes/{routeId}/select`를 호출한다.
5. `select`가 성공하면 프론트는 검색 응답에서 보관 중인 route payload로 안내를 시작한다.
6. 안내 중 프론트는 GPS 이동을 실시간으로 추적하고, WALK leg에서는 `guidanceEvents[]` 안내를 갱신하며, BUS/SUBWAY leg에서는 탑승/하차 정류장 또는 역 정보를 기준으로 안내한다.
7. 사용자가 `WALK_TO_TRANSIT` leg 진행 중 버스정류장 30m 이내 또는 지하철역 엘리베이터 25m 이내에 도착하면 프론트는 곧 탑승할 BUS/SUBWAY leg의 `sequence`로 `POST /routes/{routeId}/transit-refresh`를 호출한다.
8. 프론트는 `transit-refresh` 응답의 `arrivalStatus`와 `transits[]`로 도착예정 문구를 갱신한다. 이 응답은 route geometry를 바꾸지 않는다.
9. 현재 위치가 route geometry에서 10m 이상 이탈했고 GPS accuracy가 20m 이하이며, 이탈 상태가 2~3회 연속 확인되거나 3초 이상 지속되면 `POST /routes/reroute`를 호출하고, 응답의 `route`가 있으면 현재 안내 route를 새 route로 교체한다.
10. 사용자가 안내를 종료하거나 목적지에 도착하면 `POST /routes/{routeId}/end`를 호출한다.
11. 안내 종료 후 사용자가 평가등록 버튼을 누르면 `POST /route-ratings`를 호출한다.

#### 저시력자 즉시 안내 흐름

1. `LV-RES-01`에서 사용자가 길찾기를 선택한다.
2. 프론트는 사용자 profile과 750m 기준에 따라 `POST /routes/search/transit` 또는 `POST /routes/search/walk`를 호출한다.
3. 프론트는 기본 후보를 선택한다. 대중교통은 `RECOMMENDED`, 도보는 `SAFE`를 우선 선택하고 없으면 같은 탭의 다음 후보로 fallback한다.
4. 프론트는 선택한 기본 후보의 `routeId`와 `searchId`로 `POST /routes/{routeId}/select`를 자동 호출한다.
5. `select`가 성공하면 후보 비교 화면을 거치지 않고 `LV-NAV-01`에 진입한다.

#### 응답 사용 원칙

- `search` 응답의 `routes[]`는 후보 표시와 안내 시작 후 실제 화면/TTS 안내에 모두 사용한다.
- ROUTE-01 후보 카드의 선택 상태는 FE local state로 유지하고, API 응답 필드로 요구하지 않는다.
- `select` 응답의 `data.sessionId`는 후속 rating의 정확한 평가 대상이다. route 화면은 기존에 저장한 검색 응답 payload를 계속 사용한다.
- `transit-refresh` 응답은 버스/지하철 도착예정 정보만 갱신한다. 기존 route, leg, guidanceEvents, geometry를 덮어쓰지 않는다.
- `reroute` 응답에 `route`가 있으면 프론트는 현재 안내 중인 `routeId`, route payload, 지도 polyline, leg/guidanceEvents 진행 상태를 새 route 기준으로 교체한다.
- `reroute`로 받은 새 route는 이미 안내 가능한 route session으로 저장된 상태이므로 추가 `select`를 호출하지 않는다.
- `end` 응답의 `data.sessionId`는 종료된 안내 세션 ID다. 프론트는 이 값을 평가 요청에 사용하고 ARR-01/ARR-02 흐름으로 이동한다.
- `route-ratings` 응답에서 `ratingId`를 받으면 평가 저장 완료로 처리한다.
- 경로 북마크 저장은 북마크 API로 처리하며, 경로 API에 저장 요청을 추가하지 않는다.

#### WALK guidanceEvents 생성 원칙

- WALK leg는 출발지에서 도착지, 버스 정류장, 또는 지하철 엘리베이터까지 이어지는 하나의 leg geometry를 source of truth로 둔다.
- WALK leg에는 `steps[]`를 두지 않고, 안내가 필요한 지점을 `guidanceEvents[]` 배열로 제공한다.
- `guidanceEvents[].distanceFromLegStartMeter`와 `durationFromLegStartSecond`는 WALK leg 시작점 기준 누적값이다.
- `guidanceEvents[].distanceFromRouteStartMeter`와 `durationFromRouteStartSecond`는 route 시작점 기준 누적값이다. walk-only 경로에서는 leg 기준 누적값과 동일하고, transit 경로에서는 앞선 leg들의 거리/시간이 더해진다.
- `guidanceEvents[].geometry`는 이벤트 trigger 지점의 WKT `POINT(lng lat)`다. 횡단보도, 경사, 좁은 보도처럼 원천이 line/range인 feature도 안내 trigger는 시작 지점 `POINT`로 내려준다.
- 좌/우회전 방향은 `segment_features`가 아니라 route geometry 또는 연속 segment의 방향 변화에서 계산해 `guidanceEvents[].direction`의 `TURN_LEFT`, `TURN_RIGHT`로 제공한다.
- 방향 변화가 좌/우회전 기준에 미치지 않으면 직진으로 간주하고, 일반 직진은 별도 이벤트 없이 다음 안내 이벤트까지 기본 진행 상태로 처리한다. 횡단보도를 건넌 뒤 다음 주요 이벤트 또는 목적지까지 30m 이상 남으면 `STRAIGHT` 이벤트를 만든다.
- 접근성 이벤트가 같은 위치에 여러 개 겹치면 대표 이벤트만 선택한다. 우선순위는 `STAIR > NARROW_SIDEWALK > UNPAVED > MIDDLE_SLOPE > LOW_SLOPE > CROSSWALK > ELEVATOR`이다.
- 상호 배타 guidance event는 같은 trigger 지점에 동시에 내려주지 않는다. `direction`의 `TURN_LEFT`/`TURN_RIGHT`, `LOW_SLOPE`/`MIDDLE_SLOPE`는 각 그룹에서 하나만 선택한다. 횡단보도 부가 정보는 별도 type이 아니라 `features[]`로 함께 내려준다.
- `LOW_SLOPE`, `MIDDLE_SLOPE`, `STAIR`, `NARROW_SIDEWALK`, `UNPAVED` 같은 연속 상태형 접근성 이벤트는 같은 type이 이어지는 동안 첫 지점만 생성한다.
- type이 바뀌면 새 `guidanceEvent`를 생성하고, 일반 구간을 지나 같은 type이 다시 시작되어도 새 `guidanceEvent`를 생성한다.
- WALK leg 시작점부터 이어지는 `LOW_SLOPE`는 `0m 후` 안내 대상이 아니므로 `guidanceEvents[]`에서 제외하고 route `badges[]`에만 남긴다.
- 방향 전환은 접근성 상태와 별개로 계산하되, 같은 위치 또는 route 누적거리 기준 3m 이내에 접근성/시설 이벤트가 있으면 별도 event row를 만들지 않고 해당 event의 `direction`으로 병합한다.
- 횡단보도 시작/종료 경계 전후 10m 이내의 `TURN_LEFT`/`TURN_RIGHT` 후보는 보행 네트워크 접속부 artefact로 보고 제거한다.
- `CROSSWALK`는 지점 이벤트이므로 연속 상태 dedupe 대상에서 제외한다. 단, 동일 횡단보도가 여러 segment로 나뉘어 같은 위치에 중복 생성되는 경우에는 하나로 합친다.
- route geometry artefact로 보이는 짧은 좌/우회전 쌍은 제거한다. 연속된 `TURN_LEFT`/`TURN_RIGHT` 사이 누적 거리 차이가 5m 이하이면 둘 다 안내 이벤트에서 제외한다.
- 최종 도보가 매우 짧더라도 `STAIR` 같은 접근성 이벤트와 `DESTINATION`이 함께 있으면 그대로 제공한다. 정보 손실 없이 FE가 표시 우선순위를 결정한다.
- 접근성 요약은 route `badges[]`에만 두고, `guidanceEvents[]`에는 개별 안내 이벤트만 둔다.

### 4.1 외부 API 실패 코드 구분

`EX5020`과 `EX5040`은 모두 ODsay, BIMS, GraphHopper 같은 외부 경로 정보 연동 중 발생할 수 있다. 두 코드는 사용자에게는 비슷하게 재시도를 안내할 수 있지만, 백엔드 운영자가 봐야 할 원인이 다르므로 구분한다. 1차 구현의 지하철 `transit-refresh`는 외부 API가 아니라 DB 시간표 조회 기반이므로 시간표 row가 없으면 외부 API 실패가 아니라 `ARRIVAL_UNKNOWN`으로 처리한다.

| status | 의미 | 대표 발생 조건 | 운영 확인 방향 |
| --- | --- | --- | --- |
| `EX5020` | 외부 경로 정보를 정상 데이터로 사용할 수 없음 | ODsay 호출 실패, ODsay 인증 실패, ODsay/BIMS 5xx, 응답 파싱 실패, timeout이 아닌 네트워크/클라이언트 예외, GraphHopper profile/parameter/config/server 오류 | provider 장애, 인증키/endpoint 설정, 응답 스키마 변경, GraphHopper 데이터/config 문제 확인 |
| `EX5040` | 외부 경로 정보 응답이 제한 시간 안에 오지 않음 | ODsay/BIMS/GraphHopper 호출이 5초 timeout에 걸림 | 외부 지연, 네트워크 latency, timeout 설정, provider 응답 시간 확인 |

`EX5020`은 "답이 왔거나 호출이 실패했지만 정상 경로 정보로 쓸 수 없는 상태"이고, `EX5040`은 "기다렸지만 시간 안에 답이 오지 않은 상태"다.

아래 상황은 외부 실패 코드가 아니다.

- ODsay가 정상 응답했지만 대중교통 후보가 없으면 `RT4040`이다.
- GraphHopper가 정상 응답했지만 추천 가능한 경로가 없거나, `hints[].details`/message가 `ConnectionNotFoundException` 또는 `Connection between locations not found`인 no-route 계열이면 `RT4040`이다.
- 초기 `search/walk`, `search/transit` WALK leg와 이 흐름을 재사용하는 재탐색 경로 계산은 GraphHopper snap 거리 10m 초과를 단독 실패로 보지 않는다. GraphHopper가 선택한 가장 가까운 routeable segment snap 결과를 사용한다.
- GraphHopper가 path를 반환하면 route availability는 그 계산 결과를 기준으로 판단한다.
- `details.walk_access`는 base graph EV를 반환할 수 있으므로 `RT4040` 판정 근거로 사용하지 않는다.
- 복구/재탐색 path 자체가 없을 때만 `RT4040`이다.
- GraphHopper 400이라도 profile명 오류, 파라미터 오류, 서버 config 문제처럼 no-route 힌트가 없는 잘못된 요청/설정 오류는 `EX5020`이다.
- BIMS가 정상 응답했지만 현재 도착 차량이 없으면 검색/refresh 실패가 아니며, `remainingMinute=null` 또는 `transits=[]`로 표현한다.
- BIMS 호출 자체 실패는 후보 도착정보만 null 처리 가능한 경우 degrade하고, 필수 흐름을 막으면 `EX5020`이다.
- 대중교통 search에서 일부 후보의 WALK 연결 구간만 실패하고 다른 후보가 있으면 해당 후보만 제외하고 검색은 성공할 수 있다.
- 대중교통 search에서 모든 후보가 GraphHopper no-route로 제외되면 최종 `RT4040`이다.

## 5. POST /routes/search/walk

| 항목     | 내용                                                                                                                                                                                                                                              |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 우선순위 | P1                                                                                                                                                                                                                                                |
| Method   | `POST`                                                                                                                                                                                                                                            |
| URL      | `/routes/search/walk`                                                                                                                                                                                                                             |
| 인증     | 필요                                                                                                                                                                                                                                              |
| 설명     | 사용자가 도보 탭을 클릭했을 때 출발지와 도착지를 기준으로 도보-only 경로 후보를 탐색한다. 사용자 유형과 보행약자 세부 유형은 토큰 기반으로 백엔드가 조회한다. 응답 route의 geometry/guidanceEvents는 후보 표시뿐 아니라 선택 후 GPS 기반 실시간 도보 안내에 사용된다. |

### 프론트 사용 시점 및 응답 처리

- 호출 시점: 사용자가 출발지/도착지를 입력하거나 선택한 뒤 도보 탭을 클릭했을 때 호출한다.
- 요청 좌표: `startPoint`는 사용자의 현재 GPS 또는 사용자가 지정한 출발지이고, `endPoint`는 목적지 좌표다.
- 응답 처리: `searchId`와 `routes[]`를 프론트 상태에 저장하고, route 후보 카드와 지도 polyline을 표시한다.
- 안내 처리: 사용자가 길 안내 시작 버튼을 누르기 전까지는 route 후보 상태다. `select` 성공 후에는 저장해 둔 route의 `geometry`, `legs[]`, WALK leg의 `guidanceEvents[]`를 기준으로 현재 GPS 위치에 맞는 화면 안내와 TTS를 갱신한다.
- 이탈 처리: 안내 중 현재 위치가 route geometry에서 10m 이상 벗어났고 GPS accuracy가 20m 이하이며, 이탈 상태가 2~3회 연속 확인되거나 3초 이상 지속되면 `POST /routes/reroute`를 호출한다.
- `RT4040` 처리: 프론트는 `routes[]`가 없는 확정 빈 상태로 보고 “탐색 가능한 경로가 없습니다.”를 표시한 뒤 조건 변경 재검색 안내와 두리발 호출 CTA를 함께 노출한다.

## 6. POST /routes/search/transit

| 항목     | 내용                                                                                                                                                                                                                                                                    |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 우선순위 | P1                                                                                                                                                                                                                                                                      |
| Method   | `POST`                                                                                                                                                                                                                                                                  |
| URL      | `/routes/search/transit`                                                                                                                                                                                                                                                |
| 인증     | 필요                                                                                                                                                                                                                                                                    |
| 설명     | 사용자가 대중교통 탭을 클릭했을 때 출발지와 도착지를 기준으로 대중교통 경로 후보를 탐색한다. ODsay 후보를 기반으로 BUS/SUBWAY/WALK leg를 구성하고, WALK leg는 GraphHopper로 재계산한다. 응답 route의 leg 정보는 선택 후 GPS 기반 안내와 정류장/역 접근 판단에 사용된다. |

### 프론트 사용 시점 및 응답 처리

- 호출 시점: 사용자가 출발지/도착지를 입력하거나 선택한 뒤 대중교통 탭을 클릭했을 때 호출한다.
- 응답 처리: `searchId`와 `routes[]`를 프론트 상태에 저장하고, 환승 횟수, BUS/SUBWAY leg, 탑승/하차 정류장 또는 역 정보를 후보 카드와 상세 경로에 표시한다.
- 안내 처리: `select` 성공 후에는 저장해 둔 route의 `legs[]` 순서에 따라 GPS 기반 안내를 진행한다. `WALK_TO_TRANSIT` leg에서는 도보 이벤트와 `BUS_STOP`/`SUBWAY_ELEVATOR`를 기준으로 탑승 지점 접근 알림을 갱신하고, BUS/SUBWAY `TRANSIT` leg에서는 `ARRIVING_POINT`를 기준으로 하차 지점 안내를 갱신하며, `TRANSIT_TO_WALK` leg에서는 도보 이벤트와 `DESTINATION`을 기준으로 최종 목적지 안내를 갱신한다.
- 도착정보 갱신: 사용자가 BUS leg의 `boardingStop` 30m 이내 또는 SUBWAY leg의 엘리베이터 25m 이내에 도착하면 해당 leg의 `sequence`로 `POST /routes/{routeId}/transit-refresh`를 호출한다.
- 이탈 처리: 안내 중 현재 위치가 route geometry에서 10m 이상 벗어났고 GPS accuracy가 20m 이하이며, 이탈 상태가 2~3회 연속 확인되거나 3초 이상 지속되면 `POST /routes/reroute`를 호출한다.
- `RT4040` 처리: 프론트는 `routes[]`가 없는 확정 빈 상태로 보고 “탐색 가능한 경로가 없습니다.”를 표시한 뒤 조건 변경 재검색 안내와 두리발 호출 CTA를 함께 노출한다.

### 비고

- 대중교통 검색 단계의 시간은 도착예정 시간이 아니라 예상 소요 시간이다.
- search 응답은 BUS `laneOptions[].remainingMinute`와 SUBWAY leg의 `remainingMinute`를 초기 도착예정/대기 시간으로 제공한다. 안내 중 최신 도착예정 시간은 `/routes/{routeId}/transit-refresh`에서 다시 조회한다.
- BIMS 조회는 검색 시점 ranking에 반영한다. 단, BIMS 응답은 성공했지만 현재 도착 차량이 없으면 대중교통 검색 성공 여부에는 영향을 주지 않는다.
- BUS leg는 같은 승하차 정류장을 공유하는 여러 버스 노선으로 이동 가능할 수 있으므로 `laneOptions[]`에 후보 버스들을 함께 제공한다. BUS leg에서 `laneOptions[]`는 필수이며, leg의 `routeNo`와 `isLowFloor`는 대표 후보 요약값이다.
- 검색 단계의 `laneOptions[].remainingMinute`는 해당 버스가 탑승 정류장에 도착하기까지 남은 대기 시간이다. BIMS 조회 결과 현재 도착정보가 없거나 파싱 가능한 값이 없으면 null이다. 사용자가 정류장 근처에 도착해 `transit-refresh`를 호출하면 최신 도착분은 refresh 응답의 `transits[]`로 표시한다.
- SUBWAY leg는 `laneOptions[]`를 사용하지 않는다. 지하철 호선은 `routeNo`, 다음 열차 대기 시간은 leg의 `remainingMinute`, 행방면은 `headsign`으로 내려준다. `headsign`은 `wayCode`로 조회한 시간표의 `endStationName`에 `행`을 붙여 만든다.
- 휠체어 사용자에게 저상버스가 없는 BUS 후보는 후보에서 제외하지 않고 `routes[].warnings[]`에 `"LOW_FLOOR_BUS_UNAVAILABLE"`을 포함한다. 프론트는 이 warning code를 보면 해당 route 카드 또는 안내 화면에 두리발 호출 CTA를 노출할 수 있다.
- 환승 시간 버퍼는 BUS 환승 3분, SUBWAY 환승 5분을 기본으로 한다. 휠체어 사용자의 SUBWAY 환승은 엘리베이터 이동 여유를 반영해 8분으로 계산한다.
- BUS 탑승 준비 시간은 기본 2분, 휠체어/시각장애 사용자는 3분으로 본다. 검색/랭킹 단계에서 BUS leg당 준비 시간을 `durationSecond`에 더해 소요 시간에 반영한다.
- SUBWAY leg의 엘리베이터 좌표는 별도 `elevator` 응답 필드로 노출하지 않는다. `boardingStop`/`arrivingStop`의 `name`은 역명으로 제공하고, 좌표만 실제 탑승/하차 안내 기준인 엘리베이터 입구 좌표로 보정한다. 탑승 전 엘리베이터 접근 안내는 `WALK_TO_TRANSIT` leg의 `guidanceEvents[].type="SUBWAY_ELEVATOR"`로 표현하고, 대중교통 탑승 이후 하차 안내는 SUBWAY `TRANSIT` leg의 `guidanceEvents[].type="ARRIVING_POINT"`로 표현한다. 경로 카드 요약 표시는 `badges=["ELEVATOR"]`로 유지한다.

## 7. POST /routes/{routeId}/select

| 항목     | 내용                                                                                                                                                                  |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 우선순위 | P1                                                                                                                                                                    |
| Method   | `POST`                                                                                                                                                                |
| URL      | `/routes/{routeId}/select`                                                                                                                                            |
| 인증     | 필요                                                                                                                                                                  |
| 설명     | 사용자가 경로 후보 카드의 길 안내 시작 버튼을 눌렀을 때 특정 route를 안내 대상으로 확정한다. 선택된 route는 후속 `transit-refresh`, `reroute`, `rating`에서 참조된다. |

### 프론트 사용 시점 및 응답 처리

- 호출 시점: `search/walk` 또는 `search/transit` 응답으로 표시한 후보 중 사용자가 길 안내 시작 버튼을 눌렀을 때 호출한다.
- 요청값: path의 `routeId`는 사용자가 선택한 후보 route의 ID이고, body의 `searchId`는 해당 후보가 포함된 검색 결과 묶음 ID다.
- 성공 처리: 응답 payload의 `data.sessionId`는 후속 rating의 평가 대상 안내 세션 ID다. 프론트는 `select` 성공을 안내 세션 시작 신호로 보고, 검색 응답에서 저장해 둔 route payload로 안내를 시작한다.
- 실패 처리: `RT4041` 또는 `RT4042`를 받으면 검색 후보가 만료되었거나 일치하지 않는 상태이므로 다시 검색하도록 안내한다. `RT4041`은 Redis miss를 포함한다.

| Authorization | Y    | Bearer 액세스 토큰 |

| 이름    | 필수 | 설명                |

#### Body

| 필드명   | 타입   | 필수 | 설명                                    |
| -------- | ------ | ---- | --------------------------------------- |
| searchId | string | Y    | 해당 routeId가 포함된 검색 결과 묶음 ID |


### 비고

- 같은 사용자가 같은 `searchId + routeId`를 중복 select하면 성공으로 처리한다.
- 같은 사용자의 같은 route에 대해 ACTIVE session은 하나만 유지한다. 백엔드는 Redis 후보 조회와 snapshot 생성을 먼저 끝낸 뒤, 짧은 DB transaction에서 기존 ACTIVE session 확인 및 저장을 수행한다. DB unique 제약 충돌이 `uk_route_sessions_user_active_route`에서 발생했고 이미 ACTIVE session이 확인되는 경우에만 중복 select 성공으로 처리한다.
- 기존 DB에 `active_route_key`가 비어 있는 ACTIVE session이 남아 있으면 select 처리 경계에서 최신 ACTIVE row의 `active_route_key`를 routeId로 보정하고, 같은 사용자/route의 나머지 ACTIVE row는 `COMPLETED`로 정리한다.
- select 이후에만 `transit-refresh`, `reroute`, `rating`이 정상 동작한다.

## 8. POST /routes/{routeId}/transit-refresh

| 항목     | 내용                                                                                                                                                        |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 우선순위 | P1                                                                                                                                                          |
| Method   | `POST`                                                                                                                                                      |
| URL      | `/routes/{routeId}/transit-refresh`                                                                                                                         |
| 인증     | 필요                                                                                                                                                        |
| 설명     | 선택 확정된 대중교통 경로 안내 중 사용자가 버스정류장 30m 이내 또는 지하철역 엘리베이터 25m 이내에 도착했을 때 특정 BUS/SUBWAY leg의 도착정보를 최신화한다. |

### 프론트 사용 시점 및 응답 처리

- 호출 시점: 대중교통 경로를 `select`한 뒤 해당 route session이 `ACTIVE`인 안내 중 상태에서만 호출한다. 사용자가 BUS leg의 `boardingStop` 30m 이내 또는 SUBWAY leg의 엘리베이터 25m 이내에 도착했다고 GPS로 판단한 시점에 호출한다.
- 요청값: `legSequence`에는 현재 도착정보를 확인해야 하는 BUS/SUBWAY leg의 `sequence`를 보낸다. BUS/SUBWAY가 아닌 leg의 `sequence`를 보내면 `PT4090`이 반환된다.
- 응답 처리: `arrivalStatus`로 사용자 문구를 결정하고, `transits[]`의 `remainingMinute`를 도착예정 시간으로 표시한다. BUS leg에 `laneOptions[]`가 있으면 각 option의 도착정보를 함께 갱신할 수 있다.
- 화면 갱신 범위: 이 API 응답은 도착정보만 갱신한다. 기존 route, leg, guidanceEvents, geometry, 지도 polyline은 변경하지 않는다.
- 반복 호출: 정류장/역 근처에서 사용자가 대기 중이면 짧은 주기로 재호출할 수 있지만, 프론트는 같은 leg에 대해 최소 30초 쿨다운을 둘 수 있고 백엔드는 arrival cache TTL 안에서는 캐시된 값을 반환할 수 있다.
- BUS 도착정보: 선택 snapshot의 backend metadata에 저장된 BIMS 정류장/노선 ID를 사용한다. Redis `bims:arrival:{bstopid}:{lineid}` cache hit이면 BIMS를 다시 호출하지 않고, miss이면 BIMS를 호출한 뒤 TTL 1분으로 저장한다. BIMS arrival cache는 성능 cache이므로 Redis 조회/저장 실패가 발생해도 search 결과 cache와 달리 API 실패로 보지 않고 BIMS 직접 호출/응답을 계속 진행한다.
- SUBWAY 도착정보: 선택 snapshot의 ODsay station id와 `wayCode`로 DB의 `subway_timetables`를 조회한다. 오늘 남은 출발이 없으면 다음 날짜의 `serviceDayType`으로 첫차를 조회한다. 1차 구현에서는 refresh 요청 중 지하철 외부 API를 직접 호출하지 않는다.
- 처리 범위: route session과 snapshot 복구만 짧은 read-only transaction에서 수행하고, BIMS/Redis cache 조회와 지하철 시간표 조회는 session 복구 transaction 밖에서 수행한다.

| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 우선순위 | P1                                                                                                                                                                    |
| Method   | `POST`                                                                                                                                                                |
| URL      | `/routes/reroute`                                                                                                                                                     |
| 인증     | 필요                                                                                                                                                                  |
| 설명     | 안내 중 GPS 정확도 20m 이하인 현재 위치가 route geometry에서 10m 이상 이탈한 상태가 2~3회 연속 확인되거나 3초 이상 지속될 때 현재 위치 기준으로 새 경로를 재탐색한다. |

### 프론트 사용 시점 및 응답 처리

- 호출 시점: 도보 또는 대중교통 안내 중 GPS 현재 위치가 route geometry에서 10m 이상 벗어났고 GPS accuracy가 20m 이하이며, 이탈 상태가 2~3회 연속 확인되거나 3초 이상 지속될 때 호출한다.
- 요청값: `routeId`와 `currentPoint`만 보낸다. `currentLegSequence`는 보내지 않는다.
- 성공 처리: 응답의 `route`가 있으면 현재 안내 중인 route payload를 응답 route로 교체하고, 새 `routeId`, 지도 polyline, leg/guidanceEvents 진행 상태를 새 route 기준으로 초기화한다.
- 추가 select 여부: reroute 성공으로 받은 새 route는 이미 안내 가능한 route session으로 저장된 상태이므로 `POST /routes/{routeId}/select`를 다시 호출하지 않는다.
- 트랜잭션 범위: 기존 session 조회와 snapshot 복구, GraphHopper/ODsay/BIMS 기반 재탐색 계산은 DB write transaction 밖에서 수행한다. 새 route session 저장만 짧은 DB transaction으로 분리한다.
- 새 경로가 필요 없음: `route`가 `null`이면 기존 안내를 유지하고, 사용자에게 별도 재탐색 완료 안내를 강하게 표시하지 않아도 된다.
- `RT4040` 처리: 재탐색 가능한 새 후보가 없는 상태이므로 기존 안내를 계속 성공 상태로 오해하지 않게 추천 경로 없음 상태로 전환하고, 조건 변경 재검색 안내와 두리발 호출 CTA를 노출한다.

- 백엔드는 `currentPoint`와 저장된 route geometry를 비교해 현재 leg를 추정한다.
- 백엔드는 초기 경로 검색과 재탐색 경로 계산에서 GraphHopper snap 거리 10m 초과를 단독 실패로 처리하지 않는다. GraphHopper가 선택한 가장 가까운 routeable segment snap 결과를 사용한다.

- 기존 route와 너무 멀면 백엔드는 full reroute 또는 `RT4091`을 반환할 수 있다.
- reroute 결과로 새 route가 반환되면 항상 새 routeId를 발급한다. 새 경로가 필요 없으면 `route:null`을 반환한다.
- `rerouteType`, `newRouteId`, `reason`, `firstLeg`, `secondLeg` 같은 backend-only 판단 필드는 FE 응답 계약에 노출하지 않는다.

### 프론트 사용 시점 및 응답 처리

- 호출 시점: 사용자가 안내 종료를 누르거나, 프론트가 목적지 도착을 확정해 `ARR-01`로 이동하기 직전에 호출한다.

- 중복 호출: 같은 사용자가 같은 routeId에 대해 중복 호출하면 성공으로 처리한다.

### Request

#### Header

| 이름          | 필수 | 설명               |
| ------------- | ---- | ------------------ |
| Authorization | Y    | Bearer 액세스 토큰 |
| Content-Type  | Y    | `application/json` |

#### Path Variable

| 이름    | 필수 | 설명              |
| ------- | ---- | ----------------- |
| routeId | Y    | 종료할 선택 경로 ID |

#### Body

없음.

### Response

#### Success

- HTTP Status: `200 OK`

`data.sessionId`는 종료된 route session ID다.

```json
{
  "status": "S2000",
  "data": {
    "sessionId": "00000000-0000-0000-0000-000000000099"
  },
  "message": "안내가 종료되었습니다."
}
```

### Error

| HTTP Status | status | message                              | 발생 조건                                  |
| ----------- | ------ | ------------------------------------ | ------------------------------------------ |
| 401         | A4010  | 인증이 필요합니다.                   | 토큰 누락/만료                             |
| 403         | A4030  | 접근할 수 없는 경로입니다.           | 다른 사용자의 route 접근                   |
| 404         | RT4043 | 선택한 경로 정보를 찾을 수 없습니다. | route session 없음 또는 snapshot 복구 불가 |
| 500         | I5000  | 서버 내부 오류가 발생했습니다.       | 서버 예외                                  |

### 비고

- `end`는 route session 상태를 `COMPLETED`로 전환한다.
- route session row와 snapshot은 평가 저장과 이력 확인을 위해 유지한다.
- 기존 DB에 `active_route_key`가 비어 있거나 중복 ACTIVE session이 남아 있으면 end 처리 경계에서도 select와 같은 보정을 먼저 수행한 뒤 ACTIVE session을 종료한다.
- 경로 북마크 저장은 이 API에서 처리하지 않고 북마크 API에서 처리한다.

## 11. POST /route-ratings

| 항목     | 내용                                                                                                                                         |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| 우선순위 | P1                                                                                                                                           |
| Method   | `POST`                                                                                                                                       |
| URL      | `/route-ratings`                                                                                                                             |
| 인증     | 필요                                                                                                                                         |
| 설명     | 안내 종료 후 사용자가 평가등록 버튼을 눌렀을 때 방금 안내받은 경로에 대한 별점을 저장한다. 선택된 route session이 없으면 평가를 저장하지 않는다. |

### 프론트 사용 시점 및 응답 처리

- 호출 시점: 도착 안내가 끝난 뒤 사용자가 평가등록 버튼을 눌렀을 때 호출한다.
- 요청값: `sessionId`는 `select` 또는 `end` 성공 응답으로 받은 안내 세션 ID다. 안내 중 reroute가 발생했다면 최신 안내 세션의 `sessionId`를 사용한다.
- 성공 처리: `ratingId`가 반환되면 평가 저장 완료로 처리하고 평가 UI를 닫거나 완료 상태로 전환한다.
- 실패 처리: 인증 오류는 로그인 흐름으로 연결하고, `RR4000`은 별점 선택 누락 또는 잘못된 값으로 안내한다. `RT4043`은 선택/안내한 route session이 없거나 복구할 수 없는 상태이므로 재검색 또는 안내 재시작 흐름으로 안내한다. `RT4092`는 안내가 아직 종료되지 않은 상태이므로 먼저 `end` 흐름을 완료한 뒤 평가를 요청한다.

### Request

#### Header

| 이름          | 필수 | 설명               |
| ------------- | ---- | ------------------ |
| Authorization | Y    | Bearer 액세스 토큰 |
| Content-Type  | Y    | `application/json` |

#### Body

| 필드명    | 타입   | 필수 | 설명              |
| --------- | ------ | ---- | ----------------- |
| sessionId | string | Y    | 평가 대상 안내 세션 ID |
| score     | int    | Y    | 별점. 1~5         |

#### Request Example

```json
{
  "sessionId": "00000000-0000-0000-0000-000000000099",
  "score": 5
}
```

### Response

- HTTP Status: `201 Created`

#### Data Fields

| 필드명   | 타입 | Null 가능 | 설명                |
| -------- | ---- | --------- | ------------------- |
| ratingId | Long | N         | 저장된 경로 평가 ID |

#### Success Example

```json
{
  "status": "S2010",
  "data": {
    "ratingId": 1
  },
  "message": "생성되었습니다."
}
```

### Error

| HTTP Status | status | message                               | 발생 조건                         |
| ----------- | ------ | ------------------------------------- | --------------------------------- |
| 400         | RR4000 | 경로 평가 요청값이 올바르지 않습니다. | sessionId 누락/형식 오류 또는 score 범위 오류 |
| 401         | A4010  | 인증이 필요합니다.                    | 토큰 누락/만료                    |
| 403         | A4030  | 접근할 수 없는 경로입니다.            | 다른 사용자의 route session 평가 시도 |
| 404         | RT4043 | 선택한 경로 정보를 찾을 수 없습니다.  | route session 없음 또는 snapshot 복구 불가 |
| 409         | RT4092 | 종료된 안내 세션만 평가할 수 있습니다. | route session이 아직 ACTIVE 상태 |
| 500         | I5000  | 서버 내부 오류가 발생했습니다.        | 서버 예외                         |

### 비고

- 같은 사용자가 같은 route session에 다시 평가하면 최신 score로 갱신한다.
- route rating은 `COMPLETED` 상태의 route session에만 저장할 수 있다. 평가 시점에는 해당 `route_sessions.session_id`와 `route_snapshot_json`을 평가에 연결하며, snapshot이 없으면 저장하지 않는다.
- route session 1개에는 rating이 없거나 최대 1개만 연결된다. 중복 생성 race는 `UNIQUE(session_id)` 제약 위반을 확인한 뒤 기존 rating을 재조회해 갱신한다.

## 12. 프론트 구현 메모

- 모든 API 요청에 Authorization 헤더를 포함한다.
- 도보 탭 클릭 시 `POST /routes/search/walk`, 대중교통 탭 클릭 시 `POST /routes/search/transit`을 호출한다.
- 장소명/주소 검색은 별도 장소 검색 API를 사용하고, 경로 API에는 확정된 `startPoint`, `endPoint` 좌표만 보낸다.
- ROUTE-01 후보 카드의 선택 상태는 FE local state로 관리한다. API 응답에 선택 상태 필드를 요구하지 않는다.
- search 응답의 `searchId`와 `routes[]`를 저장하고, 사용자가 길 안내 시작 버튼을 누르면 `POST /routes/{routeId}/select`를 호출한다.
- `select` 성공 응답의 `data.sessionId`를 저장하되, route 화면은 기존에 저장한 search 응답 route payload로 안내를 시작한다.
- `estimatedTimeMinute`는 “소요 시간”으로 표시한다.
- 안내 중 남은 시간은 `durationSecond`와 geometry 진행률을 기준으로 FE가 계산한다.
- `remainingMinute`는 search 응답의 BUS `laneOptions[]`, SUBWAY leg, `transit-refresh` 응답에서 “도착예정/대기 시간”으로 표시한다.
- `arrivalStatus=NO_CURRENT_ARRIVAL`은 실패가 아니다.
- `WALK_TO_TRANSIT` 안내 중 버스정류장 30m 이내 또는 지하철역 엘리베이터 25m 이내에 도착하면 곧 탑승할 BUS/SUBWAY leg의 `sequence`로 `transit-refresh`를 호출한다.
- 안내 중 현재 위치가 route geometry에서 10m 이상 벗어났고 GPS accuracy가 20m 이하이며, 이탈 상태가 2~3회 연속 확인되거나 3초 이상 지속되면 `POST /routes/reroute`를 호출한다.
- reroute 성공 응답에 `route`가 있으면 현재 안내 route를 응답 route로 교체하고 추가 `select`는 호출하지 않는다.
- 안내 종료 또는 목적지 도착 시 `POST /routes/{routeId}/end`를 호출한다. request body는 보내지 않고, 성공 응답의 `data.sessionId`를 평가 요청에 사용한다.
- 안내 종료 후 평가등록 버튼을 누르면 `sessionId`, `score`로 `POST /route-ratings`를 호출한다.
- 경로 저장은 북마크 API로 처리한다.
- 저시력자 `LV-RES-01`의 길찾기는 search 후 기본 후보를 FE가 자동 선택하고 `select`까지 자동 호출한 뒤 `LV-NAV-01`로 진입한다.
- `RT4003`은 지도/좌표 선택 범위를 부산광역시 안으로 유도한다.
- `RT4040`은 추천 또는 재탐색 가능한 경로가 없는 확정 빈 상태다. 프론트는 “탐색 가능한 경로가 없습니다.”를 표시하고 조건 변경 재검색 안내와 두리발 호출 CTA를 함께 노출한다.
- 두리발 호출 CTA는 서버 API를 추가 호출하지 않고 프론트의 전화 연결 또는 외부 호출 흐름으로 처리한다.
- `RT4041`, `RT4043`은 재검색 흐름으로 유도한다.
- `EX5020`, `EX5040`은 외부 API 제공기관명을 노출하지 않고 “잠시 후 다시 시도”로 안내한다.
- reroute 요청에는 `currentLegSequence`를 보내지 않는다.
