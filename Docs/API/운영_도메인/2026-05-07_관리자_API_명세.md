| P1 | `POST /admin/hazard-reports/{reportId}/route-review/complete` | 제보 경로 검수 완료 |

> 관리자 API도 API 전용 서브도메인 기준으로 노출하므로 기본 path prefix로 `/api`를 붙이지 않는다.
> 관리자 기능은 로그인 사용자 중 `users.role=ADMIN`인 계정만 사용할 수 있다.
> 백엔드는 해당 사용자에게 Spring Security `ROLE_ADMIN` 권한을 부여하고 `/admin/**`를 `ADMIN` 권한으로 제한한다.
> 관리자 지도 화면은 CSV editor 서버를 호출하지 않고, 백엔드 RDB 조회 API만 사용한다.
> 구/동 selector와 보행 네트워크 필터를 사용하려면 행정동 GeoJSON을 `admin_areas`에 먼저 적재해야 한다.
> 보행 네트워크와 장소/접근성 수정 API는 `admin_area_assignments`에 지정된 담당 구/동만 수정할 수 있다.
> dev/prod 적재 스크립트는 각각 `scripts/db/load_admin_areas_dev.sh`, `scripts/db/load_admin_areas_prod.sh`를 사용한다.
> 편의시설 조회를 사용하려면 `places_erd.csv`, `place_accessibility_features_erd.csv`를 `scripts/db/load_places_dev.sh` 또는 `scripts/db/load_places_prod.sh`로 먼저 적재해야 한다.

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P1 |
| Method | `GET` |
| URL | `/admin/dashboard/bottlenecks` |
| 인증 | 필요, 관리자 권한 필요 |
| 설명 | `route_sessions.route_snapshot_json`의 도보 경로 geometry와 `hazard_reports` 근접 신고 수를 기반으로 관리자 홈 병목 후보 지도 데이터를 조회한다. |

### Request

| 변수명 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| from | date | N | 조회 시작일. 생략하면 종료일 기준 최근 7일 |

- 현재 응답은 실측 이동속도가 아니라 DB에 저장된 선택 경로의 예상 속도와 신고 근접도를 조합한 병목 후보다.
- 실제 사용자 위치 샘플/속도 텔레메트리가 도입되기 전까지 `telemetryBased=false`로 내려준다.
- 대중교통 전체 geometry는 버스/지하철 선형이 섞일 수 있으므로 `transportMode=WALK` 경로만 후보 집계에 포함한다.
- 신고 수는 후보 경로 50m 이내 `hazard_reports.report_point` 개수를 사용한다.

## 1-3. 관리자 경로/이동 통계 조회

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P1 |
| Method | `GET` |
| URL | `/admin/dashboard/route-stats` |
| 인증 | 필요, 관리자 권한 필요 |
| 설명 | `route_sessions.route_snapshot_json`과 `users` 실데이터를 기반으로 관리자 경로/이동 통계 화면 초기 데이터를 조회한다. |

### Request

| 변수명 | 타입 | 필수 | 설명 |

### 처리 규칙

- 현재 응답은 실시간 GPS telemetry가 아니라 `route_sessions.route_snapshot_json`에 저장된 거리, 소요시간, geometry를 집계한 운영 통계다.
- 이동 유형은 `users.selected_primary_user_type`, `users.selected_mobility_subtype`를 조합해 아래 네 그룹으로 정규화한다.
  - `MOBILITY_SUPPORT` = 보행약자
  - `POWER_WHEELCHAIR` = 자동휠체어
  - `MANUAL_WHEELCHAIR` = 수동휠체어
  - `VISUAL_IMPAIRMENT` = 시각장애인
- 주요 이동 구간은 동일 `geometry`를 같은 대표 이동축으로 묶어 집계한다.
- 주요 이동 구간명은 `route_snapshot_json.title`이 아니라 geometry의 시작/종료 지점을 `admin_areas`와 교차시켜 만든 행정동 기반 운영용 표기다.
- 시간대 분포와 평균 속도는 모두 세션 `created_at`을 `Asia/Seoul` 기준으로 4시간 버킷(`00-04`, `04-08`, ..., `20-24`)으로 집계한다.

## 1-4. 관리자 병목구간 통계 조회

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P1 |
| Method | `GET` |
| URL | `/admin/dashboard/bottleneck-monitoring` |
| 인증 | 필요, 관리자 권한 필요 |
| 설명 | `route_sessions.route_snapshot_json`, `hazard_reports`, `road_segments`, `admin_areas` 실데이터를 기반으로 병목구간 통계 화면 초기 데이터를 조회한다. |

### Request

| 변수명 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| from | date | N | 조회 시작일. 생략하면 종료일 기준 최근 7일 |
| to | date | N | 조회 종료일. 생략하면 오늘 |

### 처리 규칙

- 병목 후보는 `transportMode=WALK`이고 `geometry`가 있는 `route_sessions`를 동일 `geometry` 기준으로 묶어 생성한다.
- 지도/목록의 대표 이름은 `route_snapshot_json.title`이 아니라 geometry 시작점/종료점을 `admin_areas`와 교차한 행정동 기반 운영용 표기를 사용한다.
- `실제 병목구간`은 `근접 신고(report_count > 0)` 또는 `예상 평균 속도(avg_speed_mps <= 1.0)`를 만족하는 후보 수다.
- `영향을 받은 사용자`는 실제 병목구간 후보들의 `distinct user` 합계다.
- `해결 완료`는 현재 별도 해소 이력이 없어, 후보 주변 `hazard_reports` 중 `APPROVED` 또는 `REJECTED` 상태가 1건 이상 있는 후보 수를 proxy로 사용한다.

### Request Query

| 변수명 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| gu | string | Y | 담당 구 |
| dong | string | Y | 담당 동 |

### Request Body

| 필드명 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| walkAccess | string | N | `YES`, `NO`, `UNKNOWN` |
| brailleBlockState | string | N | `YES`, `NO`, `UNKNOWN` |
| audioSignalState | string | N | `YES`, `NO`, `UNKNOWN` |

### 처리 규칙

- segment 속성 수정은 DB `road_segments`에 먼저 반영한다.
- 요청에 포함된 overlay 대상 4컬럼(`walkAccess`, `stairsState`, `widthState`, `brailleBlockState`)만 `routing_segment_overrides` current-state row에 patch한다. 미포함 필드는 기존 overlay 값을 유지한다.
- 요청에 포함된 overlay 대상 필드를 patch한 결과 네 값이 모두 `NULL`이면 overlay row를 삭제한다.
- overlay 대상 필드가 하나라도 저장되면 `routing_apply_states.dirty=true`로 마킹하고 `routingApplyStatus=PENDING`을 반환한다.
- overlay 대상 필드가 하나도 없으면 `routingApplyStatus=SKIPPED`를 반환한다.
- GraphHopper runtime reload는 이 API에서 직접 호출하지 않는다. 저장된 변경은 별도의 `POST /admin/routing/overrides/apply` 실행 시점에만 반영한다.
- overlay reload는 `db_edge_id`, `walk_access`, `stairs_state`, `width_state`, `braille_block_state` encoded value가 포함된 graph-cache에서만 동작한다. 최초 1회는 해당 EV 추가 후 full rebuild가 선행되어야 한다.
- GraphHopper route 계산의 source of truth는 `INF/graphhopper/custom_models/*.json`이다. Runtime overlay는 final weight 보정이 아니라 delegate/custom model 호출 전에 `EdgeIteratorState` wrapper가 effective EV 값을 반환하는 방식이다.
- overlay wrapper는 custom model이 읽는 EV를 기준으로 적용한다. `pedestrian_*`, `visual_*`, `wheelchair_*`는 `walk_access`, `stairs_state`, `width_state` overlay를 읽고, `visual_*`는 추가로 `braille_block_state` overlay를 읽는다.
- `routingApplyStatus`는 `PENDING` 또는 `SKIPPED`를 사용한다.
- `PENDING`은 DB 저장은 끝났지만 GraphHopper runtime에는 아직 반영되지 않은 상태다.
- GraphHopper admin reload는 overlay current-state 컬럼이 없으면 실패로 응답해야 한다. Startup reload는 초기 배포 관용을 위해 빈 overlay로 시작할 수 있지만, 수동 경로 반영은 성공처럼 보이면 안 된다.
- Route calculation은 overlay effective EV를 반영하지만 GraphHopper response details/guidance가 base graph EV를 읽을 수 있다. 앱 배지/안내 문구까지 overlay current-state와 일치해야 하면 별도 response detail overlay 설계가 필요하다.

### 배포 순서

1. GraphHopper build image/config에 `db_edge_id` encoded value와 overlay reload endpoint를 반영한다.
2. 대상 slot에서 최신 DB 기준으로 graph-cache를 full rebuild 한다.
3. rebuild된 cache를 사용하는 GraphHopper runtime을 기동한다.
4. overlay reload 대상은 Redis active slot과 backend fallback previous slot이다. candidate/drain slot을 patch 대상에 포함할지는 운영 정책으로 별도 관리한다.
5. 그 다음 BE overlay reload 호출 버전을 배포한다.

### Smoke
- patch 전후 같은 OD 경로가 실제로 바뀌는지 `scripts/graphhopper/smoke_walk_access_hot_patch.py`로 검증한다.
- 이 smoke는 patch 전 route에 target `db_edge_id`가 포함되는지 확인하고, patch 후에는 해당 `db_edge_id`가 route에서 사라지거나 no-route가 되는 것을 성공으로 본다.

- smoke는 patch 전 `routing_segment_overrides` row의 4컬럼 current-state를 저장하고, 검증 후 같은 row state로 restore해야 한다. 원래 값 추출이나 restore에 실패하면 smoke도 실패로 처리한다.

## 5. 관리자 보행 네트워크 편집 반영

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P1 |
| Method | `POST` |
| URL | `/admin/road-network/edits/apply` |
| 인증 | 필요, 관리자 권한 필요 |
| 설명 | 관리자 화면의 로컬 draft를 DB `road_nodes`, `road_segments`, `segment_features`에 트랜잭션으로 반영한다. |

### 처리 규칙

- `add_segment`는 양 끝점 기준 1.0m 이내의 기존 `road_nodes`가 있으면 해당 node에 snap한다.
- 1.0m 이내 node가 없으면 새 `road_nodes`를 생성한다.
- `CROSS_WALK` 추가 시 양 끝점이 1.0m 이내 기존 node에 snap되지 않으면, 1.5m 이내 기존 `SIDE_LINE` 선 위의 가장 가까운 점으로 projection 보정한다.
- `CROSS_WALK` projection 지점이 기존 `SIDE_LINE` segment 중간이면, 기존 segment를 split하고 새 `CROSS_WALK`와 split segment가 같은 `road_nodes.vertex_id`를 공유하도록 저장한다.
- split으로 생성된 기존 segment 조각은 원래 segment의 접근성 집계 컬럼을 우선 보존하고, `segment_features`는 `source_features` 공간 매칭으로 다시 생성한다.
- 새 `road_segments.edge_id`, `road_nodes.vertex_id`, `segment_features.feature_id`는 DB sequence를 사용한다.
- 새 segment의 접근성 속성은 `source_features`와 공간 매칭해 `segment_features` 및 `road_segments` 집계 컬럼에 반영한다.
- add 작업은 요청 edit을 DB 임시 테이블에 적재한 뒤 node snap, node 생성, segment 생성, `source_features` 매칭을 set-based SQL로 처리한다.
- `delete_segment`는 `segment_features` 삭제 후 `road_segments`를 삭제한다.
- `delete_node`는 해당 node에 연결된 segment까지 삭제 대상으로 포함한다.
- 반영 마지막에 더 이상 참조되지 않는 `road_nodes`를 정리한다.
- `source_features`는 원천 참조 테이블이므로 수정하지 않는다.
- 요청한 관리자가 담당자로 지정된 구/동 안의 segment만 추가/삭제할 수 있다.

### Request

| 필드명 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| gu | string | Y | 담당 구 |
| dong | string | Y | 담당 동 |
| edits | array | Y | 반영할 draft 목록 |
| edits[].action | string | Y | `add_segment`, `delete_segment`, `delete_node` |
| edits[].segmentType | string | N | `add_segment`일 때 segment 유형. `SIDE_LINE`, `CROSS_WALK` |
| edits[].geom.type | string | N | `add_segment`일 때 `LineString` |
| edits[].geom.coordinates | array | N | `add_segment`일 때 `[lng, lat]` 좌표 배열 |
| edits[].edgeId | Long | N | `delete_segment` 대상 segment ID |
| edits[].vertexId | Long | N | `delete_node` 대상 node ID |
| edits[].reason | string | N | 관리자 화면에서 기록한 편집 사유 |

  "dong": "명지동",

        "type": "LineString",

## 17. 제보 경로 검수 완료

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P1 |
| Method | `POST` |
| URL | `/admin/hazard-reports/{reportId}/route-review/complete` |
| 인증 | 필요, 관리자 권한 필요 |
| 설명 | 진행 중인 경로 검수를 완료하고 제보 최종 상태를 갱신한다. |

### 처리 규칙

- 승인 검수 완료 시 제보 상태는 `APPROVED`가 된다.
- 원상복구 검수 완료 시 제보 상태는 `APPROVED`를 유지하고, 최신 처리 메타데이터와 완료된 복구 검수 이력만 갱신한다.
- 검수 완료 전에는 최소 1개 이상의 세그먼트 draft가 저장되어 있어야 한다.
- 완료 시 저장된 draft의 세그먼트 속성을 `road_segments` 원본 truth에 저장한다.
- `walk_access`, `stairs_state`, `width_state`, `braille_block_state` 값이 포함된 draft는 `routing_segment_overrides` current-state를 갱신한다.
- overlay 대상 draft가 하나 이상 적용되면 `routing_apply_states.dirty=true`로 마킹하고 `routingApplyStatus=PENDING`을 반환한다.
- overlay 대상 변경이 없으면 `routingApplyStatus=SKIPPED`를 반환한다.
- GraphHopper runtime reload는 검수 완료 시점에 직접 호출하지 않는다.

### Response

- HTTP Status: `200 OK`
- 응답 형식은 `latestRouteReview`와 동일하다.
- `routingApplyStatus`는 `PENDING | SKIPPED`를 사용한다.

| --- | --- |

| Method | `GET` |

## 19. 관리자 경로 반영 실행

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P1 |
| Method | `POST` |
| URL | `/admin/routing/overrides/apply` |
| 인증 | 필요, 관리자 권한 필요 |
| 설명 | 저장된 override 최신 상태 전체를 기준으로 GraphHopper runtime reload를 1회 수행한다. |

### 처리 규칙

- `routing_apply_states` singleton row를 lock해서 동시 실행을 막는다.
- stale lock 복구는 고정 시간 상수가 아니라 GraphHopper admin reload client timeout window를 기준으로 한다. 기본 설정에서는 `connect 5s + read 5s`, 최대 `2 endpoints x 2 attempts`에 buffer를 더한 뒤 그 시간을 넘긴 경우에만 `applying=true`를 stale로 간주한다.
- `dirty=false`이고 `applying=false`면 `SKIPPED`를 반환한다.
- `dirty=true`면 GraphHopper reload를 1회 호출한다.
- 성공 시 `dirty=false`, `applying=false`, `lastAppliedAt`을 갱신한다.
- 실패 시 DB 저장 내용은 롤백하지 않고 `dirty=true`를 유지한다.

### Response

- HTTP Status: `200 OK`

| 필드명 | 타입 | Null 가능 | 설명 |
| --- | --- | --- | --- |
| routingApplyStatus | string | N | `APPLIED`, `APPLIED_WITH_WARNING`, `FAILED`, `SKIPPED` |
| message | string | Y | 반영 결과 메시지 |
| dirty | boolean | N | runtime 미반영 변경 존재 여부 |
| applying | boolean | N | 현재 반영 작업 실행 중 여부 |
| lastAppliedAt | datetime | Y | 마지막 반영 완료 시각 |

### Error

| HTTP Status | status | message | 발생 조건 |
| --- | --- | --- | --- |
| 401 | A4010 | 인증이 필요합니다. | 토큰 누락/만료 |
| 403 | A4030 | 권한이 없습니다. | `users.role`이 `ADMIN`이 아님 |
| 409 | C4090 | 이미 다른 관리자가 경로 반영을 실행 중입니다. | 중복 실행 |

### Error

| HTTP Status | status | message | 발생 조건 |
| --- | --- | --- | --- |
| 400 | HR4003 | 제보 경로 검수 요청값이 올바르지 않습니다. | 저장된 세그먼트 draft가 없음 |
| 401 | A4010 | 인증이 필요합니다. | 토큰 누락/만료 |
| 403 | A4030 | 권한이 없습니다. | `users.role`이 `ADMIN`이 아님 |
