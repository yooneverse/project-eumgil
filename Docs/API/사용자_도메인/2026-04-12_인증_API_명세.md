## 2. 소셜 로그인

### 기본 정보

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P1 |
| Method | `POST` |
| URL | `/auth/social-login` |
| 인증 | 불필요 |
| 설명 | 소셜 제공자 토큰을 검증한다. 이미 가입 완료된 사용자는 서비스 토큰을 발급하고, 신규 사용자는 회원가입 완료용 `signupToken`만 반환한다. |

### Request

#### Header

| 이름 | 필수 | 설명 |
| --- | --- | --- |
| Content-Type | Y | `application/json` |

#### Body

| 필드명 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| socialProvider | string | Y | 소셜 제공자. `KAKAO`, `NAVER`, `GOOGLE` |
| socialAccessToken | string | Y | 소셜 제공자에서 발급받은 액세스 토큰 |

#### Request Example

```json
{
  "socialProvider": "KAKAO",
  "socialAccessToken": "kakao-access-token"
}
```

### Response

#### Success

- HTTP Status: `200 OK`

#### Data Fields

| 필드명 | 타입 | Null 가능 | 설명 |
| --- | --- | --- | --- |
| signupRequired | boolean | N | 회원가입 완료가 필요한 신규 사용자 여부 |
| signupToken | string | Y | 신규 사용자 회원가입 완료용 임시 토큰. `signupRequired=true`일 때만 값 존재 |
| accessToken | string | Y | 서비스 액세스 토큰. 기존 회원일 때만 값 존재 |
| refreshToken | string | Y | 서비스 리프레시 토큰. 기존 회원일 때만 값 존재 |
| userId | string(UUID) | Y | 사용자 ID. 기존 회원일 때만 값 존재 |
| selectedPrimaryUserType | string | Y | 기존 회원의 1차 사용자 유형 |
| selectedMobilitySubtype | string | Y | 기존 회원의 보행약자 세부 유형. 저시력자는 `null` |

#### 비고

- 소셜 OAuth 인증만 완료된 상태는 서비스 회원가입 완료가 아니다.
- 신규 사용자는 `users` row를 만들지 않고 `signupToken`으로 온보딩/필수 약관 단계로 이동한다.
- `POST /auth/signup`에서 필수 약관 동의와 사용자 유형이 모두 확인되면 `users` row를 생성하고 서비스 토큰을 발급한다.
- 가입 완료 사용자 테이블에는 앱 닉네임, 온보딩 완료 여부, 약관 동의 여부, TTS 설정, 경로 수집 설정을 저장하지 않는다.
- 인증/사용자 응답은 관리자 권한 정보를 포함하지 않는다. 관리자 권한은 `users.role=ADMIN` 기준으로 판정한다.
- API 응답과 JWT subject에 사용하는 사용자 식별자는 UUID 문자열이다.
- 클라이언트 요청에서 임의의 `userId`를 받아 사용자 권한을 판단하지 않는다. 내 정보 처리는 토큰 기준의 `/users/me`로 처리한다.
- 기존 회원 로그인 성공 시 브라우저 클라이언트를 위해 리프레시 토큰을 `HttpOnly` 쿠키로도 내려준다. 기존 모바일/테스트 클라이언트 호환을 위해 응답 body의 `refreshToken`은 유지한다.

#### Existing User Example

```json
{
  "status": "S2000",
  "data": {
    "signupRequired": false,
    "signupToken": null,
    "accessToken": "access-token",
    "refreshToken": "refresh-token",
    "userId": "018f7f6c-2b7e-7c3a-9f4a-8b4e3b7c9a01",
    "selectedPrimaryUserType": "MOBILITY_IMPAIRED",
    "selectedMobilitySubtype": "MANUAL_WHEELCHAIR"
  },
  "message": "정상 처리되었습니다."
}
```

#### New User Example

```json
{
  "status": "S2000",
  "data": {
    "signupRequired": true,
    "signupToken": "signup-token",
    "accessToken": null,
    "refreshToken": null,
    "userId": null,
    "selectedPrimaryUserType": null,
    "selectedMobilitySubtype": null
  },
  "message": "정상 처리되었습니다."
}
```

### Error

| HTTP Status | status | message | 발생 조건 |
| --- | --- | --- | --- |
| 400 | A4000 | 잘못된 입력입니다. | `socialProvider`, `socialAccessToken` 누락 또는 형식 오류 |
| 401 | A4011 | 인증에 실패했습니다. | 소셜 액세스 토큰이 유효하지 않음 |
| 502 | A5020 | 소셜 로그인 연동에 실패했습니다. | 소셜 제공자 API 호출 실패 |

## 3. 회원가입 완료

### 기본 정보

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P1 |
| Method | `POST` |
| URL | `/auth/signup` |
| 인증 | 불필요 |
| 설명 | 소셜 로그인에서 받은 `signupToken`과 온보딩 선택값으로 서비스 회원가입을 완료한다. |

### Request

#### Header

| 이름 | 필수 | 설명 |
| --- | --- | --- |
| Content-Type | Y | `application/json` |

#### Body

| 필드명 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| signupToken | string | Y | `POST /auth/social-login`에서 받은 회원가입 완료용 임시 토큰 |
| selectedPrimaryUserType | string | Y | 1차 사용자 유형. `LOW_VISION`, `MOBILITY_IMPAIRED` 중 하나 |
| selectedMobilitySubtype | string | 조건부 | 보행약자 세부 유형. `selectedPrimaryUserType=MOBILITY_IMPAIRED`이면 필수, `LOW_VISION`이면 `null` 또는 미전송 |
| requiredTermsAccepted | boolean | Y | 필수 약관 동의 여부. `true`만 가입 완료 가능 |

#### Request Example

```json
{
  "signupToken": "signup-token",
  "selectedPrimaryUserType": "MOBILITY_IMPAIRED",
  "selectedMobilitySubtype": "MANUAL_WHEELCHAIR",
  "requiredTermsAccepted": true
}
```

### Response

- HTTP Status: `201 Created`

| 필드명 | 타입 | Null 가능 | 설명 |
| --- | --- | --- | --- |
| accessToken | string | N | 서비스 액세스 토큰 |
| refreshToken | string | N | 서비스 리프레시 토큰 |
| userId | string(UUID) | N | 생성된 사용자 ID |
| selectedPrimaryUserType | string | N | 저장된 1차 사용자 유형 |
| selectedMobilitySubtype | string | Y | 저장된 보행약자 세부 유형. 저시력자는 `null` |

#### 비고

- 이 API가 성공한 뒤에만 `users` row가 생성된다.
- `requiredTermsAccepted`는 가입 완료 조건 검증용 요청값이다. `users` 테이블에 별도 약관 동의 여부 필드를 저장하지 않는다.
- `selectedPrimaryUserType=LOW_VISION`이면 서버는 `selectedMobilitySubtype`을 저장하지 않는다.
- `selectedPrimaryUserType=MOBILITY_IMPAIRED`이면 `selectedMobilitySubtype`이 반드시 필요하다.
- 신규 가입자는 `users.role=USER`로 가입된다. 관리자 권한은 운영 정책에 따라 `users.role=ADMIN`으로 변경된 계정에만 부여한다.
- 회원가입 성공 시 브라우저 클라이언트를 위해 리프레시 토큰을 `HttpOnly` 쿠키로도 내려준다. 기존 모바일/테스트 클라이언트 호환을 위해 응답 body의 `refreshToken`은 유지한다.

#### Success Example

```json
{
  "status": "S2010",
  "data": {
    "accessToken": "access-token",
    "refreshToken": "refresh-token",
    "userId": "018f7f6c-2b7e-7c3a-9f4a-8b4e3b7c9a01",
    "selectedPrimaryUserType": "MOBILITY_IMPAIRED",
    "selectedMobilitySubtype": "MANUAL_WHEELCHAIR"
  },
  "message": "생성되었습니다."
}
```

### Error

| HTTP Status | status | message | 발생 조건 |
| --- | --- | --- | --- |
| 400 | A4000 | 잘못된 입력입니다. | 사용자 유형 조합 오류, 필수 약관 미동의 |
| 401 | A4013 | 회원가입 토큰이 유효하지 않습니다. | `signupToken` 만료 또는 위조 |
| 409 | A4090 | 이미 가입된 사용자입니다. | 같은 소셜 계정으로 이미 가입 완료된 경우 |

## 4. 토큰 재발급

### 기본 정보

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P1 |
| Method | `POST` |
| URL | `/auth/reissue` |
| 인증 | 불필요 |
| 설명 | 리프레시 토큰으로 액세스 토큰을 재발급한다. 브라우저 클라이언트는 `HttpOnly` 쿠키의 리프레시 토큰으로 요청할 수 있다. |

### Request

#### Header

| 이름 | 필수 | 설명 |
| --- | --- | --- |
| Content-Type | N | body로 `refreshToken`을 전달할 때만 `application/json` |
| Cookie | N | `refreshToken` 쿠키. body의 `refreshToken`이 없을 때 사용 |

#### Body

| 필드명 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| refreshToken | string | N | 리프레시 토큰. 없으면 `HttpOnly` 쿠키의 `refreshToken`을 사용 |

#### Request Example

```json
{
  "refreshToken": "refresh-token"
}
```

### Response

#### Success

- HTTP Status: `200 OK`

#### Data Fields

| 필드명 | 타입 | Null 가능 | 설명 |
| --- | --- | --- | --- |
| accessToken | string | N | 새 액세스 토큰 |
| refreshToken | string | N | 새 리프레시 토큰. 브라우저 클라이언트를 위해 같은 값이 `HttpOnly` 쿠키로도 내려간다. |

#### Success Example

```json
{
  "status": "S2000",
  "data": {
    "accessToken": "new-access-token",
    "refreshToken": "new-refresh-token"
  },
  "message": "정상 처리되었습니다."
}
```

### Error

| HTTP Status | status | message | 발생 조건 |
| --- | --- | --- | --- |
| 400 | A4000 | 잘못된 입력입니다. | body가 존재하지만 `refreshToken` 형식이 잘못됨 |
| 401 | A4012 | 인증이 필요합니다. | body와 쿠키 모두에 리프레시 토큰이 없거나, 리프레시 토큰 만료 또는 위조 |

### FE refresh 처리 계약

- 보호 API에서 액세스 토큰이 없거나, 만료되었거나, 위조되었거나, 로그아웃으로 차단된 경우 HTTP `401`과 status `A4010`을 반환한다.
- FE는 `401` + `A4010`을 받으면 `POST /auth/reissue`를 1회 호출해 토큰을 갱신한 뒤 원 요청을 재시도한다.
- `POST /auth/reissue`는 body의 `refreshToken`을 우선 사용하고, body가 없으면 `HttpOnly` 쿠키 `refreshToken`을 사용한다.
- `POST /auth/reissue`가 HTTP `401` + status `A4012`를 반환하면 refresh token도 사용할 수 없는 상태로 보고 저장 토큰을 제거한 뒤 로그인 화면으로 이동한다.
- 서버 토큰 TTL은 access token `15분`, refresh token `14일`이다.

## 5. 로그아웃

### 기본 정보

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P1 |
| Method | `POST` |
| URL | `/auth/logout` |
| 인증 | 필요 |
| 설명 | 현재 로그인 사용자의 세션 또는 리프레시 토큰을 무효화하고 브라우저용 리프레시 토큰 쿠키를 제거한다. |

### Request

#### Header

| 이름 | 필수 | 설명 |
| --- | --- | --- |
| Authorization | Y | Bearer 액세스 토큰 |
| Content-Type | Y | `application/json` |

#### Body

> 없음

### Response

#### Success

- HTTP Status: `200 OK`

#### Success Example

```json
{
  "status": "S2000",
  "data": null,
  "message": "로그아웃되었습니다."
}
```

#### 비고

- 현재 access token을 blacklist에 등록하고 해당 사용자의 refresh token을 삭제한다.
- 이미 만료된 access token은 blacklist 저장을 생략할 수 있다.

### Error

| HTTP Status | status | message | 발생 조건 |
| --- | --- | --- | --- |
| 401 | A4010 | 인증이 필요합니다. | 토큰 누락/만료 |

## 6. 회원탈퇴

### 기본 정보

| 항목 | 내용 |
| --- | --- |
| 우선순위 | P2 |
| Method | `DELETE` |
| URL | `/users/me` |
| 인증 | 필요 |
| 설명 | 현재 로그인 사용자의 계정을 삭제한다. 북마크, 저장 경로, 안내 세션, 별점 평가, 제보 내역을 삭제한다. |

### Request

#### Header

| 이름 | 필수 | 설명 |
| --- | --- | --- |
| Authorization | Y | Bearer 액세스 토큰 |

#### Body

> 없음

### Response

#### Success

- HTTP Status: `200 OK`

#### Success Example

```json
{
  "status": "S2000",
  "data": null,
  "message": "회원탈퇴가 완료되었습니다."
}
```

### Error

| HTTP Status | status | message | 발생 조건 |
| --- | --- | --- | --- |
| 401 | A4010 | 인증이 필요합니다. | 토큰 누락/만료 |
| 404 | U4040 | 사용자를 찾을 수 없습니다. | 사용자 정보가 존재하지 않음 |

### 비고

- 현재 구현은 `users` row를 물리 삭제하고 남은 refresh token과 현재 access token을 무효화한다.
- FK 제약으로 사용자 삭제가 막히지 않도록 `route_ratings` -> `route_sessions` -> `bookmarks`/`favorite_routes`/`hazard_report_images`/`hazard_reports` -> `users` 순서로 사용자 종속 데이터를 먼저 삭제한다.
