# ODsay / Busan BIMS Transit Integration

## 선택 이유

- 대중교통 경로 후보 생성은 전국 단위 경로 엔진이 강한 ODsay가 적합하고, 실시간 버스 도착 정보는 부산 지역 특화 데이터인 BIMS가 더 정확하다.
- 한 공급자로는 경로 후보, 실시간성, 접근성 보강을 모두 만족시키기 어려워 역할을 분리했다.
- 도보 구간은 다시 GraphHopper로 재계산해 서비스 고유 접근성 기준을 유지할 수 있다.

## 서비스에서 맡는 역할

- ODsay: 대중교통 경로 후보와 lane 정보 제공
- BIMS: 부산 버스 도착 정보와 저상버스 여부 보강
- 내부 서비스: 도보 leg 재계산, 지하철 시간표 / 엘리베이터 정보 결합, 최종 응답 조립

## 핵심 구현 방식

### 1. ODsay로 후보를 만들고 내부 기준으로 shortlist

- `OdsayClient`가 경로 후보와 loadLane 정보를 읽어 온다.
- `TransitRouteSearchService`는 추천, 최소 환승, 최소 도보 같은 기준으로 shortlist를 만든다.
- ODsay 원본을 그대로 반환하지 않고 우리 서비스 기준 후보만 남긴다.

### 2. 도보 leg 재계산

- 각 후보의 보행 구간은 GraphHopper로 다시 계산한다.
- 이렇게 해야 대중교통 경로 안에서도 시각장애인, 휠체어 사용자 기준 안내와 badge 체계를 유지할 수 있다.

### 3. BIMS 비동기 보강

- 버스 leg는 `BusanBimsClient`로 도착 정보를 추가 조회한다.
- 다수 leg가 있는 경로에서도 응답 시간이 과도하게 늘지 않도록 비동기 보강 흐름을 사용한다.
- 저상버스 선호 슬롯과 실제 도착 순서를 함께 반영한다.

### 4. lane 캐시와 메타데이터 보존

- `OdsayLoadLaneStore`가 loadLane 결과를 저장해 같은 대중교통 구간을 매번 외부에서 다시 읽지 않게 한다.
- 최종 후보와 transit metadata는 Redis에 보관해 refresh, reroute, 재안내에 재사용한다.

## 예외 처리와 운영 이슈 대응

### 외부 공급자 실패 분리

- ODsay와 BIMS 오류를 각각 분리된 예외 코드로 감싸 장애 원인을 추적하기 쉽게 했다.
- BIMS 실패가 있더라도 전체 경로를 완전히 버리기보다 가능한 범위의 경로 응답을 유지하는 방향으로 조합한다.

### XML / JSON 파싱 안전성

- BIMS XML 파서는 DOCTYPE을 비활성화해 외부 엔티티 문제를 피한다.
- ODsay 응답은 필요한 필드만 도메인 모델로 옮기고 malformed lane은 저장 단계에서 분리 관리한다.

### 응답 시간 관리

- 실시간 도착 정보는 Redis 1분 캐시를 사용한다.
- lane geometry는 DB에 저장해 동일 구간 재조회 비용을 줄인다.
- 도보 leg 재계산은 필요한 구간에만 적용해 전체 비용을 통제한다.

## 한계와 후속 과제

- 공급자별 데이터 스키마가 다르기 때문에 파싱과 정합성 보정 코드가 늘어나는 편이다.
- 지하철 실시간성, 버스 혼잡도, 엘리베이터 장애 정보까지 통합하려면 별도 데이터 소스 확장이 필요하다.
