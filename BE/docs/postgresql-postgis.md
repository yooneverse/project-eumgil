# PostgreSQL / PostGIS

## 선택 이유

- 경로 세그먼트, 제보 위치, 장소 좌표를 동일 데이터베이스 안에서 트랜잭션과 공간 질의로 함께 다룰 수 있다.
- JPA 기반 일반 비즈니스 데이터와 공간 데이터를 분리하지 않아도 되어 개발 복잡도를 낮출 수 있다.
- 반경 검색, 영역 교차, 최근접 정렬 같은 기능을 DB 레벨에서 수행할 수 있어 애플리케이션 후처리 비용을 줄일 수 있다.

## 서비스에서 맡는 역할

- 도로 세그먼트와 장소의 geometry 저장
- 위험 제보 위치와 관리자 편집 범위 검증
- 지도 마커, 주변 시설, 행정구역 범위 질의
- 라우팅 대상 세그먼트 / 시설 후보 선별

## 핵심 구현 방식

### 1. geometry 컬럼 표준화

- `RoadSegment.geom`은 `LineString(4326)`를 사용한다.
- `Place.point`, `HazardReport.reportPoint`는 `Point(4326)`를 사용한다.
- 하나의 좌표계로 통일해 세그먼트, 장소, 제보 간 공간 연산을 단순화했다.

### 2. 공간 질의를 Repository로 끌어올림

- `RoadSegmentRepository`는 `ST_Intersects`, `ST_Buffer` 기반 범위 조회로 관리자 편집 대상을 찾는다.
- `PlaceRepository`는 `ST_DWithin`, `ST_DistanceSphere`를 사용해 주변 시설을 빠르게 추린다.
- 장소 마커는 먼저 ID 목록만 선별하고 이후 상세 로딩을 분리해 불필요한 엔티티 로딩을 줄인다.

### 3. 관리자 편집과 제보 검증에 활용

- 관리자 route review는 editable area 밖의 세그먼트가 섞이면 즉시 차단한다.
- 위험 제보 승인 후 영향 세그먼트 산정도 geometry 교차와 근접 조건으로 제한한다.

## 예외 처리와 동시성 대응

### 데이터 무결성

- 도로 편집 엔티티는 `@Version` 기반 optimistic locking을 사용해 동시 수정 충돌을 감지한다.
- 충돌이 나면 마지막 저장값으로 덮어쓰지 않고 사용자에게 재시도를 요구하는 방향으로 처리한다.

### 공간 질의 비용 관리

- 마커 조회처럼 결과 수가 많은 기능은 조건이 가벼운 ID 선별 질의와 상세 조회를 분리했다.
- bbox, 반경, 카테고리 조건을 함께 사용해 전체 geometry 스캔으로 번지지 않게 했다.

### 잘못된 요청 차단

- bbox 대각선 길이 제한, 행정구역 범위 검증 등으로 과도한 영역 요청을 초기에 막는다.
- 편집 대상 area 밖 세그먼트는 review 단계에서 배제해 운영 실수를 줄인다.

## 한계와 후속 과제

- 공간 인덱스 전략과 통계 갱신 상태에 따라 대용량 영역 조회 성능 차이가 커질 수 있다.
- 현재 좌표계는 WGS84 중심이라 정밀 거리 계산이 많은 구간은 별도 투영 좌표계 보조 전략을 검토할 수 있다.
