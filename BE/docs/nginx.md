# Nginx

## 선택 이유

- 외부 공개 진입점을 하나로 통합하면서 내부 서비스별 라우팅, TLS 종료, 프록시 헤더 정리를 한 계층에서 처리하기 쉽다.
- Jenkins, Grafana, SonarQube, 개발용 백엔드처럼 성격이 다른 서비스들을 동일 호스트 기반 규칙으로 안정적으로 노출할 수 있다.
- 웹소켓 업그레이드, 대용량 업로드, 버퍼링 제어 같은 운영 세부 조정을 애플리케이션 밖에서 관리할 수 있다.

## 서비스에서 맡는 역할

- 80 포트 요청을 443으로 강제 전환
- 도메인별 내부 서비스 프록시
- TLS 종료와 `X-Forwarded-*` 헤더 전달
- 관리자 경로, 불필요 노출 경로 차단
- Jenkins / Grafana의 websocket 및 streaming 친화 설정 제공

## 핵심 구현 방식

### 1. 호스트 기반 프록시 분기

- `busaneumgil.com`, `dev-api.busaneumgil.com`, `jenkins.busaneumgil.com`, `grafana.busaneumgil.com` 등 호스트별로 upstream을 분기한다.
- 내부 서비스는 직접 외부에 포트를 공개하지 않고 Nginx를 통해서만 접근하게 구성했다.

### 2. TLS 종료와 프록시 표준화

- Certbot 인증서를 사용해 HTTPS를 종단한다.
- 백엔드에는 `Host`, `X-Real-IP`, `X-Forwarded-For`, `X-Forwarded-Proto`를 전달해 원본 요청 정보를 유지한다.

### 3. 업로드 / 스트리밍 친화 설정

- `client_max_body_size 100m`으로 이미지 업로드와 운영 파일 전송 한도를 확보했다.
- `proxy_buffering off`, `proxy_request_buffering off`로 Jenkins 로그 스트리밍이나 장시간 응답이 끊기지 않도록 했다.
- websocket upgrade 헤더를 명시해 Jenkins와 Grafana의 실시간 연결을 유지한다.

### 4. 외부 노출 차단

- `/api/ieum/admin/`, `/ieum/admin/` 같은 관리자 경로는 외부에서 404를 반환하도록 막았다.
- 운영 정책상 공개하지 않을 서비스는 호스트 단위로 바로 404 처리한다.

## 예외 처리와 운영 이슈 대응

### 애플리케이션 보호

- 인증, 인가가 애플리케이션에도 있지만 Nginx에서 한 번 더 차단해 관리 경로 스캐닝 노출을 줄였다.
- 내부 툴은 도메인 레벨 분리로 라우팅 충돌과 잘못된 캐시 공유를 피했다.

### 장시간 연결과 업로드 안정성

- 프록시 버퍼링을 끄지 않으면 CI 로그, 대시보드 스트림, 대용량 업로드에서 지연이나 끊김이 발생할 수 있어 이를 사전에 제어했다.
- 업로드 크기 제한을 reverse proxy에서 먼저 적용해 애플리케이션까지 불필요한 트래픽이 올라가지 않도록 했다.

### 운영 단순화

- 인증서, 리다이렉트, 서비스 공개 범위를 Nginx 한 곳에서 관리해 애플리케이션별 설정 편차를 줄였다.

## 한계와 후속 과제

- 현재 규칙은 호스트/경로 기반 정적 설정 중심이라 서비스가 늘면 설정 파일이 커질 수 있다.
- rate limiting, WAF 수준 방어, access log 기반 탐지 룰은 별도로 더 강화할 수 있다.
