# Kakao Local API

## 선택 이유

- 국내 장소 검색과 주소 변환 품질이 높고, 부산 지역 서비스에 필요한 키워드 검색과 reverse geocoding을 안정적으로 제공한다.
- 내부 `places` 데이터만으로는 모든 외부 POI를 커버할 수 없기 때문에 외부 검색 백필(backfill) 계층이 필요했다.
- 장소 검색, 카테고리 검색, 좌표-주소 변환을 동일 공급자에서 일관되게 처리할 수 있다.

## 서비스에서 맡는 역할

- 키워드 기반 장소 검색
- 좌표 기반 주소 변환
- 내부 DB에 없는 외부 POI 보강
- 위험 제보 위치의 주소 표준화

## 핵심 구현 방식

### 1. 검색과 내부 데이터 매칭 결합

- `KakaoLocalClient`가 keyword search, category search, reverse geocode를 담당한다.
- `PlaceService`는 Kakao 결과를 그대로 노출하지 않고 `providerPlaceId` 기준으로 내부 `places`와 매칭한다.
- 내부 시설이면 접근성 feature, 북마크 상태 같은 로컬 도메인 정보를 함께 합성한다.

### 2. 부산 범위 필터링

- 검색 결과는 부산 주소만 남기도록 후처리한다.
- 서비스 영역 밖 POI가 섞여 들어오는 문제를 애플리케이션에서 한 번 더 제어한다.

### 3. fallback 검색 계층

- 내부 시설 DB에 없더라도 Kakao POI를 최소 기능 상세 응답으로 복구한다.
- 클릭 상세에서도 내부 시설 우선, 외부 provider fallback 순서로 응답을 조립한다.

### 4. 주소 정규화

- 위험 제보나 위치 기반 기능에서 reverse geocode 결과를 사용해 사람이 읽을 수 있는 주소를 보강한다.

## 예외 처리와 운영 이슈 대응

### 외부 API 불안정성 대응

- `KakaoLocalClient`는 짧은 connect/read timeout을 사용해 요청이 길게 묶이지 않도록 했다.
- API 키가 비어 있으면 시작 단계에서 실패하게 만들어 런타임 장애를 조기에 드러낸다.

### 외부 데이터 품질 보정

- 부산 외 지역 결과는 필터링한다.
- 내부 DB 매칭이 안 되더라도 외부 POI fallback을 남겨 검색 결과 자체가 사라지지 않게 했다.

### 응답 일관성 유지

- 내부 시설과 외부 POI의 소스가 달라도 `PlaceService`에서 공통 응답 형태로 맞춘다.
- cursor도 Kakao page를 그대로 노출하지 않고 내부 cursor로 감싸 클라이언트 계약을 고정했다.

## 한계와 후속 과제

- 외부 검색 품질은 Kakao 색인 상태에 의존한다.
- 내부 시설과 외부 POI의 중복 판정 로직은 운영하면서 지속적으로 보정할 필요가 있다.
